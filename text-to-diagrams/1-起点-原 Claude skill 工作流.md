> **本文聚焦**: Claude Code skill 这个 LLM-edits-SVG 实现的完整 7 步流程 + 它的根本性 layout 缺陷。

## 核心认知

这个 skill **不是一段会运行的代码**，而是一套**给 LLM 看的操作手册**。
所有的"逻辑"都写在 markdown 文档里，由模型在生成时按规则执行——没有解析器，没有渲染引擎，没有模板引擎。

## 文件结构

```
text-to-diagrams/
├── SKILL.md                  ← 主流程（7 步），模型每次必读
├── README.md                 ← 给用户看的介绍
├── references/
│   ├── style.md              ← 颜色/字体唯一来源（12 个 token）
│   ├── diagrams.md           ← 17 种图的目录 + 编辑规则 + 数据公式
│   └── page-shell.html       ← 多图组装的 HTML 外壳模板
├── templates/                ← 17 个 *.html，每个是一张完整的"参考实现"图
│   ├── timeline.html
│   ├── flowchart.html
│   ├── donut-chart.html
│   └── ...
├── examples/                 ← 真实运行结果（输入 + 输出）
└── tools/svg-to-png.sh       ← 唯一的脚本：rsvg-convert 转 PNG
```

## 完整流程：从文本到图

### Step 0 · 读 style.md

模型一启动这个 skill 就先读 `references/style.md`，拿到 12 个语义颜色 token：

| Token | 默认值 | 用途 |
|---|---|---|
| `paper` | `#f5f4ed` | 画布背景 |
| `near-black` | `#141413` | 主文本 |
| `olive` | `#504e49` | 次文本/箭头 |
| `stone` | `#6b6a64` | 三级文本/外部节点 |
| `brand` | `#1B365D` | **焦点描边（1–2 个元素）** |
| `brand-tint` | `#EEF2F7` | 焦点填充 |
| `up` / `down` | `#1B365D` / `#6b6a64` | K 线/瀑布图涨跌 |
| ... | | |

**为什么先读这个？** 模板里的 hex 值是默认值，如果用户改了 style.md，模型会在写 SVG 时把模板里的旧色替换成新色。这是"换肤"机制。

### Step 1 · 判断该不该画图

唯一的门槛问题：

> 一段写得好的文字，会比这张图教给读者更少吗？

如果答案是"不会"，告诉用户文本没有可视化价值，**直接停止**，不进 Step 2。

### Step 2 · 读 diagrams.md

拿到 17 种图的"选择表"：

| 内容形态 | 选 |
|---|---|
| 系统组件 + 连接 | Architecture |
| 决策分支 (if A then B else C) | Flowchart |
| 时间顺序事件 | Timeline |
| 比例分解 | Donut |
| 层级关系 | Tree |
| 双轴定位 | Quadrant |
| 排名/漏斗 | Pyramid |
| 类别比较 | Bar chart |
| 趋势 | Line chart |
| 状态机 | State machine |
| 跨角色流程 | Swimlane |
| 堆叠层 | Layer stack |
| 集合交集 | Venn |
| OHLC 价格 | Candlestick |
| 价值分解 | Waterfall |
| 数据库实体 | ER |
| 时序消息 | Sequence |

### Step 3 · 规划要画几张、画什么

**这是核心步骤**，决定了"一份文本如何拆成多张图"。

#### 3.1 列出所有"带结构的主张"

读全文，把每一句**带可视化结构**的话挑出来。"带结构"指：

- **hierarchy**（层级）："musk 带框架团队，下面有 X、Y、Z"
- **magnitude**（量级）："Q1 营收 120 万，Q2 180 万"
- **direction**（方向）："请求先到 master，再分发给 worker"
- **order**（顺序）："2019 立项，2021 上线，2023 规模化"
- **intersection**（交集）："功能 A 和 B 都包含用户认证"
- **decomposition**（分解）："总预算里 60% 是研发"

**没有结构的句子被忽略**，比如"我们认为这是个好方向"、"未来再说"——这些是策略陈述，画不出图。

#### 3.2 按"同属一个概念"把主张分组

把上一步挑出的句子放成几堆，**同一堆讲的是同一件事**。比如：

```
原文                                  → 分组
─────────────────────────────────────────────────
"master 调度 worker"                 ┐
"worker 在 Sandbox 里运行"            ├ 组 A：技术架构
"context+memory 在 Sam 处"           ┘

"Musk 带框架"                          ┐
"Sam 做 context"                      ├ 组 B：分工安排
"Yem 调研法律"                         ┘

"模型能自主调用工具？能→原生；不能→定制"  ─ 组 C：定制决策
```


#### 3.3 给每组挑一种图

每组单独查 Step 2 的选择表，看这组的**主导关系**是什么：

- 组 A：模块互相调用 → **Architecture**
- 组 B：人 → 职责的归属关系 → **Tree**
- 组 C：if A then B else C → **Flowchart**

#### 3.4 丢掉"画图反而不如文字"的组

每组再过一次 Step 1 的门槛。过不了的扔掉。例如"收费模式会改变，按时长/TOKEN 收费"——一句话讲清楚了，画图反而废话。

#### 3.5 张数由文本决定

没有上限也没有下限。短笔记可能就 1 张，长设计文档可能 10+ 张。

#### 3.6 长页面（>6 张）要分小节

如果最后留下 8 张图，**不要直接堆成一页**——那样读起来像图册/模板，不像文档。要按原文的自然小节分组，在 HTML 里插入 `<h2 class="section-title">`：

```html
<h2 class="section-title">架构</h2>
  <section class="diagram-block">[Figure 1 · Architecture]</section>
  <section class="diagram-block">[Figure 2 · Sequence]</section>

<h2 class="section-title">数据模型</h2>
  <section class="diagram-block">[Figure 3 · ER]</section>

<h2 class="section-title">流程</h2>
  <section class="diagram-block">[Figure 4 · Flowchart]</section>
  <section class="diagram-block">[Figure 5 · State machine]</section>
  ...
```

≤6 张可以不分节，平铺即可。

#### 3.7 给每张图标 anchor

如果输入是 **markdown**，给每张图标记它该插到原文哪里：

- `end-of-section "## 标题"` ← 默认，插在该小节末尾
- `after-heading "## 标题"` ← 插在标题之后（图先于文字）
- `after-paragraph "<原文前 40 字>"` ← 插在某段后面
- `append` ← 兜底，插到文末

如果输入是 **plain text**，所有图都用 `append`。

### Step 4 · 一张张画 SVG

对计划里的每张图：

1. 打开 `templates/<type>.html`
2. 提取里面的 `<svg>...</svg>` 块（保留原 viewBox）
3. **手动改写 `<text>` 和 `<rect>` 的内容和坐标**来匹配新内容
4. 把 style.md 的 hex 值套上去（如果跟模板默认值不同）
5. 应用 diagrams.md §4 的硬约束（见下）
6. 应用焦点规则：1–2 个元素穿 brand-tint 填充 + brand 描边，其他全部中性

#### diagrams.md §4 的硬约束

| 项 | 规则 |
|---|---|
| 坐标 | 全部能被 4 整除 |
| 节点宽 | 只能 128 / 144 / 160 三档 |
| 节点高 | 32（pill 圆角）或 64（标准） |
| 字号 | 7（小 mono 标签）/ 9（副标签）/ 12（节点名） |
| 箭头端点 | 必须正好落在节点边上 |
| 箭头形态 | chevron（不是实心三角） |

#### 关于"节点数怎么改"

SKILL.md 和 diagrams.md **没有**"x 轴 60→900 等分 N 段"这种公式。规则文档只规定了硬约束（坐标 ÷4、宽度 3 档等），**具体如何把模板里的 5 个事件改成 N 个事件的布局，留给模型自己判断**——它看着模板的样子（等距 + 上下交替 + 虚线连接圆点），凭设计直觉调整。

也就是说：

- **节点数** = 文本里有几个事件就画几个
- **每个节点的位置** = 模型按"模板长这样、坐标要 ÷4、看起来匀称"现场算
- **哪个是焦点** = 模型读文本判断"哪个里程碑最重要"

模板提供的是**布局算法的样例**，不是布局算法本身。

#### 数据图有公式

唯一例外：bar/line/donut/candlestick/waterfall 这五种数据图，[diagrams.md §6](#) 给了精确公式。比如 donut 的扇区角度：

```
angle_start = -90 + Σ前面百分比 × 3.6
angle_end   = angle_start + 当前百分比 × 3.6
outer_x     = 300 + 136 × cos(angle × π/180)
outer_y     = 200 + 136 × sin(angle × π/180)
```

这些必须严格按公式算，不能凭感觉。

### Step 5 · 拼总页 + 单图 SVG + markdown

总是输出三种文件：

#### A. 总 HTML 页（output.html）

复制 `references/page-shell.html`，填入：

- `{{PAGE_TITLE}}` ← 整体标题
- `{{RATIONALE}}` ← 一段斜体说明"为什么画这些图"
- `{{N}}` ← 图的总数
- 每张图复制一个 `<section class="diagram-block">` 块
- >6 张时插入 `<h2 class="section-title">`

#### B. 单图 SVG（output/figure-N-<type>.svg）

每张图额外存一份独立的 `.svg`，用户可以拖进 Figma/PPT/博客。

#### C. Markdown 文件（output.md）

- **markdown 输入** → 原文**逐字保留**，只在每个 anchor 处插入 `![Figure N](output/figure-N-x.svg)` 块
- **plain text 输入** → 重新生成一份 markdown，标题 + 各图说明

### Step 6 · 可选：转 PNG

```bash
bash tools/svg-to-png.sh output/figure-1-architecture.svg
```

只在用户明确要 PNG 时才做。

### Step 7 · 自检清单

- [ ] 每张图恰好 1–2 个焦点元素
- [ ] 焦点颜色与 caption 强调一致
- [ ] 箭头端点贴在节点边上
- [ ] 所有坐标 ÷4
- [ ] 没有 box-shadow / 渐变 / 大圆角
- [ ] 没有 emoji
- [ ] style.md 的色值都套上了
- [ ] 单图 SVG 和 output.md 都写了

## 关键设计哲学

1. **三份规则文档（SKILL/diagrams/style）+ 17 个 SVG 参考实现**——没有其他东西。
2. **模板是"长这样的图"的范例**，不是 `{{name}}` 占位符。
3. **颜色 = style.md 的 token 表**，模型负责把模板里的 hex 替换成新值。
4. **节点数 = 文本里有几个就画几个**，没有上限。
5. **焦点 1–2 个**，靠对比而不是数量传达重要性。
6. **不要加第二种强调色**（warning/success），单焦点设计是有意为之。

## 常见误解更正

| 误解 | 真相 |
|---|---|
| 模板里的事件数（如 timeline 的 5 个）= 输出会有 5 个 | 输出节点数完全由文本决定 |
| 有公式计算节点 x 坐标 | 只有数据图有公式，其他靠模型判断 |
| 颜色由模板决定 | 颜色由 style.md 决定，模板只是默认值 |
| 一张图 = 一段原文 | 一张图 = 一组"讲同一件事的句子"，可能跨段落 |
| 输出只有一个 HTML | 总是 3 种文件：HTML + 每图 SVG + markdown |

## 已知问题：节点重叠 / 错位

这是这个 skill 当前的**根本性缺陷**，不是偶发 bug。

### 真实案例

某次 swimlane 输出里，右下角 BUNDLE 框和 OPTIONAL 框重叠了：

```
BUNDLE 框：    x=720, y=332, width=144  → 占据 x ∈ [720, 864]
OPTIONAL 框：  x=760, y=332, width=144  → 占据 x ∈ [760, 904]
```

两个框在同一行，x 范围 [760, 864] 完全重叠 104px。文字叠在一起，dashed 边框压在实线框上。

挑出来检查规则：
- ✓ 720、760 都能被 4 整除（合规）
- ✓ 144 是三档宽度之一（合规）
- ✗ **没有任何规则检查"两个 rect 不能重叠"**

每个数字单独看都合规，**整体布局崩了**。

### 为什么会这样：四层原因

#### 第一层：没有约束求解器

正经的布局工具（Figma、TikZ、graphviz）都有约束求解器——告诉它"A 在 B 右边，间距 ≥ 16"，引擎自动算坐标。

这个 skill 没有任何求解器。**模型直接生成最终坐标数字**，不是在"放置元素"，而是在"写字符串"。`x="720"` 和 `x="760"` 在它眼里是两个独立的 token，要靠**心算**记住"我刚才在 720 放了一个宽 144 的框"。

#### 第二层：合规 ≠ 正确

规则文档只能验证**局部、独立**的约束：

| 约束类型 | 例子 | 模型能否自检 |
|---|---|---|
| 单点约束 | `x=143` 是否 ÷4 | ✅ 能（看一个数字） |
| 单元素约束 | `width` 是否在 {128,144,160} | ✅ 能 |
| **二元关系** | rect A 和 rect B 是否重叠 | ⚠️ 要同时记住两个元素的坐标 |
| **全局关系** | N 个元素总宽度是否超出 viewBox | ❌ 几乎做不到 |

模型每写一个新元素时，要在脑子里维护"已放置元素表"并跟新元素做碰撞检测。这事 LLM 做几个还行，做到第 8、9 个就开始忘。

#### 第三层：自回归生成的天然短板

LLM 是**一次写一个 token，从左到右**生成 SVG 字符串：

```
<rect x="720" y="332" width="144" .../>   ← 写完这行
<rect x="760" y="332" width="144" .../>   ← 写下一行时
```

写第二行的 `x="760"` 时，模型**没有"回头看"机制**——它不会把前面已生成的 rect 在脑子里栅格化、检查冲突，再决定 760 是不是合理。它只是基于"这附近大概该放在 760 左右"的统计直觉写下去。

这跟人脑画图的"先草图再细化、随时擦除"完全相反。

#### 第四层：训练数据的影响

LLM 大部分时候能画对，是因为训练数据里见过几百万张正确的 SVG，知道"两个 144 宽的框，x 应该差至少 ~160"。但这是**模式匹配的概率**，不是**逻辑保证**。

特别容易翻车的场景：
- 节点数 ≥ 6
- 同一行有多个不同宽度的元素
- 出现模板里没有的特殊布局（如"虚线浮空 OPTIONAL 框"）
- 用户改了模板的常规结构

碰撞概率随复杂度指数级上升。

### Step 7 自检清单为什么救不了

[SKILL.md:198-209](plugins/text-to-diagrams/skills/text-to-diagrams/SKILL.md#L198-L209) 的清单只检查"模型看得出来的东西"：

| 自检项 | 类型 | 模型能验证吗 |
|---|---|---|
| 焦点 1–2 个 | 计数 | ✅ |
| 颜色与 caption 一致 | 语义匹配 | ✅ |
| 箭头端点在节点边上 | 二元关系 | ⚠️ |
| 坐标 ÷4 | 单点 | ✅ |
| 没有 emoji | 关键词 | ✅ |
| **rect 不重叠** | 二元关系（O(N²)） | **❌ 没列入** |
| **总宽不超 viewBox** | 全局关系 | **❌ 没列入** |

清单没列的就不会被检查，**就算列了也未必能稳定执行**——因为这些是 LLM 的弱项。

### 系统设计取舍

skill 的作者**清楚这个弱点**，用了几个补救手段：

1. **模板法兜底**——大部分时候直接复用模板坐标，少改少错
2. **硬约束清单**（÷4、宽度三档）——把可机械验证的局部错误堵掉
3. **17 种预设构图**——每种都有典型形状，不让模型自由发挥
4. **不在 SVG 里搞复杂布局**

但**没有解决"非典型布局时坐标互相重叠"**。要真解决需要：
- 真正的约束求解器
- 生成后用浏览器/工具栅格化检查 bounding box 碰撞
- 让模型先输出抽象的"placement 计划"再翻译成坐标

skill 选择不做这些，用"够用就行 + 模板兜底"换简单。代价就是：**非常规布局会出重叠**。

### 实践建议

- 检查输出图时**重点看节点密集的区域**，特别是 6 个以上节点的图
- 出现重叠时不要让模型重画——它大概率会犯类似的错。直接告诉它具体哪两个框冲突，让它改坐标
- 复杂布局优先用 mermaid / graphviz 这种**有真正布局引擎**的工具，不要硬上这个 skill
- skill 最适合：**节点数 ≤ 6、布局符合模板典型结构**的图

### 一句话总结

**规则文档只能管"每一笔合不合规"，管不了"整张图组合起来对不对"。** LLM 写 SVG 是流式输出字符串，不是空间放置物体——所以坐标 ÷4 都对、宽度都合法，整体布局还是会崩。这是当前 LLM 的硬限制，不是这个 skill 独有的 bug。
