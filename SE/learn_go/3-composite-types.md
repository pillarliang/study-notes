# Chapter 3：复合类型（Composite Types）

本章介绍 Go 中的内置复合类型：数组、切片、字符串、map、结构体，以及配套的内置函数和最佳实践。

---

## 0. 核心原理：一切从这里推出来

### 0.1 唯一原理

> Go 里**所有的赋值和传参都是"逐字节复制变量本身"**，没有例外。函数永远拿不到调用方的变量，只拿到一份副本。

### 0.2 两条机械规则

1. **副本里的字段被改 → 调用方看不见**（副本是独立的字节）。
2. **副本里若包含指针 → 指针指向的目标是共享的**（两份指针指着同一块内存，谁通过指针写，对方都看得见）。

## 1. 数组（Array）—— 太死板，不建议直接使用

### 1.1 声明语法

**字节布局**：N 个 T 元素**紧挨着排列在变量本身的内存里**，没有任何指针字段。占 `N × sizeof(T)` 字节，因此长度大小都在编译期确定。

```text
var x [3]int            // x 的内存：[ 10 ][ 20 ][ 30 ]，共 24 字节，纯数据
```

数组所有元素必须是同一类型，长度是类型的一部分。

```go
var x [3]int                  // 长度 3，零值初始化为 [0, 0, 0]
var x = [3]int{10, 20, 30}    // 数组字面量
var x = [12]int{1, 5: 4, 6, 10: 100, 15} // 稀疏数组：仅指定非零索引
// 结果：[1, 0, 0, 0, 0, 4, 6, 0, 0, 0, 100, 15]

var x = [...]int{10, 20, 30}  // 用 ... 让编译器自动推导长度
```

#### 稀疏初始化语法详解

`索引: 值` 表示"把**值**放到指定**索引**位置"，未指定的位置自动填零值。**不带索引**的元素接在**上一个索引之后**。该语法对**数组、切片、map** 字面量都适用。

以 `[12]int{1, 5: 4, 6, 10: 100, 15}` 为例，逐元素拆解：


| 写法        | 索引  | 值   | 说明                     |
| --------- | --- | --- | ---------------------- |
| `1`       | 0   | 1   | 默认从 0 开始               |
| `5: 4`    | 5   | 4   | 显式跳到索引 5；中间索引 1~4 补 0  |
| `6`       | 6   | 6   | 紧跟上一个，索引 = 5 + 1       |
| `10: 100` | 10  | 100 | 显式跳到索引 10；中间索引 7~9 补 0 |
| `15`      | 11  | 15  | 紧跟上一个，索引 = 10 + 1      |


最终值：`[1, 0, 0, 0, 0, 4, 6, 0, 0, 0, 100, 15]`。

等价的"啰嗦写法"：

```go
var x = [12]int{
    0: 1, 1: 0, 2: 0, 3: 0, 4: 0,
    5: 4, 6: 6,
    7: 0, 8: 0, 9: 0,
    10: 100, 11: 15,
}
```

适用场景：绝大多数位置都是零值、只有少数位置有值的初始化。例如想要长度 100、只有首尾有值的切片：

```go
config := []int{42, 99: 7}  // 长度 100，[0]=42、[99]=7，中间全是 0
```

> 切片版本不需要预先写死长度，长度由"最大索引 + 1"推导得到。
> 真正稀疏的数据更多用 map，本语法常用在测试夹具和小型查找表里。

### 1.2 多维数组

Go 只有一维数组，多维通过"数组的数组"模拟：

```go
var x [2][3]int  // 长度为 2 的数组，每个元素是长度为 3 的 int 数组
```

### 1.3 比较与读写

- 长度相同且元素全等的两个数组可用 `==` / `!=` 比较。
- 索引越界：常量索引 → 编译错误；变量索引 → 运行时 panic。
- `len(x)` 返回数组长度。

### 1.4 数组的"死板"之处

> **数组长度是类型的一部分**：`[3]int` 与 `[4]int` 是两个不同类型。

由此带来三个限制：

1. 不能用变量指定长度（必须编译期确定）。
2. 不同长度的数组**不能互相类型转换**。
3. 因此无法写一个能处理任意长度数组的函数。

**结论**：除非长度在编译期就完全确定（例如加密哈希的固定输出长度），否则不要直接用数组。**数组存在的主要意义是为切片提供底层存储**。

## 2. 切片（Slice）—— Go 中最常用的序列类型

#### 切片的字节布局

切片**自己不存数据**，它在内存里就是这样一个 24 字节描述符：

```go
// 任何 []T 变量本身就是这个结构（运行时定义）
type slice struct {
    ptr *T   // 指向底层数组的某个位置（指针！）
    len int  // 已有元素数
    cap int  // 从 ptr 到底层数组末尾的容量
}
```

### 2.1 切片 vs 数组：先记住一条最简单的判别规则

**类型里写了长度的是数组，没写的是切片。**

```go
[3]int{1, 2, 3}      // 数组
[...]int{1, 2, 3}    // 数组（让编译器数长度，仍然是 [3]int）
[]int{1, 2, 3}       // 切片
var x [5]int         // 数组
var x []int          // 切片（值为 nil）
```

#### 那为什么还要分两个东西

光记上面这条规则不够用，因为它们的**运行时行为完全不同**——下面这四条是平时踩坑/做选择的核心：


| 行为         | 数组 `[N]T`                               | 切片 `[]T`        |
| ---------- | --------------------------------------- | --------------- |
| 能不能 append | ❌ 长度写在类型里，不能变                           | ✅ 可变长           |
| 长度不同的两个变量  | `[3]int` 和 `[5]int` 是**不同类型**，不能互转、不能赋值 | 都是 `[]int`，类型一样 |
| 函数传参       | 复制**全部**元素（每次都拷一份完整数据）                  | 只复制 24 字节描述符    |
| 多个变量能否共享数据 | 永远不共享（赋值即独立）                            | 经常共享底层数组，会互相改到  |


第 4 行尤其重要——切片的子切片陷阱、`append` 必须赋回原变量、`copy` 函数为什么存在，全都源于这条。

多个切片描述符可以同时指向同一段底层数组的不同区段：

```text
底层数组 A: [10, 20, 30, 40, 50]      ← 真正的数据
              ↑
              ├── 切片 s1: ptr→A[0], len=3, cap=5   "看 A 的前 3 个"
              └── 切片 s2: ptr→A[2], len=2, cap=3   "从 A[2] 开始看 2 个"
```

所以"切片传参便宜"（只复制 24 字节）、"切片之间会共享数据"（多个 ptr 指同一份数组）这两件事，是同一个机制的两面。**数组是数据本身；切片是看数据的一扇窗口**。

### 2.2 切片字面量

```go
var x = []int{10, 20, 30}                  // 注意：[] 内不写长度
var x = []int{1, 5: 4, 6, 10: 100, 15}     // 稀疏初始化
var x [][]int                              // 二维切片
```

> 区分：`[...]` 创建数组；`[]` 创建切片。

### 2.3 nil 切片

```go
var x []int   // x 此时为 nil
fmt.Println(x == nil) // true
```

- 切片的零值是 `nil`，长度和容量都为 0。
- 切片**不可比较**：除了与 `nil` 比较外，使用 `==` / `!=` 是编译错误。

### 2.4 切片相等性比较（Go 1.21+）

```go
import "slices"

x := []int{1, 2, 3, 4, 5}
y := []int{1, 2, 3, 4, 5}
slices.Equal(x, y)     // true，要求元素可比较
slices.EqualFunc(x, y, func(a, b int) bool { ... }) // 自定义比较函数
```

> `reflect.DeepEqual` 是历史遗留方案，新代码不要再用。

## 3. 切片的内置函数

### 3.1 len 与 append

```go
var x []int
x = append(x, 10)         // 单值追加
x = append(x, 5, 6, 7)    // 多值追加
y := []int{20, 30, 40}
x = append(x, y...)       // 用 ... 把切片展开
```

#### 为什么必须 `x = append(x, ...)` 把返回值赋回去

套用 [§0 核心原理](#0-核心原理一切从这里推出来)：`append(s, v)` 收到 `s` 的描述符**副本**，做两件事：

1. **写底层数组**：cap 够 → 在 `ptr+len` 槽位写入 v；cap 不够 → 分配新数组、复制旧元素、再写。
2. **返回新描述符**：`len` 比传入的多 1；若发生重新分配，`ptr` 和 `cap` 也是新的。

`append` **永远改不到调用方的 s**——它手里只有副本。只能返回新描述符，由调用方覆盖原变量。否则新元素即便已经写进底层数组，原变量的 `len` 没更新也"看不见"。

##### 实验：数据真的写进去了，但调用方看不到

```go
func foo(s []int) {
    s = append(s, 99)        // 容量够，99 直接写进底层数组
}

x := make([]int, 5, 10)      // len=5, cap=10
foo(x)
fmt.Println(len(x))          // 5                ← 描述符副本的修改没回流
fmt.Println(x)               // [0 0 0 0 0]      ← 看不到 99
fmt.Println(x[:6])           // [0 0 0 0 0 99]   ← 底层数组里 99 真的在
```

`x[:6]` 把 `len` 临时从 5 拉到 6 直接观察底层数组，能看到那个被 `len=5` 挡住的 99——它没消失，只是越过了原 `x` 的可见范围。

> Go 不允许丢弃 `append` 的返回值，单独写 `append(s, x)` 会编译报错。

### 3.2 容量（Capacity）

- **长度（length）**：当前已存元素数。
- **容量（capacity）**：底层数组预留的空间。
- 当 length == capacity 时再 append，runtime 会**分配更大的底层数组并复制数据**，原数组变成垃圾。

容量增长规则（Go 1.18 起）：


| 当前容量  | 增长方式                        |
| ----- | --------------------------- |
| < 256 | 翻倍                          |
| ≥ 256 | `(cap + 768) / 4`，逐渐收敛到 25% |


```go
// Example 3-1
var x []int
fmt.Println(x, len(x), cap(x)) // [] 0 0
x = append(x, 10)
fmt.Println(x, len(x), cap(x)) // [10] 1 1
x = append(x, 20)
fmt.Println(x, len(x), cap(x)) // [10 20] 2 2
x = append(x, 30)
fmt.Println(x, len(x), cap(x)) // [10 20 30] 3 4
x = append(x, 40)
fmt.Println(x, len(x), cap(x)) // [10 20 30 40] 4 4
x = append(x, 50)
fmt.Println(x, len(x), cap(x)) // [10 20 30 40 50] 5 8
```

#### 增长公式拆解

##### cap < 256：翻倍

新 cap = 旧 cap × 2。注意 0 是特例（0×2 还是 0），从 0 开始的第一步会跳到 1。书中 Example 3-1 完整轨迹：


| 操作         | len | cap | 说明        |
| ---------- | --- | --- | --------- |
| 起始         | 0   | 0   | nil 切片    |
| append 1 个 | 1   | 1   | 0 → 1（特例） |
| append 1 个 | 2   | 2   | 1 → 2（×2） |
| append 1 个 | 3   | 4   | 2 → 4（×2） |
| append 1 个 | 4   | 4   | 还够，不扩容    |
| append 1 个 | 5   | 8   | 4 → 8（×2） |


继续会是 8 → 16 → 32 → 64 → 128 → 256，到 256 切换到下方公式。

##### cap ≥ 256：`(cap + 768) / 4` 是"增长量"，不是新容量

```text
new_cap = old_cap + (old_cap + 768) / 4
```

验证书中两个数据：

- `cap = 512`：增长量 = (512 + 768) / 4 = 320，新 cap = 832，增长率 ≈ **63%** ✓
- `cap = 4096`：增长量 = (4096 + 768) / 4 = 1216，新 cap = 5312，增长率 ≈ **30%** ✓

##### 为什么"收敛到 25%"

把增长率公式拆开：

```text
增长率 = (cap + 768) / 4 / cap = 1/4 + 192 / cap
```

`cap` 越大，`192 / cap` 越接近 0，整体增长率就趋近 **1/4 = 25%**。


| cap     | 192 / cap | 增长率    |
| ------- | --------- | ------ |
| 256     | 0.75      | ~100%  |
| 512     | 0.375     | ~63%   |
| 1024    | 0.1875    | ~44%   |
| 4096    | 0.047     | ~30%   |
| 65536   | 0.003     | ~25.3% |
| 1000000 | 0.0002    | ~25%   |


##### 设计直觉

- 小切片**翻倍**：append 摊销到接近 O(1)，浪费的内存也很小。
- 大切片**只涨 25%**：避免一次性吃掉几十 MB 内存，增速温和但仍能保持均摊性能。

> 已知容量上限就一次性建好，避免多次扩容拷贝。

### 3.3 make

`make` 用于显式指定类型、长度、（可选）容量：

```go
x := make([]int, 5)        // len=5, cap=5，元素全为 0
x := make([]int, 5, 10)    // len=5, cap=10
x := make([]int, 0, 10)    // len=0, cap=10，可直接 append
```

> ⚠️ **常见误区**：`make([]int, 5)` 后再 `append(x, 10)`，结果是 `[0 0 0 0 0 10]`，而不是 `[10 0 0 0 0]`。`append` 永远在末尾追加。

> ⚠️ 容量不能小于长度，否则编译/运行时报错。

### 3.4 clear（Go 1.21+）

把切片所有元素置为零值，**长度不变**：

```go
s := []string{"first", "second", "third"}
clear(s)
fmt.Println(s, len(s)) // [  ] 3
```

### 3.5 切片声明的最佳实践


| 场景               | 推荐写法                              |
| ---------------- | --------------------------------- |
| 切片可能始终为空         | `var data []int`（nil 切片）          |
| 已有初始值且不会变化       | 切片字面量 `data := []int{2, 4, 6, 8}` |
| 作为 buffer 使用     | `make([]byte, n)` 指定非零长度          |
| 完全确定大小，索引赋值      | `make([]T, n)`                    |
| 大致知道大小，append 填充 | `make([]T, 0, n)` 或 `var x []T`   |


> 作者偏好：用 `var x []T` 初始化为 nil，再 append；可能稍慢但不易出 bug。

## 4. 切片的切片（Slicing）

### 4.1 基本语法

```go
x := []string{"a", "b", "c", "d"}
y := x[:2]   // [a b]
z := x[1:]   // [b c d]
d := x[1:3]  // [b c]
e := x[:]    // [a b c d]
```

### 4.2 子切片共享底层存储 ⚠️

切片的切片**不复制数据**，多个切片共享同一段底层数组。

```go
x := []string{"a", "b", "c", "d"}
y := x[:2]  // [a b]
z := x[1:]  // [b c d]
x[1] = "y"
y[0] = "x"
z[1] = "z"
fmt.Println(x) // [x y z d]
fmt.Println(y) // [x y]
fmt.Println(z) // [y z d]
```

### 4.3 子切片 + append：更隐蔽的坑

子切片的容量 = 原切片容量 − 起始偏移。append 子切片可能**覆写父切片的数据**。

```go
x := []string{"a", "b", "c", "d"}
y := x[:2]
fmt.Println(cap(x), cap(y)) // 4 4
y = append(y, "z")
fmt.Println(x) // [a b z d]   ← x 被改了！
fmt.Println(y) // [a b z]
```

#### 子切片容量公式详解

```text
子切片 cap = 原切片 cap − 子切片起始偏移
```

子切片**指向同一个底层数组**，只是起点变了。它能"看到"的容量，是从它的起点一直到底层数组末尾的距离。

以 `x := []string{"a", "b", "c", "d"}`（cap=4）为例：


| 表达式      | 起点偏移 | len | cap           |
| -------- | ---- | --- | ------------- |
| `x[:2]`  | 0    | 2   | 4 − 0 = **4** |
| `x[1:]`  | 1    | 3   | 4 − 1 = **3** |
| `x[2:3]` | 2    | 1   | 4 − 2 = **2** |
| `x[3:4]` | 3    | 1   | 4 − 3 = **1** |


```text
索引:    0   1   2   3
值:     "a" "b" "c" "d"

x[:2]   [-------]               len=2, cap=4 (一直到末尾)
x[1:]       [-----------]       len=3, cap=3
x[2:3]          [---]           len=1, cap=2
x[3:4]              [---]       len=1, cap=1
```

**底层原理**：切片在 runtime 是个三元组：

```go
type slice struct {
    ptr unsafe.Pointer  // 指向底层数组某个位置
    len int             // 当前可见元素数
    cap int             // 从 ptr 到底层数组末尾的容量
}
```

取子切片时，`ptr` 移到子切片起点，`cap` 自然变成"从新 ptr 到底层数组末尾"还剩多少格子。这正是 append 子切片会"漏出去"覆写父切片的根本原因。

### 4.4 三段切片表达式（Full Slice Expression）

第三个参数显式限定子切片的容量，避免 append 时影响父切片：

```go
x := make([]string, 0, 5)
x = append(x, "a", "b", "c", "d")
y := x[2:2:4]   // len=0, cap=2
z := x[2:4:4]   // len=2, cap=2
// 此时 y、z 各自独立扩容，不会回写到 x
```

公式变形：

```text
三段切片 cap = max − 起始偏移
```

只要把 `max` 卡在子切片的 `high` 处，append 就一定会扩容，不会回写父切片。

#### append 扩容后是否还共享内存？

**结论**：扩容后 `y` 已经指向新的底层数组，与 `x` 完全脱钩。再修改 `y[0]` 不会影响 `x`。

```go
x := []string{"a", "b", "c", "d"}
y := x[:2:2]              // len=2, cap=2，与 x 共享底层数组
y = append(y, "z")        // cap 不够 → 分配新底层数组并复制
y[0] = "Q"
fmt.Println("x:", x)      // [a b c d]   ← 完全没变
fmt.Println("y:", y)      // [Q b z]
```

内存示意：

```text
第 1 步：y := x[:2:2]
底层数组 A: ["a", "b", "c", "d"]
              ↑
              ├── x: ptr→A[0], len=4, cap=4
              └── y: ptr→A[0], len=2, cap=2     ← 此时 y[0]=... 会改到 x

第 2 步：y = append(y, "z")  扩容
底层数组 A: ["a", "b", "c", "d"]   ← x 仍指向这里
底层数组 B: ["a", "b", "z", _]     ← y 改指向新数组
              ↑
              └── y: ptr→B[0], len=3, cap=4

第 3 步：y[0] = "Q"
底层数组 A: ["a", "b", "c", "d"]   ← x 不受影响
底层数组 B: ["Q", "b", "z", _]     ← 只有 y 看得见
```

> **关键判断**：`append` **是否触发扩容**决定共享是否断开。
>
> - **没扩容** → 父子切片仍共享底层数组，互相能改到对方。
> - **扩容了** → 子切片搬家到新数组，从此互不相干。
>
> 三段切片表达式的妙处就是把 cap 卡死，**强制让下一次 append 必须扩容**，从而保证父子切片在那一刻就彻底脱钩。
>
> **建议**：取子切片后，要么完全只读，要么始终用三段表达式。

## 5. copy 函数

### 5.1 函数签名

```go
func copy(dst, src []T) int
```


| 项目  | 说明                                              |
| --- | ----------------------------------------------- |
| 功能  | 把 `src` 的元素逐个写入 `dst`，得到与源**完全独立**（不共享底层数组）的副本  |
| 输入  | `dst`：目标切片（写入位置）；`src`：源切片或源数组的切片表达式。两者元素类型必须一致 |
| 输出  | `int` —— 实际拷贝的元素数，等于 `min(len(dst), len(src))`  |


### 5.2 使用示例

```go
x := []int{1, 2, 3, 4}
y := make([]int, 4)
num := copy(y, x)         // 返回拷贝的元素数 4
fmt.Println(y, num)       // [1 2 3 4] 4
```

### 5.3 要点

- 拷贝数量只看**长度**（`len`），与容量无关。`make([]int, 0, 10)` 作为 dst 一个元素都拷不进去。
- 支持源/目标**重叠**：`copy(x[:3], x[1:])` → x 变为 `[2 3 4 4]`。
- 返回值可丢弃：`copy(y, x)` 单独成行也合法。
- 数组也能作为 src 或 dst，但需显式取切片：`copy(y, d[:])` / `copy(d[:], x)`。

## 6. 数组与切片的互转

### 6.1 数组 → 切片

```go
xArray := [4]int{5, 6, 7, 8}
xSlice := xArray[:]   // 整体转切片
y := xArray[:2]
z := xArray[2:]
```

> 注意：从数组取切片，**也共享内存**（与数组取切片同样的副作用）。

### 6.2 切片 → 数组

数组是**值类型**，转换会**复制数据**到新内存。修改切片不影响转换后的数组：

```go
xSlice := []int{1, 2, 3, 4}
xArray := [4]int(xSlice)        // 完整转换
smallArray := [2]int(xSlice)    // 只取前 2 个
xSlice[0] = 10
fmt.Println(xArray)     // [1 2 3 4]   未变
fmt.Println(smallArray) // [1 2]
```

约束：

- 数组长度必须**编译期确定**，不能用 `[...]`。
- 数组长度 > 切片**长度**（不是容量）会运行时 panic。

### 6.3 切片 → 数组指针（共享内存）

```go
xSlice := []int{1, 2, 3, 4}
xArrayPointer := (*[4]int)(xSlice)
xSlice[0] = 10
xArrayPointer[1] = 20
fmt.Println(xSlice)         // [10 20 3 4]
fmt.Println(xArrayPointer)  // &[10 20 3 4]
```

## 7. 字符串、Rune、Byte

### 7.1 字符串的本质

> Go 中的字符串是**字节序列**（不一定是 rune 序列）。源代码默认 UTF-8 编码，所以字符串字面量通常按 UTF-8 存储。

**字节布局**：字符串变量在内存里是一个 16 字节的二字段描述符，指向一段**只读**字节序列：

```go
type string struct {
    ptr *byte  // 指向字节序列（指针，但目标只读）
    len int    // 字节长度
}
```

由于字节序列是**只读**的（`s[i] = ...` 编译报错），即便 `ptr` 共享，也没人能通过它改到对方——共享只是 Go 的内部省内存优化，

支持索引和切片表达式：

```go
var s string = "Hello there"
var b byte = s[6]   // 116（小写 t 的 ASCII）
var s2 = s[4:7]     // "o t"
var s3 = s[:5]      // "Hello"
```

### 7.2 byte 和 rune 是什么

`byte` 和 `rune` 是 Go 内置的两个**类型别名**，专门服务于字符串/字符场景：


| 类型     | 实质          | 占字节 | 用来表示                      |
| ------ | ----------- | --- | ------------------------- |
| `byte` | `uint8` 的别名 | 1   | **一个字节**（数值 0–255）        |
| `rune` | `int32` 的别名 | 4   | **一个 Unicode 码点**（一个"字符"） |


起别名的目的是表达**意图**：写 `[]byte` 表示"在按字节处理"，写 `rune` 表示"在按字符处理"——比 `[]uint8` / `int32` 更易读，类型本身没区别。

#### byte 与 string 是"原子 vs 集合"的关系

`byte` 是**一个**字节（值类型），`string` 是**一段**只读字节序列（描述符）。两者不是"大小不同的同类型"，而是"原子和集合"：


| 维度      | `byte`      | `string`                        |
| ------- | ----------- | ------------------------------- |
| 表示      | 一个字节（0–255） | 任意长字节序列                         |
| 内存      | 1 字节本体      | 16 字节描述符（`ptr` + `len`），指向只读字节区 |
| 字面量     | `'A'`（单引号）  | `"A"`（双引号）                      |
| 可变性     | 变量可重新赋值     | 字节序列只读，`s[i] = ...` 编译报错        |
| `len()` | 不可用         | 字节数                             |


三者的相互转换：`s[i]` 取出 string 的第 i 个 byte；`string(b)` 把单个 byte 升成长度 1 的 string；`[]byte(s)` 把 string 拷成可变字节切片。`**string` 与 `[]byte` 表示的是同一种东西（字节序列），区别只在可变性**——所以 IO 层面频繁出现 `string ↔ []byte` 转换。

#### 字面量

```go
var b byte = 'A'        // 字符字面量用单引号，值是 ASCII/Unicode 码值 65
var r rune = '中'        // 也是单引号，值是 Unicode 码点 20013
var s string = "A"      // 字符串字面量用双引号
```

> ⚠️ `'A'`（rune）和 `"A"`（string）类型不同，不能混用。

#### 字节视角 vs 码点视角

字符串本质是 UTF-8 字节序列（§7.1），所以同一段字符串可以用两种方式"看"：

```text
"中文" 的两种视角：

按字节看：[228, 184, 173, 230, 150, 135]   ← 6 个 byte（UTF-8 编码后的字节序列）
按码点看：[20013, 25991]                    ← 2 个 rune（"中" "文" 的 Unicode 码点）
```

对应的转换：

```go
s := "中文"
bs := []byte(s)          // [228 184 173 230 150 135]，6 个字节
rs := []rune(s)          // [20013 25991]，2 个码点
fmt.Println(string(rs))  // "中文"，从码点序列还原回字符串
```

何时用哪个：

- **二进制级别**的处理（网络传输、文件 IO、UTF-8 解码） → `byte` / `[]byte`
- **字符级别**的处理（统计字符数、按字符遍历、逐字符过滤） → `rune` / `[]rune`

下一节"切坏字符串"的根源，就是把"字节视角"误当成"字符视角"。

### 7.3 索引/切片字符串的陷阱 ⚠️

回顾 §7.1：`s[i]` 取的是**第 i 个字节**，`len(s)` 是**字节数**——不是字符数。而 UTF-8 是**变长编码**，每个字符（Unicode 码点）占的字节数不固定：


| 字符类别                | UTF-8 字节数 | 例子             |
| ------------------- | --------- | -------------- |
| ASCII（英文字母、数字、半角符号） | 1         | `a` `0` `!`    |
| 拉丁扩展、希腊、阿拉伯         | 2         | `é` `α` `ع`    |
| **中日韩**             | **3**     | `中` `日` `한`    |
| Emoji、稀有字符          | 4         | `🌞` `😀` `𝕏` |


#### 直观感受

```go
s := "中文"
fmt.Println(len(s))         // 6        ← 不是 2，每个汉字 3 字节
fmt.Println(s[0])           // 228      ← 一个字节的数值，不是"中"
fmt.Println(s[:1])          // "�"      ← 把"中"字切了 1/3，得到无效 UTF-8
fmt.Println(s[:3])          // "中"     ← 正好切到完整边界，OK
```

ASCII 字符串没这个坑（每字符 1 字节，"按字节切" = "按字符切"）。一旦含中文 / emoji 就得小心。

#### 演示"切坏"

```go
s := "Hello🌞"
fmt.Println(len(s))         // 9 = "Hello"(5) + 🌞(4)
fmt.Println(s[:5])          // "Hello"      ✅ 边界正好
fmt.Println(s[:6])          // "Hello\xf0"  ❌ 切到 🌞 中间，无效字节序列
fmt.Println(s[:9])          // "Hello🌞"   ✅ 全取完整
```

#### 正确做法

按"字符"操作的三种方式：

```go
s := "你好world"

// 1. for-range：自动按码点拆，i 是字节偏移
for i, r := range s {
    fmt.Printf("%d: %c\n", i, r)
    // 0: 你   3: 好   6: w   7: o   8: r   9: l   10: d
}

// 2. 转 []rune：每个元素是一个完整码点
rs := []rune(s)
fmt.Println(len(rs))          // 7（真正的字符数）
fmt.Println(string(rs[:2]))   // "你好"

// 3. 标准库 unicode/utf8
import "unicode/utf8"
utf8.RuneCountInString(s)     // 7
```

> 仅当确定字符串只含 ASCII 字符时，才能放心用 `s[i]` / `s[a:b]`。

### 7.4 字符串与切片互转

```go
var a rune = 'x'
var s string = string(a)  // "x"
var b byte = 'y'
var s2 string = string(b) // "y"

// 字符串 ↔ []byte / []rune
var s string = "Hello, ☀"
var bs []byte = []byte(s) // UTF-8 字节序列
var rs []rune = []rune(s) // 码点序列
```

> ⚠️ **常见 bug**：`string(int(65))` 返回的是 `"A"`，**不是** `"65"`。从 Go 1.15 起，`go vet` 会拦截除 rune/byte 外的 int → string 转换。
>
> 实际项目中 `string ↔ []byte` 是**最常见**的转换——网络传输、文件 IO、序列化都是字节流。`[]rune` 反而较少用，仅在按字符索引/统计字符数/逐字符过滤时才需要。

### 7.5 UTF-8 的设计权衡

§7.3 的所有陷阱都源自一件事：**字符串按字节存储，UTF-8 是变长编码**。为什么 Go（以及现代系统）选择 UTF-8 而不是定长编码？比较三种 Unicode 编码：


| 编码     | 每字符字节数                 | 特点                                         |
| ------ | ---------------------- | ------------------------------------------ |
| UTF-32 | 固定 4 字节                | 实现最简单；Unicode 实际只用 21 位，**11 位永远是 0**，浪费严重 |
| UTF-16 | 2 或 4 字节（1~2 个 16 位序列） | ASCII 仍占 2 字节；多数语种 2 字节                    |
| UTF-8  | 1~4 字节（变长）             | ASCII = 1 字节；最差 4 字节，与 UTF-32 持平           |


UTF-8 的关键设计：

- **ASCII 兼容**：U+0000~U+007F 单字节直接复用 ASCII 字节，对英文/数字/半角符号零开销，世界上多数文本因此体积最小。
- **自同步**（self-synchronizing）：每个字节首位的 bit 模式标识它是"序列起点"还是"延续字节"。从字节流任意位置都能识别出"自己是不是切在某个字符中间"——所以 §7.3 误切到字符内部时，能立刻得到 `\xf0` 这种无效字节，而不是悄悄返回错字符。
- **无字节序问题**：UTF-8 是字节流，没有 little-/big-endian 之分（UTF-16/UTF-32 必须用 BOM 标识字节序）。
- **代价**：不能按"第 i 个字符"随机访问——必须从头扫描数过去（§7.3 那些坑的根源）。

> **冷知识**：UTF-8 由 **Ken Thompson 和 Rob Pike** 在 1992 年发明，两人都是 Go 的设计者。Go 默认要求源代码以 UTF-8 编码（除非用十六进制转义），是这套世界观的延续。

### 7.6 实践建议

- 子串与码点提取：用标准库的 `strings` 和 `unicode/utf8` 包，而不是裸切片。
- 遍历字符（按码点）：下一章会用 `for-range`。

## 8. Map

### 8.1 概念与声明

map 是 Go 内置的**键值对容器**（也叫哈希表/字典）：每个 key 唯一对应一个 value，用 key 直接查到 value。对应其他语言：Python `dict`、Java `HashMap`、JavaScript `Map`、C++ `unordered_map`。

#### 类型语法：`map[KeyType]ValueType`

map 的类型由"键类型"和"值类型"两部分组成，写在中括号里和后面：

```go
map[string]int           // 键是 string，值是 int
map[int][]string         // 键是 int，值是 []string（字符串切片）
map[string]map[int]bool  // 嵌套 map：值本身又是一个 map
```

#### 三种创建方式

```go
// ① var 声明 → nil map（零值）
var nilMap map[string]int

// ② 字面量 → 空 map 或带初值的 map
empty  := map[string]int{}
totals := map[string]int{"Orcas": 1, "Lions": 2}
teams  := map[string][]string{
    "Orcas":   []string{"Fred", "Ralph", "Bijou"},
    "Lions":   []string{"Sarah", "Peter", "Billie"},
    "Kittens": []string{"Waldo", "Raul", "Ze"},
}

// ③ make → 预分配桶数提示
ages  := make(map[int][]string)        // 等价于 map[int][]string{}
ages2 := make(map[int][]string, 10)    // 第 2 个参数是预估键值对数（hint，非上限）
```

三种方式的可读/可写性：


| 方式                        | 结果          | 读         | 写                                       |
| ------------------------- | ----------- | --------- | --------------------------------------- |
| `var m map[K]V`           | **nil map** | ✅ 返回值类型零值 | ❌ panic: assignment to entry in nil map |
| `map[K]V{}` / `make(...)` | 空 map       | ✅         | ✅                                       |


字面量里键值对用 `:` 分隔，对之间用 `,` 分隔；多行写法**最后一项后面也要逗号**（Go 的自动分号插入规则要求）。

#### 字节布局

map 变量本身**就是一个 8 字节的指针**，指向 runtime 的哈希表结构（`runtime.hmap`）：

```go
type Map = *hmap   // map[K]V 变量在内存里就是这个指针
```

`y := x` 之后 x 和 y 是两份指针、指向同一张哈希表——所以**map 的值传递天然就是"共享同一张表"**。函数里往参数 map 写键值对，调用方立刻看得见，不需要返回值（这跟切片的 `append` 必须赋回不一样）。

#### nil map vs 空 map

字节布局直接解释了两者的差异：

```text
nil map:  m → nil                   ← 指针为 nil，runtime 还没分配 hmap，桶都不存在
空 map:   m → hmap{count:0, ...}    ← hmap 已分配，只是 count=0
```

由此推出全部行为差异：


| 行为             | nil map                     | 空 map   |
| -------------- | --------------------------- | ------- |
| `m == nil`     | `true`                      | `false` |
| 读 `m[k]`       | ✅ 返回零值（runtime 看到 nil 跳过查桶） | ✅ 返回零值  |
| 写 `m[k] = v`   | ❌ **panic**（没有桶可以写入）        | ✅       |
| `delete(m, k)` | ✅ no-op                     | ✅       |
| `len(m)`       | 0                           | 0       |
| `for range m`  | ✅ 0 次循环                     | ✅ 0 次循环 |
| `clear(m)`     | ✅ no-op                     | ✅       |


**关键**：除了"写入"，nil map 几乎处处当空 map 用——读、删、`len`、`range` 都不会报错。**唯独 `m[k] = v` 会 panic**，因为底层桶根本不存在。

何时用哪个：

- **空 map**：要立刻写数据 → `map[K]V{}` 或 `make(...)`。
- **nil map**：表达"没有数据"的语义。常见场景：函数返回值（"没拿到任何键值对"，调用方用 `len(m) == 0` 处理）；struct 字段占位待后续赋值；函数参数（调用方传 nil 表示"不给配置"，函数内只读不写就安全）。

```go
func loadConfig(path string) map[string]string {
    if path == "" {
        return nil   // 调用方读它跟读空 map 一样
    }
    return parseFile(path)
}
```

### 8.2 Map 是哈希表

map 在 runtime 里是一张哈希表。哈希表运转需要两个零件：

- **哈希函数**：把 key 转成桶的下标。
- **相等性判定**：哈希冲突时确认 key 是不是真的相同。

Go 的设计是把这两件事交给编译器和 runtime——对每个可比较类型自动生成 hash 和 `==`，调用方既不写、也不能改。这跟 Java 的 `hashCode()` / `equals()`、Python 的 `__hash__` / `__eq__` 不同，后两者把这两个钩子开放给用户重写。

直接推出三条：

- **键必须是可比较类型**：slice / map / func 不可比较，编译器不会为它们生成哈希函数 → `map[[]int]string` 直接编译报错 `invalid map key type`。
- **map 本身不可比较**（除了与 `nil` 比较）。要判断两张 map 是否相等，用 `maps.Equal`（§8.6）。
- **判定永远是逐字节相等**：`"Foo"` 与 `"foo"` 不会被当作同一个 key。要这种语义，只能在写入前自己 `strings.ToLower` 归一化。

> "可比较类型"是贯穿全章的概念，完整清单与 map 键的合法/非法示例见 [§10.2 可比较类型详解](#102-可比较类型详解)。

### 8.3 读写

```go
totalWins := map[string]int{}
totalWins["Orcas"] = 1
totalWins["Lions"] = 2
fmt.Println(totalWins["Orcas"])    // 1
fmt.Println(totalWins["Kittens"])  // 0  ← 键不存在返回零值
totalWins["Kittens"]++             // 由零值 0 自增到 1
```

### 8.4 comma ok 惯用法

区分"键存在但值为零"与"键不存在"：

```go
m := map[string]int{"hello": 5, "world": 0}
v, ok := m["hello"]   // 5  true
v, ok = m["world"]    // 0  true
v, ok = m["goodbye"]  // 0  false
```

### 8.5 删除与清空

```go
delete(m, "hello")  // 不存在或 m 为 nil 都不会报错；无返回值
clear(m)            // 清空所有键值对，长度变为 0（与切片不同）
```

### 8.6 Map 比较（Go 1.21+）

```go
import "maps"
maps.Equal(m, n)
maps.EqualFunc(m, n, func(a, b V) bool { ... })
```

### 8.7 用 Map 模拟集合

Go 没有内置 set，可用 `map[T]bool`：

```go
intSet := map[int]bool{}
vals := []int{5, 10, 2, 5, 8, 7, 3, 9, 1, 2, 10}
for _, v := range vals {
    intSet[v] = true
}
fmt.Println(len(vals), len(intSet)) // 11 8
fmt.Println(intSet[5])              // true
fmt.Println(intSet[500])            // false
```

> 也可以用 `map[T]struct{}` 节省内存（空 struct 占 0 字节，bool 占 1 字节），但赋值与判断更繁琐，需要用 comma ok。仅在集合非常大时这点内存差异才值得权衡。
>
> Go 标准库**不提供**集合的并集/交集/差集运算，需要自己实现或使用第三方库。

### 8.8 切片 vs Map：何时用哪个

- **切片**：顺序敏感、按位置访问的列表型数据。
- **map**：用非整数自然键（如名字）组织的数据。

## 9. Struct（结构体）

### 9.1 声明与字面量

```go
type person struct {
    name string
    age  int
    pet  string
}

var fred person                      // 零值结构体：所有字段为字段类型的零值
bob := person{}                      // 与上等价
julia := person{                     // 位置字面量：必须按声明顺序提供所有字段
    "Julia",
    40,
    "cat",
}
beth := person{                      // 命名字面量：可乱序、可省略
    age:  30,
    name: "Beth",
}
bob.name = "Bob"                     // 字段访问用点号
```

两种字面量风格的定义：


| 风格                  | 写法                              | 顺序                    | 可省略字段       |
| ------------------- | ------------------------------- | --------------------- | ----------- |
| **位置式**（positional） | `person{"Julia", 40, "cat"}`    | 必须严格匹配 `type` 里字段声明顺序 | ❌ 所有字段必须给值  |
| **命名式**（named）      | `person{name: "Beth", age: 30}` | 任意                    | ✅ 未指定的字段填零值 |


> 同一字面量中**不能混用**两种风格——要么三个字段都写名字，要么都不写：
>
> ```go
> mix := person{"Julia", age: 40, "cat"}  // ❌ 编译报错
> ```
>
> 推荐**始终用命名式**：可读、可维护，且 struct 新增字段时旧的位置式字面量会编译报错（"too few values"），命名式不会受影响。位置式只在 struct 字段极少（2~3 个）且语义稳定时才方便。

#### 与 map 的两点区别

**作用域**：`type` 可声明在包级别、函数内、甚至任意 block 内（`if` / `for` 块也行）。函数/块内声明的 struct 类型**只在该作用域内可用**，外面引用编译报错。包级别声明则全包可见。

```go
func main() {
    type point struct{ x, y int }   // 仅 main 作用域内有效
    p := point{1, 2}
    _ = p
}
// 这里再写 var q point ❌ 编译错误：未定义 point
```

**空字面量与 var 声明完全等价**：`var bob person` 与 `bob := person{}` 没有任何区别——struct 本体就是字段紧挨排列的字节，零值已经按字段类型就位，"未赋值"和"赋空字面量"得到的是同一份零值字节。

```go
var bob person       // {"" 0 ""}
bob2 := person{}     // {"" 0 ""}  — 完全一样
```

> 与 map 对照：`var m map[K]V` 是 **nil map**，写入 panic；`m := map[K]V{}` 才是空 map，可写。差异的根源在 §0：map 变量本身是个指针，零值 nil 指向"无"；struct 变量本身就是数据，零值即已就位。

### 9.2 匿名结构体

```go
var person struct {
    name string
    age  int
    pet  string
}
person.name = "bob"
person.age = 50

pet := struct {
    name string
    kind string
}{
    name: "Fido",
    kind: "dog",
}
```

适用场景：

1. JSON / Protobuf 等数据序列化（unmarshal/marshal）。
2. 表驱动测试。

### 9.3 结构体的可比较性

- 字段全为可比较类型 → 整个结构体可比较。
- 含有 slice、map、function、channel 字段 → 不可比较。
- Go 没有运算符重载，不能像 Python 那样定义 `__eq__`，要比较只能自己写函数。

### 9.4 结构体类型转换

允许在结构体之间做类型转换的条件：**字段名、顺序、类型完全一致**。

```go
type firstPerson struct {
    name string
    age  int
}
type secondPerson struct { // 字段名/顺序/类型都相同 → 可转换
    name string
    age  int
}
type thirdPerson struct {  // 顺序不同 → 不可转换
    age  int
    name string
}
type fourthPerson struct { // 字段名不同 → 不可转换
    firstName string
    age       int
}
type fifthPerson struct {  // 多了字段 → 不可转换
    name          string
    age           int
    favoriteColor string
}
```

> **匿名结构体的特殊待遇**：当至少有一方是匿名结构体且字段一致时，可以直接 `==` 比较，也可以直接赋值，无需类型转换。

```go
type firstPerson struct {
    name string
    age  int
}
f := firstPerson{name: "Bob", age: 50}
var g struct {
    name string
    age  int
}
g = f             // 合法
fmt.Println(f == g) // 合法
```

### 9.5 含指针字段的 struct 复制行为

**字节布局**：struct 本体就是字段紧挨着排（带对齐填充），没有任何额外的描述符。复制 struct 等于把它的全部字节复制一份——但**字段是不是指针类型，决定了那段字节复制后是不是还连着原数据**。

```go
type Mixed struct {
    a int       // 8 字节，无指针 → 复制后独立
    s []int     // 24 字节切片描述符，含 ptr → 描述符独立、底层数组共享
}

x := Mixed{a: 1, s: []int{10, 20, 30}}
y := x          // 整个 struct 32 字节全部复制
y.a = 99        // ✅ 不影响 x（a 字段独立）
y.s[0] = 99     // ❌ 影响 x（s.ptr 共享同一底层数组）

fmt.Println(x)  // {1 [99 20 30]}
fmt.Println(y)  // {99 [99 20 30]}
```

同一个 struct，**不同字段的共享行为不一样**——必须**逐字段查 §0.3 的字节布局表**：含指针字段（切片、map、channel、接口、`*T`、含指针的子 struct）→ 共享指向的目标；纯数据字段（int、bool、定长数组、不含指针的子 struct）→ 完全独立。

## 10. 本章重点回顾

### 10.1 各类型概览


| 类型            | 是否可比较        | 长度可变 | 零值     | 备注           |
| ------------- | ------------ | ---- | ------ | ------------ |
| 数组 `[N]T`     | ✅（元素可比较时）    | ❌    | 元素全为零值 | 长度是类型的一部分，少用 |
| 切片 `[]T`      | ❌（仅可与 nil 比） | ✅    | `nil`  | 最常用序列容器      |
| 字符串 `string`  | ✅            | ❌    | `""`   | 不可变字节序列      |
| map `map[K]V` | ❌（仅可与 nil 比） | ✅    | `nil`  | 键必须可比较       |
| struct        | 视字段而定        | ❌    | 各字段零值  | 自定义聚合类型      |


### 10.2 可比较类型详解

定义：**能直接用 `==` / `!=` 比较的类型**。这套规则是 §1.3 数组比较、§2.3 切片不可比、§8.2 map 键、§9.3 struct 比较共用的底层概念。

**可比较类型**：


| 类型                                        | 比较行为              |
| ----------------------------------------- | ----------------- |
| 所有数值类型（int / uint / float / complex 各种宽度） | 按值                |
| `bool`                                    | 按值                |
| `string`                                  | 逐字节相等             |
| 指针 `*T`                                   | 指向同一地址才相等         |
| `channel`                                 | 同一个底层 channel 才相等 |
| `interface`                               | 动态类型 + 动态值都相同才相等  |
| 数组 `[N]T`                                 | 元素类型 T 可比较时；逐元素比  |
| struct                                    | 所有字段都可比较时；逐字段比    |


**不可比较类型**：


| 类型              | 用 `==` 会怎样 |
| --------------- | ---------- |
| `slice`         | 编译报错       |
| `map`           | 编译报错       |
| `func`          | 编译报错       |
| 含上面任一字段的 struct | 编译报错       |


唯一例外：slice / map / func **可以与 `nil` 比较**（语言特殊语法）：

```go
var s []int
s == nil      // ✅ 合法
s == []int{}  // ❌ slice can only be compared to nil
```

**为什么 slice / map / func 不可比较**：它们都含指针（slice 描述符、map 头指针、闭包），按地址比没意义（内容相同会判不等），按内容深比代价大且语义不清——Go 索性不定义。

**用在 map 键上**：

```go
map[int]V       // ✅
map[string]V    // ✅
map[*User]V     // ✅ 按指针地址作 key
map[[3]int]V    // ✅ 数组
map[Point]V     // ✅ struct，前提是 Point 字段全可比较

map[[]int]V     // ❌ slice
map[map[K]V]V   // ❌ map
map[func()]V    // ❌ func
```

**特例**：`map[interface{}]V` 编译期能过（interface 类型本身可比较），但运行时若把 slice / map / func 包进 interface 作 key，会 panic：`runtime error: hash of unhashable type`。

### 10.3 最容易踩的坑

1. `append` 一定要把返回值赋回原变量。
2. 切片的子切片**共享底层数组**，必要时用三段切片表达式或 `copy` 隔离。
3. 字符串切片以**字节**为单位，跨多字节 UTF-8 字符会切坏。
4. nil map 可读不可写，写入会 panic；nil 切片可以直接 `append`。
5. `make([]T, n)` 创建的切片已有 n 个零值元素，再 `append` 是**追加在后面**，不是覆盖。
