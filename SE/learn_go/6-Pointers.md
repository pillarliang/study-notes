# 第 6 章 指针（Pointers）

## 1. 指针的本质

### 1.1 概念

**指针是一个变量，其存储的值是另一个变量在内存中的地址。**

- 任何变量都存放在一段连续的内存地址（address）中。不同类型占用的字节数不同：`int32` 占 4 字节，`bool` 占 1 字节。
- 指针本身的大小不随被指向类型变化：现代机器上通常固定为 8 字节（64 位）。
- 指针的零值是 `nil`，表示"未指向任何位置"。
- `nil` 是 Go universe block 中的无类型标识符，不是 C 中的 `NULL`，不能与数字相互转换；同时由于定义在 universe block，可被局部变量遮蔽（不要把变量命名为 `nil`）。

> slice、map、function、channel、interface 在 Go 内部都是用指针实现的，它们的零值也是 `nil`。

### 1.2 语法骨架

```go
// 取址 & ：拿到值所在地址
var x int32 = 10
pointerX := &x          // pointerX 类型为 *int32

// 解引用 * ：拿到指针指向的值
fmt.Println(*pointerX)  // 10
z := 5 + *pointerX      // 15

// 指针类型 ：在类型名前加 *
var p *int              // p 是 *int，零值为 nil

// 内置 new ：分配一块零值内存并返回指针
// 签名：func new(T) *T —— 输入类型，输出指向该类型零值的指针
var q = new(int)        // q 指向一块新分配的 int 内存
                        // q != nil（内存已分配），*q == 0（int 零值）
```

> 对比：`var p *int` 仅声明指针变量，未分配内存，`p == nil`，解引用会 panic。
> `new` 在实际代码中很少用——struct 一般写 `&Foo{}` 更直观；它的存在意义是为"没有字面量可取地址"的场景提供统一写法。

### 1.3 安全约束

- 解引用前必须确认指针非 `nil`，否则 panic。
- Go 没有 C/C++ 那种指针算术；如果确实需要低级操作，可使用 `unsafe` 包，但极少这么做。

## 2. 取址的限制与 `makePointer` 模式

### 2.1 限制

**`&` 取的是"变量"的地址，不是"值"的地址。**

为什么字面量没有地址？分清两个时段：

- **编译期**：源代码 → 可执行文件。
- **运行期**：可执行文件被加载、CPU 执行指令。

编译器看到 `var x int = 10` 时，会"预留一段内存放 x，并生成把 10 写进去的指令"。运行期那段内存真实存在 → 有地址 → `&x` 合法。
编译器看到字面量 `10` 时，会直接把 10 **嵌进指令本身**（立即数）；整个程序中没有任何一块内存专门存放它，因此运行期"取 10 的地址"无意义。

由此推出：

- **变量**（`var y T` 或 `y := ...`）一定在内存中占据空间，即使未显式赋值，也会被初始化为类型零值，因此可取址。
- **字面量与常量**（`"hello"`、`42`、`true`）不占运行时存储，不能取址。
- **复合字面量**（`Foo{}`、`[]int{1,2}`）是例外——Go 会为其分配存储，可取址。

> 严格地说，字符串字面量在二进制里位于只读数据段、确实有物理地址；但 Go 在语法层统一禁止对所有字面量取址，以避免"有的能、有的不能"的不一致。

```go
x := &Foo{}        // OK：复合字面量被分配存储
var y string       // y 是变量，零值 ""，占内存
z := &y            // OK：取变量 y 的地址
// w := &"hello"   // 编译错误：字面量没地址
// w := &42        // 编译错误：同上
```

### 2.2 推论：结构体字段是 `*T` 时无法直接赋字面量

```go
type person struct {
    FirstName  string
    MiddleName *string
    LastName   string
}

p := person{
    FirstName:  "Pat",
    MiddleName: "Perry",   // 编译错误：cannot use "Perry" (type string) as type *string
    LastName:   "Peterson",
}
```

### 2.3 解决办法：泛型辅助函数

```go
func makePointer[T any](t T) *T {
    return &t
}

p := person{
    FirstName:  "Pat",
    MiddleName: makePointer("Perry"), // OK
    LastName:   "Peterson",
}
```

> 原理：常量传入函数时被复制成形参 `t`，`t` 是变量、有地址，因此 `&t` 合法。

## 3. 指针与"类对象"的对比

### 3.1 核心观察

**Java/JS/Python/Ruby 中的"class 实例"在底层就是指针。**

### 3.2 同一行为：Python 与 Go 对照

Python 中 class 实例的传参表现：

```python
class Foo:
    def __init__(self, x): self.x = x

def inner1(f): f.x = 20        # 修改字段
def inner2(f): f = Foo(30)     # 形参重指向新实例

f = Foo(10)
inner1(f); print(f.x)          # 20  ← 外部可见
inner2(f); print(f.x)          # 20  ← 外部不可见！
```

Go 用 `*T` 写出语义完全一致的代码：

```go
type Foo struct{ x int }

func inner1(f *Foo) { f.x = 20 }
func inner2(f *Foo) { f = &Foo{x: 30} }

f := &Foo{x: 10}
inner1(f); fmt.Println(f.x)    // 20
inner2(f); fmt.Println(f.x)    // 20
```

#### 为什么 `inner2` 改不动外部？

调用函数时，**形参拷贝的是那个指针的值**（指向同一对象）：

```text
外部 f ──┐
         ├──► Foo(x=20)
形参 f ──┘
```

`f = Foo(30)` 在堆上新建对象，把**形参指针**指向新对象；外部指针完全没动：

```text
外部 f ────► Foo(x=20)
形参 f ────► Foo(x=30)
```

函数返回后形参消失，外部 f 仍指向原对象，所以 `f.x == 20`。
这条原理在 Go 中对应 §4.3 的 `failedUpdate` vs `update`。

### 3.3 行为对照表

| 现象 | JS/Python（class 实例） | Go（`*T` 参数） |
| --- | --- | --- |
| 修改字段对调用方可见 | ✅ | ✅ |
| 形参整体重新赋值对调用方不可见 | ✅ | ✅ |
| 传 `nil/None` 后无法在函数里改变外部变量 | ✅ | ✅（见 §4） |

### 3.4 常见误解：那些语言"按引用传递"

教材常说 Python/Java 的对象是 **pass by reference**，本书反驳：

- 真按引用传递的话，`inner2` 重新赋值会影响外部——但事实并非如此。
- 实际上仍是 **pass by value**，只不过传过去的"值"本身是一个指针。

### 3.5 Go 的差异点

把上面三节的现象归纳起来：所谓"class 实例"行为本质就是指针语义，只不过其他语言把这层指针隐藏在语法背后。Go 没有发明新概念，区别只在于：

| 类别 | Java/JS/Python/Ruby | Go |
| --- | --- | --- |
| 基本类型传参 | 按值 | 按值 |
| 复合类型/对象传参 | **强制按指针**（隐式） | **可选**：值 或 `*T`（显式） |

两点变化：

- **显式化**：Go 把指针写在类型上（`*T`），读代码时立刻知道"这里会发生指针语义"，不必死记某种类型默认是值还是引用。
- **可选化**：Go 允许 struct 也按值传——其他语言对对象别无选择，必然走指针。

> 推论：日常优先用值类型——数据流更清晰、GC 压力更小；只有确需修改外部数据时才用 `*T`。这条原则在后面 §8"指针应作为最后手段"展开。

## 4. 指针标记"可变参数"

### 4.1 原则

- Go 没有 `immutable` 关键字，惯例上**用 `*T` 形参显式表示该参数会被函数修改**。
- 由于 Go 是按值传递，非指针类型（基本类型、结构体、数组）传入函数都是副本，函数无法修改原值。
- 指针传入函数时，函数得到的是**指针的副本**，但副本仍指向同一份原始数据，故可通过解引用修改原数据。

### 4.2 推论 1：传入 `nil` 指针不能让它变成非 `nil`

```go
func failedUpdate(g *int) {
    x := 10
    g = &x          // 修改的是副本 g，不影响调用方
}

func main() {
    var f *int      // f == nil
    failedUpdate(f)
    fmt.Println(f)  // nil
}
```

### 4.3 推论 2：要让赋值在函数外可见，必须解引用而非重新赋值

```go
func failedUpdate(px *int) {
    x2 := 20
    px = &x2        // 改副本，外部不可见
}

func update(px *int) {
    *px = 20        // 改 px 指向的内存，外部可见
}

func main() {
    x := 10
    failedUpdate(&x)
    fmt.Println(x)  // 10
    update(&x)
    fmt.Println(x)  // 20
}
```

#### `*px = 20` 在做什么

`px` 本身就是指针变量（类型 `*int`），`*px` 是对它施加**解引用**——顺着 px 走到它指向的 int 内存。注意 `*` 在 Go 中身兼两职：

| 出现位置 | 含义 |
| --- | --- |
| 类型位置（`var px *int`） | "...的指针"，构造类型 |
| 表达式位置（`*px = 20`） | 解引用，顺着指针走 |

放在 `=` 左边时，`*px = 20` 表示"把 20 写进 px 指向的那块内存"，并非定义新指针。

内存图（设外部 `x` 在地址 100，初值 10）：

```text
进入 update(&x)：
外部 x : 地址 100，值 10
形参 px: 地址 200，值 100   （px 也是变量，里面存着 x 的地址）

执行 *px = 20：
外部 x : 地址 100，值 20   ← 沿 px 写入这里
形参 px: 地址 200，值 100   ← 自身没动
```

对比 `failedUpdate` 的 `px = &x2`——改的是 px 自己存的内容（指向哪儿），原 x 没动：

```text
执行 px = &x2：
外部 x : 地址 100，值 10    ← 没动
形参 px: 地址 200，值 300   ← 现在指向 x2
x2    : 地址 300，值 20
```

**一句话区别：**

- `*px = ...` 顺指针走，写到指向的地方 → 外部可见
- `px  = ...` 改的是指针自己 → 仅副本变，外部不可见

## 5. 用 `nil` 区分"零值"与"未赋值"

### 5.1 问题

普通字段无法区分"用户显式赋了零值"与"根本没赋值"——`int` 字段读到 `0` 时分不清是用户写了 0 还是缺省。

例：反序列化 `{"name": "Alice"}`（没传 `age`）：

```go
type User struct {
    Name string
    Age  int
}
// 反序列化后 u.Age == 0
// 但用户究竟是没填，还是真填了 0？分辨不出
```

### 5.2 解决：用 `*T` 表示"可空字段"

```go
type User struct {
    Name string
    Age  *int   // nil = 没传；&0 = 显式传了 0
}
```

### 5.3 注意

- 因为 `*T` 同时暗示可变性，滥用会让 API 更难推理。
- **函数中如需"无返回值"含义，优先用 comma-ok 模式**（见下）；只在结构体字段需要表达"未赋值"时才用 `*T`。
- JSON 反序列化是合理使用 `*T` 表示"可空字段"的典型例外。

### 5.4 comma-ok 模式

Go 惯用的双返回值：`(值, bool)`，bool 表示该值是否有效。内置已大量使用：

```go
v, ok := m["key"]      // map 取值：ok=false 表示键不存在
v, ok := <-ch          // channel 接收：ok=false 表示已关闭
v, ok := x.(string)    // 类型断言：ok=false 表示类型不匹配
```

自定义函数同理。对比"按 ID 查用户、可能查不到"两种写法：

```go
// 方案 A：返回 *User，nil 表示没找到
func FindUser(id int) *User { /* ... */ }
u := FindUser(42)
fmt.Println(u.Name)    // 危险：忘判 nil 直接 panic

// 方案 B：comma-ok
func FindUser(id int) (User, bool) { /* ... */ }
u, ok := FindUser(42)
if !ok { return }
fmt.Println(u.Name)    // 安全：u 一定有效
```

为什么优于返回 `*T`：

| 维度 | `*T`（nil = 无值） | `(T, bool)`（comma-ok） |
| --- | --- | --- |
| 签名意图 | 含混——既可能"表示无值"也可能"想让调用方修改" | 一望即知是"可能查无此值的查询" |
| 安全性 | 忘记判 nil → panic | 拿到的是值，访问字段不会炸 |
| 与 §8 一致性 | 引入指针，对象可能逃逸到堆 | 仍然走值 |

## 6. Map 与 Slice 的差异

### 6.1 Map 在运行时是"指向 struct 的指针"

- 传 map 给函数等价于传指针；函数内对内容的任何修改都会反映到原变量。
- API 设计层面 map 不自描述、不易追踪，**公共 API 应优先使用 struct 而非 map**；只有键集合在编译期未知时 map 才是合适选择。

### 6.2 Slice 是包含三字段的 struct

```text
slice = { array *T, len int, cap int }
```

传 slice 给函数时，**复制的是这三个字段**（指针、长度、容量），但 `array` 字段仍指向同一块底层数组。由此推出：

| 在函数内的操作 | 调用方是否可见 |
| --- | --- |
| 修改下标处的元素（`s[i] = v`） | ✅ 可见 |
| `append` 且**未扩容**（在原容量内增长） | ❌ 长度变化对调用方不可见 |
| `append` 且**触发扩容**（分配新底层数组） | ❌ 全部不可见 |

> 由于这种半透明语义，**默认假设函数不会修改传入 slice 的内容**；若会修改，应在文档中显式声明。

### 6.3 为什么 Go 用 slice 而不是数组传参

**Go 的数组把长度编码进类型本身**：`[3]int` 与 `[5]int` 是**两个不同的类型**。所以函数 `func f(a [3]int)` 只能接受长度恰好为 3 的数组——既不通用，且整个数组会被按值复制。

slice 解决这两个问题：

- 类型 `[]int` 与底层数组长度无关，能接受任意大小；
- slice 实参只是 24 字节的 header，传参便宜。

> 反过来推：如果 Go 不区分数组与 slice，就要在"灵活类型"与"低成本传参"之间二选一。slice = 把"指针 + 长度 + 容量"打包成一个固定大小的 struct，正好同时满足两边。

## 7. Slice 作为缓冲区（Slices as Buffers）

避免在循环里反复分配新内存，**一次性分配 slice 后反复重用**：

```go
file, err := os.Open(fileName)
if err != nil {
    return err
}
defer file.Close()

data := make([]byte, 100) // 复用的缓冲区
for {
    count, err := file.Read(data)
    process(data[:count])
    if err != nil {
        if errors.Is(err, io.EOF) {
            return nil
        }
        return err
    }
}
```

**`Read` 是"最多读 `len(p)` 字节"，不保证读满**。所以每轮 `count` 可能小于 100——尤其是网络读取或文件末尾。`data[:count]` 切出本轮有效字节，避免把上一轮残留的数据当成新数据处理。

设文件长 250 字节，`make([]byte, 100)` 分配的同一块 100 字节内存被整个循环复用：

```text
迭代 1: Read 写入前 100 字节 → count=100
        data 内存: [b0..b99]              process(data[:100])

迭代 2: 同一块内存被覆盖写入下 100 字节 → count=100
        data 内存: [b100..b199]            process(data[:100])

迭代 3: 文件只剩 50 字节 → count=50
        data 内存: [b200..b249, 残留..]    process(data[:50])

迭代 4: count=0, err=io.EOF → 退出
```

为什么这个模式成立，分两层看：

- **机制层**：`file.Read` 通过 slice 内部的指针，把字节写到底层数组的前 `count` 个位置——对应 §6.2 表中"修改下标处元素 → 调用方可见"。函数内即便调用 `append` 改变了局部 slice 的 `len/cap`，调用方的 slice header 也不会受影响（§6.2 已说明）。
- **设计层**：`Read` 这个 API **故意只在 `len` 范围内写、不调用 append**。配合外层一次 `make(100)` 分配，整个循环复用同一块底层数组——避免了每次迭代都新分配带来的 GC 压力（同 §8.3 中 `Unmarshal` 的设计动机）。

## 8. 指针应作为"最后手段"

> 本节是**风格/惯例**层面的内容，初读可跳过；等到实际写 Go 代码、纠结"参数到底用 `Foo` 还是 `*Foo`"时再回看。

### 8.1 原则

承接前面：

- §3：Go 把"用值还是用 `*T`"的选择权交给开发者。
- §4：函数确实要修改外部数据时，**必须**用 `*T`。

**那么除此之外呢？Go 的惯用风格是——能用值就用值，指针是最后手段。**

理由：

- **数据流更可读**：`func f(x Foo)` 一望即知不会改外部状态；`*Foo` 则迫使调用方假设可能被改写。
- **GC 压力更小**：值往往随栈帧自然释放；指针更易让对象逃逸到堆（详见 §10）。
- **风格惯例**：Go 鼓励把数据像不可变那样使用，仅在必要时显式引入 `*T`。

### 8.2 反模式：把指针当"输出参数"

C/C++ 程序员常写"传入空结构体、由函数往里填字段"的模式（即 out parameter），在 Go 中不推荐：

```go
// 反模式：让调用方先造空 Foo，再传指针进来填
func MakeFoo(f *Foo) error {
    f.Field1 = "val"
    f.Field2 = 20
    return nil
}
```

问题：

- 调用方多一步——必须先 `var f Foo` 再 `MakeFoo(&f)`；
- 函数签名传达的是"我会修改你的 Foo"，但实际意图是"我给你造一个 Foo"；
- 本可以留在栈上的 Foo 因为地址被传给函数而可能逃逸到堆。

直接返回值更清晰：

```go
func MakeFoo() (Foo, error) {
    f := Foo{Field1: "val", Field2: 20}
    return f, nil
}
```

### 8.3 例外：函数期望 interface 类型（典型如 `json.Unmarshal`）

某些函数必须把结果**写回**调用方提供的变量。典型是 `json.Unmarshal`：

```go
f := struct {
    Name string `json:"name"`
    Age  int    `json:"age"`
}{}
err := json.Unmarshal([]byte(`{"name": "Bob", "age": 30}`), &f)
```

`Unmarshal` 形参类型是 `any`（interface），必须接受指针，原因有二：

1. 该 API 早于泛型，函数无法在编译期得知"要构造什么类型"——靠调用方传指针告诉它"写到这里"；
2. 调用方掌握内存分配；循环中可复用同一个目标变量，避免每次都造新对象、加重 GC。

### 8.4 返回值的总原则

- **默认返回值类型**。
- 只有当返回的数据**内部状态需要被外部修改**时才返回 `*T`。
- I/O 缓冲区（§7 已展开）、并发原语（第 12 章）这类天然就要"被多处共享"的类型，必须用指针。

## 9. 指针传递的性能

### 9.1 底层原理

- **传入**指针总是固定开销（拷贝一个机器字，约 1 ns），与数据大小无关；传入值则随数据变大线性变慢（拷贝整块内存）。
- **返回**指针会迫使被指对象逃逸到堆，**堆分配本身有固定开销**；返回值则可留在栈上、随栈帧释放，对小数据极快。

由此推出"返回小数据反而是值更快"这个反直觉结论：堆分配的固定开销 > 栈拷贝的小数据开销。当数据量大到拷贝代价超过堆分配代价时，指针才胜出。

### 9.2 经验数据（i7-8700）

| 操作 | 行为 |
| --- | --- |
| 传入指针 | 与数据大小无关，约 1 ns |
| 传入值 | 随数据增大变慢；约 10 MB 时达 0.7 ms |
| 返回 ≤10 MB 数据时返回指针 vs 返回值 | **返回指针更慢**（100 字节：30 ns vs 10 ns） |
| 返回 ≥10 MB 数据 | 返回指针快得多（约 0.5 ms vs 1.5 ms） |

> 结论：绝大多数场景两者差异可忽略；只有传/返 MB 级别数据时才考虑用指针，即便数据是不可变的。具体临界值因 CPU 架构而异，可通过 `go test ./… -bench=.` 自测。

## 10. 减轻 GC 工作量

### 10.1 栈与堆

- **栈**：连续内存，函数调用时压入 stack frame，存放局部变量与参数；通过移动 stack pointer 完成分配/释放，速度极快。
- **堆**：由 GC 管理的内存；分配开销大，回收需要运行 GC 算法。
- Go 1.17 起还会用寄存器配合栈传递参数与返回值。
- Go 的 goroutine 栈由运行时管理，可在运行中扩张/收缩；优点是初始小、占用少，缺点是扩张时需要拷贝。

### 10.2 何时分配在堆上（escape analysis）

只有满足以下全部条件才会分配在栈：

- 数据是局部变量；
- 编译期可知大小；
- 指针不会被函数返回；
- 把指针传入其他函数时，编译器仍能保证上述条件。

不满足任意一条 → 数据"逃逸（escape）到堆上"。

> Go 编译器对返回局部变量地址做了正确处理：会自动把该变量分配到堆，避免 C 中"返回局部变量指针"那种悬垂指针。
> 可用 `go build -gcflags="-m"` 查看哪些变量发生了逃逸。

### 10.3 为什么避免堆分配

1. **GC 时间成本**：Go GC 偏向低延迟（每轮 stop-the-world < 500μs）；垃圾过多时无法在一轮内清理完，整体变慢、内存占用变大。
2. **机械同情心（mechanical sympathy）**：顺序读取 RAM 远快于随机访问。
   - `[]Struct`：连续布局，缓存友好。
   - `[]*Struct` 或字段是指针的 struct：数据散落在堆上，访问慢约两个数量级。

### 10.4 与 Java 的对比

- Java 中所有对象本身都在堆上（栈只放指向对象的指针），`List` 实现是"指向指针数组"，看似线性其实跳着访问；GC 工作量极大。
- Python/Ruby/JavaScript 的顺序容器也有类似问题。
- Go 通过把 struct/基本类型直接放在栈和顺序内存里，大幅减小 GC 与缓存压力——这就是惯用 Go 风格自然带来的性能优势。

## 11. 调优 GC

### 11.1 `GOGC`

- 公式：`目标堆大小 = CURRENT_HEAP_SIZE + CURRENT_HEAP_SIZE * GOGC / 100`
- 默认 `GOGC=100`，即下次 GC 触发于堆大小翻倍之时。
- 经验法则：**`GOGC` 翻倍 → GC 占用 CPU 时间约减半**，但代价是堆更大。
- `GOGC=off` 关闭 GC，长时间运行进程会耗尽内存，通常不建议。

### 11.2 `GOMEMLIMIT`

- 类似 JVM 的 `-Xmx`，指定 Go 程序总内存上限。
- 单位支持 `B/KiB/MiB/GiB/TiB`（2 的幂）。例：`GOMEMLIMIT=3GiB` = 3,221,225,472 字节。
- 默认禁用（设为 `math.MaxInt64`）。
- 是**软限制**：当持续触顶导致 GC 颠簸（thrashing）时，运行时会主动让其超限；因此实际配置应略低于物理上限以留缓冲。

### 11.3 推荐用法

同时使用 `GOGC` 与 `GOMEMLIMIT`：前者控制节奏，后者避免被 OS 换页或 OOM。仅设 `GOMEMLIMIT` 而把 `GOGC` 关掉会从"频繁短暂停"变成"少而长的暂停"，对 Web 服务的尾延迟不利。

## 12. 速查总结

| 主题 | 关键结论 |
| --- | --- |
| 是否使用指针 | 默认用值；指针是最后手段 |
| 函数参数 | `*T` 形参 = 显式声明"会修改" |
| 函数返回 | 默认返回值；除非状态需被外部修改、或数据 ≥ MB 级 |
| `nil` 表"无值" | 仅当区别有意义时；优先 comma-ok |
| Map 入参 | 行为像指针；公共 API 优先 struct |
| Slice 入参 | 改内容可见；改长度/容量不可见；当复用缓冲区时收益明显 |
| 性能要点 | 顺序内存优于指针散落；减少堆分配以减轻 GC |
| GC 调优 | `GOGC` 控节奏，`GOMEMLIMIT` 控上限 |
