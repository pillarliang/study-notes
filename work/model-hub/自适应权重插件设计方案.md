# 自适应权重插件（Adaptive Weight）设计方案

> 本文档是 [Plaud Model Hub 完全指南](Plaud%20Model%20Hub%20完全指南.md) 的补充，描述新增的自适应权重插件的设计动机、技术方案和配置方式。

> **源码路径缩写**：`core/` 代表 `packages/core/src/model_hub_core/`，`sdk/` 代表 `packages/sdk-python/src/model_hub_sdk/`。

## 一、问题背景

### 1.1 现有限流机制的局限

当前 Model Hub 的限流器（Rate Limiter）对 429 的处理是**二值的**——endpoint 要么可用，要么完全不可用：

```
正常（weight=60）──429──→ 被动限流（不可用，Router 跳过）──过期──→ 正常（weight=60）
```

这在多 endpoint 场景下存在一个风险：

```yaml
project-summary:auto:
  endpoints:
  - provider: vertex
    model: gemini-2.5-pro
    weight: 60
  - provider: litellm
    model: gpt-4o
    weight: 30
  - provider: litellm
    model: claude-sonnet-4-20250514
    weight: 10
```

当 Provider 限流收紧（比如 Google 临时降低 Gemini 配额），429 会密集出现。限流器把 gemini endpoint 标记为不可用，所有流量突然涌向 gpt 和 claude，可能引发连锁 429。极端情况下三个 endpoint 先后被标记不可用 → **`NoAvailableEndpointError`，业务完全中断**。

### 1.2 目标

实现**渐进式降权**：429 频率升高时，不是直接排除 endpoint，而是逐步降低其路由权重，把流量平滑地导向更健康的 endpoint。

```
gemini(w=60) : gpt(w=30) : claude(w=10)     # 正常 6:3:1
       ↓ gemini 429 增多
gemini(w=33) : gpt(w=30) : claude(w=10)     # 自动降权，gpt/claude 占比上升
       ↓ gemini 429 持续
gemini(w=6)  : gpt(w=30) : claude(w=10)     # 大幅降权，流量主要走 gpt
       ↓ gemini 恢复正常
gemini(w=60) : gpt(w=30) : claude(w=10)     # 自动回升
```

核心保证：**即使所有 endpoint 都在 429，也不会出现 `NoAvailableEndpointError`**——权重降到下限但不会降为 0。

### 1.3 核心理念：429 是"太忙"不是"坏了"

429 和 5xx 的本质区别：

| | 429 Too Many Requests | 5xx Server Error |
|---|---|---|
| 含义 | "我没坏，只是你请求太多了" | "我出问题了" |
| 正确应对 | 少给它流量（降权） | 暂时别用了（熔断） |
| 恢复预期 | 减少请求后很快恢复 | 需要等修复 |

现有限流器把 429 当作"不可用"来处理，对 429 的反应过度了。自适应权重的理念是：**429 只意味着要限流，不意味着要禁用**。

### 1.4 实战场景：GSU + PayGo

一个典型的使用场景是 Google Vertex AI 的两种计费模式：

- **GSU（预购算力）**：便宜，但有固定容量上限，超了就 429
- **PayGo（按量付费）**：贵，但容量充足，很少 429

```yaml
project-summary:gemini-2.5-pro:
  endpoints:
  - provider: vertex          # GSU：主力，便宜，容量有限
    model: gemini-2.5-pro
    weight: 80
    extra: { billing: gsu }
  - provider: vertex          # PayGo：补充，贵，容量充足
    model: gemini-2.5-pro
    weight: 20
    extra: { billing: paygo }
```

#### 没有自适应权重（现有行为）

```
t=0     正常: GSU 80%, PayGo 20%

t=10    突发流量，GSU 容量打满，返回 429
        → Engine 退避重试 → 又 429 → failover 到 PayGo
        → 限流器标记 GSU is_limited=True，不可用 60 秒

t=10~70 GSU: 完全不可用（被排除）
        PayGo: 承担 100% 流量
        → PayGo 突然从 20% 跳到 100%，成本飙升
        → PayGo 也可能扛不住 → 也 429
        → 两个都不可用 → NoAvailableEndpointError !!!

t=70    GSU 限流过期，突然恢复到 weight=80
        → 80% 流量瞬间涌回 GSU → 又 429 → 又被标记不可用
        → 反复震荡
```

#### 有自适应权重（新增行为）

```
t=0     正常: GSU 80%, PayGo 20%

t=10    GSU 开始 429（error_rate=10%）
        GSU w=68(×0.85), PayGo w=20
        → 实际比例 77%:23%，轻微调整

t=30    GSU 429 增多（error_rate=30%）
        GSU w=44(×0.55), PayGo w=20
        → 实际比例 69%:31%

t=50    GSU 429 频繁（error_rate=50%）
        GSU w=20(×0.25), PayGo w=20
        → 实际比例 50%:50%
        ⚠️ WARNING: GSU weight reduced to 25%

t=70    GSU 429 持续（error_rate=60%+）
        GSU w=8(×0.10), PayGo w=20
        → 实际比例 29%:71%，大部分流量走 PayGo

t=90    流量高峰过去，GSU 429 减少
        GSU w=44, PayGo w=20
        ℹ️ INFO: GSU weight recovered to 55%

t=150   GSU 完全恢复
        GSU w=80, PayGo w=20，回到 80%:20%
        ℹ️ INFO: GSU weight fully recovered
```

**关键对比**：

| | 没有自适应权重 | 有自适应权重 |
|---|---|---|
| 流量切换 | 0% ↔ 100% 突变 | 80% → 50% → 29% 渐进 |
| PayGo 压力 | 突然承担 5 倍流量 | 逐步增加，有缓冲 |
| GSU 恢复 | 限流过期后瞬间涌回 → 再次 429 | 滑动窗口渐进恢复，不震荡 |
| 极端情况 | 两个都不可用 → 业务中断 | 两个都降权但保留 → **不中断** |
| 成本控制 | PayGo 被迫 100% → 成本不可控 | PayGo 按需承接 → 成本可预期 |

## 二、与现有机制的关系

### 2.1 三层防护体系

自适应权重是现有限流器和熔断器之间的**新增中间层**：

```
第 1 层：自适应权重（渐进式）
│  429 频率升高 → 降低 endpoint 权重 → 流量平滑转移
│  作用：预防性调度，避免问题扩大
│
第 2 层：限流器（二值式）
│  429 + Retry-After → 标记 endpoint 暂时不可用
│  作用：遵守服务端的明确限流指令
│  注意：当自适应权重启用 soft_limit_on_429 时，此层行为改变（见 2.2）
│
第 3 层：熔断器（二值式）
│  非 429 错误率超阈值 → 熔断 endpoint
│  作用：故障隔离，防止雪崩
│
第 4 层：Fallback（跨模型）
│  当前 logical model 的所有 endpoint 都失败 → 切换到另一个 logical model
│  作用：最后的安全网
```

### 2.2 Engine、限流器、自适应权重的职责划分

429 的处理涉及三个组件，各司其职：

```
┌──────────────────────────────────────────────────────────┐
│  CoreEngine（引擎）                                       │
│  📍 core/engine.py                                       │
│                                                          │
│  负责：重试循环、退避计算、failover 决策                     │
│                                                          │
│  收到 429 →                                              │
│    1. 问限流器：服务端建议等多久？（Retry-After）            │
│    2. 有建议 → 用服务端的；没有 → 用指数退避+抖动            │
│       backoff = backoff_factor × 2^attempt × random(jitter)│
│    3. sleep(backoff)                                     │
│    4. 第 1 次 429：重试同一 endpoint                       │
│       第 2 次 429：failover 到其他 endpoint                │
└──────────┬──────────────────────────┬────────────────────┘
           │                          │
           ▼                          ▼
┌─────────────────────────┐  ┌──────────────────────────┐
│  限流器（Rate Limiter）    │  │  自适应权重（Adaptive Weight）│
│  📍 core/plugins/         │  │  📍 core/plugins/            │
│     rate_limit.py         │  │     adaptive_weight.py       │
│                           │  │                              │
│  负责：                    │  │  负责：                       │
│  ① 令牌桶主动限流          │  │  ① 统计各 endpoint 429 频率   │
│  ② 解析 Retry-After 头    │  │  ② 计算权重乘数               │
│  ③ 标记 endpoint 限流状态  │  │  ③ 触发降权告警日志           │
│  ④ 告诉 Engine 等多久     │  │                              │
│    (get_suggested_backoff)│  │  关键区别：                    │
│                           │  │  不排除 endpoint，只调权重     │
└─────────────────────────┘  └──────────────────────────┘
```

| 职责 | 归属 | 说明 |
|------|------|------|
| "要不要重试" | Engine | 根据状态码和重试次数判断 |
| "等多久再试" | Engine + 限流器 | Engine 算指数退避，限流器提供 Retry-After |
| "要不要换 endpoint" | Engine | 同 endpoint 连续失败后 failover |
| "这个 endpoint 暂时别用" | 限流器 | 标记 `is_limited=True` |
| "这个 endpoint 少给点流量" | 自适应权重 | 降低权重乘数 |

### 2.3 soft_limit_on_429：与限流器的联动

**问题**：自适应权重降权是渐进的，但限流器收到 429 后会直接标记 `is_limited=True`，将 endpoint 完全排除。两者行为冲突。

**方案**：新增配置 `soft_limit_on_429`：

| `soft_limit_on_429` | 429 发生时的限流器行为 | 适用场景 |
|---|---|---|
| `false`（默认） | 标记 endpoint 不可用，Router 直接排除 | 未启用自适应权重，保持原有行为 |
| `true` | **不**标记 endpoint 不可用，仅记录配额信息 | 启用自适应权重，由降权机制平滑处理 |

当 `adaptive_weight.enabled=true` 且 `soft_limit_on_429=true` 时，工厂层会自动将限流器的 `soft_limit_on_429` 设为 `true`，无需手动配两处。

### 2.4 不影响已有配置

自适应权重作用于 `Engine._route()` 中，只对**当前请求对应的 logical model 的 endpoints** 生效：

```yaml
models:
  project-summary:auto:     # ← 3 个 endpoint，自适应权重在这里调权
    endpoints: [gemini, gpt, claude]

  project-summary:gpt:      # ← 1 个 endpoint，完全不受影响
    endpoints: [gpt-4o]
```

下游调用 `model="auto"` 走自适应路由，调用 `model="gpt"` 走直连，**两者完全独立**。

endpoint 的错误率统计按 **endpoint ID** 隔离——即使 `auto` 和 `gpt` 都用了同一个物理模型 `gpt-4o`，只要 endpoint ID 不同（配置解析时自动生成），统计互不干扰。

## 三、技术方案

### 3.1 新增 Protocol：WeightAdjustmentProvider

📍 `core/plugins/base.py`

在现有 Protocol 体系中新增权重调整接口：

```
Plugin 基础钩子           → before_request / after_response / on_error
EndpointFilterProvider   → get_unavailable_endpoints()      ← 熔断器、限流器
InvocationLifecycleHook  → on_invocation_success/failure()  ← 熔断器、限流器、自适应权重
WeightAdjustmentProvider → get_weight_multipliers()         ← 自适应权重（新增）
RetryInfoProvider        → get_suggested_backoff()          ← 限流器
RetryLifecycleHook       → on_retry_attempt/exhausted()     ← 可观测性插件
```

```python
@runtime_checkable
class WeightAdjustmentProvider(Protocol):
    def get_weight_multipliers(self) -> dict[str, float]:
        """返回 endpoint_id → 乘数（0.0~1.0），不在字典中的 endpoint 使用原始权重"""
        ...
```

### 3.2 AdaptiveWeightPlugin 实现

📍 `core/plugins/adaptive_weight.py`

**实现的接口**：
- `InvocationLifecycleHook`：追踪每次调用的成功/失败
- `WeightAdjustmentProvider`：返回权重乘数

**核心数据结构**：

```python
# 每个 endpoint 维护一个滑动窗口
_stats: dict[str, deque[tuple[float, bool]]]
#                           ↑         ↑
#                       timestamp  is_target_error（是否为 429）
```

**权重公式**：

```
error_rate = 窗口内 429 次数 / 窗口内总调用次数
multiplier = max(min_weight_ratio, 1.0 - error_rate × penalty_factor)
effective_weight = base_weight × multiplier
```

**公式效果（penalty_factor=1.5, min_weight_ratio=0.1）**：

| 429 错误率 | 乘数 | base=60 → effective | base=30 → effective |
|-----------|------|--------------------|--------------------|
| 0% | 1.0 | 60 | 30 |
| 10% | 0.85 | 51 | 25 |
| 20% | 0.70 | 42 | 21 |
| 33% | 0.50 | 30 | 15 |
| 50% | 0.25 | 15 | 7 |
| 60%+ | 0.10（下限） | 6 | 3 |

**自动恢复**：滑动窗口中的旧记录自然过期（超过 `window_size_seconds` 被清除），error_rate 随之下降，权重自动回升。无需额外恢复逻辑。

#### 错误率计算详解

📍 `core/plugins/adaptive_weight.py:123-136`

错误率是滑动窗口内的简单比值：

```
error_rate = 窗口内目标错误次数 / 窗口内总调用次数
```

**具体例子**：假设 `window_size_seconds=120`，当前时刻 t=200s：

```
窗口范围: [80s, 200s]（最近 120 秒）

记录:
  t=50s  成功  ← 超出窗口，丢弃
  t=70s  429   ← 超出窗口，丢弃
  t=90s  成功  ✓
  t=100s 429   ✓
  t=120s 成功  ✓
  t=140s 429   ✓
  t=160s 成功  ✓
  t=180s 429   ✓
  t=190s 成功  ✓
  t=200s 成功  ✓

窗口内: 8 次总调用，3 次 429
error_rate = 3/8 = 37.5%
multiplier = max(0.1, 1.0 - 0.375 × 1.5) = 0.4375
base_weight=80 → effective_weight = 80 × 0.4375 = 35
```

**数据结构**：每个 endpoint 维护一个 deque（双端队列），记录 `(timestamp, is_target_error)`。查询时从头部清理过期记录，新记录追加到尾部：

```
deque: [(90,F), (100,T), (120,F), (140,T), (160,F), (180,T), (190,F), (200,F)]
        ↑ 头部：最旧                                              尾部：最新 ↑
        过期时从这端 popleft                                  新记录 append 到这端
```

### 3.3 关于权重再分配策略的选择

**两种可选方案**：

**方案 A：自然相对变化（当前采用）**

gemini 降权后，gpt 和 claude 的绝对权重不变，相对占比自然上升：

```
gemini=33, gpt=30, claude=10 → 总和=73
gemini 占 45%，gpt 占 41%，claude 占 14%
```

**方案 B：显式重分配**

gemini 减少的 27 按比例分配给 gpt 和 claude：

```
gemini=33, gpt=50, claude=17 → 总和=100
gemini 占 33%，gpt 占 50%，claude 占 17%
```

**为什么选方案 A**：

1. **更简单**：每个 endpoint 的调整后权重只取决于自身的健康度，不存在跨 endpoint 耦合
2. **效果等价**：两种方案中，gpt 和 claude 获得的"额外流量"比例都是 3:1（等于它们的原始权重之比），区别仅在于 gemini 的最终占比不同（A=45% vs B=33%）
3. **可调**：如果觉得 A 对 gemini 惩罚不够重，提高 `penalty_factor` 即可达到同样效果，不需要引入重分配的复杂度
4. **可预测**：排查问题时，endpoint 的 effective_weight = base_weight × multiplier，一目了然

### 3.4 降权告警日志

当 endpoint 的权重乘数降到阈值以下时，输出 WARNING 级别日志。内置防重复机制，同一 endpoint 不会重复告警，恢复时输出 INFO 日志。

**告警生命周期**：

```
正常 ──降到阈值以下──→ WARNING（首次触发，记入 _alerted 集合）
  ↑                    │
  │                    ├─ 后续查询：已在 _alerted 中，不重复告警
  │                    │
  │                    ├─ 恢复到阈值以上 → INFO "weight recovered"（从 _alerted 移除）
  │                    │
  └────────────────────┘
                       └─ 完全恢复（error_rate=0）→ INFO "fully recovered"
```

**配置项**：

```python
alert_threshold: float = 0.5  # 乘数降到 50% 以下时告警
```

**日志输出示例**：

```
# 降权触发（只告警一次）
WARNING  Endpoint vertex/gemini-2.5-pro weight reduced to 25%
         (error_rate=50.0%, multiplier=0.25)

# 部分恢复（超过阈值）
INFO     Endpoint vertex/gemini-2.5-pro weight recovered to 70%
         (error_rate=20.0%, multiplier=0.70)

# 完全恢复（窗口内 429 全部过期）
INFO     Endpoint vertex/gemini-2.5-pro weight fully recovered (error_rate=0%)
```

### 3.5 Engine 集成

📍 `core/engine.py`

**改动点**：在 `_route()` 方法中，调用 `router.choose()` 之前，插入权重调整：

```
请求进来
├─ Plugin.before_request()
├─ EndpointFilterProvider.get_unavailable_endpoints()     ← 熔断器、限流器
├─ WeightAdjustmentProvider.get_weight_multipliers()      ← 自适应权重（新增）
│   → 创建 endpoint 副本，应用调整后的权重
├─ Router.choose(adjusted_endpoints)                      ← 用调整后的权重做加权随机
├─ Provider 调用
│   ├─ 成功 → InvocationLifecycleHook.on_invocation_success()
│   └─ 失败 → InvocationLifecycleHook.on_invocation_failure()
│                                                         ↑ 自适应权重在此记录 429
└─ ...
```

Router 本身**完全不变**——它只看传入的 endpoint 列表和 weight 字段，不关心权重是静态的还是动态调整的。

## 四、配置方式

### 4.1 完整配置

```yaml
plugins:
  adaptive_weight:
    enabled: true
    window_size_seconds: 120        # 滑动窗口（秒），统计最近 2 分钟的错误率
    penalty_factor: 1.5             # 惩罚系数，越大降权越快
    min_weight_ratio: 0.1           # 权重下限，不低于原始的 10%
    error_codes: [429]              # 触发降权的状态码
    soft_limit_on_429: true         # 联动限流器：429 不排除 endpoint
    alert_threshold: 0.5            # 权重乘数降到 50% 以下时告警

  rate_limit:
    enabled: true
    exclude_429_from_circuit_breaker: true
    # soft_limit_on_429 由 adaptive_weight 联动设置，无需手动配

  circuit_breaker:
    enabled: true
    # 非 429 错误仍由熔断器处理
```

### 4.2 配置参数说明

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | `false` | 是否启用 |
| `window_size_seconds` | int | `120` | 滑动窗口大小（秒） |
| `penalty_factor` | float | `1.5` | 惩罚系数。429 错误率 × penalty_factor = 权重降低的比例 |
| `min_weight_ratio` | float | `0.1` | 权重下限比例（0.0~1.0）。保证 endpoint 不会被完全降为 0 |
| `error_codes` | list[int] | `[429]` | 触发降权的 HTTP 状态码。可扩展为 `[429, 503]` |
| `soft_limit_on_429` | bool | `true` | 联动限流器。开启后 429 不再将 endpoint 标记为不可用 |
| `alert_threshold` | float | `0.5` | 告警阈值。权重乘数低于此值时输出 WARNING 日志 |

### 4.3 error_codes：自定义触发状态码

限流器固定只处理 429，而自适应权重的 `error_codes` 可以自定义：

```yaml
# 默认：只对 429 降权
error_codes: [429]

# 也可以加 503（Service Unavailable），某些 Provider 不稳定时有用
error_codes: [429, 503]
```

| | 限流器 | 自适应权重 |
|---|---|---|
| 关注的状态码 | 固定 429 | 可配置（默认 429） |
| 响应方式 | 排除 endpoint | 降低权重 |
| 用途 | 遵守服务端限流指令 | 基于错误频率调度流量 |

> **注意**：如果把 503 加入 `error_codes`，503 同时还会被熔断器统计。两者会同时生效——自适应权重降权 + 熔断器可能熔断。通常建议只配 `[429]`，让 5xx 错误由熔断器处理。

### 4.4 参数调优指南

**penalty_factor 的选择**：

| penalty_factor | 风格 | 效果 | 适用场景 |
|---|---|---|---|
| 1.0 | 温和 | 50% 错误率时权重降到原来的一半 | 429 偶发，不想过度反应 |
| 1.5 | 均衡（推荐） | 33% 错误率时权重就降到一半 | 大多数场景 |
| 2.0 | 激进 | 25% 错误率就降到一半 | 有充足的备用 endpoint |

**window_size_seconds 的选择**：

| 窗口大小 | 效果 | 适用场景 |
|---|---|---|
| 60s | 快速响应，快速恢复 | 429 通常是短暂的，几秒就恢复 |
| 120s（推荐） | 均衡 | 大多数场景 |
| 300s | 慢响应，慢恢复 | 429 通常持续较长时间 |

## 五、三 endpoint 完整行为推演

配置：gemini(w=60) : gpt(w=30) : claude(w=10)，penalty_factor=1.5

### 场景 1：只有 gemini 429

```
t=0s   gemini 开始出 429
t=30s  gemini error_rate=20% → multiplier=0.70 → w=42
       实际比例: 42:30:10 = 51%:37%:12%
       日志: (无，multiplier > alert_threshold)

t=60s  gemini error_rate=50% → multiplier=0.25 → w=15
       实际比例: 15:30:10 = 27%:55%:18%
       日志: WARNING Endpoint gemini weight reduced to 25%

t=90s  gemini 恢复正常，旧 429 记录陆续过期
t=150s gemini error_rate 降至 5% → multiplier=0.925 → w=55
       实际比例: 55:30:10 = 58%:32%:10%（接近恢复）

t=210s 窗口内 429 全部过期 → multiplier=1.0 → w=60
       完全恢复到 60:30:10
```

### 场景 2：gemini 和 gpt 同时 429

```
t=60s  gemini error_rate=50%, gpt error_rate=30%
       gemini: multiplier=0.25 → w=15
       gpt:    multiplier=0.55 → w=16
       claude: 无错误 → w=10
       实际比例: 15:16:10 = 37%:39%:24%
       日志: WARNING Endpoint gemini weight reduced to 25%

       → claude 自动承接更多流量
```

### 场景 3：全部 429（极端）

```
t=60s  三个 endpoint 都 error_rate=60%+
       gemini: multiplier=0.10 → w=6
       gpt:    multiplier=0.10 → w=3
       claude: multiplier=0.10 → w=1
       实际比例: 6:3:1 = 60%:30%:10%（退回原始比例）

       关键：没有任何 endpoint 被排除
       → 仍然正常工作，只是每个请求有较高概率 429 后重试
       → 不会 NoAvailableEndpointError
```

## 六、各机制协同总览

| 场景 | 自适应权重 | 限流器 | 熔断器 | Fallback | 结果 |
|------|-----------|--------|--------|----------|------|
| 偶发 429 | 轻微降权 | soft_limit 不排除 | 不参与（429 排除） | 不触发 | 流量微调 |
| 频繁 429 | 大幅降权 + 告警 | soft_limit 不排除 | 不参与 | 不触发 | 流量大幅转移 |
| 5xx 错误 | 不介入（仅监控 429） | 不介入 | 统计失败率，可能熔断 | 不触发 | 熔断器处理 |
| 全部 429 | 全部降到下限 | 不排除任何 endpoint | 不参与 | 不触发 | **按降权后比例分配，不中断** |
| 全部 5xx | 不介入 | 不介入 | 全部熔断 | 触发 | 切换到 fallback 模型 |
