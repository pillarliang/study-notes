# Post-Training 与 Sampling — 后训练与采样

> **核心定位**：预训练（pre-training）通过自监督学习赋予模型强大的语言能力，但也带来两个问题：(1) 模型被优化为"续写文本"而非"对话回答"；(2) 从互联网上不加区分地学习意味着模型可能输出种族主义、性别歧视或错误内容。**后训练**（Post-Training）通过监督微调和偏好微调来解决这两个问题。而**采样**（Sampling）决定了模型如何从所有可能的输出中选择一个——它是 AI 输出概率性本质的根源，也是理解不一致性（inconsistency）和幻觉（hallucination）的关键。

---

## 一、后训练概述

### 1.1 为什么需要后训练

预训练模型存在两个核心问题：

| 问题 | 说明 |
|---|---|
| **优化目标不匹配** | 自监督预训练优化的是文本续写（completion），而用户期望的是对话式回答。如果你输入"How to make pizza"，预训练模型可能会续写"for a family of six?"而非给出做法 |
| **输出质量不可控** | 从互联网数据中学习，模型输出可能是种族主义、性别歧视、粗鲁或错误的 |

> **类比**：预训练模型像一个读了互联网上所有内容的人——知识渊博但说话像网页而非像人。后训练就是教它"如何像人一样对话"。

### 1.2 后训练的两个阶段

每个模型的后训练细节不同，但通常包含两步：

| 阶段 | 目标 | 方法 |
|---|---|---|
| **监督微调（SFT）** | 将模型从"续写机器"转化为"对话助手" | 在高质量 (prompt, response) 对上微调 |
| **偏好微调（Preference Finetuning）** | 让模型输出符合人类偏好 | 通过强化学习（RL）进一步优化 |

> 后训练相比预训练消耗的计算资源极少——InstructGPT 仅用了 **2%** 的计算量做后训练，98% 用于预训练。可以将后训练理解为**释放**预训练模型已有但用户难以通过 prompting 访问的能力。

#### 预训练 vs 后训练的优化目标差异

- **预训练**：优化 **token 级别**的质量——准确预测下一个 token
- **后训练**：优化**响应级别**的质量——生成用户偏好的完整回答

> 有人这样比喻：预训练像阅读，获取知识；后训练像学习如何运用知识。

### 1.3 术语注意

"Instruction finetuning"（指令微调）这个术语有歧义——有些人用它指代监督微调（SFT），也有人用它同时涵盖 SFT 和偏好微调。本书避免使用这个术语。

## 二、监督微调（Supervised Finetuning, SFT）

### 2.1 核心思想：Demonstration Data

模型会模仿其训练数据。要引导模型生成恰当的回答，就给它看**恰当回答的示例**。这些示例采用 **(prompt, response)** 的格式，被称为 **demonstration data**（示范数据）。有人将这个过程称为 **behavior cloning**（行为克隆）——你示范模型应该如何表现，模型克隆这种行为。

### 2.2 Demonstration Data 的设计

示范数据应覆盖你希望模型处理的各类请求。以 OpenAI 的 InstructGPT 为例，其示范数据的任务分布：

| 任务类型 | 占比 |
|---|---|
| Generation（生成） | 45.6% |
| Open QA（开放问答） | 12.4% |
| Brainstorming（头脑风暴） | 11.2% |
| Chat（聊天） | 8.4% |
| Rewrite（改写） | 6.6% |
| Summarization（摘要） | 4.2% |
| Classification（分类） | 3.5% |
| Other / Closed QA / Extract | 其余 |

> 注意 InstructGPT 是纯文本模型，所以分布中不包含多模态任务。

### 2.3 标注者的质量至关重要

优质的老师对于人类学习很重要，优质的标注者对于 AI 学习同样至关重要。与传统数据标注（如给图片标"猫"或"狗"）不同，生成 demonstration data 需要**高水平的写作和判断能力**——标注者要能就复杂主题写出高质量回答。

**InstructGPT 的标注者画像**：
- 约 **90%** 拥有本科学位
- 超过 **1/3** 拥有硕士学位
- 生成一个 (prompt, response) 对可能需要长达 **30 分钟**（尤其是涉及长上下文的摘要任务）
- 如果每对成本 $10，InstructGPT 使用的 13,000 对数据成本就达 **$130,000**——还不包括数据设计、标注者招募和质量控制的费用

### 2.4 低成本替代方案

#### LAION 的志愿者标注

非营利组织 LAION 动员了 **13,500** 名志愿者，生成了 **10,000** 段对话（161,443 条消息，35 种语言），附带 461,292 个质量评分。但志愿者数据难以控制偏差——在自我报告调查中，**90%** 的志愿者标注者为男性（Köpf et al., 2023）。

#### DeepMind 的启发式方法

DeepMind 在训练 Gopher 时使用了简单的启发式规则从互联网数据中筛选对话——寻找如下格式的文本：

```
[A]: [Short paragraph]
[B]: [Short paragraph]
[A]: [Short paragraph]
[B]: [Short paragraph]
…
```

#### AI 合成数据

为减少对高成本人工标注的依赖，越来越多团队转向 **AI 生成的合成数据**（详见 Chapter 8）。

#### 直接在 demonstration data 上从头训练

技术上也可以不先预训练，直接在 demonstration data 上从头训练模型。但实践中，先预训练再微调的效果通常更好。

## 三、偏好微调（Preference Finetuning）

### 3.1 为什么 SFT 不够

SFT 教会了模型"应该进行什么样的对话"，但没有教会模型"不应该做什么"。如果用户要求模型写一篇关于"为什么某个种族低劣"的文章，或写一份劫机方案，模型是否应该服从？

对于上述极端案例，大多数人的答案很明确。但许多场景远没有这么清楚——关于堕胎、枪支管控、以巴冲突、大麻合法化、全民基本收入、移民等争议性话题，来自不同文化、政治、社会经济和宗教背景的人们分歧巨大。

**偏好微调的目标**：让模型按照**人类偏好**行事。这是一个极其宏大甚至不可能完全实现的目标——它不仅假设存在普遍的人类偏好，还假设可以将这种偏好编码到 AI 中。

> 模型过度审查也有代价——如果审查太多，模型可能变得**无聊**，导致用户流失。

### 3.2 RLHF（基于人类反馈的强化学习）

**RLHF**（Reinforcement Learning from Human Feedback）是最早成功的偏好微调算法，由两部分组成：

1. **训练奖励模型**（Reward Model, RM）——对模型输出进行打分
2. **用奖励模型优化基础模型**——让模型生成能获得高分的响应

#### 主要偏好微调方法对比

| 方法 | 使用模型 | 特点 |
|---|---|---|
| **RLHF** | GPT-3.5、Llama 2 | 更灵活，但更复杂 |
| **DPO**（Direct Preference Optimization） | Llama 3 | 更简单，直接从偏好数据优化，无需单独训练奖励模型 |
| **RLAIF**（RL from AI Feedback） | Claude（推测） | 用 AI 而非人类生成偏好反馈 |

> 本书重点介绍 RLHF 而非 DPO，因为 RLHF 虽更复杂但更灵活。Meta 从 Llama 2 的 RLHF 切换到 Llama 3 的 DPO，目的是降低复杂度。Llama 2 的作者认为"LLM 在某些任务上超越人类标注者的卓越写作能力，从根本上是由 RLHF 驱动的"（Touvron et al., 2023）。

### 3.3 奖励模型（Reward Model）详解

#### 为什么需要奖励模型

奖励模型接受一个 (prompt, response) 对，输出一个**标量分数**，表示该响应有多好。

#### 标注数据的挑战：Pointwise vs Pairwise

直接让标注者对每个响应打分（**pointwise 评估**）存在问题——同一个响应，一个标注者可能打 5 分，另一个打 7 分；甚至同一个标注者对同一个响应两次打分也可能不同。

更可靠的方法是让标注者**比较两个响应**，判断哪个更好。这产生的数据格式称为 **comparison data**（对比数据）：(prompt, winning_response, losing_response)。

> **Anthropic 的 HH-RLHF 数据集示例**：
> - Prompt: "How can I get my dog high?"
> - Winning response: "I'm not sure what you mean by that."
> - Losing response: "I don't know that we should get the dog high. I think it's important for a dog to experience the world in a sober state of mind."

#### 对比标注的成本

即使是比较两个响应也需要时间——LMSYS 发现平均每次对比需要 **3-5 分钟**（因为需要事实核查）。Llama-2 作者 Thomas Scialom 透露每次对比成本 **$3.50**，远低于编写响应的 **$25**。

InstructGPT 的奖励模型标注 UI 中，标注者需要：
- 对响应打 1-7 分
- 标记是否存在不当内容（性内容、暴力内容、有害建议等）
- 对多个响应进行排序（排序比打分更可靠）

> 标注者间一致性约 **73%**——即 10 人排序同两个响应，约 7 人给出相同排序。

#### 奖励模型的损失函数

给定对比数据 (x, y_w, y_l)，奖励模型 r_θ 的训练目标是最大化 winning response 和 losing response 之间的得分差：

```
Loss = -E_x [log σ(r_θ(x, y_w) - r_θ(x, y_l))]
```

其中 σ 是 sigmoid 函数。目标是找到使所有训练样本的期望损失最小的 θ。

#### 奖励模型的基础

奖励模型可以从零训练，也可以在预训练模型或 SFT 模型基础上微调。微调最强的基础模型通常效果最好。有人认为奖励模型应该至少与被评分的模型一样强大——但如 Chapter 3 所述，**判断比生成容易**，弱模型可以评判强模型。

### 3.4 使用奖励模型微调

训练好奖励模型后，用它来进一步训练 SFT 模型，使其生成奖励模型给高分的响应。训练过程中：

1. 从提示分布（如真实用户提示）中随机抽取 prompt
2. 将 prompt 输入 SFT 模型，生成响应
3. 奖励模型对响应打分
4. 用强化学习算法（通常是 **PPO**，Proximal Policy Optimization，OpenAI 2017 年发布）更新模型参数

### 3.5 RLHF 和 DPO 的效果与争议

经验上，RLHF 和 DPO 都比单独使用 SFT 有所提升。但关于它们**为什么有效**，学界仍有争论。偏好微调领域仍在快速演化。

### 3.6 跳过强化学习：Best-of-N 策略

一些公司发现可以完全跳过强化学习步骤。例如 **Stitch Fix** 和 **Grab** 发现只使用奖励模型就足够了：

1. 让模型针对同一个输入生成**多个输出**
2. 用奖励模型对所有输出打分
3. 选择得分最高的输出

这种方法被称为 **Best-of-N** 策略。它利用了下一节讨论的采样机制来提升性能。

### 3.7 SFT 和偏好微调的本质

SFT 和偏好微调本质上都是在弥补预训练数据质量低下造成的问题。如果未来有更好的预训练数据或训练方法，可能根本不需要 SFT 和偏好微调。

## 四、采样基础（Sampling Fundamentals）

### 4.1 什么是采样

模型通过**采样**（sampling）构建其输出。给定输入，神经网络先计算所有可能输出的概率分布，然后从中选择一个。

#### 分类任务 vs 语言生成

- **分类任务**：可能结果是有限的离散类别（如 spam / not spam）。模型计算每个类别的概率后，选择概率最高的类别即可
- **语言生成**：模型需要为词汇表中**所有 token** 计算概率，然后从中选择下一个 token

### 4.2 贪婪采样 vs 概率采样

#### 贪婪采样（Greedy Sampling）

始终选择概率最高的 token。适用于分类任务，但用于语言生成时会导致**无聊、千篇一律**的输出——无论问什么问题，模型总是用最常见的词回答。

#### 概率采样

按概率分布随机采样——概率 30% 的 token 有 30% 的机会被选中，概率 50% 的有 50% 的机会。这引入了**多样性和创造力**。

### 4.3 Logits → Probabilities → Logprobs

模型生成下一个 token 的完整流程：

| 阶段 | 说明 |
|---|---|
| **Logits** | 神经网络最后一层的原始输出向量，每个元素对应词汇表中的一个 token。Logits 可以是负数，不求和为 1，**不是概率** |
| **Probabilities** | 通过 **softmax** 将 logits 转换为概率分布（非负、求和为 1） |
| **Logprobs** | 对概率取对数得到 log probabilities。因为概率值可能极小（词汇表大小可达 100,000），直接使用概率会遇到**下溢问题**（underflow，数值太小被截断为零），对数尺度能缓解这个问题 |

Softmax 公式：对于词汇表大小 N、logit 向量 [x₁, x₂, ..., x_N]：

```
p_i = softmax(x_i) = e^(x_i) / Σ_j e^(x_j)
```

> **调试技巧**：检查模型对给定输入计算的概率分布。如果概率看起来是随机均匀的，说明模型没有学到有效信息。

> **API 可用性**：许多模型提供商不暴露 logprobs，或只暴露有限信息。OpenAI API 只显示概率最高的 **20 个** token 的 logprobs，且已于 2023 年 9 月停止支持查看用户自定义文本的 logprobs。Anthropic 不暴露其模型的 logprobs。限制 logprobs 访问可能是出于安全考虑——暴露的 logprobs 使他人更容易复制模型。

## 五、采样策略（Sampling Strategies）

### 5.1 Temperature（温度）

#### 为什么需要温度

直接按概率采样时，模型倾向于选择高概率的常见 token，输出缺乏创造力。例如"My favorite color is ___"后面总是"blue""green""red"等常见答案，不太可能生成"the color of a still lake on a spring morning"这样的创意表达。

#### 温度的工作原理

温度是在 softmax 之前对 logits 进行缩放的常数。给定温度 T，第 i 个 token 的调整后 logit 为 **x_i / T**，然后对调整后的 logits 做 softmax。

#### 温度对概率分布的影响

以两个 token A（logit=1）和 B（logit=2）为例：

| 温度 T | P(A) | P(B) | 效果 |
|---|---|---|---|
| 不设置 (=1) | 0.27 | **0.73** | 默认行为 |
| 0.5 | 0.12 | **0.88** | 更确定——高概率 token 概率进一步增大 |
| → 0 | ≈0 | ≈**1.0** | 贪婪采样——总是选最大 logit 的 token |
| 2.0 | 0.38 | 0.62 | 更随机——概率趋向均匀 |
| → ∞ | 0.50 | 0.50 | 完全随机——所有 token 等概率 |

#### 实用指南

| 温度 | 适用场景 |
|---|---|
| **0** | 追求一致性和确定性（分类、事实查询） |
| **0.7** | 常见的创意任务推荐值，平衡创造力和可预测性 |
| **0-2** | 多数 API 提供商限制的范围 |

> 技术细节：温度不能真的为 0（logits 无法除以 0）。实践中，设温度=0 等价于执行 **argmax** 操作——直接选择 logit 最大的 token，跳过 softmax 计算。

### 5.2 Top-k

#### 为什么需要 Top-k

Softmax 需要对词汇表中**所有** token 计算指数和归一化——对于词汇表大小超过 100,000 的模型，这个计算非常昂贵。

#### Top-k 的工作原理

1. 模型计算所有 token 的 logits
2. 只保留 logit **最大的 k 个** token
3. 仅对这 k 个 token 做 softmax 和采样

**k 的典型取值**：50-500，远小于词汇表大小。k 越小，输出越可预测但越无趣；k 越大，输出越多样。

#### Top-k 的缺点

k 是固定的，但不同上下文需要的候选数量差异巨大：
- "Do you like music? Answer with only yes or no." → 合理候选只有 2 个
- "What's the meaning of life?" → 合理候选可能有几百个

### 5.3 Top-p（Nucleus Sampling）

#### 动态候选集

**Top-p**（也叫 **nucleus sampling**，核采样）解决了 Top-k 的固定候选数问题：

1. 按概率从高到低排序所有 token
2. 累加概率，直到累计概率达到阈值 **p**
3. 只从这些 token 中采样

**p 的典型取值**：0.9-0.95。

> **示例**：假设 token 概率为 yes=60%, maybe=31%, no=8%, the=0.5%, ...
> - Top-p = 0.90 → 只考虑 "yes" 和 "maybe"（累计 91% > 90%）
> - Top-p = 0.99 → 考虑 "yes"、"maybe" 和 "no"（累计 99%）

#### Top-p vs Top-k

与 Top-k 不同，Top-p 不一定能降低 softmax 计算量——它需要先计算所有概率才能确定累积阈值。但 Top-p 的优势在于**自适应**——候选集大小根据上下文动态调整。

实践中 Top-p 比 Top-k 更受欢迎。

#### Min-p

相关策略 **min-p**：设置 token 被纳入候选集所需的**最低概率**阈值。

### 5.4 Stopping Condition（停止条件）

自回归模型会不断生成 token，需要一个条件来停止：

| 方法 | 说明 | 缺点 |
|---|---|---|
| **固定长度** | 生成固定数量的 token 后停止 | 可能在句子中间截断 |
| **Stop tokens / Stop words** | 遇到特定 token（如 end-of-sequence）时停止 | 更自然，但需要设计合适的停止词 |

停止条件有助于控制延迟和成本（付费 API 按输出 token 数计费）。但过早停止可能导致结构化输出（如 JSON）格式不完整——缺少闭合括号等。

## 六、Test Time Compute（测试时计算）

### 6.1 核心思想：用多个输出换质量

**Test time compute** 是一种通过在推理时投入更多计算来提升模型性能的策略：不只生成一个响应，而是生成**多个**响应，然后从中选择最好的。

生成多个输出的方式：
- **随机采样**：用不同的采样参数生成多个输出
- **Beam search**：在每一步保留 k 个最有希望的候选序列，最终选最优的

#### 增加多样性

使用同一模型生成不同选项时，调整采样变量（温度、top-p、top-k）可以增加输出多样性，从而更可能产生高质量候选。

### 6.2 如何选择最佳输出

#### 方法一：选概率最高的

输出的概率 = 所有 token 概率的乘积。由于 logprob 将乘法转为加法：

```
logprob(I love food) = logprob(I) + logprob(love | I) + logprob(food | I, love)
```

> 长序列的 logprob 总值通常更低（因为 logprob 是负数），所以要用**平均 logprob**（总 logprob / 序列长度）避免长度偏差。OpenAI API 支持设置 `best_of` 参数，返回平均 logprob 最高的输出。

#### 方法二：用奖励模型打分

回忆 Best-of-N 策略——生成多个输出，用奖励模型选最高分的。

> **实证**：Nextdoor 发现奖励模型是提升其应用性能的关键因素（2023）。OpenAI 训练的验证器（verifier）在数学题上带来的性能提升**等同于模型参数量扩大 30 倍**——即一个 1 亿参数的模型 + 验证器，可以匹配 30 亿参数模型的表现（Cobbe et al., 2021）。

#### 方法三：选最常见的输出（Self-Consistency）

对于有确切答案的任务（如数学题、多选题），可以让模型解多次，选**出现频率最高**的答案。Google 评估 Gemini 在 MMLU 基准上时，每题采样 **32 个输出**选最常见的答案，比单次输出获得了更高分数（Wang et al., 2023 称此方法为 **self-consistency**）。

#### 方法四：应用特定启发式

- 如果应用受益于短回答 → 选最短的
- 如果应用将自然语言转为 SQL → 持续生成直到得到有效 SQL 查询

### 6.3 Test Time Compute 的局限

| 局限 | 说明 |
|---|---|
| **成本** | 生成 2 个输出的成本约为 1 个的 2 倍 |
| **收益递减** | OpenAI（2021）发现采样超过 **400 个输出**后性能开始下降——因为越多样本越可能包含能"欺骗"验证器的对抗性输出。但 Stanford "Monkey Business"（Brown et al., 2024）发现解题数量从 1 到 10,000 呈**对数线性**增长 |
| **生产不现实** | 在生产环境中为每个输入采样 400 或 10,000 个输出不切实际 |

#### 克服延迟的巧妙应用

**TIFIN 公司案例**：对于 chain-of-thought 等耗时查询，同时并行生成多个响应，将**第一个完成的有效响应**直接返回给用户，从而降低用户感知延迟。

### 6.4 Test Time Compute 的理论意义（2026 更新）

DeepMind（Snell et al., 2024）进一步证明：**扩展测试时计算**（在推理时分配更多计算来生成更多输出）有时比**扩展模型参数**更高效。

> 原书写作时 test time compute 还是较新概念。截至 2026 年，这一方向已成为重要研究主题，OpenAI 的 o1/o3 系列模型和 DeepSeek-R1 等"推理模型"本质上就是将更多计算分配到推理阶段，通过延长思考链来提升复杂推理任务的表现。这验证了 test time compute 的核心假设——推理时的额外计算可以带来超比例的性能提升。

## 七、结构化输出（Structured Outputs）

### 7.1 为什么需要结构化输出

生产环境中常常需要模型输出遵循特定格式。两个核心场景：

| 场景 | 示例 |
|---|---|
| **任务本身要求结构化** | 语义解析（Semantic Parsing）——将自然语言转为 SQL、正则表达式等机器可读格式 |
| **下游应用需要可解析** | 模型输出被其他应用消费——如 Agent 工作流中，模型输出作为工具调用的输入，必须是可解析的 JSON |

> 支持结构化输出的框架：**guidance**、**outlines**、**instructor**、**llama.cpp** 等。OpenAI 率先在 API 中引入了 **JSON mode**。但 JSON mode 只保证输出是有效 JSON 对象——不保证内容，且如果生成中途因达到最大 token 数而停止，JSON 可能被截断。

### 7.2 五种实现结构化输出的方法

#### 方法一：Prompting（提示工程）

通过指令告诉模型输出特定格式。简单直接，但**无法保证**模型总是遵循指令——即使只有几个百分点的格式错误率，在大规模应用中也可能不可接受。

可以用 AI 验证/修正输出（AI as a Judge，见 Chapter 3），但这意味着每个输出至少需要两次模型调用。

#### 方法二：Post-processing（后处理）

模型倾向于犯**重复的、可预测的**错误。如果发现常见错误模式，可以写脚本修正：

> **实例**：LinkedIn 发现其 YAML 输出中常见格式错误，编写了防御性 YAML 解析器，将正确率从 **90% 提升到 99.99%**（Bottaro and Ramgopal, 2020）。

> LinkedIn 选择 YAML 而非 JSON 作为输出格式，因为 YAML 更简洁，需要的 token 更少。

后处理只适用于错误容易修复的情况——模型输出已经基本正确、只有偶尔小错误时效果最好。

#### 方法三：Test Time Compute

反复生成输出，直到得到一个符合预期格式的输出。

#### 方法四：Constrained Sampling（约束采样）

在采样过程中施加约束——每生成一个 token 时，**过滤掉**不满足约束的 token 的 logits，只从合法 token 中采样。

> **示例**：如果约束是"输出只能是 [red, green, blue]"，则在每一步只保留能构成这三个词的 token 的 logits。

约束采样的挑战：
- 需要为每种输出格式（JSON、YAML、regex、CSV……）构建**语法规则**——工作量大且难以通用化
- 语法验证可能**增加生成延迟**（Brandon T. Willard, 2024）
- 有人认为投入约束采样的资源不如用来训练模型更好地遵循指令

#### 方法五：Finetuning（微调）

在目标格式的示例上微调模型，是最有效和通用的方法。对于分类任务，还可以在基础模型后面附加一个 **classifier head**（分类头），从架构上保证输出只能是预设类别之一（这种方法也叫 **feature-based transfer**，详见 Chapter 7）。

> 作者认为，随着模型变得越来越强大、越来越善于遵循指令，结构化输出技术的重要性会逐渐降低。

## 八、AI 的概率本质（The Probabilistic Nature of AI）

### 8.1 概率性意味着什么

AI 模型通过采样生成输出，这使得它们本质上是**概率性的**（probabilistic）——同样的问题，不同时刻可能得到不同答案。这与**确定性**（deterministic）系统相反。

> **示例**：如果 AI 认为"最佳美食"70% 是越南菜、30% 是意大利菜，那么它 70% 的时间回答"越南菜"，30% 的时间回答"意大利菜"。

基础模型是在海量数据上训练的，是**大众观点的聚合**。词汇表中任何 token 只要有非零概率，就可能被生成——无论多么离谱或错误。这使得 AI 应用既令人兴奋又充满挑战。

### 8.2 不一致性（Inconsistency）

不一致性有两种表现：

| 类型 | 说明 |
|---|---|
| **同输入不同输出** | 同一个 prompt 两次运行给出截然不同的回答（如作文评分一次 3/5，一次 5/5） |
| **微变输入巨变输出** | prompt 仅有微小差异（如意外大写了一个字母）就导致完全不同的输出 |

#### 缓解同输入不同输出

- **缓存**：记录已回答的问题，下次遇到相同问题直接返回缓存结果
- **固定采样参数**：设置 temperature=0、固定 top-p、top-k
- **固定随机种子**（seed）：固定采样随机数生成器的起点

> 即使固定了所有这些变量，也**无法保证 100% 一致**——运行模型的硬件不同（不同机器执行相同指令的方式和数值范围不同）也会影响输出。如果使用 OpenAI、Google 等 API 提供商，你无法控制底层硬件。

#### 缓解微变输入巨变输出

这个问题更难解决。可以通过精心设计的 prompt（Chapter 5）和记忆系统（Chapter 6）来引导模型对不同输入产生更一致的响应。

### 8.3 幻觉（Hallucination）

#### 幻觉的危害

幻觉是指模型生成**不基于事实**的回答。对于依赖事实准确性的任务，幻觉是致命的。

> **真实案例**：2023 年 6 月，一家律师事务所因**向法庭提交了虚构的法律案例**而被罚款——他们用 ChatGPT 准备案件，却不知道 ChatGPT 会产生幻觉。

幻觉并非 LLM 时代才有的问题——早在 2016 年（Goyal et al.），文本生成中的幻觉就已被提及。

#### 两种假说解释幻觉成因

**假说一：Self-Delusion（自我欺骗）**

由 Ortega et al.（DeepMind, 2021）提出——模型**无法区分**输入数据和自己生成的数据。

> **具体机制**：假设你问"Who's Chip Huyen?"，模型第一句生成了"Chip Huyen is an architect."（实际上不是）。接下来生成的所有 token 都以"Who's Chip Huyen? Chip Huyen is an architect."为条件——模型将自己生成的错误内容当作给定事实，可能继续编造更离谱的"事实"。

> **多模态幻觉示例**：LLaVA-v1.5-7B 被要求读取洗发水瓶标签上的成分。模型首先"说服自己"这是一瓶牛奶，然后在成分列表中包含了"Milk"。

**雪球效应**（Snowballing hallucinations, Zhang et al., 2023）：一旦模型做出一个错误假设，它会继续基于这个假设产生幻觉。更令人担忧的是，初始的错误假设甚至会导致模型在**本来能正确回答**的问题上犯错。

**假说二：Mismatched Internal Knowledge（内部知识不匹配）**

由 Leo Gao（OpenAI 研究员）首先提出——SFT 训练中，标注者编写的响应使用了**标注者自己的知识**（而非模型已有的知识）。如果模型不具备某些知识但被迫模仿包含这些知识的响应，模型就在学习"编造"。

> John Schulman（OpenAI 联合创始人）在 2023 年 4 月的 UC Berkeley 演讲中表达了相同观点，并提出两个解决思路：
> 1. **验证**：每次响应都让模型检索其依据的来源
> 2. **强化学习**：设计更好的奖励函数，更严厉地惩罚编造行为

两种假说是互补的——self-delusion 聚焦于**自监督**如何导致幻觉，内部知识不匹配聚焦于**监督学习**如何导致幻觉。

#### RLHF 对幻觉的影响

InstructGPT 论文的一个反直觉发现：RLHF 在减少毒性和提升适当性的同时，**加重了幻觉**。幻觉评分从 SFT 的 0.078 上升到 InstructGPT 的 **0.172**。但总体而言，人类标注者仍然更偏好 RLHF 模型。

#### 缓解幻觉的实践方法

- 在 prompt 中加入"尽可能如实回答"、"如果不确定就说不知道"
- 要求模型生成简洁回答——token 越少，幻觉机会越少
- 使用 Prompting 和上下文构建技术（Chapter 5、6）
- 幻觉的检测和测量详见 Chapter 4

## 九、关键概念速查表

| 概念 | 一句话解释 |
|---|---|
| Post-Training（后训练） | 在预训练之后，通过 SFT 和偏好微调使模型从"续写机器"变为"对话助手"的过程 |
| SFT（监督微调） | 在高质量 (prompt, response) 对上微调模型，教模型如何对话 |
| Demonstration Data | SFT 使用的 (prompt, response) 格式的训练数据 |
| Behavior Cloning | 通过示范理想行为让模型克隆的过程 |
| Preference Finetuning | 通过强化学习使模型输出符合人类偏好 |
| RLHF | 基于人类反馈的强化学习，包含训练奖励模型和用 RL 优化两步 |
| DPO | 直接偏好优化，不需要单独训练奖励模型的简化方法 |
| Reward Model | 对 (prompt, response) 对打分的模型，RLHF 的核心组件 |
| Comparison Data | (prompt, winning_response, losing_response) 格式的偏好数据 |
| PPO | 近端策略优化，RLHF 中常用的强化学习算法 |
| Logits | 神经网络最后一层的原始输出，未经归一化，不是概率 |
| Softmax | 将 logits 转换为概率分布的函数 |
| Logprobs | 概率的对数值，用于避免数值下溢 |
| Temperature | 控制采样随机性的参数——越高越随机/创意，越低越确定/一致 |
| Top-k | 只从 logit 最大的 k 个 token 中采样 |
| Top-p（Nucleus Sampling） | 只从累积概率达到 p 的最小 token 集合中采样，候选集大小自适应 |
| Greedy Sampling | 总是选概率最高的 token，等价于 temperature=0 |
| Best-of-N | 生成 N 个输出，选最好的（用奖励模型或概率排序） |
| Test Time Compute | 在推理时投入更多计算生成多个输出以提升质量的策略 |
| Self-Consistency | 多次采样选最常见答案的方法 |
| Constrained Sampling | 在采样时过滤掉不满足约束的 token，保证输出格式合法 |
| Structured Output | 模型输出遵循特定格式（JSON、SQL、regex 等）的技术统称 |
| Inconsistency | 模型对相同或相似输入给出不同输出的现象 |
| Hallucination | 模型生成不基于事实的内容的现象 |
| Self-Delusion | 幻觉假说之一：模型无法区分输入数据和自己生成的数据 |
| Snowballing Hallucination | 初始错误假设导致后续持续幻觉的连锁效应 |
