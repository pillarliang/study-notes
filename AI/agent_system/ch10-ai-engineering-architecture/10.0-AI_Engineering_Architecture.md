# Chapter 10: AI Engineering Architecture and User Feedback — 开篇与 AI 工程架构

> **核心命题**：前面各章分别讨论了提示工程、RAG、Agent、微调、数据集工程、推理优化等技术。本章的目标是把这些技术**串联起来**，构建出真正可用的产品。架构从最简单的形式出发，逐步添加组件，最终形成一个完整的 AI 应用平台。

---

## 一、章节导读

### 1.1 为什么需要讨论架构？

面对众多 AI 工程技术和工具，选择合适的组合方式容易让人不知所措。本章采用**渐进式方法**：

1. 从最简单的基础模型应用架构开始
2. 指出该架构面临的挑战
3. 逐步添加组件来解决这些挑战

### 1.2 用户反馈的特殊地位

对于 AI 应用，用户反馈不仅用于指导产品开发（这在所有产品中都很重要），还有一个更关键的作用——**作为改进模型的数据源**。对话式界面使用户更容易给出反馈，但也使开发者更难从中提取信号。本章后半部分将讨论如何设计反馈收集系统。

### 1.3 关于验证

判断一个应用是否成功，唯一的方法是把它放到用户面前。无论花多少时间做理论推演，都不如实际部署后观察用户行为来得有价值。

## 二、AI 工程架构（AI Engineering Architecture）

本节提出的架构已在多家公司得到验证，适用于大部分 AI 应用，但个别应用可能会有偏差。

### 2.1 最简架构

最简形式下，AI 应用只有两个要素：

```
用户 ──Query──▶ Model API ──Response──▶ 用户
                (Generation)
```

- 没有上下文增强、没有护栏、没有优化
- **Model API** 既指第三方 API（OpenAI、Google、Anthropic），也指自托管模型
- 自托管推理服务器的搭建已在第 9 章讨论

### 2.2 渐进式扩展路径

从最简架构出发，按需添加组件的典型顺序如下：

| 步骤 | 组件 | 解决的问题 |
|------|------|-----------|
| **Step 1** | 增强上下文（Context Construction） | 让模型能访问外部数据源和工具 |
| **Step 2** | 添加护栏（Guardrails） | 保护系统和用户安全 |
| **Step 3** | 模型路由与网关（Router & Gateway） | 管理多模型的复杂度与成本 |
| **Step 4** | 缓存（Caching） | 优化延迟与成本 |
| **Step 5** | Agent 模式（Agent Patterns） | 支持复杂逻辑和写操作 |

> 这个顺序是作者在实践中常见的渐进过程，但每个团队的需求不同，应按照对自身最有意义的顺序来演进。

## 三、Step 1：增强上下文（Enhance Context）

### 3.1 什么是上下文构建？

上下文构建是为模型提供生成输出所需的相关信息的过程。手段包括：

- **文本检索**（如 RAG）
- **图像检索**
- **表格数据检索**（如 SQL 查询）
- **工具调用**（如网页搜索、天气 API、新闻 API）

> **核心类比**：上下文构建之于基础模型，就像特征工程之于传统机器学习。它为模型提供产出高质量输出所必需的信息，在系统质量中扮演核心角色。

### 3.2 不同提供商的上下文构建能力差异

| 维度 | 差异表现 |
|------|---------|
| 文档类型与数量 | 专用 RAG 方案可容纳大量文档，通用 API 可能限制较多 |
| 检索算法与配置 | 不同框架使用不同的分块大小（chunk size）等参数 |
| 工具使用 | 支持的工具类型和执行模式不同（并行执行 vs 长时间运行任务） |

### 3.3 架构变化

添加上下文构建后，架构演变为：

```
         ┌──────────────────────┐
         │   Context Construction │
         │ (RAG, Agent, 查询改写)  │
         └──────────┬───────────┘
                    │
  ┌─────────────────▼──────────────────┐
  │  Read-only actions    Databases    │      Model API
  │  (向量搜索, SQL,     (文档, 表,     │──▶  (Generation)
  │   网页搜索)          聊天记录,      │
  │                      向量库)        │
  └────────────────────────────────────┘
```

## 四、Step 2：添加护栏（Put in Guardrails）

护栏用于缓解风险、保护开发者和用户。只要存在风险暴露点，就应部署护栏。护栏分为**输入护栏**和**输出护栏**两大类。

### 4.1 输入护栏（Input Guardrails）

输入护栏防范两类风险：

#### 风险一：隐私信息泄露

使用外部模型 API 时，敏感数据可能被发送到组织外部。泄露路径包括：

- 员工把公司机密或用户隐私粘贴到第三方 API
- 开发者在 system prompt 中暴露内部策略和数据
- 工具从内部数据库检索出隐私信息并添加到上下文

**没有绝对防漏的方法**，但可以通过护栏缓解。常见的敏感数据类别：

- 个人信息（身份证号、电话号码、银行账户）
- 人脸图像
- 公司知识产权相关的特定关键词和短语

**处理方式**：检测到敏感信息后，可以选择**屏蔽整个查询**或**移除敏感部分**。例如，用 `[PHONE NUMBER]` 占位符替换电话号码。如果模型响应中包含该占位符，再通过一个**可逆 PII 映射表（reversible PII map）**将其还原为原始信息。

#### 风险二：提示注入攻击

攻击者通过恶意 prompt 操控系统行为。防御手段已在第 5 章讨论。

### 4.2 输出护栏（Output Guardrails）

输出护栏有两个核心功能：

1. **捕获输出失败**
2. **定义不同失败模式的处理策略**

#### 常见失败类型

| 类别 | 失败表现 |
|------|---------|
| **质量** | 格式错误（如期望 JSON 但输出非法 JSON）、事实幻觉、质量低下 |
| **安全** | 有毒内容（种族歧视、色情、违法）、泄露隐私信息、触发远程工具/代码执行、品牌风险（歪曲自身公司或竞争对手） |

> **重要提醒**：安全护栏不能只关注安全失败率，还要关注**误拦截率（false refusal rate）**。一个过于严格的系统会拦截合法请求，反而破坏用户体验。

#### 失败处理策略

最简单的应对方式是**重试**——AI 模型是概率性的，相同查询可能得到不同结果：

- 响应为空 → 重试 X 次直到获得非空响应
- 格式错误 → 重试直到格式正确

**重试的权衡**：每次重试都会增加额外的延迟和成本。优化手段：
- **并行调用**：同时发送两次请求，选择更好的那个（避免串行等待）
- **人工兜底**：将特定查询转交人类处理（如检测到用户愤怒情绪、对话轮次过多陷入循环等）

### 4.3 护栏实现的权衡

#### 可靠性 vs 延迟（Reliability vs Latency Trade-off）

这是护栏面临的核心权衡。一些团队因为护栏**显著增加延迟**而选择跳过。

**流式模式下的特殊挑战**：输出护栏在流式模式中不太好用。流式模式下 token 生成后立即推送给用户，难以在推送前对完整响应进行安全评估，不安全的部分可能已经被用户看到。

#### 自托管 vs 第三方 API

| 部署方式 | 护栏需求 |
|----------|---------|
| 第三方 API | API 提供商已内建许多护栏，但仍需额外输入护栏防止数据泄露 |
| 自托管 | 不需要担心数据外发，但需要自建全部护栏 |

#### 现成的护栏工具

- Meta 的 **Purple Llama**
- NVIDIA 的 **NeMo Guardrails**
- Azure 的 **PyRIT** / **AI Content Filters**
- Google 的 **Perspective API**
- OpenAI 的 **Content Moderation API**

> **评分器（Scorers）**：护栏中常使用评分器来判断输入/输出是否合规。评分器通常是 AI 驱动的，比生成模型更小、更快。在架构图中，评分器被放在 Model API 内部，但也可放在输出护栏中。

## 五、Step 3：模型路由与网关（Model Router and Gateway）

当应用涉及更多模型时，路由器和网关帮助管理多模型的复杂度与成本。

### 5.1 路由器（Router）

路由器的核心思想：**不同类型的查询使用不同的解决方案**。

#### 路由器的收益

| 收益 | 说明 |
|------|------|
| **性能提升** | 专用模型在特定任务上可能优于通用模型（如技术故障排查专用模型、账单处理专用模型） |
| **成本降低** | 简单查询路由到便宜模型，只有复杂查询使用昂贵模型 |

#### 路由器的实现：意图分类器（Intent Classifier）

路由器的核心组件是**意图分类器**——预测用户想做什么，然后将查询路由到合适的方案。

**客户支持聊天机器人的路由示例**：

| 用户意图 | 路由目标 |
|----------|---------|
| 重置密码 | FAQ 页面 |
| 纠正账单错误 | 人类客服 |
| 技术故障排查 | 专用技术 chatbot |
| 超出范围的问题 | 礼貌拒绝（不浪费 API 调用） |
| 模糊查询 | 请求用户澄清 |

#### 路由器的高级用途

- **下一步动作预测器（Next-action Predictor）**：用于 Agent 场景，路由器决定模型下一步应使用代码解释器还是搜索 API
- **记忆系统路由**：预测应从记忆层次结构的哪个部分提取信息
- **上下文容量管理**：当查询被路由到上下文窗口较小的模型时，可能需要截断或重新路由

#### 实现与部署

- 许多团队使用 GPT-2、BERT、Llama 7B 等较小模型作为意图分类器，也有团队训练更小的分类器
- 路由器应**快速且便宜**，以便可以多次使用而不显著增加延迟和成本
- 路由通常发生在检索**之前**（判断查询是否在范围内、是否需要检索），但也可发生在检索之后（判断是否需要人工介入）
- 常见的应用模式：**路由 → 检索 → 生成 → 评分**

### 5.2 网关（Gateway）

模型网关（Model Gateway）是一个中间抽象层，提供**统一接口**来对接不同的模型（包括自托管模型和商业 API）。

```
┌────────────────┐                    ┌──────────────────┐
│ 客户支持 chatbot │                    │  Self-hosted 模型  │
│ 产品评价摘要     │──▶ Model Gateway ──▶│  OpenAI APIs     │
│ Text-to-SQL    │    (Unified API)    │  Google APIs     │
│ ...            │                    │  ...             │
└────────────────┘                    └──────────────────┘
```

#### 网关的核心价值

| 功能 | 说明 |
|------|------|
| **统一接口** | API 变更时只需更新网关，无需修改所有下游应用 |
| **访问控制与成本管理** | 不直接给团队发 API token，而是通过网关集中管控谁能用哪个模型，监控和限制 API 用量 |
| **故障降级（Fallback）** | 主 API 不可用时自动切换到备选模型，或短暂等待后重试 |
| **负载均衡、日志、分析** | 请求和响应已经流经网关，天然适合做这些 |

#### 现成的网关工具

Portkey AI Gateway、MLflow AI Gateway、Wealthsimple LLM Gateway、TrueFoundry、Kong、Cloudflare 等。

## 六、Step 4：缓存降低延迟（Reduce Latency with Caches）

缓存是降低延迟和成本的经典手段。AI 应用中有两种主要的系统级缓存机制（注意：KV cache 和 prompt cache 是推理级缓存，由模型 API 提供商实现，在第 9 章讨论）。

### 6.1 精确缓存（Exact Caching）

**原理**：只在请求与缓存中的项**完全匹配**时才命中。

**适用场景**：
- 产品摘要请求 → 缓存该产品的摘要
- 向量搜索 → 缓存查询-结果对，避免重复检索
- 多步骤查询（chain-of-thought、检索、SQL 执行、网页搜索等）→ 这些耗时操作的结果尤其值得缓存

**实现要点**：

| 要素 | 选项与说明 |
|------|----------|
| 存储 | 内存缓存（快但容量有限）、数据库缓存如 PostgreSQL/Redis/分层存储 |
| 淘汰策略 | LRU（最近最少使用）、LFU（最不常用）、FIFO（先进先出） |
| 缓存时效 | 取决于查询被再次调用的可能性——用户特定的查询（"我的订单状态？"）不适合缓存；时间敏感的查询（"今天天气？"）也不适合；通用且无时效性的查询最值得缓存 |

> **缓存与数据泄露风险**：缓存处理不当会导致数据泄露。例如电商网站中，用户 X 的退货政策查询结果被缓存，但该结果依赖 X 的会员等级。当用户 Y 问同样的问题时，系统返回了基于 X 信息的缓存结果，从而泄露了 X 的信息。许多团队训练一个分类器来预测查询是否应被缓存。

### 6.2 语义缓存（Semantic Caching）

**原理**：即使查询**措辞不同但语义相似**，也可以复用缓存结果。

**示例**：
- 用户 A："越南的首都是哪？" → 回答："河内"
- 用户 B："越南的首都城市是？" → 语义相似 → 复用 A 的缓存

**实现步骤**：

1. 对每个查询生成 embedding
2. 在向量数据库中找到相似度最高的已缓存 embedding（得到相似度分数 X）
3. 如果 X 超过阈值 → 返回缓存结果；否则 → 正常处理并缓存新结果

**语义缓存的局限性**：

> *与其他缓存技术相比，语义缓存的价值更加存疑，因为它的多个组件都容易出问题。*

| 风险 | 说明 |
|------|------|
| 依赖链过长 | 需要高质量 embedding、可靠的向量搜索、合理的相似度阈值——任一环节出问题都会导致返回错误结果 |
| 阈值难调 | 相似度阈值需要大量试错才能确定合适值 |
| 错误匹配 | 如果系统错误地将不同查询判断为相似，返回的缓存结果就是错的 |
| 耗时耗力 | 向量搜索本身也需要时间和计算资源，其速度和成本取决于缓存的 embedding 规模 |

**建议**：只有当缓存命中率足够高时，语义缓存才值得投入。在引入之前，需评估其效率、成本和性能风险。

## 七、Step 5：添加 Agent 模式（Add Agent Patterns）

### 7.1 从顺序流到循环流

前述步骤中的应用仍是**顺序执行**的流水线。但如第 6 章所述，真实的应用流程可能包含**循环、并行执行和条件分支**。

**示例**：系统生成输出后，发现尚未完成任务，需要再次检索更多信息。原始响应连同新检索到的上下文被重新送入模型——这就形成了一个循环。

### 7.2 写操作（Write Actions）

模型的输出还可以触发**写操作**，如发送邮件、下单、发起银行转账等。写操作让系统能力大增，但也显著增加风险暴露面。必须**极度谨慎**地授予模型写权限。

### 7.3 复杂度与可调试性的权衡

> 如果你已经完成了所有 5 个步骤，架构会变得相当复杂。虽然复杂系统能解决更多任务，但它们也引入了更多失败模式，使系统更难调试——因为潜在的故障点太多。

## 八、监控与可观测性（Monitoring and Observability）

可观测性不是事后补充，而应**内建于产品设计**中。产品越复杂，可观测性就越关键。

### 8.1 监控的目标

| 目标类型 | 具体内容 |
|----------|---------|
| **缓解风险** | 应用故障、安全攻击、数据/模型漂移 |
| **发现机会** | 应用改进、性能提升、成本节省 |
| **保持问责** | 让你对系统性能有全面的可见性 |

### 8.2 可观测性质量的三个指标（来自 DevOps 社区）

| 指标 | 定义 | 含义 |
|------|------|------|
| **MTTD**（Mean Time to Detection） | 从出问题到被发现的平均时间 | 发现问题的速度 |
| **MTTR**（Mean Time to Response） | 从发现问题到解决的平均时间 | 修复问题的速度 |
| **CFR**（Change Failure Rate） | 变更或部署中导致失败（需修复或回滚）的比例 | 变更的稳定性 |

> CFR 高不一定说明监控系统差，可能说明**评估管线**需要改进——应在部署前就拦截问题。评估指标应与监控指标高度相关：评估时表现好的模型在监控中也应表现好，反之亦然。

### 8.3 Monitoring vs Observability

| 概念 | 定义 | 区别 |
|------|------|------|
| **Monitoring**（监控） | 跟踪系统的外部输出 | 不假设外部输出能反映内部状态——有时你看到输出异常，但不一定能定位内部原因 |
| **Observability**（可观测性） | 通过外部输出推断系统内部状态 | 更强的假设——当出问题时，应能通过日志和指标找到原因，而无需发布新代码 |

> 本书中，"monitoring" 指追踪系统信息的行为，"observability" 指包含仪表化（instrumentation）、追踪和调试在内的整个过程。

### 8.4 指标（Metrics）

#### 指标设计原则

指标本身不是目的——它的价值在于**告诉你什么时候出了问题**以及**发现改进机会**。

设计思路：先确定你想捕获的**失败模式**，然后围绕这些失败模式设计指标。

#### 常见指标分类

| 类别 | 指标示例 |
|------|---------|
| **质量** | 格式错误率（如无效 JSON 比例）、事实一致性、生成质量（简洁度、创造性、正面性）——多数可用 AI Judge 计算 |
| **安全** | 有毒内容率、隐私信息泄露率、护栏触发率、系统拒绝率、异常查询检测 |
| **用户行为** | 用户中途停止生成的频率、每次对话平均轮数、每次输入/输出的平均 token 数及其随时间的分布变化 |
| **延迟** | TTFT、TPOT、总延迟（详见第 9 章） |
| **成本** | 查询量、输入/输出 token 量（TPS）、每秒请求数（特别是对有速率限制的 API） |
| **组件级** | RAG 中的上下文相关性/精度、向量库的存储量和查询耗时 |

#### 指标计算策略

| 策略 | 说明 |
|------|------|
| **抽样检查（Spot Check）** | 对数据子集采样，快速发现问题 |
| **全量检查（Exhaustive Check）** | 对每个请求都计算，提供全面视图 |

建议两者结合。计算指标时，应按**用户、版本、prompt/chain 版本、prompt/chain 类型、时间**等维度分解，以便定位具体问题。

#### 与业务北极星指标的关联

技术指标应与业务**北极星指标（North Star Metrics）**建立关联，如：

- DAU（日活跃用户）
- 会话时长
- 订阅数

与北极星强相关的技术指标 → 指出改进方向；不相关的指标 → 指出不必过度优化的方向。

### 8.5 日志与追踪（Logs and Traces）

#### 日志

指标告诉你"5 分钟前出了问题"，但**日志告诉你"发生了什么"**。

**日志记录原则**：尽可能多地记录——因为你不知道将来需要查看什么。应记录的内容包括：

- 所有配置（API endpoint、模型名称、temperature/top-p/top-k/停止条件、prompt 模板）
- 用户原始查询、发送给模型的最终 prompt、模型输出、中间输出
- 工具调用及其输出
- 组件的启动、结束、崩溃时间
- 每条日志附带 tag 和 ID，标明来源

> 日志量会增长极快，需要自动化日志分析和异常检测工具（许多已由 AI 驱动）。

**手动日志审查的价值**：虽然自动化处理是必须的，但定期**手动检查**生产日志也很有价值。Shankar et al. (2024) 发现，开发者对"什么是好/坏的输出"的认知会随着他们接触更多数据而改变，从而促使他们改写 prompt 并更新评估管线。

#### 追踪（Traces）

日志是一系列离散事件，而追踪将相关事件**串联成一条完整的时间线**——从用户发送查询到最终响应返回的全过程。

一条好的追踪应记录：
- 系统采取的每个动作
- 检索到的文档
- 发送给模型的最终 prompt
- 每一步的耗时和成本

> **目标**：能够逐步追踪每个查询在系统中的变换过程。如果查询失败，应能精确定位是处理错误、检索内容无关，还是模型生成了错误响应。

**工具**：LangSmith 等平台提供追踪的可视化，展示每个 Chain/LLM/Tool 步骤的耗时和 token 消耗。

### 8.6 漂移检测（Drift Detection）

系统的组成部分越多，可能发生变化的因素就越多。AI 应用中常见的三类漂移：

| 漂移类型 | 说明 | 应对 |
|----------|------|------|
| **System Prompt 变更** | prompt 可能基于模板构建，模板被同事修改后你可能毫不知情 | 简单逻辑即可检测 prompt 是否发生变化 |
| **用户行为变化** | 用户会随时间学会如何更好地使用系统——例如学会写更简洁的指令，导致响应长度逐渐下降。仅看指标可能看不出原因 | 需要深入调查根因，不能只看指标变化 |
| **底层模型变更** | 使用 API 时，模型提供商可能在 API 不变的情况下更新底层模型，且不一定公告。Chen et al. (2023) 发现同一 API 的不同版本在基准测试中有显著差异；Voiceflow 报告切换 GPT-3.5-turbo 版本后性能下降 10% | 持续监控模型输出质量 |

## 九、AI 管线编排（AI Pipeline Orchestration）

### 9.1 编排器的角色

当 AI 应用变得复杂（多模型、多数据库、多工具），编排器（Orchestrator）帮助你指定各组件如何协作，形成端到端的管线。

编排器在高层面做两件事：

#### 1）组件定义（Components Definition）

告诉编排器系统使用哪些组件：不同模型、外部数据源、可用工具，以及评估和监控工具。模型网关可以简化这一步。

#### 2）链式组合（Chaining）

链式组合本质上是**函数组合**——将不同组件（函数）串联起来。一个典型流程：

1. 处理原始查询
2. 基于处理后的查询检索相关数据
3. 将原始查询与检索数据组合成模型期望的 prompt 格式
4. 模型基于 prompt 生成响应
5. 评估响应质量
6. 如果合格 → 返回给用户；否则 → 路由到人工

编排器负责组件间的数据传递，并提供工具确保上一步的输出格式符合下一步的输入预期。理想情况下，它应在数据流被打断时通知你（如组件故障或数据格式不匹配）。

> **AI 管线编排器** 不同于通用工作流编排器（如 Airflow、Metaflow）。

### 9.2 编排的性能考量

对于有严格延迟要求的应用，应尽量让组件**并行执行**。例如路由组件（决定查询去向）和 PII 移除组件可以同时运行。

### 9.3 常见的编排工具

LangChain、LlamaIndex、Flowise、Langflow、Haystack 等。由于检索和工具使用是常见的应用模式，许多 RAG 和 Agent 框架本身也是编排工具。

### 9.4 关于编排器的建议

> *你可能想在没有编排器的情况下先开始构建应用。*

任何外部工具都会带来额外的复杂度。编排器可能抽象掉系统运作的关键细节，使系统变得更难理解和调试。在早期阶段，直接编写代码可能是更好的选择。

## 十、本节全景图

```
AI Engineering Architecture
    │
    ├── 最简架构：Query → Model API → Response
    │
    ├── Step 1: 增强上下文
    │     └── 文本/图像/表格检索 + 工具调用
    │
    ├── Step 2: 添加护栏
    │     ├── 输入护栏（PII 脱敏、提示注入防御）
    │     └── 输出护栏（格式/质量/安全检查 + 重试 + 人工兜底）
    │
    ├── Step 3: 模型路由与网关
    │     ├── 路由器：意图分类 → 查询分发（专用模型 / 便宜模型 / 人工）
    │     └── 网关：统一接口 + 访问控制 + 故障降级 + 负载均衡
    │
    ├── Step 4: 缓存
    │     ├── 精确缓存（完全匹配，可靠）
    │     └── 语义缓存（语义相似匹配，价值存疑）
    │
    ├── Step 5: Agent 模式
    │     ├── 循环 / 并行 / 条件分支
    │     └── 写操作（邮件、下单、转账——需极度谨慎）
    │
    ├── 监控与可观测性
    │     ├── 三指标：MTTD / MTTR / CFR
    │     ├── 指标设计：围绕失败模式 + 关联北极星
    │     ├── 日志与追踪：尽量多记录 + 端到端追踪
    │     └── 漂移检测：prompt 变更 / 用户行为 / 底层模型
    │
    └── AI 管线编排
          ├── 组件定义 + 链式组合
          └── 建议：先不用编排器，直到复杂度真正需要它
```

> **核心 Takeaway**：AI 应用架构是一个渐进演化的过程——从最简单的"查询→生成→响应"开始，根据实际需求逐步添加上下文构建、护栏、路由/网关、缓存和 Agent 模式。监控与可观测性应从第一天就内建于设计中，而编排器则应在复杂度真正需要时才引入。不要过早增加复杂度，但也不要在需要时犹豫不决。
