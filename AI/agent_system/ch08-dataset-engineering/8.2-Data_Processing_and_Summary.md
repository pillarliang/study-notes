# Chapter 8: Dataset Engineering — 数据处理与总结

> **核心命题**：数据需要根据具体用例进行处理。数据处理步骤包括检查数据、去重、清洗过滤和格式化。阅读公开模型论文中对数据集细节的披露，往往能获得极有价值的实操经验。

---

## 一、数据处理效率建议

在大规模数据上，每个处理步骤都可能耗费数小时甚至数天。以下是提升效率的实操建议：

| 建议 | 说明 |
|------|------|
| **灵活调整步骤顺序** | 将耗时短的步骤放在前面。例如清洗单条数据比去重耗时更长，则先去重再清洗；反之亦然 |
| **先做试运行（Trial Run）** | 在全量数据上运行处理脚本之前，先在小样本上验证脚本行为符合预期 |
| **保留原始数据副本** | 就地修改数据有风险——其他团队可能需要不同方式处理同一数据；脚本 bug 可能损坏数据 |

## 二、检查数据（Inspect Data）

### 2.1 为什么要手动检查？

> **Greg Brockman（OpenAI 联合创始人）**："手动检查数据可能是机器学习中**价值与声望比最高**的活动。"

作者在每个项目中的经验：**花 15 分钟盯着数据看**，往往能发现节省数小时排错时间的洞察。

### 2.2 手动检查清单

| 检查项 | 具体操作 |
|--------|----------|
| **样本合理性** | 查看样本是否说得通；如果是标注数据，自己试着标注几条，看是否与给定标注一致 |
| **事实核查** | 检查回复内容是否正确 |
| **唯一性** | 是否有相同 query 对应不同 response？是否有相同 response 对应不同 query？ |
| **可信度** | 标注结果是否可信？标注指南是否清晰？ |

### 2.3 统计分析

手动检查之外，应通过统计手段系统了解数据分布：

**应绘制的分布图**：

- **Token 分布**：哪些 token 最常见？
- **输入长度和回复长度分布**
- **是否包含特殊 token？**
- **话题和语言分布**：这些话题和语言与你的任务相关吗？

**创造性统计方法**：微软研究者（Peng et al., 2023）使用**（动词，直接宾语，名词）三元组频率分布**和**回复长度分布**来比较 GPT-3 和 GPT-4 在相同指令集上的输出差异。结果发现 GPT-4 的动词-名词搭配**更多样**，且倾向于生成**更长的回复**。

### 2.4 分维度切片分析

按不同维度切片绘制分布图，发现潜在问题：

| 切片维度 | 关注点 |
|----------|--------|
| **数据来源** | 不同来源的数据质量是否一致？ |
| **时间** | 是否存在时间相关的质量趋势？ |
| **标注员** | 某些标注员是否倾向于给更短的回复或更高的评分？ |

**标注员一致性检查**：

- 如果分数应遵循正态分布，检查所有标注员的评分是否遵循该分布
- 如果每条样本有多个标注，计算**标注者间不一致度（inter-annotator disagreement）**
- 检查冲突标注并决定如何解决

> **研究发现**：Kern et al. (2024) 发现标注者在**第二个标注会话中**做出的标注质量更低，很可能是由于**厌倦或疲劳**。

## 三、数据去重（Deduplicate Data）

### 3.1 为什么去重至关重要？

重复数据会**歪曲数据分布、引入偏见**：

**示例**——一个产品价格数据集：

| 序号 | Input (Product description) | Output (Price) |
|------|----------------------------|----------------|
| 1 | {item: pencil, color: red} | $20 |
| 2 | {item: compass, color: green} | $2 |
| 3 | {item: pencil, color: red} | $20 |
| 4 | {item: pencil, color: red} | $20 |
| 5 | {item: pencil, color: green} | $1 |

第 1、3、4 条重复 → 模型可能错误学到"所有红色物品都是贵的"。

**重复数据的其他危害**：

- 导致**测试集污染**：切分数据时，一条数据可能同时出现在训练集和测试集中
- **浪费计算资源**：即使不影响性能，重复数据也增加训练时间和成本

**量化影响**：Anthropic 研究（Hernandez et al., 2022）发现，仅重复 **0.1%** 的数据 100 次，就能让一个 800M 参数模型的性能退化到 **400M 参数模型**的水平——尽管其余 90% 的训练 token 仍然是唯一的。

### 3.2 重复的类型

| 类型 | 说明 | 检测难度 |
|------|------|----------|
| **整文档重复** | 同一文档出现多次 | 较容易 |
| **文档内重复** | 同一段落在一篇文档中出现两次 | 中等 |
| **跨文档重复** | 同一句名言出现在多篇文档中 | 较难 |

### 3.3 去重的定义问题

在实施去重前，需要先回答几个定义问题：

- 在什么**粒度**上去重？（文档/段落/句子/token 级别）
- **完全匹配**才算重复，还是 **80% 重叠**就算？
- 两个列表包含相同元素但**顺序不同**，算重复吗？

### 3.4 去重技术

| 技术 | 方法 | 适用场景 | 局限 |
|------|------|----------|------|
| **两两比较（Pairwise Comparison）** | 计算每对样本的相似度（精确匹配、n-gram、模糊匹配、语义相似度） | 小数据集 | 大数据集上**成本极高** |
| **哈希（Hashing）** | 将样本哈希到桶中，只比较同桶内的样本；方法包括 **MinHash** 和 **Bloom Filter** | 大数据集 | 可能有哈希冲突 |
| **降维（Dimensionality Reduction）** | 先降维，再在低维空间做两两比较；可复用向量搜索技术（Chapter 6） | 大数据集 | 降维可能丢失信息 |

**去重工具库**：dupeGuru、Dedupe、datasketch、TextDistance、TheFuzz、deduplicate-text-datasets、lazyNLP（作者的开源库，支持 Bloom Filter 去重）。

## 四、清洗与过滤数据（Clean and Filter Data）

### 4.1 清洗——移除干扰 token

- 移除**多余的格式化 token**（HTML 标签、Markdown 标记等），除非模型需要学习这些标记
- **Databricks** 发现移除多余 Markdown 和 HTML token 后，模型准确率**提升 20%**，输入 token 长度**减少 60%**

### 4.2 清洗——合规性处理

移除不合规内容：

- **PII（个人身份信息）**：姓名、邮编等
- **受版权保护的内容**
- **有毒内容**

### 4.3 过滤——移除低质量数据

| 方法 | 说明 |
|------|------|
| **数据验证技术** | 使用前文"数据验证"部分讨论的 AI 验证器和启发式方法 |
| **人工检查** | 最直接有效——盯着数据看，发现低质量模式 |
| **启发式规则** | 移除空样本、过短/过长样本、截断或删除超长样本 |
| **按元数据过滤** | 按关键词、作者、来源、创建日期等筛选 |

> **研究发现**：Kern et al. (2024) 发现标注会话**后半段**的标注质量明显下降（标注员疲劳/厌倦）。因此检查标注时间维度的质量分布是有价值的。

### 4.4 数据过多时的高级选择策略

当数据量超过预算（计算资源）所能承受时，可以用更精细的技术选择最有价值的子集：

| 技术 | 说明 |
|------|------|
| **主动学习（Active Learning）** | 选择对模型学习**最有帮助**的样本 |
| **重要性采样（Importance Sampling）** | 找到对任务**最重要**的样本 |

Meta 研究者在数据剪枝论文（Sorscher et al., 2022）中指出：发现好的数据剪枝指标可以**显著降低**现代深度学习的资源成本。

## 五、格式化数据（Format Data）

### 5.1 匹配模型期望的格式

去重和清洗完成后，数据需要转换为模型微调所期望的格式：

- 每个模型有特定的 **tokenizer** 和 **chat template**
- **使用错误的 chat template 会导致奇怪的 bug**

### 5.2 从 Prompt Engineering 到 Finetuning 的指令转换

微调用的 instruction 与 prompt engineering 中的 instruction **可以不同**：

**Prompt Engineering 阶段**（3-shot 食物分类 prompt）：

```
Label the following item as either edible or inedible.

Item: burger
Label: edible

Item: car
Label: inedible

Item: mushroom
Label: edible

Item: {INPUT}
Label:
```

**微调阶段**：不需要任务描述和 few-shot 示例，直接将示例转化为训练数据：

| Example ID | Input | Output |
|------------|-------|--------|
| 1 | burger --> | edible |
| 2 | car --> | inedible |
| 3 | mushroom --> | edible |

**微调后的推理 prompt** 可以简化为：

```
{INPUT} -->
```

> 这比基座模型使用的 prompt **短得多**，如果你对输入 token 的成本敏感，微调可以作为一种降低成本的方式。

### 5.3 格式一致性的重要性

**推理时的 prompt 格式必须与微调数据的格式一致**。如果训练数据使用 `burger -->` 格式，以下推理 prompt 都会出问题：

| 错误 prompt | 问题 |
|-------------|------|
| `"burger"` | 缺少结尾箭头 `-->` |
| `"Item: burger -->"` | 多了 `Item:` 前缀 |
| `"burger --> "` | 箭头后多了空格 |

> **关键实操建议**：不同的微调数据格式会影响模型性能。建议通过**实验**确定最佳格式。

## 六、Chapter 8 总结（Summary）

### 6.1 核心原则

> 尽管创建训练数据的**实际过程极其繁琐**，但构建数据集的**原则出奇地简单**：先想清楚你希望模型学会哪些行为，然后设计一个**展现这些行为**的数据集。

### 6.2 数据集设计的三大核心标准

无论是预训练、指令微调还是偏好微调，数据集设计共享相同的三大标准：

| 标准 | 说明 |
|------|------|
| **质量（Quality）** | 少量高质量数据可以胜过大量噪声数据 |
| **覆盖 / 多样性（Coverage）** | 许多团队发现增加数据集多样性是提升模型性能的关键 |
| **数量（Quantity）** | 重要但优先级通常低于质量和覆盖 |

### 6.3 合成数据的崛起

- 获取高质量人类标注数据的难度促使团队转向**合成数据**
- 以编程方式生成数据一直是目标，但直到 AI 能够生成**逼真且复杂**的数据后，合成数据才真正成为实用方案
- 合成数据与真实数据一样，**必须先验证质量**才能用于训练
- 评估 AI 生成数据的难度等同于评估其他 AI 输出——**人们更倾向于使用可以可靠评估的数据**

### 6.4 数据工作的本质——无法完全自动化

作者对数据工作本质的深刻洞察：

```
标注数据很难，
  创建标注指南更难，
    自动化数据生成更更难，
      自动化验证生成的数据更更更难。
```

合成数据可以帮助生成更多数据，但你**无法自动化**以下环节：

- **思考你到底需要什么数据**
- **创建标注指南**
- **关注细节**

### 6.5 创造力是关键

作者在研究本章时最深的感触是：数据集设计中蕴含着**巨大的创造力**——人们在构造和评估数据方面有无穷的创意。本章讨论的各种合成和验证技术，正是这种创造力的体现。

```

    │
    ├── 数据策展（Data Curation）

    │     ├── 数据验证（功能正确性、AI 验证器）
```

> **终极 Takeaway**：数据集工程的核心是一个看似简单的循环——**想清楚目标行为 → 设计展现这些行为的数据 → 验证数据质量 → 迭代改进**。工具和技术可以帮助加速这个循环，但无法替代对数据本身的深入思考和创造力。下一章将讨论如何为训练好的模型优化推理的延迟和成本。
