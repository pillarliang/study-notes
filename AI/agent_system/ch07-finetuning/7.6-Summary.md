### 1.2 核心主线

本章的主线可以用一句话概括：**大模型微调的核心挑战是内存瓶颈，几乎所有微调技术的发展都围绕着如何在有限内存下达到最佳效果**。

| 技术方向 | 解决什么 | 代表方法 |
|----------|----------|----------|
| **减少可训练参数** | 梯度 + 优化器状态的内存 | PEFT（LoRA、adapter） |
| **降低数值精度** | 模型权重的内存 | 量化（QLoRA、NF4、INT8/INT4） |
| **不保存激活值** | 前向传播中间结果的内存 | 梯度检查点（用计算换内存） |
| **组合多个模型** | 多任务场景下的效率 | 模型合并（SLERP、TIES、层堆叠） |

## 二、关键结论

### 2.1 关于"何时微调"

- **先穷尽 Prompt 方案 → 再 RAG → 最后才微调**
- 微调管"形式"（form），RAG 管"事实"（facts）
- 微调不是一劳永逸——需要持续的数据、训练知识和运维投入
- 通用模型的能力在快速提升，领域专用模型的价值窗口在缩短

### 2.2 关于 PEFT 与 LoRA

- 微调的过程本身不难——框架已经封装了大部分逻辑
- LoRA 是当前最流行的 PEFT 方法，同时具备**参数高效**和**样本高效**的特性
- LoRA 之所以有效，是因为预训练后的模型具有**很低的内在维度**——模型训练得越好，低秩近似就越有效
- LoRA 的模块化特性使得**多模型部署和组合**变得高效

### 2.3 关于模型合并

- 模型合并是微调的互补技术，允许组合多个专用模型
- 应用广泛：多任务微调、端侧部署、模型上缩放、联邦学习
- 合并前**剪枝冗余参数**（TIES/DARE）可显著提升合并质量
- 仍然是实验性较强的领域，大规模 LLM 上的效果不够稳定

### 2.4 关于实践策略

- 从 LoRA 开始，效果不够再全参微调
- 基座模型选择：先验证可行性（最强模型），再优化性价比
- 微调数据是最难的部分——获取高质量标注数据成本高昂

## 四、速查卡片

### 方法选择速查

```
我的场景是什么？
    │
    ├── 模型回答事实错误 → RAG
    ├── 模型行为/格式不对 → 微调
    │       │
    │       ├── 数据少（< 1000 条）→ LoRA / QLoRA
    │       ├── 数据多（> 10000 条）→ 可考虑全参微调
    │       └── GPU 内存有限 → QLoRA
    │
    ├── 需要多个专用模型 → 各自 LoRA 微调 → 模型合并
    └── 端侧部署 → 量化 + 模型合并
```

### 内存估算速查

```
推理内存 ≈ N × M × 1.2

训练内存 = N × M  （模型权重）
         + N_train × 3 × M  （梯度 + Adam 优化器状态）
         + 激活值（可能超过权重本身）

其中：N = 参数量，M = 每参数字节数，N_train = 可训练参数量
```

### LoRA 配置速查

| 配置项 | 推荐 |
|--------|------|
| 应用矩阵 | 所有注意力矩阵 + 前馈层（预算允许时） |
| 秩 r | 4~64（通常足够） |
| α:r 比值 | 1:8 到 8:1 |
| 部署方式 | 单模型→合并部署；多模型→分离部署 |
