# Prompt Engineering Best Practices（Prompt 工程最佳实践）

> **核心定位**：早期的 prompt 技巧——如写 "Q:" 而非 "Questions:"、在 prompt 中加入 "I'll tip you $300 for the right answer" 等情感激励语句来诱导模型产出更好的回答——随着模型变强会迅速过时。本节聚焦**跨模型、跨时间通用的 prompt 工程原则**，来自 OpenAI、Anthropic、Meta、Google 的官方指南以及成功落地的生产级应用团队的实战经验。每个模型仍有自己的 quirks，使用时应查阅其专属的 prompt 指南。

---

## 一、Write Clear and Explicit Instructions（清晰明确的指令）

### 第一性原理

与 AI 沟通和与人沟通遵循同一法则：**清晰度决定沟通效率**。模型无法猜测你的意图，所有隐含假设都必须显式化。

### 1.1 消除歧义

明确告诉模型**做什么、怎么做、遇到不确定时如何处理**。

- 要评分？说明评分区间（1-5 还是 1-10）
- 不确定时？输出 "I don't know" 还是尽力给出答案？
- 观察到不期望行为后**立即修补** prompt——例如模型输出了小数分数 4.5，就追加"只输出整数分数"

### 1.2 角色设定（Persona）

Persona 帮助模型切换**评判视角**，同一输入在不同角色下可能得到截然不同的输出。

> 示例：作文"I like chickens. Chickens are fluffy and they give tasty eggs."——默认模型给 2/5 分；切换为一年级教师 persona 后给 4/5 分。

### 1.3 提供示例（Few-Shot Examples）

示例是消除输出方式歧义的最直接手段。

**案例**：一个面向儿童的聊天机器人，被问到"圣诞老人会送我礼物吗？"时，默认模型会回答"圣诞老人是虚构人物"。通过提供一个关于牙仙的示例（"当然！把牙齿放在枕头下……"），模型学会以符合儿童期待的方式回应。

**Token 效率**：在性能相当的前提下，优先选择更紧凑的示例格式：

| 格式 | 示例 | Token 数（GPT-4） |
|---|---|---|
| `Input/Output` 格式 | `Input: chickpea` / `Output: edible` | 38 |
| `-->` 格式 | `chickpea --> edible` | **27** |

### 1.4 指定输出格式

- **要求简洁**：冗长输出不仅增加成本（按 token 计费）还增加延迟。如果模型总是以"Based on the content of this essay, I'd give it a score of..."开头，明确禁止 preamble
- **指定结构**：需要 JSON 时，明确 key 名称；必要时给示例
- **使用 markers 标记输入结束**：对分类等结构化输出任务，如果不标记输入的结束位置，模型可能**继续追加输入而非生成输出**

| 有无 Marker | Prompt（末尾） | 模型行为 |
|---|---|---|
| 无 marker | `pineapple pizza --> edible` / `cardboard --> inedible` / `chicken` | 模型继续追加 `tacos --> edible` 等新输入 |
| 有 marker | `chicken -->` | 模型正确输出 `edible` |

> 选择不太可能出现在输入中的字符串作为 marker，否则模型会被混淆。

## 二、Provide Sufficient Context（提供充分上下文）

### 核心原理

就像参考资料帮助学生在考试中表现更好，充分的上下文帮助模型产出更好的结果。**缺乏必要信息时，模型只能依赖内部知识——这往往不可靠，从而导致幻觉。**

### 上下文获取的两种方式

| 方式 | 说明 |
|---|---|
| **直接提供** | 在 prompt 中嵌入相关文档、数据 |
| **赋予工具** | 给模型检索能力（RAG pipeline、web search），由模型自行获取上下文（详见 Chapter 6） |

获取上下文的过程称为 **context construction**。

### 限制模型知识范围的难度梯度

在角色扮演、模拟等场景中，通常希望模型**只使用给定上下文中的信息**回答。实现难度递增：

| 方法 | 可靠性 | 说明 |
|---|---|---|
| **Prompting** | 低 | 指令"仅基于给定上下文回答" + 给出不应回答的问题示例 + 要求引用来源。但无法保证模型完全遵从 |
| **Finetuning** | 中 | 在目标语料上微调，但 pre-training 数据仍可能泄露进响应 |
| **仅在目标语料上训练** | 高 | 最安全，但通常不可行——语料量太少，训练不出高质量模型 |

## 三、Break Complex Tasks into Simpler Subtasks（任务分解）

### 核心模式

不要用一个巨大的 prompt 一次性完成整个复杂任务，而是将其拆为多个子任务，**每个子任务用独立的 prompt 单独调用一次 LLM**。前一步的输出作为后一步的输入，形成链式调用（prompt chaining）。

### 实例：客服聊天机器人

分解为两步：

```text
Step 1: Intent Classification
  → 输入: 用户查询
  → 输出: {primary: "Technical Support", secondary: "Troubleshooting"}

Step 2: Response Generation（每种意图一个专用 prompt）
  → 输入: 用户查询 + 意图
  → 输出: 针对性响应
```

10 种意图 = 1 个分类 prompt + 10 个响应 prompt。分解粒度取决于 **performance / cost / latency 三方权衡**。

### 四大优势

| 优势 | 说明 |
|---|---|
| **Monitoring** | 不仅监控最终输出，还能监控所有中间输出 |
| **Debugging** | 定位出问题的步骤，独立修复，不影响其他步骤 |
| **Parallelization** | 独立子任务可并行执行（如同时生成三个不同阅读水平的故事版本），显著降低延迟 |
| **Effort** | 写简单 prompt 比写复杂 prompt 容易得多 |

### 两大代价

| 代价 | 说明 |
|---|---|
| **延迟增加** | 中间步骤不可见时，用户需等待更长时间才能看到首个输出 token |
| **成本增加** | 更多 API 调用。但分解后每个 prompt 更短 → 单次 token 更少；且简单步骤可用更便宜的模型（如用弱模型做意图分类、强模型做响应生成） |

### 实战案例

GoDaddy (2024) 发现其客服聊天机器人的 prompt 在一轮迭代后膨胀到 **1,500+ tokens**。将 prompt 分解为针对不同子任务的小 prompt 后，**性能提升且 token 成本降低**。

## 四、Give the Model Time to Think（给模型思考时间）

### Chain-of-Thought (CoT)

**核心思想**：显式要求模型逐步推理，而非直接输出答案。由 Wei et al. (2022) 在论文 "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models" 中提出，比 ChatGPT 发布早近一年。

**实验证据**：CoT 在 LaMDA、GPT-3、PaLM 上均显著提升了 MAWPS、SVAMP、GSM-8K 等数学推理 benchmark 的性能。LinkedIn 发现 CoT 也能**降低幻觉**。

### CoT 的四种变体

| 变体 | Prompt 示例 |
|---|---|
| **Zero-shot CoT（自由推理）** | "Which animal is faster: cats or dogs? **Think step by step** before arriving at an answer." |
| **Zero-shot CoT（解释优先）** | "... **Explain your rationale** before giving an answer." |
| **Zero-shot CoT（指定步骤）** | "... **Follow these steps**: 1. Determine the speed of the fastest dog breed. 2. ..." |
| **One-shot CoT（示例推理）** | 提供一个完整的推理过程示例（如 sharks vs dolphins），模型据此模仿推理 cats vs dogs |

哪种变体最有效取决于具体应用，需要实验确定。

### Self-Critique（自我批判）

让模型**检查并修正自己的输出**，也称 self-eval（详见 Chapter 3）。与 CoT 类似，self-critique 促使模型对问题进行批判性思考。

### 代价

CoT 和 self-critique 都会**增加延迟和成本**。尤其当模型自行决定推理步骤数时，步骤序列可能很长，导致延迟难以预测且成本可能过高。

## 五、Iterate on Your Prompts（迭代优化）

### 核心理念

Prompt engineering 是一个**反复迭代**的过程：

```text
理解模型 → 写/改 prompt → 观察输出 → 发现问题 → 修改 prompt → ...
```

例如，让模型推荐"最好的电子游戏"，模型可能回答"众说纷纭，没有绝对最好的"。观察到这一行为后，修改 prompt 为"即使观点不同，也请选一个"。

### 了解模型的 Quirks

| 维度 | 差异示例 |
|---|---|
| 能力偏好 | 有的模型擅长数字理解，有的擅长角色扮演 |
| 指令位置 | 有的偏好 system instructions 在开头，有的偏好在结尾 |
| 探索方式 | 阅读模型开发者的 prompt 指南、搜索他人经验、使用 playground、在不同模型上对比同一 prompt |

### 系统化实验

- **版本管理 prompt**——每次修改都记录
- **使用 experiment tracking 工具**——统一管理实验
- **统一评估指标和评估数据**——确保不同 prompt 之间可比
- **在完整系统上下文中评估**——一个 prompt 可能提升子任务性能但**恶化系统整体性能**

## 六、Evaluate Prompt Engineering Tools（评估 Prompt 工程工具）

### 工具的三个层次

| 层次 | 代表工具 | 工作方式 |
|---|---|---|
| **全流程自动化** | OpenPrompt (Ding et al., 2021)、DSPy (Khattab et al., 2023) | 指定输入输出格式 + 评估指标 + 评估数据 → 自动搜索最优 prompt（类似 autoML） |
| **AI 辅助生成** | Promptbreeder (DeepMind)、TextGrad (Stanford) | 用 AI 模型写 prompt、critique prompt、生成 in-context examples |
| **局部辅助** | Guidance、Outlines、Instructor | 引导模型生成结构化输出；扰动 prompt 寻找最优变体 |

**Promptbreeder** 采用进化策略：初始 prompt → AI 生成变异 → 选择最优变异 → 再变异 → 直到满足标准。

**AI 辅助的最简形式**：直接让模型写 prompt（如"帮我写一个给大学作文打 1-5 分的 prompt"），或让模型 critique 和改进已有 prompt。

### 风险警告

| 风险 | 说明 |
|---|---|
| **隐藏 API 调用** | 10 个 prompt 变体 x 30 个评估样本 = 300 次调用；每次调用可能还需额外调用来验证格式和打分。如果工具自由设计 prompt chain，调用量更不可控 |
| **工具开发者错误** | 模板与模型不匹配、用 token 拼接而非原始文本构造 prompt、prompt 模板中的拼写错误（如 LangChain 默认 critique prompt 中的 typo） |
| **无预警变更** | 工具可能随时更换 prompt 模板或重写默认 prompt，使用的工具越多，系统越复杂，出错概率越高 |

### 核心原则

> **KISS（Keep It Simple）**：先不用任何工具，手写 prompt。这能帮你深入理解模型和需求。如果使用工具，务必 **inspect 工具产出的 prompt**，并 **追踪 API 调用量**。

## 七、Organize and Version Prompts（组织与版本管理）

### Prompt 与代码分离

将 prompt 放入独立文件（如 `prompts.py`），代码中引用：

```python
# prompts.py
GPT4o_ENTITY_EXTRACTION_PROMPT = "[YOUR PROMPT]"

# application.py
from prompts import GPT4o_ENTITY_EXTRACTION_PROMPT

def query_openai(model_name, user_prompt):
    completion = client.chat.completions.create(
        model=model_name,
        messages=[
            {"role": "system", "content": GPT4o_ENTITY_EXTRACTION_PROMPT},
            {"role": "user", "content": user_prompt}
        ]
    )
```

**四大好处**：

| 好处 | 说明 |
|---|---|
| **Reusability** | 多个应用复用同一 prompt |
| **Testing** | 代码与 prompt 可独立测试，代码可用不同 prompt 进行测试 |
| **Readability** | 分离后两者各自更易阅读 |
| **Collaboration** | 领域专家可专注于 prompt，无需理解代码 |

### Prompt 元数据

当 prompt 数量增多，为每个 prompt 附加元数据便于检索和管理：

```python
from pydantic import BaseModel

class Prompt(BaseModel):
    model_name: str
    date_created: datetime
    prompt_text: str
    application: str
    creator: str
```

可扩展字段：模型 endpoint URL、sampling parameters（temperature、top-p）、输入 schema、输出 schema。

### 专用 Prompt 文件格式

多种工具提出了 `.prompt` 文件格式：Firebase Dotprompt、Humanloop、Continue Dev、Promptfile。

**Firebase Dotprompt 示例**：

```yaml
---
model: vertexai/gemini-1.5-flash
input:
  schema:
    theme: string
output:
  format: json
  schema:
    name: string
    price: integer
    ingredients(array): string
---

Generate a menu item that could be found at a {{theme}} themed restaurant.
```

### 版本管理策略对比

| 策略 | 优点 | 缺点 |
|---|---|---|
| **Git 版本化**（prompt 文件与代码同仓库） | 简单直接，利用现有 Git 工作流 | prompt 更新后**所有依赖应用被迫同步更新**，无法让某个应用停留在旧版本 |
| **独立 Prompt Catalog** | 显式版本化每个 prompt，不同应用可使用不同版本 | 需要额外基础设施 |

一个完善的 Prompt Catalog 应具备：

- 每个 prompt 的元数据和搜索能力
- 显式版本号
- **依赖追踪**：记录哪些应用使用了哪个 prompt
- **版本通知**：当 prompt 有新版本时，通知依赖它的应用负责人

## 小结

| 最佳实践 | 核心要点 |
|---|---|
| **清晰指令** | 消除歧义 + 角色设定 + 示例 + 输出格式（含 markers） |
| **充分上下文** | 直接提供或赋予检索工具；限制知识范围的可靠性：prompting < finetuning < 专属训练 |
| **任务分解** | 大 prompt → 多个子 prompt 链式执行；四大优势（监控/调试/并行/简化），代价是延迟和成本 |
| **思考时间** | CoT（逐步推理）+ Self-critique（自我检查），显著提升推理任务性能，但增加延迟和成本 |
| **迭代优化** | 反复实验，系统化管理：版本控制 + experiment tracking + 统一评估 + 全系统视角评估 |
| **工具评估** | KISS 原则——先手写，再用工具；警惕隐藏 API 调用、工具错误和无预警变更 |
| **组织与版本** | Prompt 与代码分离 + 元数据管理 + 独立 Prompt Catalog 实现跨应用版本控制 |
