# Chapter 9: Inference Optimization — 开篇与理解推理优化

> **核心命题**：无论模型多好，如果推理太慢，用户会失去耐心；如果推理太贵，投资回报率不成立。前面各章讨论了如何让模型更好，本章聚焦于如何让模型**更快、更便宜**。

---

## 一、章节导读

### 1.1 为什么推理优化重要？

推理优化解决的是"最后一公里"问题——即使训练出了世界上最好的模型，也需要让它在生产环境中高效运行：

- **速度**：预测次日股价的模型如果需要两天才能出结果，就毫无价值
- **成本**：模型运行开销过高会侵蚀商业价值，使投资回报率为负

### 1.2 优化的三个层面

推理优化可以在三个层面进行，书中用**射箭**做类比：

| 优化层面 | 类比 | 具体手段 |
|----------|------|----------|
| **模型级（Model-level）** | 制作更好的箭 | 减小模型体积、改进架构（如消除注意力机制中的计算瓶颈） |
| **硬件级（Hardware-level）** | 训练更强壮的弓箭手 | 设计更强大的芯片和加速器 |
| **服务级（Service-level）** | 优化整个射击过程（含瞄准条件） | 针对特定硬件优化模型、高效分配资源、根据流量模式降低延迟和成本 |

### 1.3 优化的权衡

优化并非总能同时降低成本和提升速度——有时两者可以兼得（如降低精度既缩小模型又加速），但更多时候需要**取舍**（如更好的硬件能加速，但价格更高）。

### 1.4 本章聚焦

本章主要讨论**模型级**和**服务级**优化，并概述 AI 加速器。硬件设计本身不在讨论范围内。

### 1.5 开源模型与推理服务

随着开源模型越来越多，越来越多团队开始自建推理服务。即使不亲自实现这些优化技术，理解它们也有助于：

- **评估**推理服务和框架
- **诊断**应用的延迟和成本问题
- **选择**合适的优化方案

## 二、推理概述（Inference Overview）

### 2.1 训练 vs 推理

AI 模型生命周期有两个截然不同的阶段：

| 阶段 | 定义 | 谁需要关心 |
|------|------|-----------|
| **训练（Training）** | 构建模型的过程（包含前向传播和反向传播） | 训练或微调模型的团队 |
| **推理（Inference）** | 用模型对给定输入计算输出的过程（仅前向传播） | 几乎所有使用模型的团队 |

> **训练成本 vs 推理成本**：一个有趣的关系式——设 T 为总训练成本，p 为单次推理收费，N 为可售出的推理调用次数。只有当 T ≤ p × N 时，开发模型才有经济意义。模型使用量越大，推理成本就越能被摊薄。但这不适用于在开源模型上售卖推理调用的第三方 API 提供商。

### 2.2 推理服务器架构

在生产环境中，运行模型推理的组件叫做 **推理服务器（inference server）**：

```
                    ┌─────────────────────┐
   Requests ───────▶│   Inference Server   │
                    │  ┌───┐ ┌───┐ ┌───┐  │
                    │  │M1 │ │M2 │ │M3 │  │◀──── Apps
                    │  └───┘ └───┘ └───┘  │
   Responses ◀──────│                     │
                    └─────────┬───────────┘
                              │
                    ┌─────────▼───────────┐
                    │     Hardware        │
                    │  (GPUs, TPUs, etc.) │
                    └─────────────────────┘
```

- 推理服务器托管可用模型，访问必要的硬件资源
- 根据应用请求（如用户 prompt），分配资源执行相应模型，返回响应
- 推理服务器是更广义的**推理服务（inference service）**的一部分，后者还负责接收、路由请求，以及预处理

**两种使用场景**：

- 使用 OpenAI、Google 等提供的 **Model API**——它们本身就是推理服务，本章大多数优化技术无需自行实现
- 自行托管模型——需要自己构建、优化、维护推理服务

## 三、计算瓶颈（Computational Bottlenecks）

优化的本质是**识别瓶颈并消除它**。推理服务器需要针对其工作负载的两类主要计算瓶颈进行设计：

### 3.1 计算受限（Compute-bound）

**定义**：任务的完成时间取决于硬件能执行多少次计算。计算能力是限制因素。

**类比**：密码破解就是典型的 compute-bound 任务——需要大量数学计算来尝试各种可能。

**优化方向**：分摊到更多芯片上、使用更高 FLOP/s 的芯片。

### 3.2 内存带宽受限（Memory Bandwidth-bound）

**定义**：任务的完成时间取决于数据在内存和处理器之间的传输速率。数据搬运的速度是限制因素。

**类比**：如果数据存在 CPU 内存中，但模型在 GPU 上训练，CPU→GPU 的数据传输就可能成为瓶颈。

**优化方向**：使用带宽更高的芯片。

### 3.3 术语辨析：Memory-bound vs Bandwidth-bound

> **注意**："Memory-bound"一词有**两种含义**，在不同社区的使用习惯不同：
>
> | 使用者群体 | "Memory-bound"的含义 |
> |------------|---------------------|
> | 系统/GPU 工程师 | 内存**带宽**受限（= bandwidth-bound） |
> | AI/ML 工程师 | 内存**容量**受限（即 OOM，out-of-memory） |
>
> **容量受限本质上也是带宽受限**：如果 GPU 内存装不下整个模型，需要将模型拆分到 GPU 和 CPU 内存中。拆分会导致数据在 CPU-GPU 之间传输的开销。但如果数据传输足够快（带宽足够高），拆分带来的性能损失可以忽略。因此，内存容量限制实际上可以归结为内存带宽限制。

### 3.4 Arithmetic Intensity（算术强度）与 Roofline 图

**如何判断**一个操作是 compute-bound 还是 memory bandwidth-bound？通过它的 **算术强度（arithmetic intensity）**——即每字节内存访问对应的算术操作次数（FLOP/byte）。

这个概念源自 Williams et al. (2009) 的 **"Roofline"论文**。NVIDIA Nsight 等性能分析工具可以生成 **Roofline 图**来直观显示工作负载的瓶颈类型：

```
Performance
(FLOP/s)
    │          ┌──────────────── 峰值计算能力（天花板）
    │         ╱
    │        ╱    Compute-bound region
    │       ╱
    │      ╱
    │     ╱  Memory bandwidth-
    │    ╱   bound region
    │   ╱
    │──╱─────────────────────────────
    └──────────────────────────────── Arithmetic intensity (FLOP/byte)
```

- 图的左侧（低算术强度）→ memory bandwidth-bound
- 图的右侧（高算术强度）→ compute-bound
- 拐点处即为从带宽受限转为计算受限的临界点

### 3.5 不同模型/工作负载的瓶颈差异

| 工作负载 | 主要瓶颈类型 | 原因 |
|----------|-------------|------|
| Stable Diffusion 等图像生成 | Compute-bound | 需要大量并行计算 |
| 自回归语言模型 | Memory bandwidth-bound | 逐 token 生成，需反复加载大量权重矩阵 |

### 3.6 LLM 推理的两阶段：Prefill 与 Decode

基于 Transformer 的自回归语言模型推理分为两个步骤，它们的**计算特性截然不同**：

| 阶段 | 做什么 | 瓶颈类型 | 原因 |
|------|--------|----------|------|
| **Prefill（预填充）** | 并行处理所有输入 token，填充初始 KV cache | **Compute-bound** | 可同时处理的 token 数受限于硬件计算能力 |
| **Decode（解码）** | 逐个生成输出 token | **Memory bandwidth-bound** | 每步需将大型矩阵（模型权重）加载到 GPU，受限于硬件加载数据的速度 |

```
  Input                        Output
┌──────────────────┐   ┌──────────────────────────────┐
│ Who wrote Chaos? │   │ James │ Gleick │ did │ . │<eos>│
└────────┬─────────┘   └──┬──────┬──────┬───┬──┬──────┘
         │                │      │      │   │  │
    ┌────▼────┐      ┌────▼─┐┌──▼──┐┌──▼┐┌▼┐┌▼──┐
    │ Prefill │─────▶│Decode││Decode││Dec││D││Dec│
    └─────────┘      └──────┘└─────┘└───┘└─┘└───┘
    Compute-bound    ◄── Memory bandwidth-bound ──►
```

**关键影响因素**：

- 影响 prefill 和 decode 计算量的因素包括：上下文长度、输出长度、请求批处理策略
- 长上下文通常增加 memory bandwidth-bound 的工作负载，但巧妙的优化技术可以缓解
- 由于两阶段的计算特性不同，生产中常将 prefill 和 decode **解耦到不同机器**上运行

**当前现状**：由于 Transformer 架构和现有加速器技术的特点，当前大多数 AI 和数据工作负载是 **memory bandwidth-bound**。未来软硬件进步有望使其转变为 compute-bound。

## 四、在线与批量推理 API（Online and Batch Inference APIs）

许多推理服务提供商提供两种 API 类型：

| API 类型 | 优化目标 | 特点 | 典型延迟 |
|----------|----------|------|----------|
| **Online API** | **延迟（Latency）** | 请求到达后立即处理 | 秒级 |
| **Batch API** | **成本（Cost）** | 请求可积攒后批量处理，更多优化空间 | 小时到分钟级 |

> **价格差异**：Google Gemini 和 OpenAI 的 batch API 提供 **50% 的成本折扣**（截至写作时）。

### 4.1 适用场景

| 场景 | 推荐 API |
|------|----------|
| 聊天机器人、代码生成等用户交互型 | Online API |
| 合成数据生成 | Batch API |
| 定期报告（如汇总 Slack 消息、舆情分析） | Batch API |
| 新客户上线时批量处理文档 | Batch API |
| 迁移到新模型时重新处理全部数据 | Batch API |
| 生成个性化推荐或通讯 | Batch API |
| 知识库重建索引 | Batch API |

### 4.2 基础模型 vs 传统 ML 的批量推理差异

| | 传统 ML | 基础模型 |
|---|---------|---------|
| Online 推理 | 请求到达**后**计算预测 | 同左 |
| Batch 推理 | 请求到达**前**预计算预测（如提前为所有用户生成推荐） | 无法预计算（因为用户 prompt 是开放式的，不可预测） |

### 4.3 Streaming Mode（流式模式）

自回归解码可能耗时很长，用户不愿等待完整响应。许多 online API 提供 **streaming mode**——每个 token 生成后立即返回给用户：

- **优点**：减少用户等待首个 token 的时间，提升体验
- **缺点**：无法在展示前对完整响应进行质量评分/过滤，用户可能看到低质量中间结果
- **补救**：可以在检测到风险时，事后更新或移除响应

## 五、推理性能指标（Inference Performance Metrics）

从用户角度看，核心关切是**延迟**（响应质量是模型本身的属性，不属于推理服务优化范围）。但开发者还需关注**吞吐量**和**利用率**以控制成本。

### 5.1 延迟（Latency）、TTFT 与 TPOT

#### TTFT——Time to First Token（首 Token 延迟）

**定义**：用户发送查询后，生成第一个 token 所需的时间。

- 对应 prefill 步骤的耗时
- 受输入长度影响
- 不同应用对 TTFT 的容忍度不同：
  - 聊天机器人 → 应接近即时
  - 文档摘要 → 用户愿意多等一会

#### TPOT——Time per Output Token（每 Token 输出延迟）

**定义**：第一个 token 之后，每个后续输出 token 的生成时间。

- 如果 TPOT = 100 ms，1000 token 的响应需要 100 秒
- 在 streaming mode 下，TPOT 应快于人类阅读速度，但不必快太多
- 快速阅读者约 120 ms/token，因此 TPOT ≈ 120 ms（即 6-8 tokens/s）对大多数场景已足够

> **变体名称**：TBT（Time Between Tokens，LinkedIn 使用）和 ITL（Inter-Token Latency，NVIDIA 使用）本质上测量的是同一件事——两个输出 token 之间的间隔时间。

#### 总延迟计算

```
总延迟 = TTFT + TPOT × 输出 token 数
```

#### TTFT vs TPOT 的权衡

相同总延迟下，不同的 TTFT/TPOT 组合会带来**不同的用户体验**：

- 更低的 TTFT + 稍高的 TPOT → 用户快速看到首字，后续生成稍慢
- 稍高的 TTFT + 更低的 TPOT → 首字等待稍长，但后续token飞速生成

哪种更好？取决于具体场景，需要做用户研究。技术上，可以通过调整 prefill 和 decode 的计算资源分配来调节两者。

#### "Time to Publish" 概念

在 CoT（思维链）或 Agent 场景中，模型生成大量**用户看不到的中间步骤**（如规划、执行动作、记录日志），最后才生成用户可见的响应。此时：

- 模型内部的 TTFT 是第一步计划生成的时间
- 但**用户感知的 TTFT** 是最终可见响应的第一个 token 出现的时间，远长于模型内部 TTFT

一些团队为此使用 **time to publish** 指标来明确区分。

#### 延迟的统计分析

- 平均延迟**容易产生误导**——少数异常值（如网络错误、超长 prompt）会严重拉高平均值
- 应使用**百分位数（percentiles）**：
  - **p50（中位数）**：半数请求在此延迟以下
  - **p90 / p95 / p99**：关注尾部延迟，帮助发现异常
- 建议将 TTFT 与输入长度对照绘图，观察延迟分布

### 5.2 吞吐量与 Goodput

#### 吞吐量（Throughput）

**定义**：推理服务每秒能为所有用户和请求生成的输出 token 总数。

| 度量维度 | 指标 | 说明 |
|----------|------|------|
| Token 级 | **TPS**（tokens/s） | 最常用，无修饰词时默认指输出 token |
| Token 级（多用户） | **tokens/s/user** | 评估随用户数增长的扩展能力 |
| 请求级 | **RPS**（requests/s）或 **RPM**（requests/min） | 基础模型请求可能耗时数秒，因此常用 RPM |

**注意事项**：

- 由于 prefill 和 decode 有不同计算瓶颈且常在现代推理服务器中解耦，输入和输出吞吐量应**分别统计**
- 不同模型有不同的 tokenizer，直接比较 TPS 只是近似值——更好的比较指标是**每请求成本**

#### 吞吐量与成本的关系

吞吐量直接关联计算成本——更高的吞吐量意味着更低的单位成本：

```
假设：
  系统成本 = $2/h，吞吐量 = 100 tokens/s
  → 1M output tokens 成本 ≈ $5.56

  Prefill 成本：$2/h，100 requests/min
  → 1K requests prefill 成本 ≈ $0.33

  总成本 1K 请求（每请求平均 200 output tokens）：
  → $1.11（decode）+ $0.33（prefill）= $1.44
```

#### 延迟/吞吐量权衡

与大多数软件系统一样，AI 应用存在**延迟-吞吐量权衡**：

- 批处理（batching）等技术可以**提高吞吐量**但可能**增加延迟**
- LinkedIn AI 团队在部署生成式 AI 产品一年后的反思中提到：为了追求吞吐量而牺牲 TTFT 和 TPOT，将吞吐量**翻倍甚至三倍**是常见的做法

#### Goodput

**问题**：仅关注吞吐量可能导致糟糕的用户体验——100 RPM 的吞吐量中，如果大量请求不满足 SLO（服务等级目标），这个吞吐量毫无意义。

**定义**：Goodput 衡量满足 SLO 的每秒完成请求数。这是一个从网络领域借鉴的概念。

**示例**：

```
SLO 要求：TTFT ≤ 200ms，TPOT ≤ 100ms
系统吞吐量：100 RPM
满足 SLO 的请求：30 个
→ Goodput = 30 RPM
```

### 5.3 利用率（Utilization）、MFU 与 MBU

#### GPU 利用率的误区

NVIDIA 官方监控工具 `nvidia-smi` 显示的 **GPU utilization** 指标容易产生误解：

- 它衡量的是 GPU **"在处理任务"的时间占比**
- 例如：GPU 运行 10 小时，其中 5 小时在处理任务 → 50% utilization
- **但**：即使 100% utilization，也可能效率极低——一个能做 100 次操作/秒的 GPU 即使只做 1 次操作/秒，`nvidia-smi` 也会报告 100%

> **核心观点**：利用率的目标**不是让芯片尽可能忙**，而是让工作**更快更便宜地完成**。利用率上升但成本和延迟都变大，等于无用。

#### MFU——Model FLOP/s Utilization

**定义**：实际观测到的吞吐量（tokens/s）与系统以峰值 FLOP/s 运行时的理论最大吞吐量的**比值**。

- 如果芯片标称 100 tokens/s，实际推理只能生成 20 tokens/s → MFU = 20%
- MFU 这个术语由 PaLM 论文（Chowdhery et al., 2022）引入

**不同模型和加速器的 MFU 参考**：

| 模型 | 参数量 | 加速器 | MFU |
|------|--------|--------|-----|
| GPT-3 | 175B | V100 | 21.3% |
| Gopher | 280B | 4096 TPU v3 | 32.5% |
| Megatron-Turing NLG | 530B | 2240 A100 | 30.2% |
| PaLM | 540B | 6144 TPU v4 | 46.2% |

- 训练的 MFU 通常高于推理（因为训练的工作负载更可预测，能做更多优化如更好的 batching）
- Prefill 阶段的 MFU 通常高于 decode 阶段（因为 prefill 是 compute-bound，decode 是 memory bandwidth-bound）
- 训练中 MFU 50% 以上被认为不错，但在特定硬件上可能很难达到

> **"Peak FLOP/s Hacking"**：芯片厂商可能在特定条件下（如使用稀疏矩阵）测试以提高其标称 FLOP/s，使实际 MFU 看起来更低。

#### MBU——Model Bandwidth Utilization

**定义**：实际使用的内存带宽占芯片理论峰值内存带宽的百分比。

**计算方法**：

```
实际带宽使用 = 参数量 × 每参数字节数 × tokens/s

MBU = 实际带宽使用 / 理论峰值带宽
```

**示例**：7B 参数模型，FP16（每参数 2 字节），100 tokens/s：

```
实际带宽 = 7B × 2 bytes × 100 = 700 GB/s
在 A100-80GB GPU（理论带宽 2 TB/s）上：
MBU = 700 GB/s / 2000 GB/s = 35%
```

**MFU 与 MBU 的关系**：

| 工作负载类型 | MFU | MBU |
|-------------|-----|-----|
| Compute-bound | 较高 | 较低 |
| Bandwidth-bound | 较低 | 较高 |

> **量化的重要性**在此显现：每参数字节数越少（如 INT8 vs FP16），消耗的带宽越少，MBU 越低，吞吐量潜力越大。

## 六、AI 加速器（AI Accelerators）

### 6.1 AI 与硬件的共生关系

AI 模型的发展与硬件能力始终紧密交织：

| 时间点 | 事件 | 影响 |
|--------|------|------|
| 1970s | 计算能力不足 | 导致第一次 AI 寒冬 |
| 2012 | AlexNet 成功使用 GPU 训练神经网络（Krizhevsky et al., 2012） | 深度学习复兴——此前训练同等规模模型需要数千个 CPU，GPU 大幅降低了门槛 |
| 2020s | NVIDIA GPU 主导 AI 硬件市场 | GPU 成为 AI 发展的核心经济驱动力 |

### 6.2 什么是加速器？

**加速器（Accelerator）** 是为加速特定类型计算而设计的芯片。**AI 加速器**专为 AI 工作负载设计。

#### CPU vs GPU 的核心区别

| 特性 | CPU | GPU |
|------|-----|-----|
| 设计目标 | 通用计算 | **并行处理** |
| 核心数 | 少量强力核心（通常 ≤ 64） | **数千个**较弱核心 |
| 擅长 | 高单线程性能、操作系统管理、I/O、复杂顺序逻辑 | 大量独立的小型计算（如矩阵乘法，占 ML 浮点运算的 90%+） |
| 并行化代价 | — | 对内存设计和功耗提出更高要求 |

#### AI 加速器生态

NVIDIA GPU 的成功催生了大量 AI 加速器：

| 厂商 | 加速器 |
|------|--------|
| AMD | GPU（新一代） |
| Google | **TPU**（Tensor Processing Unit） |
| Intel | **Habana Gaudi** |
| Graphcore | **IPU**（Intelligent Processing Unit） |
| Groq | **LPU**（Language Processing Unit） |
| Cerebras | **WSE**（Wafer-Scale Quant Processing Unit） |

**重要趋势**：专用于推理的芯片正在兴起。据 Desislavov et al. (2023) 调查，推理成本可超过训练成本，在已部署的 AI 系统中推理占机器学习总成本的**高达 90%**。

#### 训练 vs 推理的硬件需求差异

| 需求 | 训练 | 推理 |
|------|------|------|
| 内存需求 | **更高**（反向传播需要存储梯度等中间状态） | 较低 |
| 精度要求 | **更高**（低精度下反向传播更困难） | 较低（可以大幅量化） |
| 优化重点 | **吞吐量** | **延迟** |
| 芯片设计偏好 | 大内存容量 | **快速内存访问**和低精度优化 |

推理专用芯片包括 Apple Neural Engine、AWS Inferentia、MTIA（Meta）等。边缘计算芯片（如 Google Edge TPU、NVIDIA Jetson Xavier）也通常面向推理场景。

### 6.3 计算原语（Compute Primitives）

不同硬件架构有不同的专用计算单元，针对不同数据类型进行优化：

| 计算原语 | 操作对象 | 示例硬件 |
|----------|----------|----------|
| **Scalar** | 单个数值 | 传统 CPU |
| **Vector** | 一维数组 | GPU（传统优势） |
| **Tensor** | 多维矩阵 | GPU Tensor Core、**TPU**（以 tensor 运算为主要计算原语） |

> 要让模型在特定硬件上高效运行，需要同时考虑硬件的**内存布局**和**计算原语**。

### 6.4 GPU 关键特性详解

评估芯片时有三个核心问题：能否运行工作负载？速度多快？成本多少？三个关键指标帮助回答前两个问题：

#### 1）计算能力（Computational Capabilities）

- 衡量指标：**FLOP/s**（每秒浮点运算次数），常写作 FLOPS
- 标称的是**峰值** FLOP/s，实际应用几乎不可能达到
- 精度越高，能执行的操作越少（如 32 位加法的计算量约为 16 位的两倍）

**NVIDIA H100 SXM 不同精度的 FLOP/s**：

| 精度格式 | teraFLOP/s（含稀疏性） |
|----------|----------------------|
| TF32 Tensor Core | 989 |
| BFLOAT16 Tensor Core | 1,979 |
| FP16 Tensor Core | 1,979 |
| FP8 Tensor Core | 3,958 |

#### 2）内存大小与带宽（Memory Size and Bandwidth）

GPU 通常与三层内存交互，形成**内存层次结构**：

```
              ┌──────────┐
              │ GPU SRAM │  ~20 MB, ~10 TB/s
              │ (on-chip)│  （极快但极小）
              ├──────────┤
              │ GPU HBM  │  ~40-80 GB, 256 GB/s ~ 1.5 TB/s
              │          │  （快且较大）
              ├──────────┤
              │ CPU DRAM │  ~16 GB-1 TB+, 25-50 GB/s
              │          │  （慢但很大）
              └──────────┘
```

| 内存层级 | 类型 | 典型容量 | 典型带宽 | 用途 |
|----------|------|----------|----------|------|
| **CPU DRAM** | DDR SDRAM（2D 结构） | 16-64 GB（笔记本）~ 1 TB+（工作站） | 25-50 GB/s | 系统内存，主机端数据存储 |
| **GPU HBM** | 高带宽内存（3D 堆叠结构） | 24-80 GB | 256 GB/s ~ 1.5+ TB/s | GPU 专用内存，存储模型权重和中间结果 |
| **GPU SRAM** | 片上缓存（L1/L2/L3） | ~40 MB 以下 | 10+ TB/s | 极高速缓存，存储频繁访问的数据和指令 |

> GPU 内存比 CPU 内存**更贵**的原因之一：需要更高的带宽和更低的延迟，因此采用更先进的内存技术。
>
> 大量 GPU 优化工作的核心就是如何**充分利用内存层次结构**。这推动了 AI 研究者学习 GPU 编程语言（如 CUDA、OpenAI Triton、ROCm）的热潮，因为 PyTorch/TensorFlow 尚未提供对内存的细粒度控制。

#### 3）功耗（Power Consumption）

芯片功耗通常以两个指标衡量：

| 指标 | 定义 | 说明 |
|------|------|------|
| **Maximum Power Draw** | 满载时芯片可能消耗的峰值功率 | 实际上限 |
| **TDP**（Thermal Design Power） | 冷却系统需要散发的最大热量 | 并非精确的功耗值，而是预期功耗的近似指示；GPU 最大功耗约为 TDP 的 1.1-1.5 倍 |

**能耗的现实影响**：

- NVIDIA A100 有 **540 亿**个晶体管，H100 有 **800 亿**个
- 一块 NVIDIA H100 满载运行一年约消耗 **7,000 kWh**——接近美国普通家庭年均用电量（10,000 kWh）
- 电力供应已成为扩展计算的**关键瓶颈**之一，大规模数据中心选址的核心考量就是电力供应

#### 选择加速器的决策框架

| 考量因素 | 对应指标 |
|----------|----------|
| 硬件能否运行你的工作负载？ | 内存容量、FLOP/s |
| 运行需要多长时间？ | FLOP/s、内存带宽 |
| 成本多少？ | 云服务按使用计费；自购硬件看购买价格 + 持续功耗 |

- Compute-bound 工作负载 → 优先选择 **高 FLOP/s** 的芯片
- Memory bandwidth-bound 工作负载 → 优先选择 **高带宽、大内存** 的芯片
