# Chapter 9: Inference Optimization — 优化技术

> **核心命题**：推理优化可以在模型级、硬件级和服务级三个层面进行。模型级优化修改模型本身（可能改变输出质量），服务级优化不修改模型（只改变服务方式）。实际生产中往往需要**多个层面的技术组合使用**。

---

## 一、优化技术总览

### 1.1 三层优化的类比

| 层面 | 射箭类比 | 核心特点 |
|------|----------|---------|
| **模型级** | 制作更好的箭 | 修改模型本身，可能改变输出质量 |
| **硬件级** | 训练更强的弓箭手 | 设计更强大的芯片（本章不深入讨论） |
| **服务级** | 优化整个射击流程 | 不修改模型，只改变资源分配和服务方式 |

### 1.2 优化可能导致质量退化

理想情况下，优化模型的速度和成本不应改变输出质量。但实际上，**许多优化技术会导致模型退化**。

**实验证据**：同一个 Llama 模型在不同推理服务提供商（Fireworks、Together、Groq、Cerebras）上运行时，在 MMLU、GPQA、MGSM、MATH、DROP 等 benchmark 上的表现存在**明显差异**（Cerebras, 2024）。原因是不同提供商使用了不同的优化技术。

---

## 二、模型级优化（Model Optimization）

模型级优化的目标是让模型更高效，通常需要**修改模型本身**。当前 Transformer 架构的基础模型有三个使推理资源密集的特征：**模型体积**、**自回归解码**和**注意力机制**。下面分别讨论对应的优化策略。

### 2.1 模型压缩（Model Compression）

模型压缩通过减小模型体积来提速——更小的模型不仅存储更少，运行也更快。三种主要技术：

| 技术 | 原理 | 特点 |
|------|------|------|
| **量化（Quantization）** | 降低模型参数的数值精度（如 FP32→FP16→INT8→INT4） | **最流行、最易用、效果最好**。精度从 32 位降到 16 位，内存占用减半。目前接近量化极限（不可能低于 1 bit/value） |
| **蒸馏（Distillation）** | 训练小模型模仿大模型的行为 | 可以得到体积远小于原模型但性能接近的模型 |
| **剪枝（Pruning）** | 移除模型中对预测贡献最小的参数或结构 | 理论上前景好（Frankle & Carbin 2019 证明可减少 90%+ 非零参数），但实践中**不如量化和蒸馏常用** |

#### 剪枝的两种含义

| 类型 | 做什么 | 效果 |
|------|--------|------|
| **结构化剪枝** | 移除整个神经网络节点/层 | 改变模型架构，减少参数总数 |
| **非结构化剪枝** | 将不重要的参数设为零 | 不改变参数总数，但减少非零参数数量，使模型**稀疏化** |

**剪枝在实践中不流行的原因**：

- 需要深入理解原始模型架构
- 性能提升通常不如其他方法
- 产生的稀疏模型需要硬件支持稀疏计算才能充分发挥优势
- 不是所有硬件架构都为稀疏性做了优化

> **当前共识**：纯权重量化（weight-only quantization）是迄今为止**最实用**的模型压缩方法——易用、适用于大多数模型、效果显著。但量化已接近极限，蒸馏因此成为重要的补充方案。

### 2.2 克服自回归解码瓶颈

#### 为什么自回归解码是瓶颈？

自回归语言模型逐个生成 token——每生成一个 token 都需要将模型的全部参数从 HBM 加载到计算单元。这一过程：

- **速度慢**：100 ms/token 的速度下，100 token 的回答需要 10 秒
- **成本高**：跨模型 API 提供商，一个输出 token 的成本约为输入 token 的 **2-4 倍**
- **效率低**：每次生成仅产生一个 token，但需要传输整个模型的参数，导致 FLOP 利用率极低

> Anyscale 实验发现：**一个输出 token** 对整体延迟的影响约等于 **100 个输入 token**。

#### 技术一：推测解码（Speculative Decoding）

**核心思想**：用一个更小更快的**草稿模型（draft model）**先快速生成一串 token，然后让**目标模型（target model）**并行验证这些 token 是否正确。

**工作流程**：

```
1. 草稿模型生成 K 个候选 token：x_{t+1}, x_{t+2}, ..., x_{t+K}
2. 目标模型并行验证这 K 个 token（一次前向传播）
3. 目标模型接受从左到右最长的正确子序列（假设接受了 j 个）
4. 目标模型额外生成一个新 token x_{t+j+1}
5. 循环回到步骤 1
```

**关键洞察——为什么这有效**：

1. **验证比生成快**：验证 K 个 token 可以并行化（类似 prefill），而生成是串行的。推测解码将 decode 的计算特征转变为类似 prefill 的特征
2. **许多 token 容易预测**：如常见短语、代码模式等，弱模型也能猜对，接受率很高
3. **解码阶段有空闲算力**：解码是 memory bandwidth-bound，GPU 有空闲的 FLOP 可以"免费"用于验证

> **注意**：如果 MFU 已经接近极限，推测解码的效果会减弱——因为没有空闲算力可用。

**实际效果**：

| 实验 | 草稿模型 | 目标模型 | 延迟降低 |
|------|----------|---------|---------|
| DeepMind（Chen et al., 2023） | 4B 参数同架构模型 | Chinchilla-70B | **>50%**（1.8 ms/token vs 14.1 ms/token） |
| Leviathan et al. (2022) | — | T5-XXL | 类似的加速效果 |

**接受率**是领域相关的——结构化文本（如代码）的接受率更高。K 值越大意味着对目标模型的调用越少，但接受率会降低。草稿模型理想情况下应与目标模型共享**相同的词汇表和 tokenizer**。

**生态支持**：推测解码已被集成到 vLLM、TensorRT-LLM、llama.cpp 等主流推理框架中，PyTorch 中仅需约 50 行代码即可实现。

#### 技术二：参考推理（Inference with Reference）

**核心思想**：当模型的输出很可能直接复制输入中的内容时，直接从输入中"抄写" token，而非让模型逐个生成。

**适用场景**：

- 基于文档回答问题（回答中大量引用原文）
- 修复代码 bug（输出与原始代码大部分重复）
- 检索系统、多轮对话

**与推测解码的对比**：

| 特性 | 推测解码 | 参考推理 |
|------|---------|---------|
| 草稿 token 来源 | 额外的草稿模型生成 | **直接从输入文本中选取** |
| 是否需要额外模型 | 是 | **否** |
| 适用范围 | 通用 | 仅在输入与输出**高度重叠**时有效 |

**效果**：Yang et al. (2023) 在检索系统和编程等场景中实现了 **2 倍**的生成加速。

#### 技术三：并行解码（Parallel Decoding）

**核心思想**：打破自回归的严格顺序依赖，尝试**同时生成多个未来 token**。这意味着在还不知道 x_{t+1} 的情况下就尝试预测 x_{t+2}。

**为什么可行**：对于已有上下文，下文通常有足够的可预测性。例如给定"the cat sits"，即使不知道下一个词是"on""under"还是"behind"，你仍可能正确预测后面会出现"the"。

**两种主要方法**：

##### Lookahead Decoding（Fu et al., 2024）

- 由**同一个解码器**并行生成多个未来 token
- 使用 **Jacobi 方法**（一种可以同时独立更新多个解的迭代算法）验证生成的 token
- 流程：
  1. 并行生成 K 个未来 token
  2. 验证这些 token 与上下文的连贯性和一致性
  3. 如果某些 token 验证失败，只重新生成/调整这些失败的 token（而非丢弃全部）
  4. 反复迭代直到全部 token 通过验证

##### Medusa（Cai et al., 2024）

- 在原始模型上添加**多个额外的解码头（Medusa heads）**
- 每个 head 负责预测不同位置的未来 token（第 k 个 head 预测 x_{t+k+1}）
- 这些 head 与原始模型一起训练，但**原始模型权重冻结**
- 使用**树状注意力机制**组织候选 token，从中选择最优组合
- NVIDIA 声称 Medusa 在 HGX H200 GPU 上将 Llama 3.1 的 token 生成速度提升了 **1.9 倍**

**并行解码的局限**：虽然打破顺序依赖的思路很有吸引力，但实现较复杂，且 Medusa 等技术需要额外的训练步骤。

### 2.3 注意力机制优化

#### 为什么注意力机制需要优化？

生成每个新 token 需要计算它与**所有前序 token** 的注意力分数。这意味着：

- 注意力计算量随序列长度呈 **O(n²)** 增长
- 需要存储所有前序 token 的 Key 和 Value 向量
- 这些向量存储在 **KV cache** 中，避免每步重复计算

#### KV Cache 大小计算

```
KV cache 内存 = 2 × B × S × L × H × M
```

| 符号 | 含义 |
|------|------|
| 2 | Key 和 Value 两组向量 |
| B | batch size |
| S | 序列长度 |
| L | Transformer 层数 |
| H | 模型维度 |
| M | 数值表示所需的内存（如 FP16 = 2 bytes） |

**示例**：LLama 2 13B（40 层，model dimension 5120），batch size 32，序列长度 2048，FP16：

```
2 × 32 × 2,048 × 40 × 5,120 × 2 bytes = 54 GB
```

这个数字可以超过模型权重本身的大小！

#### 优化方向一：重新设计注意力机制

这些技术**改变注意力的工作方式**，因此只能在训练或微调时应用：

| 技术 | 原理 | KV cache 缩减效果 |
|------|------|-------------------|
| **Local Windowed Attention**（Beltagy et al., 2020） | 每个 token 只关注固定窗口内的邻近 token，而非全部前序 token | 窗口 1,000、序列 10,000 → 缩减 **10 倍** |
| **Cross-layer Attention**（Brandon et al., 2024） | 相邻层共享 KV 向量（如 3 层共享一组） | 缩减 **3 倍** |
| **Multi-query Attention（MQA）**（Shazeer, 2019） | 所有 query head 共享同一组 KV 向量 | 大幅缩减（但灵活性受限） |
| **Grouped-query Attention（GQA）**（Ainslie et al., 2023） | MQA 的泛化——将 query head 分组，组内共享 KV 向量 | 在 query 数量和 KV 对数量之间取得**灵活平衡** |

**Local windowed attention** 可以与 global attention 交替使用——local attention 捕获局部上下文，global attention 捕获全局任务信息。

**实际案例**：Character.AI（聊天机器人应用）的平均对话长度达 180 条消息。通过组合使用 MQA、local+global attention 交替、cross-layer attention，他们将 KV cache 缩减了**超过 20 倍**，使得内存不再是大批量请求的瓶颈。

#### 优化方向二：优化 KV Cache 管理

这些技术不改变注意力机制本身，而是更高效地**管理 KV cache 的内存**：

| 技术 | 原理 |
|------|------|
| **PagedAttention**（vLLM, Kwon et al., 2023） | 将 KV cache 划分为非连续的块（类似操作系统的虚拟内存分页），减少碎片化，支持灵活的内存共享 |
| **KV Cache 量化**（Hooper et al., 2024; Kang et al., 2024） | 降低 KV cache 中数值的精度 |
| **自适应 KV Cache 压缩**（Ge et al., 2023） | 根据重要性动态压缩 KV cache |
| **选择性 KV Cache**（Liu et al., 2024） | 只保留对当前生成最重要的 KV 对 |

> vLLM 的 PagedAttention 是其快速崛起的关键创新，被视为推理框架领域的标志性技术。

#### 优化方向三：编写注意力计算内核

不改变机制设计也不改变存储方式，而是**优化注意力分数的计算过程本身**，使其在特定硬件上运行更快。

**FlashAttention**（Dao et al., 2022）是最著名的注意力优化内核：

- 将 Transformer 中常见的多个操作（Matmul → Mask → Softmax → Dropout → Matmul）**融合（fuse）**为单个内核
- 原始 PyTorch 实现需要多次内存读写，FlashAttention 将其减少为一次
- 最初为 NVIDIA A100 优化，后来 FlashAttention-3（Shah et al., 2024）为 H100 做了适配

### 2.4 内核与编译器（Kernels and Compilers）

#### 什么是内核？

**内核（Kernel）**是为特定硬件加速器（GPU/TPU）优化的代码片段，用于执行需要反复、并行运行的计算密集型操作。

**常见的 AI 操作**——矩阵乘法、注意力计算、卷积——都有专用内核来提升特定硬件上的性能。

#### 编写内核的四种常用技术

| 技术 | 原理 | 通用性 |
|------|------|--------|
| **向量化（Vectorization）** | 一次处理内存中连续的多个数据元素（而非逐个处理），减少 I/O 操作 | 通用 |
| **并行化（Parallelization）** | 将数组/矩阵分割为独立块，在不同核心/线程上同时处理 | 通用 |
| **循环切片（Loop Tiling）** | 优化循环中数据的访问顺序，适配硬件的内存布局和缓存（CPU 和 GPU 的最优切片策略不同） | **硬件相关** |
| **算子融合（Operator Fusion）** | 将多个操作合并为一次执行，减少冗余内存读写（如两个循环操作同一数组→合并为一个循环） | 需要理解**模型特定架构** |

> 向量化和并行化可以广泛适用于不同模型，但算子融合需要对模型的具体算子和架构有深入了解，因此更需要优化工程师的参与。

#### 内核编程语言

| 语言 | 用途 |
|------|------|
| **CUDA** | NVIDIA GPU 专用 |
| **Triton** | OpenAI 开发，用于编写自定义内核 |
| **ROCm** | AMD GPU（AMD 的 CUDA 开源替代方案） |

这些语言提供对线程管理和内存访问的细粒度控制，但学习门槛高于 Python。

> 内核编写曾经是少数优化工程师的"黑魔法"。许多公司将自研内核视为**商业机密**——因为能让模型运行得比竞争对手更快更便宜就是核心竞争力。但随着推理优化需求爆发，越来越多 AI 工程师开始学习内核编写。

#### 编译器

**编译器（Compiler）**的作用是将模型代码"降级（lower）"为目标硬件兼容的语言，并在可能时转换为专用内核以提速。

| 编译器 | 类型 |
|--------|------|
| **Apache TVM** | 独立工具 |
| **MLIR** | 独立工具（Multi-Level Intermediate Representation） |
| **torch.compile** | PyTorch 内置 |
| **XLA** | 由 TensorFlow 开发，有开源版本 OpenXLA |
| **TensorRT** | NVIDIA GPU 专用编译器 |

内核与硬件紧密绑定——每当新硬件架构发布，就需要开发新内核。例如 FlashAttention 最初为 A100 设计，后来需要为 H100 适配 FlashAttention-3。

#### PyTorch 优化案例研究

PyTorch 团队对 Llama-7B 在 A100-80GB GPU 上逐步叠加优化技术，吞吐量变化如下：

| 优化步骤 | 吞吐量（tok/s/user） | 相对提升 |
|----------|---------------------|---------|
| 原始（eager mode） | 25.5 | — |
| + `torch.compile` | 107.0 | **4.2×** |
| + INT8 量化 | 157.4 | **1.5×** |
| + INT4 量化 | 202.1 | **1.3×** |
| + 推测解码 | 244.7 | **1.2×**（+21%） |

> 从 25.5 到 244.7 tok/s/user，总计提升近 **10 倍**。但需要注意：实验未说明这些优化对输出质量的影响。

## 三、推理服务级优化（Inference Service Optimization）

服务级优化聚焦于**资源管理**——给定固定的计算和内存资源，以及来自多用户、多模型的动态推理请求，目标是高效分配资源以优化延迟和成本。

**关键区别**：与模型级技术不同，服务级技术**不修改模型本身**，因此不应改变输出质量。

### 3.1 批处理（Batching）

批处理是降低成本最简单有效的方式之一。将同时到达的多个请求**合并处理**，而非逐个处理，可以显著提高吞吐量。

**类比**：处理每个请求就像每人开一辆车；批处理就像让大家上同一辆公交车——能运更多人，但每个人的行程可能稍长。如果调度得当，对延迟的影响可以很小。

#### 三种批处理策略

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **Static Batching** | 等待凑满固定数量的请求后一起处理 | 实现简单 | 早到的请求需等待最后一个请求，**延迟不可控** |
| **Dynamic Batching** | 设置最大等待时间窗口（如 100ms），到时间或凑满就发送 | 延迟可控（早到的请求不会被无限拖延） | batch 可能不满，计算利用率不如 static |
| **Continuous Batching** | 请求完成后立即返回，空出的位置**立即填入新请求** | **最优效率**——无需等待整个 batch 完成 | 实现最复杂 |

```
Static batching（等满发车）:
  Requests:  ○○○    ○ ○○○   ○ ○○   ○      ○
  Batches:   [○○○]  [○ ○○○ ○]  [○○ ○]  [○]

Dynamic batching（定时发车或满即发）:
  Requests:  ○○○    ○ ○○○   ○ ○○   ○      ○
  Batches:   [○○○]  [○ ○○○]  [○ ○○ ○]  [○]  [○]
```

**Continuous Batching**（也叫 **in-flight batching**，Yu et al., Orca, 2022）的特殊之处：

- 传统批处理中，所有请求必须等最长的请求完成后才能统一返回
- 一个 10 token 的响应和一个 1,000 token 的响应在同一 batch 中→短响应必须等 1,000 token 生成完
- Continuous batching 让**完成的请求立即返回**，空位立即被新请求填入
- 类似公交车在每站都下客上客，持续保持高载客率

### 3.2 Prefill 与 Decode 解耦

**问题**：Prefill 是 compute-bound，decode 是 memory bandwidth-bound。在同一台机器上同时运行两者会导致**资源竞争**：

- GPU 正在满负载做 prefill + decode
- 新请求到达，触发一个 prefill 任务
- 这个 prefill 任务抢占 decode 的计算资源
- 导致已有请求的 **TPOT 显著劣化**

**解决方案**（DistServe, Zhong et al., 2024; "Inference Without Interference", Hu et al., 2024）：将 prefill 和 decode **分配到不同的 GPU 实例**上。

**效果**：对多种主流 LLM 和应用，解耦后在满足延迟要求的前提下**显著提高了处理请求的总量**。

**通信开销**：解耦需要在 prefill 实例和 decode 实例之间传输中间状态，但现代 GPU 集群内的高带宽连接（如 NVLink）使这一开销可忽略不计。

**Prefill:Decode 实例比例**取决于工作负载特征和延迟优先级：

| 场景 | 推荐比例 |
|------|---------|
| 输入序列长（需要大量 prefill 计算）、优先降低 TTFT | 2:1 到 4:1 |
| 输入序列短、优先降低 TPOT | 1:2 到 1:1 |

### 3.3 Prompt Caching（提示词缓存）

**问题**：同一应用中的大量请求往往共享相同的文本段——最典型的是 **system prompt**。没有 prompt cache 时，每个请求都要重新处理完整的 system prompt。

**解决方案**：将重复出现的文本段的处理结果缓存起来，后续请求直接复用。

```
  ┌─ System prompt ─────────────┐  ← 缓存并复用
  │ Your task is to identify ... │
  │ ...                          │
  │ ...                          │
  ├─ Example ───────────────────┤  ← 也可以缓存
  │ Text: "Brave New World..."  │
  │ Entities: Brave New World,  │
  │           Aldous Huxley      │
  ├─ Task ──────────────────────┤
  │ Text: ${USER_INPUT}          │  ← 每次不同
  │ Entities:                    │
  └─────────────────────────────┘
```

**Prompt cache 也叫 context cache 或 prefix cache**。

**适用场景**：

- 长 system prompt 的应用
- 基于同一长文档的多轮查询（如读书、代码库）
- 长对话（缓存早期消息的处理结果）

**成本与延迟影响**（来自 Anthropic, 2024）：

| 场景 | 无缓存延迟（TTFT） | 有缓存延迟 | 成本节省 |
|------|-------------------|----------|---------|
| 读书对话（100K token cached prompt） | 11.5 s | 2.4 s（−79%） | **−90%** |
| Many-shot prompting（10K token prompt） | 1.6 s | 1.1 s（−31%） | **−86%** |
| 多轮长对话（10 轮，长 system prompt） | ~10 s | ~2.5 s（−75%） | **−53%** |

**生态支持**：

- Prompt cache 由 Gim et al. 于 2023 年 11 月引入，随后被快速集成到各大模型 API 中
- Google Gemini：缓存 token 享受 **75% 折扣**（但需额外付费存储缓存，$1.00/百万 token/小时）
- Anthropic：承诺最高 **90% 成本节省**和 **75% 延迟降低**

**注意事项**：

- Prompt cache 本质上也是 KV cache，会占用大量内存
- 除非使用的 API 已内置此功能，否则自行实现需要**大量工程投入**

### 3.4 并行策略（Parallelism）

加速器天然为并行处理设计。推理中常用的并行策略按通用性分为两大类：

#### 通用并行策略

##### Replica Parallelism（副本并行）

**最简单**的策略——创建模型的多个副本，分布到不同芯片上处理更多请求。

**挑战**：如何将不同大小的模型放到不同容量的 GPU 上——本质上是一个**装箱问题（bin-packing problem）**。

示例：有 8B、13B、34B、70B 四种模型，以及 24GB、40GB、48GB、80GB 四种 GPU，全部使用 8-bit 精度。如何分配副本和 GPU 以最大化性能指标（或最小化成本）？

##### Model Parallelism（模型并行）

当模型太大无法放入单机时，需要将模型**拆分到多台机器**上。

**Tensor Parallelism**（张量并行 / intra-operator parallelism）：

- 将单个操作（如矩阵乘法）中的张量**按列切分到多个设备**上，并行计算
- 两个好处：
  1. 使超大模型能运行在单机装不下的场景
  2. **降低延迟**（单个操作被分摊到多个设备）
- 缺点：设备间通信开销可能抵消延迟收益

**Pipeline Parallelism**（流水线并行）：

- 将模型按**层**切分为多个阶段，每个阶段分配到不同设备
- 数据在设备间流水传递——当设备 1 处理第二个 micro-batch 时，设备 2 可同时处理第一个 micro-batch 的后续层
- 优点：可在多台机器上运行超大模型
- 缺点：每个请求的**总延迟增加**（因为流水线各阶段间有通信开销）
- 因此在**延迟敏感**的推理场景中，pipeline parallelism **不如 replica parallelism 常用**
- 但在训练中广泛使用（训练更关注吞吐量）

#### LLM 专用并行策略

| 策略 | 原理 |
|------|------|
| **Context Parallelism** | 将输入序列本身切分到不同设备上分别处理（如前半在机器 1，后半在机器 2） |
| **Sequence Parallelism** | 将处理整个输入所需的不同操作切分到不同设备上（如 attention 在机器 1，feedforward 在机器 2） |

这两种策略都是为处理**长输入序列**而设计的优化。

> 实际生产中，一个优化方案**通常组合使用多种并行策略**。

## 四、章节总结与技术选择指南

### 4.1 模型级 vs 服务级优化对比

| 维度 | 模型级优化 | 服务级优化 |
|------|----------|----------|
| 是否修改模型 | **是** | **否** |
| 是否影响输出质量 | **可能** | **不应该** |
| 代表技术 | 量化、蒸馏、推测解码、注意力优化、内核编写 | 批处理、prefill/decode 解耦、prompt caching、并行策略 |

### 4.2 技术选择取决于工作负载

| 工作负载特征 | 推荐优化重点 |
|-------------|------------|
| 长上下文 | **KV cache 优化**（重要性远高于短上下文场景） |
| 长且重复的 prompt 段、多轮对话 | **Prompt caching** |
| 低延迟优先 | **Replica parallelism**（增加副本，每台机器处理更少请求→更快响应） |
| 高吞吐量优先 | **Tensor parallelism**（减少延迟同时支持大模型）、**批处理** |

### 4.3 跨场景最具影响力的四项技术

根据原书 Summary 部分，在各种使用场景中最具普遍影响力的技术是：

1. **量化（Quantization）**——通用性最强，适用于几乎所有模型
2. **Tensor Parallelism**——既降低延迟又支持大模型服务
3. **Replica Parallelism**——最简单直接，易于实现
4. **注意力机制优化**——对 Transformer 模型加速效果显著

## 五、全景图

```
推理优化技术（Inference Optimization）
    │
    ├── 模型级优化（Model Optimization）
    │     ├── 模型压缩
    │     │     ├── 量化（最流行，接近极限）
    │     │     ├── 蒸馏
    │     │     └── 剪枝（不如前两者常用）
    │     │
    │     ├── 克服自回归解码瓶颈
    │     │     ├── 推测解码（草稿模型 + 目标模型验证）
    │     │     ├── 参考推理（从输入中复制 token）
    │     │     └── 并行解码（Lookahead / Medusa）
    │     │
    │     ├── 注意力机制优化
    │     │     ├── 重新设计：Local Windowed / Cross-layer / MQA / GQA
    │     │     ├── KV Cache 管理：PagedAttention / 量化 / 压缩 / 选择性
    │     │     └── 计算内核：FlashAttention
    │     │
    │     └── 内核与编译器
    │           ├── 四种技术：向量化 / 并行化 / 循环切片 / 算子融合
    │           └── 编译器：TVM / MLIR / torch.compile / XLA / TensorRT
    │
    └── 服务级优化（Inference Service Optimization）
          ├── 批处理：Static → Dynamic → Continuous（最优）
          ├── Prefill/Decode 解耦（不同实例处理不同阶段）
          ├── Prompt Caching（缓存重复文本段）
          └── 并行策略
                ├── 通用：Replica / Tensor / Pipeline Parallelism
                └── LLM 专用：Context / Sequence Parallelism
```

> **核心 Takeaway**：推理优化不是单一技术的选择，而是**多种技术的组合与权衡**。PyTorch 的案例研究表明，逐步叠加 compile + 量化 + 推测解码可以实现近 10 倍的吞吐量提升。选择哪些技术取决于你的工作负载特征（长/短上下文、延迟/吞吐量优先）、硬件条件和可接受的质量退化程度。对大多数应用开发者而言，使用模型 API 提供商已内置的优化（prompt caching、batching 等）是最直接有效的方式，但理解底层技术有助于做出更好的架构决策和成本估算。
