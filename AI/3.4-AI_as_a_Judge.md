# AI as a Judge（AI 评审）

> **核心定位**：当开放式任务无法通过精确评估（功能正确性、相似度度量）自动衡量时，**用 AI 评估 AI** 成为最实用的自动评估手段。AI as a Judge 属于**主观评估**——评分结果取决于所用模型和 prompt 的选择。截至目前，它已成为生产环境中最常用的评估方法之一。

---

## 为什么需要 AI as a Judge

### 从第一性原理出发

评估的终极瓶颈在于：**开放式任务的正确答案是无限的**。精确评估（exact match、BLEU 等）依赖有限的参考答案集，天然不完备。人类评估虽然灵活，但**慢、贵、不可规模化**。

AI as a Judge 的本质就是：**用一个足够强的模型去模拟人类评审者的判断能力**，以获得接近人类判断质量、但速度和成本远优于人类的评估方案。

### 核心优势

| 优势 | 说明 |
|---|---|
| **快速** | 秒级出评分，远快于人类标注 |
| **便宜** | 相对人类评估成本低几个数量级 |
| **无需参考数据** | 可直接评估生成结果本身的质量，适用于生产环境中无标准答案的场景 |
| **灵活** | 可按任意标准评估：正确性、流畅性、毒性、幻觉、角色一致性…… |
| **可解释** | AI 不仅给分，还能解释评分理由，便于审计 |

### 与人类评估的相关性

- Zheng et al. (2023) 在 MT-Bench 上发现 GPT-4 与人类评审的一致率达 **85%**，甚至高于人类之间的一致率（81%）
- AlpacaEval 的 AI 评审与 LMSYS Chatbot Arena 人类排行榜的相关性达 **0.98**

### 行业采用情况

- LangChain 2023 年报告：其平台上 **58%** 的评估由 AI Judge 完成
- 2023–2024 年几乎所有 AI 评估创业公司的产品 demo 都采用了 AI as a Judge

## 如何使用 AI as a Judge

### 三种基本评估模式

| 模式 | 输入 | 输出 | 适用场景 |
|---|---|---|---|
| **单独评分** | (问题, 回答) | 1–5 分 | 对单个回答进行质量评估 |
| **参考对比** | (问题, 参考答案, 生成答案) | True / False | 替代手工相似度度量，判断生成答案是否与参考一致 |
| **成对比较** | (问题, 回答A, 回答B) | A 或 B | 生成偏好数据（用于 RLHF 对齐）、排名模型 |

#### 示例 Prompt

**模式一：单独评分**

```
Given the following question and answer, evaluate how good the answer is
for the question. Use the score from 1 to 5.
- 1 means very bad.
- 5 means very good.
Question: [QUESTION]
Answer: [ANSWER]
Score:
```

**模式二：参考对比**

```
Given the following question, reference answer, and generated answer,
evaluate whether this generated answer is the same as the reference answer.
Output True or False.
Question: [QUESTION]
Reference answer: [REFERENCE ANSWER]
Generated answer: [GENERATED ANSWER]
```

**模式三：成对比较**

```
Given the following question and two answers, evaluate which answer is
better. Output A or B.
Question: [QUESTION]
A: [FIRST ANSWER]
B: [SECOND ANSWER]
The better answer is:
```

### 评估标准的灵活性

AI Judge 可以按**任意自定义标准**评估。举例：

- 角色扮演聊天机器人 → "这个回答听起来像甘道夫会说的话吗？"
- 商品推广图片生成 → "从 1 到 5，这张图片中产品的可信度如何？"

#### 常见工具的内置评估标准

| 工具 | 内置标准 |
|---|---|
| **Azure AI Studio** | Groundedness、Relevance、Coherence、Fluency、Similarity |
| **MLflow.metrics** | Faithfulness、Relevance |
| **LangChain Criteria** | Conciseness、Relevance、Correctness、Coherence、Harmfulness、Helpfulness 等 |
| **Ragas** | Faithfulness、Answer Relevance |

> **关键警告**：不同工具对同一标准（如 Relevance）的定义、prompt 和评分体系完全不同，分数**不可跨工具比较**。

---

### Prompt 设计三要素

一个好的 AI Judge prompt 应清晰包含：

1. **任务描述**：告诉模型它要做什么（如"评估生成答案与问题的相关性"）
2. **评估标准**：越详细越好（如"主要关注生成答案是否包含足够信息来回答问题"）
3. **评分体系**：

| 评分体系类型 | 示例 | 适用性 |
|---|---|---|
| **分类** | good / bad、relevant / irrelevant | 语言模型最擅长，效果最好 |
| **离散数值** | 1–5 分 | 效果较好；范围越窄越好，典型为 1–5 |
| **连续数值** | 0–1 之间 | 效果最差；模型对数字的感知不如文本 |

> **最佳实践**：在 prompt 中包含**每个分值对应的示例**，以及**给出该分值的理由**。带示例的 prompt 表现显著优于不带示例的。

#### Azure AI Studio 的 Relevance Prompt 示例（节选）

```
Your task is to score the relevance between a generated answer and the
question based on the ground truth answer in the range between 1 and 5,
and please also provide the scoring reason.

Your primary focus should be on determining whether the generated answer
contains sufficient information to address the given question according
to the ground truth answer. …

If the generated answer contradicts the ground truth answer, it will
receive a low score of 1-2.

For example, for the question "Is the sky blue?" the ground truth answer
is "Yes, the sky is blue." and the generated answer is "No, the sky is
not blue." …

This inconsistency would result in a low score of 1–2, and the reason
for the low score would reflect the contradiction between the generated
answer and the ground truth answer.
```

### 核心认知

> **AI Judge 不只是一个模型——它是一个由模型 + prompt 构成的系统。** 改变模型、prompt 或采样参数中的任何一个，就会产生一个不同的 Judge。

## AI as a Judge 的局限性

### 1. 不一致性（Inconsistency）

**问题**：AI Judge 是概率性的，同一个 Judge 对同一输入多次运行可能给出不同分数。不同的 prompt 更会导致不同结果。

**缓解方法**：
- 通过采样参数（如 temperature = 0）提高确定性
- 在 prompt 中加入评估示例：Zheng et al. (2023) 发现这能将 GPT-4 的一致性从 **65% 提升到 77.5%**
- 但代价是更长的 prompt → 更高的推理成本（他们的 GPT-4 开销增加了 **4 倍**）

**深层隐患**：高一致性不一定意味着高准确性——Judge 可能一致地犯同样的错误。

### 2. 标准歧义（Criteria Ambiguity）

**问题**：AI Judge 的评估标准没有统一规范。不同工具对同一标准名称的实现完全不同。

**以 Faithfulness 为例**：

| 工具 | Prompt 核心内容 | 评分体系 |
|---|---|---|
| **MLflow** | 评估输出与上下文的事实一致性；完全忽略输入 | 1–5 分 |
| **Ragas** | 判断每个陈述是否可由上下文验证 | 0 和 1 |
| **LlamaIndex** | 判断信息是否被上下文支持；只要任何上下文支持即为 YES | YES / NO |

同一个 `(context, answer)` 对，MLflow 给 3 分、Ragas 给 1、LlamaIndex 给 NO——**应该以谁为准？**

**时间维度的歧义**：

AI Judge 本身也是 AI 应用，会随时间变化。如果上月应用的 coherence 得分 90%、本月 92%——是应用变好了，还是 Judge 的 prompt 或模型变了（比如同事修了个 typo，导致 Judge 更宽松了）？当应用团队和 Judge 团队分属不同部门时，问题更突出。

> **铁律：如果你看不到 AI Judge 使用的模型和 prompt，就不要信任它的评分。**

### 3. 成本与延迟增加（Increased Costs and Latency）

**成本问题**：

- 用 GPT-4 同时生成和评估 → API 调用量翻倍 → 成本翻倍
- 评估 3 个标准（质量、事实一致性、毒性）→ API 调用量变为 4 倍
- 某些场景下评估成本甚至超过生成成本

**缓解方法**：
- **使用更弱/更便宜的模型**做 Judge
- **抽样评估（Spot-checking）**：只评估一部分响应。抽样比例越高 → 信心越高 → 成本越高，需在两者间权衡

**延迟问题**：

如果在返回响应给用户之前进行评估（作为 guardrail），会增加用户等待时间。对延迟敏感的应用可能无法接受。

> 尽管有成本，AI Judge 仍远比人类评估便宜。

### 4. AI Judge 的偏差（Biases）

| 偏差类型 | 表现 | 缓解方法 |
|---|---|---|
| **自我偏好（Self-bias）** | 模型倾向于给自己生成的回答打更高分。GPT-4 自我偏好导致 win rate 高 10%，Claude-v1 高 25% | 避免用同一模型同时生成和评估 |
| **位置偏差（Position bias）** | AI 倾向于偏好成对比较中**第一个**出现的答案 | 多次测试并打乱顺序；用精心设计的 prompt 缓解 |
| **冗长偏差（Verbosity bias）** | AI 倾向于偏好更长的回答，即使更短的回答更正确。Wu & Aji (2023)：GPT-4 和 Claude-1 偏好约 100 词带事实错误的回答，而非约 50 词的正确回答 | 更强的模型偏差更小（GPT-4 < GPT-3.5） |
| **通用 AI 局限** | 隐私与知识产权风险：使用闭源模型做 Judge 需将数据发送给模型提供商；无法确认 Judge 模型的训练数据是否存在商业安全问题 | 使用开源模型或有数据保护协议的服务 |

> **注意**：AI Judge 的位置偏差与人类相反——人类有**近因效应（Recency bias）**，倾向于偏好最后看到的答案。

> GPT-4 的冗长偏差比 GPT-3.5 小，暗示随着模型能力增强，部分偏差可能自然减弱。

## 什么模型可以做 Judge？

### 三种 Judge 配置

| 配置 | 说明 | 优势 | 劣势 |
|---|---|---|---|
| **更强的模型** | 用 GPT-4 评估 GPT-3.5 的输出 | 判断质量高；可帮助弱模型改进 | 最强模型没有 Judge；需要其他方法确定谁"最强" |
| **自我评估** | 模型评估自己的输出 | 适合做健全性检查；可推动模型自我修正 | 受自我偏好影响 |
| **更弱的模型** | 用小模型评估大模型 | 成本低、速度快 | 评判能力可能不足 |

### 更强模型做 Judge

直觉上合理——"阅卷老师应该比考生更懂"。但存在两个实际原因让更强模型做 Judge 而非直接做生成：

1. **成本**：不可能对所有请求都用最贵的模型生成响应，但可以用它评估少量响应（如 1%）
2. **延迟**：强模型可能太慢不适合实时生成，但可以在后台做异步评估。如果发现弱模型回答不好，可采取补救措施（如用强模型的回答替换）

> 反向模式也常见：用强模型生成，弱模型在后台做评估。

### 自我评估（Self-evaluation）

虽然受自我偏好影响，但**如果模型认为自己的回答是错的，那回答确实可能不靠谱**——这是有价值的信号。

自我评估还可以驱动模型**自我修正**（self-critique）：

```
Prompt [用户]: What's 10+3?
First response [AI]: 30
Self-critique [AI]: Is this answer correct?
Final response [AI]: No it's not. The correct answer is 13.
```

### 更弱的模型做 Judge

核心论点：**评判比生成更容易**。不是所有人都能写出好歌，但任何人都能判断一首歌好不好听。

- Zheng et al. (2023) 发现更强的模型与人类偏好的相关性更高，因此实践中人们倾向于使用能承担的最强模型
- 但这不意味着弱模型完全不行——**小型专用 Judge** 是一个重要方向

## 专用 Judge（Specialized Judges）

通用 Judge 用一个大模型处理所有评估标准。专用 Judge 则是**为特定评估任务训练的小模型**，在特定判断上可能比大型通用 Judge 更可靠。

### 三类专用 Judge

#### 1. 奖励模型（Reward Model）

| 属性 | 说明 |
|---|---|
| **输入** | (prompt, response) |
| **输出** | 0–1 之间的质量分 |
| **代表模型** | Google Cappy（3.6 亿参数，远小于通用基础模型） |
| **核心应用** | RLHF 训练中对生成响应打分 |

#### 2. 参考基准 Judge（Reference-based Judge）

| 属性 | 说明 |
|---|---|
| **输入** | (候选响应, 参考响应) 或 (prompt, 生成响应, 参考响应, 评分标准) |
| **输出** | 相似度分数 或 质量分 |
| **代表模型** | **BLEURT**：输出候选与参考之间的相似度分（范围约 -2.5 到 1.0）；**Prometheus**：输出 1–5 分（假设参考响应得 5 分） |
| **定位** | 用专用模型替代手工相似度度量（BLEU、ROUGE 等） |

#### 3. 偏好模型（Preference Model）

| 属性 | 说明 |
|---|---|
| **输入** | (prompt, response_1, response_2) |
| **输出** | 哪个响应更好（用户更偏好） |
| **代表模型** | **PandaLM**（Wang et al., 2023）、**JudgeLM**（Zhu et al., 2023） |
| **核心价值** | 偏好数据对 AI 对齐至关重要但获取昂贵，偏好模型可大规模自动化生成偏好信号 |

PandaLM 不仅输出哪个响应更好，还输出判断理由。

## 总结

```
                        ┌─ 单独评分：(问题, 回答) → 分数
    AI as a Judge 模式 ─┼─ 参考对比：(问题, 参考, 生成) → 是否一致
                        └─ 成对比较：(问题, 回答A, 回答B) → 谁更好

                        ┌─ 不一致性：概率性导致不可复现
    四大局限 ──────────┼─ 标准歧义：同名标准在不同工具中定义不同
                        ├─ 成本与延迟：额外 API 调用开销
                        └─ 偏差：自我偏好、位置偏差、冗长偏差

                        ┌─ 通用 Judge（强模型/自我评估/弱模型）
    Judge 类型 ────────┤
                        └─ 专用 Judge（奖励模型 / 参考基准 Judge / 偏好模型）
```

### 核心原则

1. **AI Judge = 模型 + Prompt**，改变任何一个就是不同的 Judge
2. **不可信任看不到模型和 prompt 的 AI Judge 分数**
3. **AI Judge 分数不可跨工具比较**——即使标准名称相同
4. **AI Judge 应与精确评估和/或人类评估互补**，而非单独使用
5. 专用小模型在特定评判任务上可能优于通用大模型，是降低成本和提高可靠性的重要方向
