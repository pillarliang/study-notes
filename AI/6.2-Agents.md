# Agents（智能体）

> **核心定位**：RAG 让模型从外部数据源**检索**信息来构造上下文，而 Agent 模式让模型能**使用工具**（如搜索 API、代码解释器、SQL 执行器）来收集信息和执行操作。Agent 不仅能构造上下文，还能**直接与外部世界交互**——创建网站、发送邮件、操作数据库、下单交易等——这使得它在能力上远超 RAG。RAG 可以被看作 Agent 的一个特例：检索器就是模型可用的一个工具。

---

## 一、Agent 概述

### 什么是 Agent？

Russell 和 Norvig 的经典教材《Artificial Intelligence: A Modern Approach》（1995）将**人工智能研究**定义为"对理性智能体的研究与设计"。Agent（智能体）是一个贯穿 AI 领域的核心概念。

**定义**：Agent 是任何能够**感知环境（perceive）**并**作用于环境（act）**的实体。因此，一个 Agent 由两个要素刻画：


| 要素                   | 说明                     | 示例                           |
| -------------------- | ---------------------- | ---------------------------- |
| **环境（Environment）**  | Agent 运行的世界，由其用例决定     | 游戏（Minecraft、围棋）、互联网、厨房、道路系统 |
| **动作集合（Action Set）** | Agent 能执行的所有操作，通过工具来扩展 | 搜索网页、执行代码、生成图片、编辑文件          |


环境与工具之间存在**双向依赖**：

- 环境决定了 Agent 能潜在使用哪些工具（棋盘上只能走合法棋步）
- 工具反过来限制了 Agent 能操作的环境（只能游泳的机器人被限制在水中）

### Agent 的工作方式

在 AI Agent 中，**AI 模型是"大脑"**——它处理接收到的信息（任务描述、环境反馈），规划实现任务的行动序列，并判断任务是否完成。

**实例**：以一个电商（Kitty Vogue）的 Agent 为例，它有三个动作：生成回答、生成 SQL 查询、执行 SQL 查询。当用户问"预测 Fruity Fedora 未来三个月的销售收入"时，Agent 的工作流程如下：

1. **推理**：要预测未来销售，先需要过去五年的销售数据
2. **调用 SQL 生成**：生成查询过去五年销售数据的 SQL
3. **调用 SQL 执行**：执行查询
4. **推理**：数据有缺失值，不足以做可靠预测，还需要过去的营销活动信息
5. **调用 SQL 生成**：生成查询营销活动的 SQL
6. **调用 SQL 执行**：执行查询
7. **推理**：信息足够，生成销售预测
8. **推理**：任务完成

### 为什么 Agent 需要更强的模型？

相比非 Agent 场景，Agent 对底层模型的要求更高，原因有二：


| 原因                          | 说明                                                                                 |
| --------------------------- | ---------------------------------------------------------------------------------- |
| **复合错误（Compound Mistakes）** | Agent 需要执行多步才能完成任务，整体准确率随步数增加而**指数级下降**。假设每步准确率 95%，10 步后整体准确率降到 60%，100 步后仅剩 0.6% |
| **更高风险（Higher Stakes）**     | Agent 能通过工具执行高影响力操作（如发起银行转账、删除数据库），一旦出错后果更严重                                       |


多步任务需要时间和金钱，但如果 Agent 能自主完成，节省的人力成本往往远超 API 开销。

### Agent 成功的两大支柱

给定一个环境，Agent 的成功取决于两个因素：

1. **工具库（Tool Inventory）**——Agent 能使用哪些工具
2. **AI 规划能力（Planning）**——模型能否有效地分析任务、规划步骤、纠正错误

## 二、Tools（工具）

Agent 不一定需要外部工具，但没有工具的 Agent 能力非常有限——LLM 只能生成文本，图像模型只能生成图片。外部工具让 Agent **能力倍增**。

工具帮助 Agent 做两件事：

- **感知环境**（read-only actions，只读动作）——如检索文档、读取邮件、查看数据库
- **作用于环境**（write actions，写入动作）——如发送邮件、修改数据库、发起转账

### 2.1 知识增强工具（Knowledge Augmentation）

这类工具帮助 Agent **获取完成任务所需的上下文信息**，前面章节讨论的检索器（text retriever、image retriever）和 SQL 执行器都属于这一类。其他例子包括：

- **内部工具**：人员搜索、库存 API、Slack 检索、邮件阅读器等——增强模型对组织内部私有流程和信息的了解
- **外部工具（Web Browsing）**：泛指所有访问互联网的工具，包括浏览器和各种 API（搜索 API、新闻 API、GitHub API、社交媒体 API 等）

**Web Browsing 的价值**：它是最早被集成到 ChatGPT 等聊天机器人中的能力之一。模型的训练数据有截止日期，如果上周刚训练完成，就无法回答本周的天气、新闻、股价等问题——除非这些信息通过上下文提供。Web Browsing 防止模型"过时（go stale）"。

> **注意**：Web Browsing 在带来最新信息的同时，也可能将互联网上的垃圾信息引入 Agent。选择互联网 API 时需要谨慎。

### 2.2 能力扩展工具（Capability Extension）

这类工具**弥补 AI 模型的固有缺陷**，是提升模型表现最简单直接的方式。

| 工具 | 解决的问题 |
|---|---|
| **计算器** | AI 模型不擅长数学运算。199,999 ÷ 292 对模型很难，但给它一个计算器就轻而易举 |
| **日历 / 时区转换器** | 时间相关的精确计算 |
| **单位转换器** | 如磅 ↔ 千克 |
| **翻译器** | 翻译模型不擅长的语言 |
| **代码解释器** | 让模型能执行代码、返回结果、分析错误。使 Agent 能充当编程助手、数据分析师、甚至研究助手（写代码跑实验并报告结果）。但自动执行代码有**代码注入攻击**的风险 |
| **文生图 / 图像标注** | 让纯文本模型变成多模态——ChatGPT 用 DALL-E 生成图片就是这个原理 |
| **LaTeX 编译器 / 浏览器** | 渲染数学公式、从 HTML 渲染网页 |
| **OCR** | 让模型能读取 PDF 中的文字 |
| **转录工具** | 让模型能处理音频 |

> **2024-2026 更新**：上表中部分工具的必要性已随模型进化而变化：
>
> - **数学能力**：推理模型（OpenAI o1/o3、DeepSeek-R1）的数学和逻辑能力已大幅提升，简单算术不再需要外部计算器，但复杂数值计算仍需工具辅助
> - **原生多模态**：GPT-4o、Claude 3.5+、Gemini 2.0+ 等模型已原生支持图像/音频输入和理解，不再需要单独的图像标注或转录工具来"感知"这些模态。但**生成**图像仍需外部工具（如 DALL-E、Stable Diffusion）
> - **代码解释器**已成为主流模型的标准内置功能（ChatGPT Code Interpreter、Claude Artifacts 等），不再是需要额外集成的外部工具
> - 新兴的能力扩展工具还包括 **Computer Use**（操控计算机 GUI，详见下文 2.4 节）


**实证数据**：Chameleon（Lu et al., 2023）让一个基于 GPT-4 的 Agent 使用 13 个工具（知识检索、查询生成器、图像标注器、文本检测器、Bing 搜索等），在多个 benchmark 上**大幅超越单独的 GPT-4**：

- ScienceQA（科学问答）：比最佳 few-shot 结果提升 **11.37%**
- TabMWP（表格数学问题）：准确率提升 **17%**

### 2.3 写入动作（Write Actions）

前面讨论的都是**只读**工具。但工具也能执行**写入**操作——改变数据源：

- SQL 执行器可以检索表（读），也可以修改或删除表（写）
- 邮件 API 可以读邮件，也可以回复邮件
- 银行 API 可以查余额，也可以发起转账

写入动作使得**端到端自动化**成为可能：调研潜在客户 → 找到联系方式 → 起草邮件 → 发送初始邮件 → 阅读回复 → 跟进 → 提取订单 → 更新数据库。

**安全隐患**：赋予 AI 自动改变现实世界的能力令人不安。正如你不会让实习生拥有删除生产数据库的权限，你也不应让不可靠的 AI 发起银行转账。对系统能力和安全措施的**信任**至关重要。AI 系统即使不进入物理世界也能造成危害——操纵股市、窃取版权、侵犯隐私、传播错误信息等。

> 大多数模型提供商已支持**函数调用（Function Calling）**功能。配合丰富工具集的函数调用已成为大多数模型的标准能力。

### 2.4 Computer Use 与 GUI Agent（2024-2026 新兴领域）

传统工具通过 API 与 Agent 交互——Agent 调用函数、传递参数、接收结果。但现实世界中大量软件**没有 API**，只有图形用户界面（GUI）。**Computer Use** 是一种全新的工具类型，让 AI Agent 能像人类一样通过**视觉识别和鼠标/键盘操作**来控制计算机。

| 产品 | 提供商 | 工作方式 |
|---|---|---|
| **Claude Computer Use** | Anthropic（2024.10 公测） | 通过截图"看到"屏幕内容 → 识别 UI 元素 → 执行精确的鼠标点击和键盘输入。可控制用户自己的桌面、原生应用和网站 |
| **Computer Using Agent (CUA)** | OpenAI（2025） | 基于 GPT-4o，在安全的**虚拟环境**中进行浏览器自动化和 UI 操作 |

**技术原理**：Claude Computer Use 结合了高级视觉能力和坐标精确交互——模型捕获屏幕截图、分析视觉信息识别界面元素、以像素级精度执行操作。值得注意的是，Claude 仅在计算器和文本编辑器等简单应用上训练，却能泛化到从未见过的复杂软件上。

**安全考量**：Computer Use 有独特的安全风险，尤其是与互联网交互时。Anthropic 建议：对重要决策要求人类确认、限制互联网访问到白名单域名、避免访问敏感登录信息、使用最小权限的专用虚拟机。系统内置自动分类器来检测和防御 prompt 注入攻击。

**意义**：Computer Use 极大扩展了 Agent 的工具边界——理论上任何人类能通过电脑完成的任务，Agent 都可以尝试。这将 Agent 的环境从"API 可达的系统"扩展到了"所有有 GUI 的软件"。

### 2.5 工具互操作协议（2024-2026 新兴标准）

随着 Agent 工具生态的爆发，一个关键问题浮现：**如何标准化 Agent 与工具、Agent 与 Agent 之间的通信？** 两个互补协议正在成为行业标准：

#### MCP（Model Context Protocol）——Anthropic，2024.11

MCP 标准化了 LLM 与**工具和数据源**的连接方式。它定义了一套基于 JSON 的通用接口，让任何工具只需实现一次 MCP 适配器就能被所有支持 MCP 的 Agent 使用。

核心能力：prompt 编排、函数路由与调用、从外部资源注入上下文数据。支持企业认证标准（OAuth2、mTLS）。

**解决的问题**：在 MCP 之前，每个 Agent 框架都有自己的工具定义格式，同一个工具要为不同框架写不同的适配器。MCP 让工具开发者只需"写一次，到处用"。

#### A2A（Agent-to-Agent Protocol）——Google，2025.4

A2A 解决的是**Agent 之间**的通信问题。它使用 JSON-RPC 2.0 over HTTPS，让不同平台、不同框架的 Agent 能互相发现、协商和协作。

**MCP 与 A2A 的关系**：Google 和 Anthropic 共同推荐的方案是——**"MCP 连接工具，A2A 连接 Agent"**。MCP 将 Agent 连接到结构化工具和资源，A2A 实现 Agent 之间的对话和协作。理想的架构是将 A2A Agent 建模为 MCP 资源。

| 协议 | 负责连接 | 类比 |
|---|---|---|
| **MCP** | Agent ↔ 工具/数据源 | USB 接口——让各种外设（工具）用统一接口连接电脑（Agent） |
| **A2A** | Agent ↔ Agent | HTTP 协议——让不同电脑（Agent）之间互相通信 |

**当前挑战**（2025-2026）：两个协议的组合集成仍面临语义互操作性、组合安全风险、跨协议调试困难等问题，但它们共同为可扩展的多 Agent 系统提供了必要的架构基础。

> Gartner 预测：到 2026 年，40% 的商业应用将集成特定任务的 Agent，远高于 2025 年的不到 5%。

## 三、Planning（规划）

### 3.1 为什么规划很重要？

Agent 的核心是负责解决任务的模型。一个任务由**目标（goal）**和**约束（constraints）**定义。例如："规划一次从旧金山到印度的两周旅行，预算 5,000 美元"——两周旅行是目标，5,000 美元是约束。

复杂任务需要规划。**规划的输出是一个计划（plan）**——一份概述完成任务所需步骤的路线图。有效的规划需要模型理解任务、考虑不同方案、选择最优方案。

**效率差异**：对于查询"没有收入的公司中，有多少融资超过 10 亿美元？"，有两种解法：

1. 先找出所有没有收入的公司 → 再按融资额过滤
2. 先找出融资超过 10 亿的公司 → 再按收入过滤

方案 2 明显更高效——没有收入的公司远多于融资超过 10 亿的公司。一个智能 Agent 应该选择方案 2。

### 3.2 规划与执行解耦

如果把规划和执行耦合在一个 prompt 中（如让模型 "think step by step" 然后直接执行），模型可能生成一个 1,000 步的计划却根本无法达成目标——在你发现之前已经浪费了大量时间和 API 费用。

**正确做法**：将规划与执行**解耦**。流程如下：

```text
任务输入
    ↓
┌─────────────────────────┐
│  Plan Generation（计划生成）│
└────────────┬────────────┘
             ↓
┌─────────────────────────┐
│  Plan Validation（计划验证）│ ← 启发式规则 / AI 评判
│  好计划 → 执行             │
│  坏计划 → 重新生成          │
└────────────┬────────────┘
             ↓
┌─────────────────────────┐
│  Execution（执行）         │ → 函数调用
└────────────┬────────────┘
             ↓
┌─────────────────────────┐
│  Output Evaluation（输出评估）│
└─────────────────────────┘
```

**计划验证的方法**：

- **启发式规则**：排除包含无效动作的计划（如计划需要 Google 搜索但 Agent 没有该工具）；排除步骤数超过阈值的计划
- **AI 评判**：让模型评估计划是否合理，或如何改进

系统此时有三个组件：生成计划、验证计划、执行计划。如果把每个组件看作一个 Agent，这就是一个**多 Agent 系统（multi-agent system）**。实际上大多数 Agent 工作流都涉及多个组件，因此大多数 Agent 本质上都是多 Agent 的。

为了加速，可以**并行生成多个计划**，让评估器选出最优方案。代价是额外的 API 调用成本。

#### 多 Agent 编排框架（2024-2026 更新）

多 Agent 系统已从学术概念发展为**企业级基础设施**——2025 年市场规模达 75.5 亿美元，预计 2026 年增长至 108.6 亿美元，超过 70% 的新 AI 项目使用编排框架。主流框架各有侧重：

| 框架 | 设计理念 | 适用场景 |
|---|---|---|
| **LangGraph** | 基于图的状态机——将工作流建模为节点、边和条件路由构成的有状态图 | 复杂的有状态工作流、需要精确控制流的场景。支持检查点、重试策略和分布式追踪 |
| **CrewAI** | 基于角色的团队协作——独特的"Crews and Flows"设计模式，为每个 Agent 分配角色 | 分工明确的流程（如研究团队中有调研员、分析师、作者）。提供结构化任务交接 |
| **AutoGen**（Microsoft） | 事件驱动的多 Agent 对话——将一切建模为 Agent 之间的对话 | 需要灵活 Agent 间协商的场景。异步事件驱动架构 |

**生产环境的现实**：多 Agent 系统解决了单 Agent 的关键缺陷（错误放大、非确定性路由、上下文窗口崩溃），但 95% 的 AI 试点项目未能突破 PoC（Proof of Concept，概念验证——即小规模可行性测试）阶段。企业级部署要求状态管理、容错、人类监督、可观测性和合规性。2025 年 EU AI Act（欧盟《人工智能法案》，全球首部 AI 综合监管法律）正式生效执行，使得人类监督机制成为架构级的硬性要求。

### 3.3 意图分类与人类参与

**意图分类器（Intent Classifier）**通常用来帮助 Agent 规划。它可以是一个额外的 prompt 或专门训练的分类模型。例如客服场景中：

- 查询关于**账单** → Agent 需要访问支付记录的工具
- 查询关于**密码重置** → Agent 需要访问文档检索的工具
- 查询**不在范围内** → 分类为 IRRELEVANT，Agent 礼貌拒绝，避免浪费计算资源

**人类参与**：规划的三个阶段（生成、验证、执行）都可以引入人类：

- **人类提供计划**：对于复杂任务，人类专家提供高层次计划，Agent 在此基础上细化
- **人类验证计划**：涉及高风险操作（如更新数据库、合并代码）时要求人类明确批准
- **人类执行操作**：让人类执行某些敏感操作

需要为每个动作**明确定义 Agent 的自动化级别**。

### 3.4 Agent 解决任务的完整流程

1. **计划生成（Plan Generation / Task Decomposition）**：将任务分解为一系列可执行的动作
2. **反思与纠错（Reflection & Error Correction）**：评估计划，如果是坏计划则重新生成
3. **执行（Execution）**：执行计划中的动作，通常涉及函数调用
4. **反思与纠错**：收到动作结果后，评估是否达成目标、识别并纠正错误。未完成则生成新计划

> 要求模型"think step by step"本质上就是要求它做任务分解。要求模型"verify if your answer is correct"本质上就是要求它反思。

### 3.5 基础模型能否规划？

#### 早期争论（2023）

这曾是一个激烈的开放问题。多位研究者认为基础模型（至少是自回归语言模型）**无法真正规划**：

- Meta 首席 AI 科学家 Yann LeCun（2023）明确表示自回归 LLM 无法规划
- Kambhampati（2023）在"Can LLMs Really Reason and Plan?"中认为：LLM 擅长提取知识但不擅长规划。声称 LLM 能规划的论文混淆了"从 LLM 提取的通用规划知识"和"可执行的计划"

**书中的分析**：

规划本质上是一个**搜索问题**——在不同路径中搜索、预测每条路径的结果、选择最有前景的路径。搜索常常需要**回溯（backtracking）**：在一个节点选择了动作 A，发现前景不好，需要回退到之前的状态选择动作 B。

有人认为自回归模型只能"向前"生成，无法回溯，因此无法规划。但这不一定成立——模型执行完路径 A 后，如果发现不合理，可以修改为路径 B，实际效果等同于回溯。模型也可以从头开始选择另一条路径。

Hao et al.（2023）在论文"Reasoning with Language Model is Planning with World Model"中提出：LLM 包含大量世界知识，有能力预测每个动作的结果，将这种结果预测纳入生成过程可以产生连贯的计划。

> 即使 AI 不能独立规划，它仍然可以**作为规划器的一部分**。可以给 LLM 增加搜索工具和状态追踪系统来辅助其规划。

#### 推理模型的突破（2024-2026 更新）

2024-2025 年，**推理模型（Reasoning Models / Large Reasoning Models, LRMs）**的出现显著改变了这一争论的走向。推理模型通过在推理阶段分配**额外的计算资源（test-time compute）**来进行深度思考，大幅提升了规划和推理能力：

| 模型 | 发布时间 | 关键特点 |
|---|---|---|
| **OpenAI o1** | 2024.9 | 首个商用推理模型，通过隐式 Chain-of-Thought 在推理时"思考更久"来解决复杂问题 |
| **DeepSeek-R1** | 2025.1 | 开源推理模型，通过 RL 训练获得长链推理能力，性能媲美 o1 |
| **OpenAI o3 / o4-mini** | 2025 | 进一步提升推理能力，o3 在 SWE-bench Verified 达到 69.1% |
| **Gemini 3.1 Pro** | 2026.2 | 100 万 token 上下文 + ARC-AGI-2 推理 77.1%，面向自主 Agent 优化 |

**对规划能力的影响**：推理模型本质上将规划中所需的"搜索-评估-回溯"过程**内化到了推理链中**——模型在生成最终答案之前，会在隐式思维链中探索多条路径、评估可行性、自我纠错。这部分回应了早期"自回归模型无法回溯"的批评。

**最新研究进展**：

- **SPIRAL**（2025.12）：将三个专业化 LLM Agent（Planner、Simulator、Critic）嵌入蒙特卡洛树搜索（MCTS），将搜索过程转化为引导式自纠正推理，在 DailyLifeAPIs 上达到 83.6% 准确率，比竞争框架高出 16 个百分点
- **PlanGEN**（2025.2）：通过约束引导的迭代验证和自适应算法选择来解决复杂规划中的验证缺口，在 NATURAL PLAN 上提升约 8%，OlympiadBench 上提升约 4%
- **LogiPlan**（2025.6）：在测试中发现推理增强模型在简单任务上表现良好，但在需要深层逻辑规划和关系推理的复杂配置上仍然**挣扎**

**当前共识**：LLM 的规划能力已经有了**质的提升**，但并未完全解决问题。推理模型的一个已知缺陷是**效率问题**——它们经常生成过长的推理链，包含大量冗余内容，导致成本和延迟显著增加。当前研究的重点已经从"LLM 能否规划"转向"如何高效规划"——即在保持规划质量的同时压缩推理链的长度和成本。

### FM Agent vs RL Agent


| 维度       | FM Agent（基础模型 Agent）                     | RL Agent（强化学习 Agent）       |
| -------- | ---------------------------------------- | -------------------------- |
| **相似点**  | 都由环境和可能的动作刻画                             | 都由环境和可能的动作刻画               |
| **规划器**  | 基础模型本身就是规划器，通过 prompt 或微调来提升规划能力         | 规划器由 RL 算法训练，训练通常需要大量时间和资源 |
| **资源需求** | 通常更少                                     | 通常更多                       |
| **未来趋势** | FM Agent 可以融合 RL 算法来提升性能，长远来看两者可能会**合并** | —                          |


### 3.6 计划生成（Plan Generation）

#### 通过 Prompt Engineering 生成计划

最简单的方法是通过 prompt 将模型变成计划生成器。示例——为 Kitty Vogue 创建一个有三个外部工具的 Agent：

```text
SYSTEM PROMPT

Propose a plan to solve the task. You have access to 5 actions:

get_today_date()
fetch_top_products(start_date, end_date, num_products)
fetch_product_info(product_name)
generate_query(task_history, tool_output)
generate_response(query)

The plan must be a sequence of valid actions.

Examples
Task: "Tell me about Fruity Fedora"
Plan: [fetch_product_info, generate_query, generate_response]

Task: "What was the best selling product last week?"
Plan: [fetch_top_products, generate_query, generate_response]

Task: {USER INPUT}
Plan:
```

**注意事项**：

- 计划中只列出函数名，参数由 Agent 在运行时根据前一步的输出推断
- `generate_query` 接收任务历史和最近的工具输出，生成送入回答生成器的查询

**参数的不确定性**：模型经常需要**猜测**参数值。例如用户问"最畅销产品的平均价格"——"最畅销"是指上周、上月还是历史上的？要看多少个产品？这些都不明确。因为动作序列和参数都由 AI 生成，它们都可能产生**幻觉**——调用无效函数或用错误参数调用有效函数。

#### 提升规划能力的方法


| 方法                   | 说明               |
| -------------------- | ---------------- |
| **改进 System Prompt** | 添加更多示例，写更清晰的指令   |
| **改进工具描述**           | 让模型更好地理解每个工具及其参数 |
| **重构工具**             | 将复杂函数拆分为更简单的函数   |
| **使用更强的模型**          | 更强的模型通常规划能力更好    |
| **微调模型**             | 针对计划生成任务微调       |


#### 函数调用（Function Calling）

大多数模型提供商已支持**工具使用（Tool Use）**，即函数调用。基本流程：

**Step 1：创建工具库** —— 声明所有可能使用的工具，每个工具包含：执行入口（函数名）、参数、文档（功能说明和参数说明）

**Step 2：指定可用工具** —— 不同查询可能需要不同工具，API 通常允许按查询指定工具列表，并支持以下控制模式：


| 模式         | 说明           |
| ---------- | ------------ |
| `required` | 模型必须使用至少一个工具 |
| `none`     | 模型不应使用任何工具   |
| `auto`     | 模型自行决定使用哪些工具 |


**Step 3：模型自动生成工具调用** —— 给定查询，Agent 自动决定调用哪个工具及其参数。某些函数调用 API 能确保只生成**有效的函数名**，但无法保证参数值正确。

示例：用户问"40 磅是多少千克？"，Agent 生成：

```python
response = ModelResponse(
   finish_reason='tool_calls',
   message=chat.Message(
       content=None,
       role='assistant',
       tool_calls=[
           ToolCall(
               function=Function(
                   arguments='{"lbs":40}',
                   name='lbs_to_kg'),
               type='function')
       ])
)
```

> **实践建议**：使用 Agent 时，始终要求系统**报告每次函数调用使用的参数值**，以便检查正确性。

#### 规划粒度（Planning Granularity）

计划可以有不同的**粒度**——就像年度规划可以按季度、按月或按周。存在**规划/执行的权衡**：


|          | 细粒度计划 | 粗粒度计划      |
| -------- | ----- | ---------- |
| **生成难度** | 更难    | 更容易        |
| **执行难度** | 更容易   | 更难（需要更多推断） |


**解决方案——层次化规划（Hierarchical Planning）**：先生成高层次计划（如按季度），再为每个季度生成细化计划（如按月）。

**函数名 vs 自然语言**：

前面的示例用的是精确函数名（如 `get_time()`、`fetch_top_products()`），但这有两个问题：

1. 工具 API 变化时（如 `get_time()` 改名为 `get_current_time()`），需要更新所有 prompt 和示例
2. 如果之前已微调模型来生成基于旧工具库的计划，工具变化后需要重新微调
3. 精确函数名使得规划器难以跨不同用例复用

**替代方案**：用**自然语言**生成计划（如 "get current date" 而非 `get_time()`），然后用一个**翻译模块**将自然语言动作转译为可执行命令。翻译比规划简单得多，可以用较弱的模型完成，幻觉风险也更低。

#### 复杂控制流（Complex Control Flows）

前面的计划示例都是**顺序执行**的，但实际的计划可能需要更复杂的控制流：


| 控制流                    | 说明                       | 示例                                         |
| ---------------------- | ------------------------ | ------------------------------------------ |
| **Sequential（顺序）**     | 任务 B 在任务 A 完成后执行（B 依赖 A） | SQL 查询必须在从自然语言翻译完成后才能执行                    |
| **Parallel（并行）**       | 任务 A 和 B 同时执行            | 查询"售价低于 100 的畅销品"→ 先检索 100 个畅销品，再并行查询每个的价格 |
| **If Statement（条件分支）** | 根据上一步的输出决定执行 B 还是 C      | 先查看 NVIDIA 财报 → 根据结果决定买入还是卖出               |
| **For Loop（循环）**       | 重复执行任务 A 直到满足条件          | 持续生成随机数直到得到一个质数                            |


在传统软件工程中，控制流的条件是**精确的**。在 AI Agent 中，**AI 模型决定控制流**。非顺序控制流的计划更难生成，也更难翻译为可执行命令。

> **评估 Agent 框架时**，需要检查它支持哪些控制流。例如，如果需要浏览十个网站，能否并行执行？并行执行可以**显著降低用户感知的延迟**。

### 3.7 反思与纠错（Reflection & Error Correction）

反思不是 Agent 的必要组件，但对成功**至关重要**。反思在任务的多个节点都有用：

- 收到用户查询后 → 评估请求是否可行
- 初始计划生成后 → 评估计划是否合理
- 每个执行步骤后 → 评估是否在正确轨道上
- 整个计划执行完成后 → 判断任务是否已完成

反思和纠错是**相辅相成**的两个机制：反思产生洞察，洞察帮助发现需要纠正的错误。

#### ReAct 框架

ReAct（Yao et al., 2022）首次提出了**交替进行推理和行动（interleaving reasoning and action）**的模式，这已成为 Agent 的通用范式。在每一步，Agent 被要求：解释思考（规划）→ 执行动作 → 分析观察（反思），循环直到任务完成：

```text
Thought 1: ...
Act 1: ...
Observation 1: ...

... [continue until task is finished] ...

Thought N: ...
Act N: Finish [Response to query]
```

#### Reflexion 框架

Reflexion（Shinn et al., 2023）将反思拆分为两个独立模块：


| 模块                        | 职责               |
| ------------------------- | ---------------- |
| **Evaluator（评估器）**        | 评估执行结果           |
| **Self-Reflection（自我反思）** | 分析**为什么**失败、如何改进 |


工作流程：每一步评估和反思后，Agent 提出新的计划（论文中称为 trajectory）。

**示例**：代码生成任务 → 评估器发现生成的代码在 ⅓ 的测试用例上失败 → Agent 反思失败原因是没有考虑全负数数组 → 重新生成代码，处理全负数情况。

**代价**：Thought、Observation 和 Action 都需要大量 token 来生成，这增加了**成本和延迟**，尤其是中间步骤多的任务。ReAct 和 Reflexion 都在 prompt 中使用大量示例来引导格式，这进一步增加了输入 token 成本并减少了可用的上下文空间。

## 四、Tool Selection（工具选择）

工具选择需要仔细考虑。给 Agent 的工具取决于环境、任务和底层 AI 模型。

**核心矛盾**：更多工具 = 更强能力，但也意味着更难高效使用。就像人类也很难精通大量工具。工具描述也需要占用上下文空间。

### 选择工具的方法


| 方法                       | 说明                                                    |
| ------------------------ | ----------------------------------------------------- |
| **对比实验**                 | 用不同工具集测试 Agent 的表现                                    |
| **消融实验（Ablation Study）** | 逐个移除工具，看性能下降多少。如果移除某工具后性能不降，就去掉它                      |
| **分析错误模式**               | 找出 Agent 经常出错的工具。如果经过 prompt 优化甚至微调后仍然难以使用，就换一个更简单的工具 |
| **绘制工具调用分布**             | 看哪些工具最常用、哪些最少用                                        |


### 不同模型的工具使用差异

Chameleon（Lu et al., 2023）的实验揭示了两个重要发现：

1. **不同任务需要不同工具**：ScienceQA（科学问答）更依赖知识检索工具，TabMWP（表格数学问题）则不同
2. **不同模型有不同的工具偏好**：GPT-4 倾向选择更广泛的工具集，ChatGPT 偏好图像标注；GPT-4 偏好知识检索

### 工具组合与进化

Chameleon 还研究了**工具转换（Tool Transition）**：使用工具 X 之后，Agent 接下来调用工具 Y 的概率。如果两个工具经常一起使用，可以将它们**组合成一个更强大的工具**。

Voyager（Wang et al., 2023）提出了**技能管理器（Skill Manager）**：Agent 在完成任务过程中创建新技能（本质是代码程序），当技能管理器判断新技能有用时（如成功帮助完成任务），将其加入**技能库**供后续复用。这让 Agent 能像人类一样——不仅使用现有工具，还能**创造新工具**。

> **评估 Agent 框架时**：评估它支持哪些规划器和工具。不同框架可能聚焦不同工具类别——AutoGPT 侧重社交媒体 API（Reddit、X、Wikipedia），Composio 侧重企业 API（Google Apps、GitHub、Slack）。同时评估扩展新工具的难度，因为需求会随时间变化。

## 五、Agent 失败模式与评估

评估的核心是**检测失败**。Agent 执行的任务越复杂，潜在的失败点越多。除了所有 AI 应用共有的失败模式（如幻觉、偏见等），Agent 还有**规划、工具执行和效率**方面的独特失败。

> 评估方法：识别 Agent 的所有失败模式，度量每种失败模式的发生频率。

### 5.1 规划失败（Planning Failures）

#### 工具使用失败


| 失败类型                                                   | 说明                     | 示例                                |
| ------------------------------------------------------ | ---------------------- | --------------------------------- |
| **无效工具（Invalid Tool）**                                 | 计划中使用了工具库中不存在的工具       | 生成的计划包含 `bing_search`，但工具库中没有     |
| **有效工具，无效参数（Valid Tool, Invalid Parameters）**          | 调用了正确的工具但传入了错误数量或类型的参数 | `lbs_to_kg` 只需要一个参数 `lbs`，但传了两个参数 |
| **有效工具，错误参数值（Valid Tool, Incorrect Parameter Values）** | 调用了正确的工具和正确的参数，但值是错误的  | `lbs_to_kg(lbs=100)` 但正确值应该是 120  |


#### 目标失败（Goal Failure）

Agent 未能实现目标。可能原因：

- 计划**没有解决任务**（如要求规划从旧金山到河内的行程，但 Agent 规划了到胡志明市的行程）
- 计划**没有满足约束**（如预算 5,000 美元的旅行计划实际花费远超预算）

#### 时间约束失败

许多情况下 Agent 的耗时很关键。如果你让 Agent 准备一份科研基金申请，它在截止日期之后才完成，那就毫无意义。

#### 反思错误

Agent 确信自己已经完成任务，但实际上没有。例如：要求将 50 人分配到 30 个酒店房间，Agent 只分配了 40 人却坚称任务已完成。

#### 评估规划失败的方法

创建一个规划数据集，每个样本是一个 `(task, tool_inventory)` 元组。对每个任务生成 K 个计划，计算以下指标：

1. 所有生成的计划中，有多少是**有效的**？
2. 对于给定任务，平均需要生成多少个计划才能得到一个**有效计划**？
3. 所有工具调用中，有多少是**有效的**？
4. **无效工具**被调用的频率？
5. **有效工具**被以**无效参数**调用的频率？
6. **有效工具**被以**错误参数值**调用的频率？

分析输出中的**模式**：Agent 在哪类任务上失败最多？哪些工具出错最频繁？某些工具可能对 Agent 来说"太难用"——可以通过更好的 prompt、更多示例或微调来改进。如果都不行，就换一个更容易使用的工具。

#### 现代 Agent Benchmark（2024-2026 更新）

Agent 评估已从简单的问答准确率发展为**端到端任务完成率**评估。几个重要的 benchmark：

| Benchmark | 评估内容 | 说明 |
|---|---|---|
| **SWE-bench Verified** | 编程 Agent 的代码修复能力 | 金标准。包含 500 个来自真实 GitHub 项目（Django、Flask、scikit-learn 等）的 issue，要求 Agent 在庞大代码库中导航、理解复杂依赖、修复 bug 且不破坏现有测试。2026 年领先成绩：Claude 4 Sonnet 77.2%、Claude 4 Opus 72.5%、OpenAI o3 69.1% |
| **Berkeley Function Calling Leaderboard** | 函数调用准确性 | 评估模型选择正确工具和生成正确参数的能力 |
| **TravelPlanner** | 复杂规划能力 | 评估 Agent 在约束条件下的多步规划 |
| **ARC-AGI-2** | 通用推理能力 | 评估 Agent 解决新颖抽象推理问题的能力。Gemini 3.1 Pro 达到 77.1% |
| **Terminal-Bench** | 编码 Agent 终端操作 | Letta Code 在此 benchmark 排名第一 |

### 5.2 工具失败（Tool Failures）

工具失败发生在**正确的工具被调用但输出错误**的情况：


| 失败类型       | 说明                                           |
| ---------- | -------------------------------------------- |
| **工具输出错误** | 图像标注器返回错误描述，SQL 生成器生成错误的 SQL                 |
| **翻译错误**   | 如果 Agent 生成的是高层次计划，翻译模块在将自然语言动作转译为可执行命令时出错   |
| **缺少工具**   | Agent 没有完成任务所需的工具。如任务需要查询实时股价但 Agent 无法访问互联网 |


**评估方法**：

- 工具失败是**工具相关**的，每个工具需要**独立测试**
- 始终**打印每次工具调用及其输出**以便检查
- 如果有翻译模块，为其创建专门的 benchmark
- 如果 Agent 在某个领域频繁失败，可能是缺少该领域的工具——与领域专家合作，观察人类会使用什么工具

### 5.3 效率（Efficiency）

Agent 可能生成了有效计划、使用了正确工具、完成了任务，但可能**效率低下**。应跟踪的指标：


| 指标       | 说明                          |
| -------- | --------------------------- |
| **平均步数** | Agent 完成任务平均需要多少步？          |
| **平均成本** | 完成任务平均花费多少？                 |
| **每步耗时** | 每个动作通常需要多长时间？有没有特别耗时或昂贵的动作？ |


> **注意**：比较 AI Agent 和人类时，两者有非常不同的操作模式。对人类来说低效的事（如浏览 100 个网页）对 AI Agent 可能轻而易举（可以并行访问）。反之亦然。

## 六、Agent 产品化落地（2024-2026 新增）

Agent 已从研究演示走向了规模化产品。以下是几个标志性的落地方向：

### 编程 Agent

编程是 Agent 最成功的落地领域之一，因为编程环境**高度结构化**（代码文件、终端、测试用例），且可以通过测试自动验证结果。

| 产品 | 说明 |
|---|---|
| **Cursor / Windsurf** | IDE 内嵌的编程 Agent，能感知整个代码库、执行终端命令、编辑多个文件来完成复杂编码任务 |
| **Claude Code**（Anthropic） | 终端中运行的编程 Agent，可导航代码库、编辑文件、执行命令、使用 Git |
| **Devin**（Cognition） | 号称"第一个 AI 软件工程师"，能端到端完成软件开发任务 |
| **OpenHands** | 开源编程 Agent 平台 |
| **SWE-agent** | 开源 SOTA 编程 Agent，mini-SWE-agent 仅用 100 行 Python 即在 SWE-bench Verified 上达到 65% |

### Deep Research Agent

**Deep Research** 是 Agent 在信息研究领域的落地——能自主浏览网页、评估来源、综合信息、生成结构化研究报告。

| 产品 | 特点 |
|---|---|
| **OpenAI Deep Research** | 基于 o3/o4-mini，侧重分析深度和学术严谨性。能生成 40+ 页报告，引用 50-75 个来源 |
| **Gemini Deep Research** | 基于 Gemini 2.5 Pro，侧重广度、交互性和速度。集成 Google Workspace，支持多模态报告 |

### Anthropic 高级工具使用（2025.11）

Anthropic 发布了三个重要的工具使用增强功能，解决了 Agent 在实际使用中的痛点：

| 功能 | 解决的问题 |
|---|---|
| **Programmatic Tool Calling** | 让 Claude 通过**代码执行**而非自然语言来调用工具，减少上下文消耗并支持并行工具执行。例如处理数千行电子表格时不再使上下文窗口过载 |
| **Tool Search Tool** | 按需发现工具，而非预先加载所有工具定义。一个有 58 个工具的五服务器配置原本在开始工作前就消耗约 55K token，使用 Tool Search 后节省了约 19 万 token 的上下文 |
| **Tool Use Examples** | 提供工具使用示例的通用标准，帮助模型学习工具的使用模式和惯例 |

这些功能直接回应了书中提到的核心矛盾——"更多工具 = 更强能力 = 更难管理"：Programmatic Tool Calling 提升效率，Tool Search 解决工具数量爆炸的上下文压力。

## 小结


| 主题 | 核心要点 |
|---|---|
| **Agent 定义** | Agent = 感知环境 + 作用于环境。由环境和动作集合刻画。AI 是"大脑"，负责处理信息、规划和判断 |
| **RAG 与 Agent 的关系** | RAG 是 Agent 的特例——检索器是 Agent 的一个工具。Agent 的能力远超 RAG，能直接与世界交互 |
| **工具** | 知识增强（检索器、Web Browsing）、能力扩展（计算器、代码解释器）、写入动作（修改数据库、发送邮件）、**Computer Use（GUI 操控）** |
| **工具生态标准化** | **MCP**（Agent ↔ 工具）+ **A2A**（Agent ↔ Agent）两个互补协议正在成为行业标准 |
| **规划** | 复杂任务的核心。规划与执行应解耦。支持层次化规划、自然语言计划、多种控制流（顺序/并行/条件/循环） |
| **推理模型对规划的影响** | o1/o3/DeepSeek-R1 等推理模型通过 test-time compute 大幅提升规划能力，但效率（推理链过长）仍是待解决的问题 |
| **ReAct & Reflexion** | 交替推理-行动-观察的范式。Reflexion 将评估和反思拆分为独立模块，让 Agent 能从错误中学习 |
| **多 Agent 编排** | LangGraph（图状态机）、CrewAI（角色团队）、AutoGen（事件对话）。2026 年市场规模超 100 亿美元 |
| **工具选择** | 更多工具 = 更强能力 = 更难掌握。Anthropic Tool Search 等方案解决工具数量爆炸问题 |
| **评估** | SWE-bench Verified（编程 Agent 金标准，SOTA 77.2%）、ARC-AGI-2（通用推理）、Berkeley FCL（函数调用） |
| **失败模式** | 规划失败（无效工具/参数/目标）、工具失败（输出错误/翻译错误/缺少工具）、效率问题 |
| **产品化落地** | 编程 Agent（Cursor、Claude Code、Devin）、Deep Research Agent（OpenAI、Gemini） |
| **安全与信任** | 写入动作和 Computer Use 使能力与风险并增。EU AI Act 2025 使人类监督成为架构级硬性要求 |
