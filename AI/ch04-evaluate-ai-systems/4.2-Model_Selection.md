# Model Selection（模型选择）

> **核心定位**：有了评估标准后，下一步是用这些标准从众多模型中选出最适合你应用的那一个。模型选择不是一次性决策，而是贯穿应用开发全周期的迭代过程。本节涵盖**选择工作流、Build vs Buy 决策、公开 Benchmark 导航、数据污染问题**四大主题。

---

## 模型选择概述

### 为什么模型选择是迭代的

你真正关心的不是"哪个模型最好"，而是"哪个模型**对你的应用**最好"。在应用开发过程中，随着不同适配技术（prompt engineering → finetuning 等）的推进，你需要**反复进行模型选择**：

- **Prompt engineering 阶段**：可能先从最强模型开始验证可行性，再往下试小模型
- **Finetuning 阶段**：可能先从小模型开始验证代码和流程，再往上试最大的能装进你 GPU 的模型

### 选择过程的两个核心步骤

1. **找到最佳性能上限**：确定在某技术路线下能达到的最佳表现
2. **在成本-性能轴上映射模型**：选择性价比最高的模型

实际选择过程远比这两步复杂，需要区分**硬属性**和**软属性**。

### 硬属性 vs 软属性

| 类型 | 定义 | 来源 | 示例 |
|---|---|---|---|
| **硬属性（Hard Attributes）** | 你无法或不愿改变的约束 | 模型提供商的决策（许可证、模型大小、训练数据）或你自身的政策（隐私、控制） | 许可证不允许商用、必须自托管、模型太大装不进你的 GPU |
| **软属性（Soft Attributes）** | 可以通过努力改善的指标 | 模型当前表现 | 准确率、毒性、事实一致性 |

**关键洞察**：同一属性在不同场景下可以是硬的也可以是软的。例如，延迟——如果你能控制模型（自托管），延迟是软属性（可优化）；如果你用第三方 API，延迟就是硬属性（你改不了）。

对软属性的改进幅度也需要**保持务实**：有时初始准确率只有 20%，但分解任务后能跳到 70%；有时花了数周调优也无法使模型可用。

## 四步评估工作流

模型评估流程包含四个步骤，且是**迭代**的——后续步骤的发现可能让你回退修改前面的决策：

```
┌──────────────┐    ┌──────────────────┐    ┌──────────────────┐    ┌──────────────┐
│ 1. 硬属性过滤  │ →  │ 2. 公开 Benchmark │ →  │ 3. 私有评估       │ →  │ 4. 线上监控    │
│   排除不合格   │    │   缩小候选范围     │    │   找到最佳模型     │    │   持续改进     │
└──────────────┘    └──────────────────┘    └──────────────────┘    └──────────────┘
                          ↕                        ↕
                    ┌──────────────────────────────────┐
                    │  平衡三要素：模型质量 / 成本延迟 / 易用性 │
                    └──────────────────────────────────┘
```

1. **硬属性过滤**：排除硬属性不符合的模型（许可证、隐私、是否使用 API 等）——这一步取决于你的内部策略
2. **公开评估**：利用公开 benchmark 和排行榜缩小候选模型范围，平衡质量、延迟和成本
3. **私有评估**：用你自己的评估流水线找到最佳模型，再次平衡所有目标
4. **线上监控**：部署后持续监控，检测故障并收集反馈

## Model Build Versus Buy（自建 vs 购买）

### 核心问题

对大多数公司而言，不会从头训练 foundation model。真正的问题是：**用商业 API 还是自托管开源模型？** 这个决策会大幅缩小候选模型池。

### 开源模型的概念辨析

"开源模型"这个术语已变得有争议。严格来说存在不同层次的"开放"：

| 术语 | 含义 |
|---|---|
| **Open weight（开放权重）** | 模型权重公开，但训练数据不公开（绝大多数"开源模型"属于此类） |
| **Open model（开放模型）** | 模型权重 + 训练数据均公开（真正的开源） |

**为什么这很重要**：模型的性能很大程度上取决于其训练数据。开放数据允许更灵活的使用（如从头训练、修改训练流程），也便于审计模型是否使用了被盗或受版权保护的数据。但大多数模型开发者故意隐藏训练数据信息，以避免公众审查和潜在诉讼。

### 开源模型的许可证问题

开源模型使用各种不同的许可证：MIT、Apache 2.0、GPL、BSD、Creative Commons 等。许多模型还有自己的专属许可证（如 Meta 的 Llama 2 Community License Agreement、Llama 3 Community License Agreement，Hugging Face 的 BigCode Open RAIL-M v1 等）。

选择模型时必须检查的许可证问题：

1. **是否允许商业使用？** Meta 的第一代 Llama 使用非商业许可证；Llama-2/3 允许商用，但限制月活超 7 亿的应用需要 Meta 的特别许可
2. **是否允许用模型输出训练其他模型？** 合成数据（用已有模型生成数据训练新模型）是重要的数据来源。模型蒸馏（用大模型教小模型）也依赖于此。Llama 许可证在写作时仍不允许这一用途
3. **即使许可证允许，也可能间接受限**：如果模型 X 使用了 ChatGPT 的输出数据训练，而 ChatGPT 的使用条款不允许此用途，那么 X 的数据血统就存在问题

### 开源模型 vs Model API

模型要服务用户，需要一台机器来托管和运行它。托管模型、接收查询、运行推理并返回结果的服务称为**推理服务（Inference Service）**，用户与之交互的接口称为 **Model API**。

模型开发者可选择将模型开源、提供 API、或两者都做。当前格局：

- **商业模型提供商**（OpenAI、Anthropic 等）：通常最强模型只通过 API 提供
- **开源友好提供商**（Meta、Mistral、Cohere 等）：部分模型开源 + 部分模型只提供 API
- **同一模型可通过多个 API 获取**：GPT-4 可通过 OpenAI 和 Azure 获取；开源模型可通过 AWS、Azure、GCP 以及 Databricks Mosaic、Anyscale 等第三方获取

**注意**：同一模型通过不同 API 提供时，性能可能存在差异（不同 API 可能使用不同优化技术），切换 API 时务必重新测试。

对于**商业模型提供商**，模型就是其竞争优势；对于**没有自研模型的 API 提供商**，API 本身才是竞争优势——后者更有动力提供更好的 API 和更优惠的价格。

### 七维 Build vs Buy 分析

#### 1. 数据隐私

使用第三方 API 意味着**必须将数据发送到外部**，这对有严格数据隐私政策的公司可能不可接受。

典型案例：2023 年三星员工将公司机密信息输入 ChatGPT，导致信息泄露。事件严重到三星在 2023 年 5 月禁止使用 ChatGPT。

风险点：
- 部分 API 提供商的政策声称不会用你的数据训练模型，但政策可能变更（2023 年 8 月 Zoom 悄悄修改条款允许使用用户数据训练 AI）
- 研究表明模型可以记忆训练样本——例如 Hugging Face 的 StarCoder 记住了 8% 的训练集
- 某些国家法律要求数据不得出境，API 提供商需要在这些国家设立本地服务器

#### 2. 数据血统与版权

数据血统和版权问题可以将公司推向不同方向——开源、闭源或完全不用。

现状：
- 大多数模型对训练数据**缺乏透明度**（Google Gemini 技术报告只说"所有数据标注工人获得至少当地最低工资"，OpenAI CTO 被问到训练数据来源时无法给出满意答案）
- AI 相关的知识产权法仍在演变：美国专利商标局在 2024 年明确"AI 辅助发明不可获得专利"；模型是否在受版权保护的数据上训练 → 是否可用于捍卫你的产品 IP → 仍不清楚
- 部分公司（如游戏和电影工作室）在 AI 相关 IP 法律明确前**拒绝使用 AI** 创作产品

使用开源模型在版权方面的**风险更低**（侵权方更可能追责模型开发者而非用户）；使用商业模型时，合同可能提供数据血统风险保护。

#### 3. 性能

公开 benchmark 显示开源模型和闭源模型的差距在**持续缩小**。在 MMLU benchmark 上，开源模型（如 Llama 3.1 405B）已接近闭源模型（如 Claude 3.5 Sonnet、GPT-4o）的水平。

但作者认为**最强开源模型可能始终落后于最强闭源模型**，因为激励机制不对等——拥有最强模型的公司更倾向于将其闭源以变现或竞争，而非开源让他人受益。

另一个开源模型落后的原因：开源后模型开发者**无法获得用户反馈**来改进模型，而商业模型可以。

对许多不需要最强模型的用例来说，开源模型**已经足够好**。

#### 4. 功能性

围绕模型需要许多功能才能使其在实际应用中可用：

| 功能 | 说明 |
|---|---|
| **可扩展性** | 推理服务能否支撑你的应用流量并保持延迟和成本 |
| **Function calling** | 模型能否调用外部工具（RAG 和 Agent 场景的关键能力） |
| **结构化输出** | 如 JSON 格式输出 |
| **Output guardrails** | 降低输出风险（如防止种族歧视、性别歧视内容） |
| **Logprobs** | 对分类任务、评估和可解释性非常有用 |

API 方式的限制：
- 受限于 API 提供的功能集——许多 API 不暴露 logprobs
- 只能在 API 提供商允许的范围内微调模型

自托管方式的优势：
- 完全访问 logprobs 和中间输出
- 可以自由微调、量化、优化模型

#### 5. API 成本 vs 工程成本

| 方式 | 成本构成 |
|---|---|
| **Model API** | 按 token 计费，使用量大时可能非常昂贵 |
| **自托管** | 人才、时间、工程投入来优化、托管、维护模型和推理服务（可通过使用第三方模型托管服务缓解） |

通常 API 更容易上手和扩展，而开源模型的组件更容易操作和定制。

无论选择哪种方式，都建议遵循**标准 API 接口**，以便后续切换模型。当前许多 API 提供商模仿 OpenAI 的 API 格式。

#### 6. 控制、访问与透明度

a16z 2024 年调查显示，企业关心开源模型的首要原因是**控制（37%）和可定制性（37%）**，其次才是成本（26%）。

使用 API 的风险：
- 依赖第三方的 SLA（服务等级协议），如果 API 不可靠会影响你的业务
- 模型可能**不透明地发生变化**（版本更新、性能波动），且不一定预先通知——你的 prompt 可能突然失效
- 模型提供商可能停止服务、被禁、或所在国家禁止某些用途（例如意大利在 2023 年短暂禁止 OpenAI）
- 商业模型的安全护栏倾向于**过度审查**（over-censoring），可能限制某些合法用例（如 Convai 构建 3D AI 角色时，商业模型不断回复"作为 AI 模型我没有物理能力"，最终改用开源模型）

使用自托管模型的优势：
- 可以冻结模型版本，保持行为稳定
- 可以审查模型变更
- 完全掌控模型的使用方式

#### 7. 端侧部署（On-Device Deployment）

如需在设备端运行模型（无网络、隐私敏感、特定地区无网络覆盖等），第三方 API 不可行，只能自托管。

### 综合对比表

| 维度 | 使用 Model API | 自托管模型 |
|---|---|---|
| **数据** | 需将数据发送给提供商（有泄露风险） | 数据不出门；但需自行处理数据血统/版权 |
| **性能** | 最强模型通常是闭源的 | 开源最强模型可能落后于闭源，但差距在缩小 |
| **功能** | 更容易支持扩展、function calling、结构化输出；logprobs 通常受限 | 可访问 logprobs 和中间输出，但 function calling 等功能需自建 |
| **成本** | API 按使用量计费 | 需投入人才和时间优化、托管、维护（可通过第三方托管服务缓解） |
| **微调** | 只能在提供商允许范围内微调 | 可自由微调、量化、优化（如果许可证允许），但有技术门槛 |
| **控制与透明度** | 有限：受制于提供商的 SLA、模型变更不透明、有被停服的风险 | 更高：可冻结模型、审查变更，但需自行建设和维护 API |
| **边缘用例** | 无法离线运行 | 可在设备端运行（但技术难度大） |

## Navigate Public Benchmarks（导航公开 Benchmark）

### Benchmark 生态概览

Benchmark 数量极其庞大——Google 的 BIG-bench (2022) 一个项目就包含 **214 个 benchmark**。而且随着 AI 用例增长和旧 benchmark 饱和，新 benchmark 不断涌现。

**评估工具（Evaluation Harness）** 帮助你在多个 benchmark 上评估模型：
- EleutherAI 的 **lm-evaluation-harness**：支持 400+ benchmark
- OpenAI 的 **evals**：约 500 个 benchmark，覆盖数学、谜题、ASCII 艺术识别等

### Benchmark 选择与聚合

从数千个 benchmark 中选择和聚合结果来排名模型面临两个核心问题：

1. **选择哪些 benchmark？**
2. **如何聚合结果来排名模型？**

例如：模型 A 在代码 benchmark 上优于模型 B，但在毒性 benchmark 上劣于模型 B，你选谁？同类 benchmark 之间（两个不同的代码 benchmark）结果不一致时又怎么办？

### 公开排行榜（Public Leaderboards）

#### 运作方式

公开排行榜基于多个 benchmark 的聚合表现来排名模型。它们非常有用但**远非完美**：

- 受算力限制，排行榜只能包含少量 benchmark（例如 HELM Lite 因成本排除了信息检索 benchmark MS MARCO；Hugging Face 因 HumanEval 的高算力需求而排除了它）
- 不同排行榜选择不同 benchmark，导致难以跨排行榜比较

#### Hugging Face Open LLM Leaderboard 演变

**初始版本（2023 年底）**——6 个 benchmark 的简单平均：

| Benchmark | 评估能力 |
|---|---|
| **ARC-C** (Clark et al., 2018) | 复杂的小学科学问答 |
| **MMLU** (Hendrycks et al., 2020) | 57 个学科的知识和推理（数学、历史、计算机科学、法律等） |
| **HellaSwag** (Zellers et al., 2019) | 句子/场景补全（常识和日常活动理解） |
| **TruthfulQA** (Lin et al., 2021) | 生成真实且不误导的回答 |
| **WinoGrande** (Sakaguchi et al., 2019) | 挑战性代词消歧（需复杂常识推理） |
| **GSM-8K** (OpenAI, 2021) | 小学数学应用题 |

**更新版本（2024 年 6 月）**——旧 benchmark 饱和，全面更换为更具挑战性的 benchmark：

| 新 Benchmark | 替代关系 / 说明 |
|---|---|
| **MATH lvl 5** | 替代 GSM-8K，只包含竞赛数学中最难的题目 |
| **MMLU-PRO** (Wang et al., 2024) | 替代 MMLU，更高难度 |
| **GPQA** (Rein et al., 2023) | 研究生级别 Q&A |
| **MuSR** (Sprague et al., 2023) | 链式思维、多步推理 |
| **BBH (BIG-bench Hard)** (Srivastava et al., 2023) | 推理 benchmark |
| **IFEval** (Zhou et al., 2023) | 指令遵从 benchmark |

Hugging Face 表示选择这些 benchmark 是因为"它们测试了跨多个领域的各种能力和通用知识"。

#### Stanford HELM Leaderboard

与 Hugging Face 同期使用 10 个 benchmark，其中只有 MMLU 和 GSM-8K 与 Hugging Face 重叠。另外 8 个包括：

- 竞赛数学（MATH）、法律（LegalBench）、医学（MedQA）、翻译（WMT）
- 阅读理解（NarrativeQA、OpenBookQA）
- 通用问答（Natural Questions，有/无 Wikipedia 两种设置）

#### 公开排行榜的局限性

公开排行榜试图用少量 benchmark 覆盖广泛的能力（推理、事实一致性、领域特定能力等），但覆盖的定义并不清晰。例如：为什么 HELM Lite 包含医学和法律但不包含一般科学？为什么有两个数学测试但没有代码测试？为什么没有摘要、工具使用、毒性检测、图像搜索的测试？

**排行榜的价值在于了解模型的广泛表现趋势，但它不能告诉你哪个模型最适合你的应用。**

### Benchmark 相关性（Benchmark Correlation）

一个经常被忽视的重要维度。如果两个 benchmark 高度相关，同时使用两者会**放大偏见**（bias）。

Hugging Face 排行榜 6 个 benchmark 的 Pearson 相关系数（2024 年 1 月计算）：

| | ARC-C | HellaSwag | MMLU | TruthfulQA | WinoGrande | GSM-8K |
|---|---|---|---|---|---|---|
| **ARC-C** | 1.0 | 0.4812 | **0.8672** | 0.4809 | **0.8856** | 0.7438 |
| **HellaSwag** | 0.4812 | 1.0 | 0.6105 | 0.4842 | 0.4842 | 0.3547 |
| **MMLU** | **0.8672** | 0.6105 | 1.0 | 0.5507 | **0.9011** | 0.7936 |
| **TruthfulQA** | 0.4809 | 0.4228 | 0.5507 | 1.0 | 0.4550 | 0.5009 |
| **WinoGrande** | **0.8856** | 0.4842 | **0.9011** | 0.4550 | 1.0 | 0.7979 |
| **GSM-8K** | 0.7438 | 0.3547 | 0.7936 | 0.5009 | 0.7979 | 1.0 |

关键发现：
- **WinoGrande、MMLU、ARC-C 高度相关**（都测试推理能力）
- **TruthfulQA 与其他 benchmark 仅中度相关**——提升推理和数学能力并不一定提升事实一致性

### 聚合方法

| 方法 | 代表 | 特点 |
|---|---|---|
| **简单平均** | Hugging Face | 所有 benchmark 等权重——但 TruthfulQA 80 分的难度可能远高于 GSM-8K 80 分 |
| **平均胜率（Mean Win Rate）** | HELM | 计算模型在不同场景中胜出的比例 |

两种方法各有利弊，简单平均的缺点是隐含了"所有 benchmark 同等重要"的假设。

### 自定义排行榜（Custom Leaderboards with Public Benchmarks）

为你的特定应用创建私有排行榜：

1. **收集与应用相关的 benchmark**（如构建编码 Agent → 代码类 benchmark；写作助手 → 创意写作类 benchmark）
2. 随着新 benchmark 不断出现、旧 benchmark 饱和，应**持续关注最新 benchmark**
3. 验证 benchmark 的可靠性——任何人都可以创建 benchmark，不一定准确测量其声称的能力

**并非所有模型在所有 benchmark 上都有公开分数**。如果你关心的模型在某 benchmark 上没有分数，你需要自己运行评估。评估成本可能很高——Stanford 花了约 $80,000–$100,000 在 HELM 完整套件上评估 30 个模型。

### 模型是否在变差？——一个评估的启示

Stanford 和 UC Berkeley 的研究（Chen et al., 2023）发现 GPT-3.5 和 GPT-4 在 2023 年 3 月到 6 月间的多个 benchmark 上性能**显著变化**。例如：

- GPT-4 在代码生成任务上直接可执行率从 52.0% 降至 10.0%
- GPT-3.5 在同一任务上从 22.0% 降至 2.0%

这并不意味着 OpenAI 故意发布更差的模型。一个可能的原因是**评估本身就很难**——没有人（包括 OpenAI）能绝对确定模型是变好了还是变差了。这也强化了一个核心观点：**整体排名最好的模型不一定是你应用的最佳模型。**

## Data Contamination with Public Benchmarks（公开 Benchmark 的数据污染）

### 什么是数据污染

数据污染（Data Contamination）也称 **data leakage**、**training on the test set** 或 **cheating**——指模型在其评估数据上训练过。如果发生了，模型可能只是"记住"了评估答案而非真正具备能力，导致评估分数虚高。

经典案例：Stanford 博士生 Rylan Schaeffer 在 2023 年的讽刺论文 *"Pretraining on the Test Set Is All You Need"* 中，仅用 **100 万参数**的模型、只在 benchmark 数据上训练，就在多个 benchmark 上获得**接近满分**并超过了大得多的模型。

### 数据污染如何发生

| 途径 | 说明 |
|---|---|
| **无意间包含** | 大多数模型在互联网数据上训练，爬取过程会意外拉取公开 benchmark 的数据。这也是旧 benchmark 快速饱和的原因之一 |
| **间接包含** | 训练数据和评估数据来自相同来源（如同一套数学教材） |
| **故意包含** | 为提升模型表现，开发者可能先排除 benchmark 数据选模型，再在 benchmark 数据上继续训练最佳模型后发布 |

一个朋友的玩笑话："Benchmark 一旦公开就不再有用了。"

### 处理数据污染

#### 检测方法

| 方法 | 原理 | 优缺点 |
|---|---|---|
| **N-gram 重叠** | 如果评估样本中连续 13 个 token 也出现在训练数据中，该样本被视为"脏的" | 更准确，但需要与完整训练数据比对，**耗时且需访问训练数据** |
| **Perplexity** | 如果模型对评估数据的困惑度异常低，说明可能在训练时见过 | 不需要训练数据，但**准确性较低** |

#### 应对策略

传统 ML 的做法是将评估样本从训练数据中移除，以保证 benchmark 标准化、跨模型可比。但 foundation model 时代面临新挑战：

- 大多数人**无法控制训练数据**
- 即使能控制，也**未必想移除** benchmark 数据——因为高质量 benchmark 数据有助于提升模型整体性能
- 新 benchmark 总是在模型训练后才出现，所以**永远会有**受污染的评估样本

**最佳实践（对模型开发者）**：
- 训练前从训练数据中移除关心的 benchmark 数据
- 报告模型性能时披露：benchmark 数据在训练集中的占比、模型在完整 benchmark 和"干净"子集上的分数差异

**实际情况**：检测和清除污染需要大量工作，很多人选择直接跳过。

#### GPT-3 的污染案例

OpenAI 分析 GPT-3 的训练数据（Brown et al., 2020），发现其常用 benchmark 中有 **13 个 benchmark 至少 40%** 的数据存在于训练集中。在"干净"样本上评估 vs 完整 benchmark 评估的性能差异如下例：

- **Quac**：完整 44.3 → 干净 54.1（差异 20%）
- **DROP**：完整 36.5 → 干净 29.5（差异 -21%）
- **BoolQ**：完整 76.0 → 干净 76.3（差异 0%）

部分 benchmark 受污染后分数反而更高，部分反而更低（可能因为"干净"样本更简单），部分影响不大。

#### 排行榜的应对措施

为对抗数据污染，排行榜可以：
- 绘制模型在各 benchmark 上的标准差来**识别异常值**
- 保留部分数据为私有，并提供工具让模型开发者在私有 hold-out 数据上自动评估

### 核心结论

公开 benchmark 能帮你**筛掉差模型**，但**不能帮你找到最佳模型**。通过公开 benchmark 缩小候选范围后，你需要运行自己的评估流水线——使用你自己的 prompt 和 metric——来找到最适合你应用的模型。

---

## 总结：模型选择的方法论

1. **模型选择是迭代过程**，贯穿应用开发全周期——不同技术阶段需要重新选择
2. **先用硬属性过滤**，大幅缩小候选池（许可证、数据隐私、是否 API / 自托管）
3. **Build vs Buy 是核心决策**，需从数据隐私、数据血统、性能、功能、成本、控制、端侧部署七个维度权衡
4. **公开 benchmark 是有用的起点**，但有三大局限：覆盖不全、benchmark 相关性被忽视、数据污染普遍
5. **创建自定义排行榜**——选择与你应用相关的 benchmark，注意 benchmark 间的相关性
6. **公开 benchmark 之后必须做私有评估**——用你自己的数据和标准做最终决策
7. **持续监控**——模型会变化，你的评估标准也需随之演进
