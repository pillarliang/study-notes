# Chapter 7: Finetuning — 开篇与概述

> **核心定义**：微调（Finetuning）是通过进一步训练模型的全部或部分参数，将其适配到特定任务的过程。与 Prompt 方法通过指令、上下文和工具来适配模型不同，微调直接**调整模型权重**。微调最常用于改善模型的**指令遵循能力**，尤其是确保其遵守特定的输出风格和格式。

## 二、Finetuning Overview（微调概述）

### 2.1 迁移学习（Transfer Learning）

微调是**迁移学习**的一种实现方式。迁移学习由 Bozinovski 和 Fulgosi 于 1976 年首次提出，核心思想是将一个任务中获得的知识迁移到新的相关任务上，加速学习过程。

**类比**：就像会弹钢琴的人更容易学习其他乐器。

**早期大规模成功案例**：Google 多语言翻译系统（Johnson et al., 2016）——模型从"葡→英"和"英→西"翻译中迁移知识，直接实现"葡→西"翻译，尽管训练数据中**没有葡西互译样本**。

### 2.2 迁移学习的核心价值

| 价值点 | 说明 |
|--------|------|
| **解决数据稀缺** | 在数据丰富的任务上训练基座模型，再迁移到数据稀缺的目标任务 |
| **样本效率（Sample Efficiency）** | 从零训练法律问答模型可能需要数百万样本，微调好的基座模型可能只需**几百条** |
| **释放已有能力** | OpenAI InstructGPT 论文（2022）的观点：微调不是教会模型新知识，而是**解锁模型已有但难以通过 prompt 访问的能力** |

> **Feature-based Transfer**（基于特征的迁移）是另一种迁移学习方式：用一个模型提取特征（通常是 embedding 向量），再供另一个模型使用。在计算机视觉中非常常见（如 2010 年代后期用 ImageNet 预训练模型提取图像特征用于目标检测、图像分割等下游任务）。

### 2.3 微调的类型

微调是预训练之后的训练过程，可以有多种形式：

#### 1）自监督微调 / 继续预训练（Continued Pre-training）

在用昂贵的任务数据微调之前，先用**廉价的任务相关数据**进行自监督微调。

| 目标任务 | 继续预训练数据 | 后续微调数据 |
|----------|----------------|--------------|
| 法律问答 | 原始法律文档 | 标注的 (question, answer) 对 |
| 越南语书籍摘要 | 大量越南语文本 | 标注的摘要数据 |

#### 2）监督微调（Supervised Finetuning, SFT）

使用高质量 **(input, output)** 标注数据训练模型，使其行为与人类使用习惯和偏好对齐。

- **开放式输出**：如书籍摘要
- **封闭式输出**：如分类任务
- 高质量指令数据创建**成本高昂**，尤其是需要事实一致性、领域专业知识或政治正确性的任务

**自回归 vs 填空**：

| 微调方式 | 说明 | 典型用途 |
|----------|------|----------|
| **Next Token Prediction** | 根据前文预测下一个 token | 文本生成 |
| **Infilling（填空）** | 根据前后文补全中间内容 | 文本编辑、代码调试 |

> 即使模型是自回归预训练的，也可以进行 infilling 微调。

#### 3）偏好微调（Preference Finetuning / RLHF）

通过强化学习优化模型，使其生成最大化人类偏好的回复。

- 数据格式：**(instruction, winning response, losing response)**

#### 4）长上下文微调（Long-Context Finetuning）

通过修改模型架构（如调整位置编码）来扩展上下文长度。

- 更长的序列意味着更多可能的 token 位置，位置编码需要能处理它们
- 比其他微调技术**更难实施**
- 可能在短序列上出现**性能退化**

> **案例：Code Llama**（Rozière et al., 2024）从 Llama 2 基座出发，通过不同微调技术构建多个版本：
> - 长上下文微调：最大上下文从 4,096 → **16,384 tokens**（以容纳更长的代码文件）
> - 指令微调（即 SFT）：提升指令遵循能力

### 2.4 模型开发者 vs 应用开发者

| 角色 | 微调方式 |
|------|----------|
| **模型开发者** | 在发布前使用多种微调技术进行后训练；可能发布多个版本（微调程度不同）供选择 |
| **应用开发者** | 通常在已后训练的模型上微调（而非从预训练模型开始）；基座模型越精炼、与任务越相关，所需适配工作越少 |

## 三、章节全景图

```
Pre-training（自监督，海量无标注数据）
    │
    ▼
Continued Pre-training（自监督，任务相关廉价数据）   ← 可选
    │
    ▼
Supervised Finetuning（标注的 instruction-output 数据）
    │
    ▼
Preference Finetuning / RLHF（偏好比较数据）
    │
    ▼
Long-Context Finetuning（架构修改 + 长序列数据）   ← 可选
    │
    ▼
应用开发者进一步微调（特定任务数据）
```

> **核心takeaway**：微调的本质不是"教"模型新知识，而是**解锁和精炼**模型在预训练中已经获得的能力，使其更好地服务于特定任务。
