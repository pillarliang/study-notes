# The AI Engineering Stack（AI 工程技术栈）

> **核心定位**：AI 工程的快速增长伴随着大量的工具、技术和新概念涌现，令人应接不暇。与其追逐每一个新工具，不如理解 AI 工程的**基本构建块**。本节将 AI 应用栈分为三层（应用开发、模型开发、基础设施），逐层拆解每层在 AI 工程与传统 ML 工程中的差异，并阐明 AI 工程正在向全栈工程靠拢的趋势。

---

## 一、不变的原则与变化的实践

尽管基础模型带来了前所未有的兴奋和创造力，构建 AI 应用的**许多核心原则**仍然不变：

| 不变的原则 | 说明 |
|---|---|
| 解决业务问题 | 企业 AI 应用仍需解决真实的商业问题 |
| 业务指标 ↔ ML 指标映射 | 仍然需要将业务目标映射到可量化的 ML 指标 |
| 系统化实验 | 传统 ML 调超参数，AI 工程调模型选择、prompt、检索算法、采样变量等 |
| 推理优化 | 仍然追求更快更便宜的模型运行 |
| 反馈闭环 | 仍然需要用生产数据持续迭代改进应用 |

**意义**：过去十年 ML 工程师积累的经验大部分仍然适用，为所有人进入 AI 工程提供了集体知识基础。但在这些不变的原则之上，AI 工程也带来了许多**独特的创新**。

---

## 二、AI Engineering vs ML Engineering：三大核心差异

| 维度 | 传统 ML Engineering | AI Engineering |
|---|---|---|
| **模型来源** | 为每个应用从头训练自有模型 | 使用别人训练好的基础模型 → 重心从"建模和训练"转向**模型适配** |
| **计算规模** | 模型较小，资源需求较低 | 模型更大，消耗更多算力，延迟更高 → 对高效训练和推理优化的压力更大，需要更多懂 GPU 和大型集群的工程师 |
| **输出性质** | 输出通常是封闭式的（如 spam/not spam） | 输出是**开放式的** → 灵活性更高但**评估难度剧增** |

> 一句话总结：AI Engineering = 更少的模型开发 + 更多的模型适配和评估。

### 模型适配的两大类别

| 类别 | 是否更新模型权重 | 特点 |
|---|---|---|
| **Prompt-based 技术**（含 prompt engineering） | 否 | 通过指令和上下文适配模型，不改变模型本身。更容易上手、数据需求少、方便尝试更多模型。但对复杂任务或严格性能要求可能不够 |
| **Finetuning** | 是 | 通过修改模型权重来适配。更复杂、需要更多数据、成本更高，但可以改善质量、延迟和成本，且能让模型学会训练时未接触过的新任务 |

## 三、AI 技术栈的三层架构

| 层级 | 职责 | 具体内容 |
|---|---|---|
| **Application Development**（应用开发） | 在模型之上构建用户可用的应用 | AI 界面、Prompt Engineering、上下文构造、评估 |
| **Model Development**（模型开发） | 开发和调优模型本身 | 推理优化、数据集工程、建模与训练、评估 |
| **Infrastructure**（基础设施） | 支撑以上两层的底层设施 | 算力管理、数据管理、模型部署（Serving）、监控 |

### GitHub 生态数据（2024 年 3 月，500 星以上的 AI 仓库，共 920 个）

- 2023 年 Stable Diffusion 和 ChatGPT 发布后，AI 工具数量出现**爆发式增长**
- 增长最快的两个类别是：**应用（Applications）**和**应用开发工具（AI Engineering）**
- 基础设施层也有增长，但幅度远小于其他层——因为核心基础设施需求（资源管理、部署、监控）本质上没有变化

## 四、模型开发层详解

### 4.1 建模与训练（Modeling and Training）

指设计模型架构、训练模型、微调模型的完整过程。代表工具：TensorFlow、Hugging Face Transformers、PyTorch。

**传统 ML 的必备知识**：
- ML 算法（聚类、逻辑回归、决策树、协同过滤）
- 神经网络架构（前馈、循环、卷积、Transformer）
- 训练原理（梯度下降、损失函数、正则化）

**基础模型时代的变化**：ML 知识不再是构建 AI 应用的**硬性前提**——许多成功的 AI 应用开发者对梯度下降完全不感兴趣。但 ML 知识仍然极其有价值——它扩展了可用工具集，在模型表现不佳时帮助排查问题。

#### Training / Pre-training / Finetuning / Post-training 辨析

"训练"始终涉及改变模型权重，但并非所有模型权重的改变都叫训练（例如量化改变权重精度，但不算训练）。

| 术语 | 定义 | 资源需求 | 谁来做 |
|---|---|---|---|
| **Pre-training** | 从零开始训练模型（权重随机初始化），通常训练文本补全能力 | **极高**——InstructGPT 的预训练占整体计算和数据资源的 **98%**。一个小错误可能造成巨额损失和严重延期 | 极少数组织（已成为一门"手艺"） |
| **Finetuning** | 在已预训练的模型基础上继续训练。模型已有一定知识基础，因此数据和算力需求比预训练少得多 | 中等 | 模型开发者或应用开发者 |
| **Post-training** | 概念上等同于 finetuning，但通常特指**模型开发者**在预训练后、发布前对模型进行的训练（如让模型更好地遵循指令） | 中等 | 模型开发者 |

> **常见误区**：有人将 prompt engineering 称为 "training"。虽然日常语言中"训练模型做某事"说得通，但技术上 prompt engineering 是通过上下文输入教模型做事，并**不改变模型权重**，因此不是训练。反之，也有人将 finetuning 称为 prompt engineering，这同样不准确。

### 4.2 数据集工程（Dataset Engineering）

指为训练和适配 AI 模型而进行的数据筹备、生成和标注工作。

| 维度 | 传统 ML | AI Engineering |
|---|---|---|
| **数据类型** | 主要是**表格数据**（tabular） | 主要是**非结构化数据**（文本、图像、音频） |
| **核心工作** | 特征工程（feature engineering） | 去重、分词、上下文检索、质量控制（去除敏感信息和有害数据） |
| **标注难度** | 封闭式输出（spam/not spam）相对容易标注 | 开放式输出（如写一篇文章）标注难度大得多 |
| **数据的重要性** | 重要 | **更加重要**——模型已成为商品，数据成为核心差异化因素 |

数据需求与适配技术相关：从头训练 > finetuning > prompt engineering。

无论需要多少数据，理解数据的专业能力都非常有价值——训练数据揭示了模型的优势和弱点。

### 4.3 推理优化（Inference Optimization）

让模型运行得更快更便宜。这在传统 ML 中就很重要，在基础模型时代**更加重要**——因为模型规模更大，推理成本和延迟更高。

**特殊挑战**：基础模型通常是自回归的——token 是顺序生成的。如果每个 token 需要 10ms，生成 100 个 token 就需要 1 秒，更长的输出需要更多时间。用户对延迟越来越不耐烦，典型互联网应用的可接受延迟约为 **100ms**。

推理优化技术包括：量化（quantization）、蒸馏（distillation）、并行化（parallelism），详见 Chapters 7-9。

### 模型开发层变化总结

| 类别 | 传统 ML | 基础模型时代 |
|---|---|---|
| 建模与训练 | ML 知识是从零训练模型的必备技能 | ML 知识是加分项而非必需（有争议） |
| 数据集工程 | 偏重特征工程，表格数据为主 | 偏重去重/分词/上下文检索/质量控制，非结构化数据为主 |
| 推理优化 | 重要 | **更加重要** |

## 五、应用开发层详解

在传统 ML 中，团队用自有模型构建应用，模型质量本身就是差异化。在基础模型时代，许多团队使用相同的模型，**差异化必须通过应用开发过程来实现**。

### 5.1 评估（Evaluation）

评估贯穿模型适配的全流程：选择模型、基准测试、判断是否上线、检测生产问题和改进机会。

**为什么 AI 工程中评估更难？**

基础模型的开放式输出和更广泛的能力使得评估难度剧增：
- 封闭式任务（如欺诈检测）有明确的预期输出可供对比
- 开放式任务（如聊天机器人回复）不可能穷举所有合理回复来比较

**评估方法也影响结论**：2023 年 12 月 Google 发布 Gemini 时声称在 MMLU 基准上超越 ChatGPT。但 Google 使用了 **CoT@32**（给模型 32 个示例）来评估 Gemini，而 ChatGPT 只给了 5 个示例：

| 模型 | MMLU（CoT@32） | MMLU（5-shot） |
|---|---|---|
| Gemini Ultra | 90.04% | 83.7% |
| GPT-4 | 87.29%（CoT@32, via API）/ 86.4%（5-shot, reported） | 86.4% |

当条件统一为 5-shot 时，GPT-4 的表现反而更好。这个案例充分说明了 **prompt engineering 对模型性能的巨大影响**——同一模型在不同 prompt 策略下可以从 83.7% 跳到 90.04%。

### 5.2 Prompt Engineering 与上下文构造

**Prompt Engineering** 是在**不改变模型权重**的前提下，通过输入来引导模型行为的技术。

核心工作不仅是告诉模型做什么，还包括：
- 提供模型完成任务所需的**上下文和工具**
- 对于长上下文任务，可能需要**记忆管理系统**（Memory）来帮助模型追踪历史
- Prompt engineering 详见 Chapter 5，上下文构造详见 Chapter 6

### 5.3 AI 界面（AI Interface）

**AI 界面**是最终用户与 AI 应用交互的方式。

**传统 ML 时代**：只有拥有足够资源开发模型的组织才能构建 AI 应用 → 应用通常嵌入到组织已有产品中（如 Stripe/PayPal 的欺诈检测、Netflix/TikTok 的推荐系统）。

**基础模型时代**：任何人都能构建 AI 应用 → 可以作为**独立产品**或**嵌入其他产品**：

| 形态 | 示例 |
|---|---|
| 独立产品 | ChatGPT、Perplexity |
| 插件/Add-on | GitHub Copilot（VSCode 插件） |
| 浏览器扩展 | Grammarly（Google Docs 扩展） |
| 平台集成 | Midjourney（Discord 集成） |
| 聊天集成 | Slack、Discord、WeChat、WhatsApp 中的聊天机器人 |
| 语音界面 | 语音助手 |
| 具身 AI | 增强现实、虚拟现实中的 AI |

AI 界面也带来了新的**用户反馈收集方式**——对话界面让用户更容易用自然语言给出反馈，但这类反馈也更难结构化地提取和分析（详见 Chapter 10）。

### 应用开发层变化总结

| 类别 | 传统 ML | 基础模型时代 |
|---|---|---|
| AI 界面 | 不太重要（嵌入已有产品） | **重要**（需为独立产品设计界面） |
| Prompt Engineering | 不适用 | **重要**（核心适配技术） |
| 评估 | 重要 | **更加重要**（开放式输出使评估更难） |

## 六、AI Engineering vs Full-Stack Engineering

AI 工程越来越接近**全栈工程**（Full-Stack Engineering）。

### 6.1 界面重要性提升 → 前端工程师涌入

- 传统 ML 工程以 **Python** 为中心（TensorFlow、PyTorch 等框架都是 Python API）
- AI 工程时代，**JavaScript** 的支持在增长：LangChain.js、Transformers.js、OpenAI Node library、Vercel AI SDK
- 越来越多的 AI 工程师来自 Web 开发或全栈背景，而非传统 ML 背景
- 全栈工程师的优势：能够快速将想法变成 demo、获取反馈并迭代

### 6.2 工作流的根本颠倒

| 传统 ML 工作流 | AI Engineering 工作流 |
|---|---|
| Data → Model → Product | **Product → Data → Model** |
| 先收集数据、训练模型，最后构建产品 | 先构建产品（用现成模型），产品有前景后再投资数据和模型优化 |

**意义**：AI 工程的新工作流奖励那些**能快速迭代的人**。

### 6.3 AI 工程师 vs ML 工程师的角色演变

- 许多公司将 AI 工程和 ML 工程归入**同一职位**（LinkedIn 上的岗位标题常见"ML Engineer, Generative AI"）
- 但也有越来越多公司设立**独立的 AI Engineer 岗位**
- 两者角色有大量重叠——现有 ML 工程师可以扩展技能进入 AI 工程，也有完全没有 ML 背景的 AI 工程师
- 传统 ML 中，模型开发和产品开发往往是**脱节的流程**，ML 工程师很少参与产品决策；基础模型时代，AI 工程师**更深度地参与产品构建**

## 七、Summary（全章总结）

本章服务于两个目的：

1. **解释 AI 工程作为一个学科的崛起**——基础模型的可用性如何催生了 AI 工程
2. **概述在这些模型之上构建应用所需的流程**

### 全章回顾的关键脉络

```
语言模型 → (自监督) → 大语言模型 → (多模态) → 基础模型 → (模型即服务) → AI Engineering
```

### 核心要点

| 要点 | 说明 |
|---|---|
| **应用场景丰富且仍在早期** | 已有大量 AI 应用投入生产，但我们仍处于 AI 工程的早期阶段，更多创新尚未出现 |
| **构建前先问"该不该做"** | 在动手之前，评估用例、理解 AI 的角色、设定期望、规划里程碑和维护——这些规划工作常被忽视 |
| **AI 工程脱胎于 ML 工程** | ML 工程的许多原则仍然适用，但 AI 工程也带来了新挑战和新方案 |
| **技术栈三层变化不均** | 应用开发层变化最大（界面、prompt engineering、评估），基础设施层变化最小 |
| **社区能量惊人但也令人焦虑** | AI 工程社区的集体能量、创造力和人才密度令人难以置信，新技术、新发现不断涌现，不可能全部跟上 |
| **框架比工具重要** | 工具会变，框架帮助你在不断变化的环境中导航。本书的目标就是提供这样一个框架 |

> "一个令人欣慰的事实是：AI 非常擅长信息聚合，它可以帮助我们汇总和消化这些新动态。但工具只能帮到一定程度——空间越是让人应接不暇，拥有一个框架来导航就越重要。"
