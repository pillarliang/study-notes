# Modeling — 模型架构与规模

> **核心定位**：训练模型之前，开发者需要做出两类关键决策：模型应该采用什么**架构**（architecture）？应该有多少**参数**（parameters）？这些决策不仅决定了模型的能力上限，还直接影响下游应用中模型的**可用性**——例如，一个 7B 参数的模型在部署便捷性上远优于 175B 参数的模型，而针对延迟优化 Transformer 和优化其他架构的方法也截然不同。

---

## 一、模型架构（Model Architecture）

### 1.1 Transformer：当前的主导架构

截至本书写作时，语言基础模型中最主流的架构是 **Transformer**（Vaswani et al., 2017），其核心是**注意力机制**（attention mechanism）。Transformer 解决了此前架构的许多局限性，但它自身也有局限。本节将分析 Transformer 及其替代方案。

### 1.2 前身：Seq2seq 架构

要理解 Transformer 解决了什么问题，需要先看它的前身——**seq2seq**（sequence-to-sequence，序列到序列）架构。

#### Seq2seq 是什么

Seq2seq 于 2014 年被提出并迅速流行，在当时的两大挑战性任务——**机器翻译**和**文本摘要**——上取得了显著突破。2016 年，Google 将 seq2seq 集成到 Google Translate 中，声称这带来了"有史以来机器翻译质量最大的改进"。Seq2seq 因此成为序列任务的首选架构。

#### Seq2seq 的工作原理

Seq2seq 由**编码器**（encoder）和**解码器**（decoder）两部分组成，两者都使用 RNN（循环神经网络，Recurrent Neural Network）：

1. **编码器**：逐个处理输入 token，将整个输入压缩为一个**最终隐藏状态**（final hidden state）向量
2. **解码器**：基于这个隐藏状态和已生成的 token，逐个生成输出 token

> **类比**：就像只用一本书的摘要来回答关于这本书的所有问题——信息被高度压缩，必然有大量细节丢失。

#### Seq2seq 的两个根本缺陷

| 缺陷 | 说明 |
|---|---|
| **信息瓶颈** | 解码器只能看到编码器的最终隐藏状态，输入的丰富信息被压缩为一个固定大小的向量。输入越长，信息损失越严重 |
| **顺序处理** | RNN 必须逐 token 处理输入和生成输出——如果输入有 200 个 token，必须等第 1 个处理完才能处理第 2 个，速度极慢 |

> RNN 还特别容易出现**梯度消失和梯度爆炸**问题：梯度在多步传播中要么不断缩小趋近于零（难以学习），要么不断膨胀导致训练不稳定。

### 1.3 Transformer 如何解决这些问题

#### 注意力机制的核心思想

Transformer 用**注意力机制**替代了 seq2seq 的"压缩为单一向量"策略。注意力机制允许模型在生成每个输出 token 时，**直接权衡所有输入 token 的重要性**。

> **类比**：如果 seq2seq 是"只读书的摘要来答题"，那么 Transformer 就是"可以翻阅书中任何一页来答题"。

> 注意力机制虽然常与 Transformer 关联，但它实际上比 Transformer 早三年被提出（Bahdanau et al., 2014）。Google 在 2016 年就将注意力机制用于其 seq2seq 架构的 GNMT（Google Neural Machine Translation）模型。但直到 Transformer 论文证明注意力机制可以**完全脱离 RNN 独立使用**，它才真正起飞。

#### 并行处理输入

Transformer 完全抛弃了 RNN。输入 token 可以被**并行处理**，极大地加速了输入处理速度。但需要注意：Transformer 解决了输入端的顺序瓶颈，但基于 Transformer 的**自回归语言模型**在输出端仍然是顺序的——每次只生成一个 token。

#### 推理的两个阶段

Transformer 语言模型的推理分为两个阶段：

| 阶段 | 说明 |
|---|---|
| **Prefill（预填充）** | 并行处理所有输入 token，生成中间状态（包括所有输入 token 的 key 和 value 向量），为生成第一个输出 token 做准备 |
| **Decode（解码）** | 逐个生成输出 token，每次生成一个 |

> Prefill 的可并行性和 Decode 的顺序性，驱动了 Chapter 9 中讨论的许多推理优化技术。

### 1.4 注意力机制详解

#### Key、Value、Query 向量

注意力机制的核心组件是三种向量——**Key（K）**、**Value（V）**、**Query（Q）**：

| 向量 | 角色 | 类比 |
|---|---|---|
| **Query（Q）** | 代表解码器在当前解码步的状态——"我在找什么信息？" | 一个正在寻找信息来撰写摘要的人 |
| **Key（K）** | 代表每个先前 token 的"标识"——"我是谁？" | 书中每一页的页码 |
| **Value（V）** | 代表每个先前 token 的实际内容——"我包含什么信息？" | 书中每一页的具体内容 |

> 注意：在给定解码步中，"先前 token"包括所有输入 token **和**已经生成的输出 token。

#### 注意力的计算过程

1. **计算 K、V、Q 向量**：给定输入 x，通过三个权重矩阵 W_K、W_V、W_Q 分别计算：
   ```
   K = x · W_K
   V = x · W_V
   Q = x · W_Q
   ```

2. **计算注意力分数**：Query 向量与每个 Key 向量做**点积**（dot product），得分越高表示该 token 的内容（Value）对当前生成越重要

3. **加权求和**：用 softmax 归一化的注意力分数对 Value 向量加权求和，得到最终输出

数学公式：

```
Attention(Q, K, V) = softmax(Q · K^T / √d) · V
```

其中 d 是向量维度，除以 √d 是为了防止点积值过大导致 softmax 梯度消失。

#### 多头注意力（Multi-Head Attention）

实际中，注意力几乎总是**多头的**（multi-headed）。多头注意力允许模型**同时关注不同组的先前 token**——就像多个人同时阅读同一本书的不同部分。

具体做法是将 Q、K、V 向量切分为更小的子向量，每组对应一个注意力头（attention head）：

> **实例**：Llama 2-7B 的隐藏维度（hidden dimension）为 4096，有 32 个注意力头。每个 K、V、Q 向量维度为 4096，被切分为 32 个维度为 128 的子向量（4096 / 32 = 128）。

所有注意力头的输出被拼接后，通过一个**输出投影矩阵**（output projection matrix）做变换，得到的结果维度与模型的隐藏维度相同。

### 1.5 Transformer Block（Transformer 块）

Transformer 架构由多个**Transformer 块**堆叠而成。每个 Transformer 块包含两个核心模块：

#### Attention 模块

包含四个权重矩阵：Query、Key、Value 和输出投影矩阵。

#### MLP 模块（多层感知机）

由**线性层**（也叫前馈层，feedforward layer）和**非线性激活函数**（nonlinear activation function）交替组成：

- **线性层**：一个权重矩阵，执行线性变换
- **激活函数**：引入非线性，让模型能学习复杂的非线性模式

常见的激活函数：

| 激活函数 | 使用模型 | 数学定义 |
|---|---|---|
| **ReLU**（Rectified Linear Unit） | GPT-2 | ReLU(x) = max(0, x) |
| **GELU** | GPT-3 | 更平滑的 ReLU 变体 |

> 激活函数为什么这么简单就够了？研究社区曾经竞相设计复杂的激活函数，但结果发现更精致的函数并没有带来更好的效果。模型只需要一个非线性函数来打破前馈层的线性——**越简单越好**，因为复杂函数会消耗更多计算和内存。

### 1.6 Transformer 模型的完整结构

Transformer 语言模型在多个 Transformer 块的前后，还各有一个模块：

#### Transformer 块之前：Embedding 模块

包含两个矩阵：

| 矩阵 | 作用 |
|---|---|
| **Embedding matrix** | 将 token 转换为嵌入向量 |
| **Positional embedding matrix** | 将位置索引转换为位置向量，编码 token 在序列中的位置信息 |

> 位置索引的数量朴素地决定了模型的**最大上下文长度**。例如，如果模型维护 2048 个位置索引，其最大上下文长度就是 2048。但存在技术手段可以在不增加位置索引数量的情况下扩展上下文长度。

#### Transformer 块之后：Output Layer（输出层）

将模型的输出向量映射为 token 概率，用于采样生成下一个 token。这个模块也叫 **unembedding layer**（反嵌入层）或 **model head**（模型头）。

### 1.7 决定模型大小的关键维度

Transformer 模型的大小由以下构建块的维度决定：

| 维度 | 影响 |
|---|---|
| **模型维度（d_model）** | 决定 K、Q、V 和输出投影矩阵的大小 |
| **Transformer 块数量** | 通常被称为模型的"层数"（number of layers） |
| **前馈层维度（d_ff）** | MLP 模块中前馈层的宽度 |
| **词汇表大小（V）** | 影响 embedding 和 output 层的大小 |

#### 不同 Llama 模型的维度对比

| 模型 | Transformer 块数 | 模型维度 | 前馈维度 | 词汇表大小 | 上下文长度 |
|---|---|---|---|---|---|
| Llama 2-7B | 32 | 4,096 | 11,008 | 32K | 4K |
| Llama 2-13B | 40 | 5,120 | 13,824 | 32K | 4K |
| Llama 2-70B | 80 | 8,192 | 22,016 | 32K | 4K |
| Llama 3-7B | 32 | 4,096 | 14,336 | 128K | 128K |
| Llama 3-70B | 80 | 8,192 | 28,672 | 128K | 128K |
| Llama 3-405B | 126 | 16,384 | 53,248 | 128K | 128K |

> 注意：上下文长度的增加会影响模型的**内存占用**，但**不影响**模型的总参数量。

## 二、其他模型架构（2026 更新）

### 2.1 Transformer 的统治地位与挑战者

自 2012 年 AlexNet 复兴深度学习以来，架构潮流更迭频繁：seq2seq 火了四年（2014-2018），GAN 也火了约五年（2014-2019）。相比之下，Transformer 自 2017 年起已主导超过八年——这在深度学习领域极为罕见。

为什么新架构难以取代 Transformer？

- Transformer 自 2017 年以来经过了**极其深入的优化**
- 新架构要取代它，必须在**人们关心的规模**和**人们使用的硬件**上表现更好
- Google 最初为 TPU（Tensor Processing Unit）设计了 Transformer，后来才针对 GPU 优化

### 2.2 RWKV

**RWKV**（Peng et al., 2023）是一种基于 RNN 的模型，但可以**并行训练**。由于其 RNN 本质，理论上不受 Transformer 那样的上下文长度限制。但实践中，没有上下文长度限制并不等于在长上下文上表现良好。

### 2.3 状态空间模型（SSM）系列

**SSM**（State Space Models，状态空间模型）是另一类有前景的架构（Gu et al., 2021a），在**长程记忆**（long-range memory）方面展现出 Transformer 所缺乏的优势。SSM 的发展历程：

| 模型 | 论文 | 关键贡献 |
|---|---|---|
| **S4** | Gu et al., 2021b | "Efficiently Modeling Long Sequences with Structured State Spaces"——提高了 SSM 的效率 |
| **H3** | Fu et al., 2022 | "Hungry Hungry Hippos"——引入跨序列回忆和比较 token 的机制，类似注意力但更高效 |
| **Mamba** | Gu and Dao, 2023 | 将 SSM 扩展到 30 亿参数。Mamba-3B 在语言建模上**超越了同等大小的 Transformer，并匹配两倍大小的 Transformer**。推理计算量随序列长度**线性增长**（Transformer 是二次方增长），在百万级长度序列上仍有提升 |
| **Jamba** | Lieber et al., 2024 | **混合架构**——交替堆叠 Transformer 层和 Mamba 层，并引入 MoE。发布了 52B 总参数（12B 活跃参数）的 MoE 模型，可放入单个 80GB GPU。在标准基准和 256K 上下文评测上均表现强劲 |

> 截至 2026 年，Mamba 和基于 SSM 的混合架构已经获得更广泛的采用。Mamba-2（Dao and Gu, 2024）进一步优化了训练效率，而 Jamba 等混合架构的思路被更多模型采纳——将 Transformer 层与 SSM 层混合使用，兼顾注意力的表达能力和 SSM 的长序列效率。但 Transformer 仍然是主流选择，尤其在最大规模的模型中。

### 2.4 架构变革的影响

即使未来某个架构取代了 Transformer，本书讨论的大部分模型适配技术——Prompt Engineering、RAG、微调等——仍然适用。正如从 ML 工程到 AI 工程的范式转变没有改变许多基本方法，底层架构的改变也不会颠覆 AI 工程的基本方法论。

## 三、模型大小（Model Size）

### 3.1 参数量：模型规模的基本衡量

近年来 AI 的很大一部分进展可以归因于**模型规模的增大**。参数量通常附在模型名称之后——例如 Llama-13B 表示 Llama 家族中拥有 **130 亿参数**的版本。

#### 参数量的意义

- **学习容量**：参数越多，模型可以学习的模式越复杂。同一家族中，13B 参数的版本通常远优于 7B 的版本
- **估算计算资源**：参数量可以直接估算推理所需的 GPU 内存。例如，7B 参数模型如果每个参数用 2 字节（16 位）存储，推理至少需要 **14 GB** GPU 内存
- **代际进步**：随着训练技术的进步，新一代同等大小的模型会优于老一代。例如 Llama 3-8B（2024）在 MMLU 基准上**超越了** Llama 2-70B（2023）

### 3.2 稀疏模型与混合专家（MoE）

参数量可能具有**误导性**——如果模型是**稀疏**（sparse）的。

#### 稀疏模型

稀疏模型中有大量参数值为零。例如，一个 7B 参数的模型如果 90% 稀疏，实际只有 **7 亿**非零参数。稀疏性允许更高效的存储和计算——大型稀疏模型可能比小型密集模型消耗更少的计算资源。

#### 混合专家模型（Mixture-of-Experts, MoE）

**MoE**（Shazeer et al., 2017）是近年来流行的一种稀疏模型。其核心思想：

- 模型参数被分为多组，每组称为一个**专家**（expert）
- 对每个 token，只有**部分专家被激活**（active），其余专家不参与计算

> **实例：Mixtral 8x7B**
> - 8 个专家，每个专家 7B 参数
> - 如果没有参数共享，总参数应为 8 × 7B = 56B
> - 实际因部分参数共享，总参数为 **46.7B**
> - 每个 token 在每层只激活 **2 个专家** → 活跃参数仅 **12.9B**
> - 因此其推理成本和速度等同于一个 12.9B 的密集模型

### 3.3 更大不一定更好

大模型在数据不足时也会表现不佳：

> **极端例子**：一个 13B 参数的模型如果只在"I like pineapples"一句话上训练，其表现会远不如在大量数据上训练的小得多的模型。

讨论模型大小时，必须同时考虑**训练数据的规模**。

### 3.4 数据集大小的衡量

#### 样本数 vs Token 数

对语言模型而言，一个训练样本可以是一个句子、一篇维基百科文章、一段对话或一本书——差异巨大。因此**训练样本数**不是好的度量指标，**token 数**更合适。

但 token 数也不完美——不同模型的分词方式不同，同一个数据集在不同模型中的 token 数不一样。尽管如此，token 是模型实际操作的单位，知道数据集的 token 数可以帮助估算模型能从中学到多少。

#### 当前 LLM 的训练数据规模

LLM 的训练数据已达到**万亿级 token**：

| 模型系列 | 训练 token 数 |
|---|---|
| Llama 1 | 1.4 万亿 |
| Llama 2 | 2 万亿 |
| Llama 3 | 15 万亿 |

开源数据集 RedPajama-v2 有 **30 万亿** token——相当于约 4.5 亿本书，或维基百科的 5,400 倍。但 RedPajama-v2 是不加区分地收集的，其中高质量数据的比例要低得多。

#### 训练 Token 数 ≠ 数据集 Token 数

**训练 token 数**衡量的是模型实际训练时处理的 token 总量。如果一个数据集有 1 万亿 token，模型在该数据集上训练了 2 个 **epoch**（一个 epoch 指对整个数据集完整遍历一次），那么训练 token 数就是 **2 万亿**。

> 大模型通常只在预训练数据上训练 **1 个 epoch**。

#### 各模型的参数量与训练 token 数对比

| 模型 | 参数量 | 训练 token 数 |
|---|---|---|
| LaMDA | 137B | 168B |
| GPT-3 | 175B | 300B |
| Jurassic | 178B | 300B |
| Gopher | 280B | 300B |
| MT-NLG 530B | 530B | 270B |
| **Chinchilla** | **70B** | **1.4T** |

> Chinchilla 是一个关键转折点——参数量只有 70B，但训练了 1.4 万亿 token，远超同期其他模型。这引出了接下来讨论的 Scaling Law。

> 数据的三个黄金标准：**数量**（quantity）、**质量**（quality）和**多样性**（diversity），详见 Chapter 8。

## 四、计算量（Compute）

### 4.1 FLOP 与 FLOP/s

预训练大模型需要大量**计算资源**（compute）。衡量计算量的标准化单位是 **FLOP**（Floating Point Operation，浮点运算）——一次浮点数学运算。

| 概念 | 含义 | 衡量的是什么 |
|---|---|---|
| **FLOPs**（复数） | 多次浮点运算 | 一个任务的**计算需求** |
| **FLOP/s** | 每秒浮点运算数 | 一台机器的**计算峰值性能** |

> 注意 FLOP/s 和 FLOPS 看起来很像，容易混淆。为避免混淆，一些公司（包括 OpenAI）使用 **FLOP/s-day** 来衡量计算需求：1 FLOP/s-day = 60 × 60 × 24 = **86,400 FLOPs**。

#### 实际训练的计算量

| 模型 | 训练所需 FLOPs |
|---|---|
| PaLM-2（Google 最大版本） | 10²² |
| GPT-3-175B | 3.14 × 10²³ |

#### 从 FLOPs 估算训练时间

以 NVIDIA H100 NVL GPU 为例：峰值性能 **60 TeraFLOP/s**（即 6 × 10¹³ FLOP/s），每天可执行 5.2 × 10¹⁸ FLOPs。

假设你有 256 块 H100，以最大算力运行、无任何训练错误：
```
训练 GPT-3-175B 所需时间 = (3.14 × 10²³) / (256 × 5.2 × 10¹⁸)
                        ≈ 236 天 ≈ 7.8 个月
```

### 4.2 利用率（Utilization）

机器不可能始终以峰值算力运行。**利用率**衡量的是实际使用的峰值算力比例：

| 利用率 | 评价 |
|---|---|
| 50% | 还行 |
| 70%+ | 很好 |

以 70% 利用率和 $2/小时/H100 估算训练 GPT-3-175B 的成本：

```
$2/H100/hour × 256 H100 × 24 hours × 256 days / 0.7 = $4,142,811
```

> 即训练一个 GPT-3 级别的模型**成本超过 400 万美元**。

### 4.3 模型规模的三个信号

三个数字共同刻画了一个模型的规模：

| 指标 | 衡量的是什么 |
|---|---|
| **参数量** | 模型的**学习容量** |
| **训练 token 数** | 模型**学到了多少** |
| **FLOPs** | **训练成本** |

## 五、Scaling Law（缩放定律）

### 5.1 Chinchilla Scaling Law

前面我们知道了三个事实：
1. 模型性能取决于模型大小和数据集大小
2. 更大的模型和数据集需要更多计算
3. 计算需要花钱

现实中，团队通常从一个**计算预算**出发——不是先选模型大小再算成本，而是先确定能花多少钱，再优化配置。

**核心问题**：给定固定的 FLOPs 预算，什么样的模型大小和数据集大小能获得最佳性能？能在固定计算预算下达到最佳性能的模型叫做 **compute-optimal**（计算最优）的。

**Chinchilla Scaling Law** 由 DeepMind 在论文 "Training Compute-Optimal Large Language Models"（2022）中提出。他们训练了 **400 个语言模型**（参数量从 7000 万到 160 亿以上），发现：

> 对于计算最优训练，训练 token 数应约为模型参数量的 **20 倍**。

也就是说：
- 3B 参数模型 → 需要约 **60B** 训练 token
- 模型大小每翻一倍，训练 token 数也应翻倍

这个定律还能预测：给定 FLOPs 预算，最优的参数量和 token 数各是多少，以及预期的训练损失。

### 5.2 Scaling Law 的局限

| 局限 | 说明 |
|---|---|
| **针对密集模型** | 该定律是基于在人类生成数据上训练的密集模型推导的，对稀疏模型（如 MoE）和合成数据的适用性仍是开放问题 |
| **计算最优 ≠ 实用最优** | Scaling Law 优化的是给定计算预算下的模型质量。但在生产环境中，模型质量不是唯一考量——**可用性**同样重要。Meta 选择训练较小的 Llama 模型而非更大但计算最优的模型，因为小模型更容易部署和推理，有助于更广泛的采用 |
| **Sardana et al.（2023）修正** | 修改了 Chinchilla Scaling Law，在计算最优 LLM 参数和预训练数据大小时纳入了**推理需求**的考量 |

### 5.3 性能提升的代价在增加

虽然达到**相同性能水平**的成本在下降（2019-2021 年间，ImageNet 达到 93% 准确率的成本减半），但**提升性能**的边际成本在上升：

> Meta 的论文 "Beyond Neural Scaling Laws: Beating Power Law Scaling via Data Pruning" 指出：一个 **2% 错误率**的模型所需的数据、计算或能源，可能比 **3% 错误率**的模型多出一个数量级。

语言建模中，将交叉熵损失从 3.4 降到 2.8 nats 需要 **10 倍**的训练数据。损失的微小改善虽然在指标上看起来不大，但可以导致下游应用质量的显著差异。

## 六、Inverse Scaling（逆缩放）

### 更大的模型是否总是更好？

通常是的，但不绝对。2022 年，Anthropic 发现了一个反直觉的现象：

> 更多的对齐训练（alignment training）反而导致模型**更不**符合人类偏好（Perez et al., 2022）。经过更多对齐训练的模型"更可能表达特定的政治观点（支持持枪权和反移民）、宗教观点（佛教）、自我报告的意识体验和道德自我价值感，以及不愿被关闭的倾向"。

2023 年，纽约大学等机构发起了 **Inverse Scaling Prize**（逆缩放奖），悬赏寻找更大模型反而表现更差的任务：
- 三等奖 $5,000 × 11 个，二等奖 $20,000，一等奖 $100,000
- 收到 99 份提交，11 份获三等奖
- 发现大模型在**需要记忆的任务**和**有强先验的任务**上有时表现更差
- 但未颁发二等奖或一等奖——因为提交的任务虽然在小测试集上显示退步，但**在真实场景中并未表现出实际失败**

## 七、Scaling Extrapolation（缩放外推）

### 7.1 超参数的挑战

模型性能高度依赖**超参数**（hyperparameters）的设置。

#### Parameter vs Hyperparameter

| 类型 | 由谁确定 | 示例 |
|---|---|---|
| **Parameter** | 训练过程中由模型自动学习 | 权重矩阵中的值 |
| **Hyperparameter** | 由用户在训练前手动设定 | 层数、模型维度、词汇表大小、batch size、epoch 数、学习率等 |

小模型可以通过多次训练尝试不同超参数组合，选最好的。但大模型的训练极其昂贵——可能只有**一次**设置超参数的机会。

### 7.2 Scaling Extrapolation（超参数迁移）

**Scaling extrapolation**（也叫 **hyperparameter transferring**）是一个新兴研究方向：

1. 在**远小于目标模型**的小模型上，研究不同超参数的影响
2. **外推**这些结论到目标大模型

> Microsoft 和 OpenAI 2022 年的论文证明，可以将超参数从 **4000 万参数**的模型迁移到 **67 亿参数**的模型。

但这种外推存在困难：
- 超参数数量多且相互作用复杂（10 个超参数就有 1,024 种组合）
- **涌现能力**（emergent abilities, Wei et al., 2022）——只在大规模才出现的能力——使得小模型上的结论可能不适用于大模型

## 八、Scaling Bottlenecks（缩放瓶颈）

### 8.1 数据枯竭

模型规模每增长一个数量级，性能都有所提升：
- GPT-1 → GPT-2：参数从 1.17 亿增长到 15 亿（约 10 倍）
- GPT-2 → GPT-3：15 亿增长到 1,750 亿（约 100 倍）
- 2018-2021 年间，模型大小增长了 **3 个数量级**

但增长能持续多久？两个关键瓶颈：

#### 训练数据即将耗尽

训练数据集的增长速度远超新数据的产生速度（Villalobos et al., 2022）。互联网公开可用的数据正在被逐渐用完——**如果你在互联网上发布过任何内容，你应该假设它已经或将被纳入某个语言模型的训练数据**。

更严峻的是：
- 有人故意将数据发布到互联网上，企图**注入**未来模型的训练数据，以操纵模型行为（也可用于 prompt injection 攻击）
- 互联网正被 **AI 生成的内容**快速填充。如果用这些 AI 生成的数据训练新模型，可能导致"**模型崩溃**"——逐步丧失原始数据模式（Shumailov et al., 2023），但实际影响比预期更复杂（详见 Chapter 8）
- 2023 年 12 月，xAI 的 Grok 被发现拒绝请求时使用了"违反 OpenAI 使用政策"的措辞，引发了 Grok 可能使用 ChatGPT 输出进行训练的猜测

#### 公开数据耗尽后的出路

一旦公开数据耗尽，最可行的方向是**私有数据**——版权书籍、翻译文档、合同、医疗记录、基因组序列等。这就是 OpenAI 与 Axel Springer、Associated Press 等出版商谈判数据协议的原因。

同时，许多公司（如 Reddit、Stack Overflow）已修改数据使用条款，限制其他公司抓取其数据训练模型。Longpre et al.（2024）发现，2023-2024 年间，网络数据来源的限制措施急剧增加，公共数据集 C4 中 **28% 最关键的来源已被完全限制使用**，加上爬取限制，C4 已有 **45% 被限制**。

### 8.2 能源瓶颈

机器需要**电力**运行。截至本书写作时，数据中心消耗全球电力的 **1-2%**，预计到 2030 年将达到 **4-20%**（Patel, Nishball, and Ontiveros, 2024）。在找到新的电力生产方式之前，数据中心最多只能增长约 50 倍——不到两个数量级。这引发了对近期电力短缺的担忧，并将推高电力成本。

## 九、关键概念速查表

| 概念 | 一句话解释 |
|---|---|
| Transformer | 基于注意力机制的模型架构，自 2017 年起主导语言模型领域 |
| Seq2seq | Transformer 的前身，使用 RNN 的编码器-解码器架构，有信息瓶颈和顺序处理两大缺陷 |
| Attention Mechanism | 允许模型在生成每个 token 时直接权衡所有先前 token 重要性的机制 |
| Q/K/V 向量 | 注意力机制的三个核心组件：Query（当前需求）、Key（token 标识）、Value（token 内容） |
| Multi-Head Attention | 将注意力分为多个头并行计算，允许模型同时关注不同位置的信息 |
| Transformer Block | Transformer 的基本构建块，包含 Attention 模块和 MLP 模块 |
| Prefill / Decode | Transformer 推理的两个阶段：并行处理输入 / 顺序生成输出 |
| SSM（State Space Model） | 状态空间模型，在长序列处理上具有线性复杂度优势的替代架构 |
| Mamba | 最成功的 SSM 变体，推理复杂度随序列长度线性增长 |
| MoE（Mixture-of-Experts） | 混合专家模型，总参数量大但每个 token 只激活部分专家，推理成本等同于小模型 |
| Sparse Model | 含大量零值参数的模型，可实现更高效的存储和计算 |
| FLOP | 浮点运算，衡量计算任务需求量的标准单位 |
| FLOP/s | 每秒浮点运算数，衡量硬件计算能力的指标 |
| Utilization | GPU 实际使用的峰值算力比例，70%+ 被认为很好 |
| Chinchilla Scaling Law | 给定计算预算下，最优训练 token 数约为参数量的 20 倍 |
| Compute-optimal | 在固定计算预算下能达到最佳性能的模型配置 |
| Inverse Scaling | 更大模型在某些任务上反而表现更差的现象 |
| Scaling Extrapolation | 在小模型上研究超参数影响，外推到大模型的技术 |
| Emergent Abilities | 只在大规模模型中出现的能力，小模型上不可观测 |
| Parameter | 训练过程中模型自动学习的内部变量 |
| Hyperparameter | 用户在训练前手动设定的配置，控制模型结构和训练过程 |
