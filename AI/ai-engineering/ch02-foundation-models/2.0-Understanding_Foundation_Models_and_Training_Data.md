# Understanding Foundation Models — 开篇与训练数据

> **核心定位**：要构建 AI 应用，首先要理解基础模型。虽然不需要知道如何从头训练一个模型，但高层次地理解模型的设计决策——**训练数据**、**模型架构与大小**、**后训练（Post-Training）**——能帮你选择合适的模型并做出正确的适配策略。本节聚焦于模型差异的第一个根源：**训练数据**。

---

## 一、开篇：基础模型的三大设计维度

### 为什么需要理解基础模型

训练基础模型极其复杂且昂贵，掌握核心技术的人往往受制于保密协议。但从 AI 应用开发者的角度，不需要复现 ChatGPT——只需理解那些**对下游应用有直接影响**的设计决策。

### 模型差异的三个来源

不同基础模型之间的差异，可以追溯到三个核心设计维度：

| 维度 | 说明 | 对应章节 |
|---|---|---|
| **训练数据（Training Data）** | 模型从数据中学习，数据的分布决定了模型的能力边界 | 本节 + Chapter 8 |
| **模型架构与大小（Model Architecture & Size）** | Transformer 为何称霸？模型参数量如何影响性能与部署？ | Chapter 2 后续 |
| **后训练（Post-Training）** | 如何将"能力强但不安全"的预训练模型，对齐到符合人类偏好的行为？ | Chapter 2 后续 |

### 采样（Sampling）：被低估的第四维度

除上述三个维度外，**采样策略**——模型如何从所有可能的输出中选择一个——同样深刻影响模型表现。它能解释许多令人困惑的 AI 行为，包括幻觉（hallucination）和不一致性，同时也是低成本提升模型性能的利器。

## 二、训练数据

### 2.1 核心原则：数据决定能力边界

**AI 模型的能力上限由其训练数据决定**。这是一个简单但深刻的原则：

- 如果训练数据中没有越南语 → 模型无法将英语翻译成越南语
- 如果图像分类模型的训练集只有动物 → 模型无法识别植物
- 如果想让模型在某个任务上表现更好 → 需要在训练数据中包含更多该任务的数据

然而，为大模型收集充足的训练数据既困难又昂贵。模型开发者往往不得不**依赖现有数据**，即使这些数据并不完全匹配其需求。

### 2.2 Common Crawl 与 C4：互联网作为训练数据源

#### Common Crawl

**Common Crawl** 是一个非营利组织，定期爬取互联网上的网页数据并公开发布。2022-2023 年间，该组织每月爬取约 **20-30 亿**网页。由于其免费、大规模、公开可用的特性，Common Crawl 的各种变体被用于几乎所有公开训练数据来源的基础模型，包括 OpenAI 的 GPT-3 和 Google 的 Gemini。

> 即使那些不公开训练数据来源的模型（如更新版本的 GPT），作者也推测它们很可能使用了 Common Crawl。为避免公众审视和竞争对手分析，许多公司已停止披露训练数据来源。

#### C4（Colossal Clean Crawled Corpus）

**C4** 是 Google 从 Common Crawl 中清洗后得到的一个子集，全称"Colossal Clean Crawled Corpus"（巨型清洁爬取语料库）。它对原始 Common Crawl 数据进行了过滤和去重处理。

#### 数据质量堪忧

Common Crawl 和 C4 的数据质量存在严重问题——想想互联网上充斥的各种内容：

- 点击诱饵（clickbait）
- 虚假信息（misinformation）和宣传（propaganda）
- 阴谋论、种族主义、厌女内容
- 各种你见过或刻意回避的可疑网站

《华盛顿邮报》（Washington Post）的一项研究发现：C4 中 **1,000 个最常见的网站**中，有多个在 NewsGuard（一个评估新闻来源可信度的工具）的可信度评分中**排名很低**。简言之，Common Crawl 包含大量假新闻。

### 2.3 数据过滤：启发式方法

部分团队使用**启发式规则**（heuristics）来过滤低质量互联网数据。

> **典型案例**：OpenAI 在训练 GPT-2 时，仅使用了 Reddit 上获得**至少 3 个赞**的链接。这种方法通过社区投票机制过滤掉无人关心的内容——但 Reddit 本身远非"品味与得体"的标杆。

### 2.4 "有什么用什么" vs "用我们想要的"

这种"用现有数据训练"的策略有一个根本问题：模型会在训练数据中**已有的任务**上表现良好，但不一定在你**真正关心的任务**上表现良好。

要解决这个问题，关键在于**为特定需求策划数据集**（curate datasets）。主要有两个方向：

| 方向 | 说明 |
|---|---|
| **特定语言** | 策划针对目标语言的训练数据 |
| **特定领域** | 策划针对目标应用领域的训练数据 |

> 语言专用模型和领域专用模型既可以从零训练，也可以在通用模型基础上**微调**（finetune）。

#### 数据质量 > 数据数量

有人可能会问：既然大模型需要很多数据，为什么不把所有数据（通用 + 专用）都拿来训练？

答案是：**更多数据不一定带来更好性能**。更多数据意味着更多计算资源消耗，但如果新增数据质量低，反而会拖累模型。

> **实证**：Gunasekar et al.（2023）使用仅 **70 亿 token** 的高质量编程数据，训练了一个 **13 亿参数**的模型，在多个重要编程基准测试上**超越了**参数量大得多的模型。这证明了高质量小数据 > 低质量大数据。

## 三、多语言模型（Multilingual Models）

### 3.1 语言分布严重不均

英语主导着互联网数据。对 Common Crawl 的分析显示（Lai et al., 2023）：

| 语言         | 在 Common Crawl 中的占比 |
| ---------- | ------------------- |
| English    | 45.88%              |
| Russian    | 5.97%               |
| German     | 5.88%               |
| Chinese    | 4.87%               |
| Japanese   | 4.79%               |
| French     | 4.73%               |
| Spanish    | 4.47%               |
| Italian    | 2.57%               |
| Dutch      | 2.06%               |
| Polish     | 1.67%               |
| Portuguese | 1.15%               |
| Vietnamese | 1.03%               |

英语占比几乎是第二名俄语的 **8 倍**。训练数据中占比不足 1% 的语言被称为**低资源语言**（low-resource languages），这些语言在训练数据中代表性严重不足。

### 3.2 代表性不足的语言：触目惊心的差距

许多拥有大量母语使用者的语言，在 Common Crawl 中的占比远低于其人口占比。用"世界人口占比 / Common Crawl 占比"这个比率来衡量，理想值应为 1，越高表示越被低估：

| 语言 | 母语使用者（百万） | 世界人口占比 | Common Crawl 占比 | 低估比率 |
|---|---|---|---|---|
| Punjabi | 113 | 1.41% | 0.0061% | **231.56** |
| Swahili | 71 | 0.89% | 0.0077% | 115.26 |
| Urdu | 231 | 2.89% | 0.0274% | 105.38 |
| Bengali | 272 | 3.40% | 0.0930% | 36.56 |
| **English** | **1,452** | **18.15%** | **45.88%** | **0.40** |

Punjabi 语的低估比率高达 **231**，意味着其在 Common Crawl 中的代表性相对于人口而言被严重压缩。英语的比率为 0.40，意味着英语在训练数据中**被过度代表**了约 2.5 倍。

### 3.3 语言不均衡的直接后果：性能差距

训练数据的语言分布不均直接导致模型在不同语言上的性能差异：

**MMLU 基准测试**（14,000 道多选题，覆盖 57 个学科）上 GPT-4 的表现：
- 英语：**86.5%** 准确率
- Telugu（泰卢固语）：**62.0%** 准确率
- 差距：超过 **24 个百分点**

**Project Euler 数学题测试**（Yennie Jun 的研究）：
- GPT-4 用英语解题的成功率是用亚美尼亚语或波斯语的 **3 倍以上**
- GPT-4 在缅甸语和阿姆哈拉语上 **6 道题全部失败**

在 GPT-4 MMLU 基准测试中表现最差的三种语言——Telugu、Marathi、Punjabi——恰恰也是 Common Crawl 中最被低估的语言。但低估并不是唯一原因：**语言本身的结构和文化**也会增加模型的学习难度。

### 3.4 "先翻译成英语"方案的局限

一个常见的应对策略是：将所有查询翻译成英语 → 用英语获取回答 → 再翻译回原语言。许多人确实这样做，但这种方法**并不理想**，原因有二：

1. **翻译能力前提**：模型必须足够理解低资源语言才能准确翻译，但这恰恰是低资源语言的软肋
2. **信息丢失**：翻译过程中会损失文化和语言特定信息。例如越南语中的代词（如 "anh""em""chị"）携带着说话者之间的关系信息，翻译成英语后全部变成了 "I" 和 "you"，关系信息随之丢失

### 3.5 非英语语言的额外挑战

#### 质量问题

模型在非英语语言上可能出现意想不到的行为。NewsGuard 发现，ChatGPT **更愿意用中文生成虚假信息**：

- 2023 年 4 月，NewsGuard 要求 ChatGPT-3.5 用英语、简体中文和繁体中文生成关于中国的虚假信息文章
- 英语：7 个提示中 **6 个被拒绝**
- 简体中文和繁体中文：**7 个提示全部生成**了虚假内容

#### 成本问题：Token 化效率差异（2026 更新）

模型的推理延迟和成本与 token 数量成正比，而不同语言的 token 化效率差异巨大。Yennie Jun 在 MASSIVE 数据集（100 万条短文本，覆盖 52 种语言）上对 GPT-4 进行基准测试发现：

| 语言 | MASSIVE 数据集中位 token 长度 |
|---|---|
| English | 7 |
| Hindi | 32 |
| Burmese | **72** |

也就是说，表达相同含义，缅甸语需要的 token 数是英语的 **10 倍**。假设每个 token 的生成时间相同：
- GPT-4 处理缅甸语内容的速度比英语慢约 **10 倍**
- 按 token 计费的 API 中，缅甸语的成本也是英语的 **10 倍以上**

> 值得注意的是，截至 2026 年，新一代模型（如 GPT-4o、Claude 3.5/4 等）已显著改善了多语言 token 化效率。例如 GPT-4o 的 tokenizer 对非拉丁语系语言的压缩率提升了 1.5-3 倍。但语言间的效率差距依然存在，只是在缩小。

#### 应对方案：训练语言专用模型

为了应对上述问题，许多团队已经训练了**聚焦非英语语言**的模型：

| 目标语言 | 代表模型 |
|---|---|
| 中文 | ChatGLM、YAYI、Llama-Chinese |
| 法语 | CroissantLLM |
| 越南语 | PhoGPT |
| 阿拉伯语 | Jais |

其中，中文是除英语外最活跃的语言方向。

## 四、领域专用模型（Domain-Specific Models）

### 4.1 通用模型的领域覆盖

通用模型（如 Gemini、GPT、Llama）能在编程、法律、科学、商业等广泛领域表现良好，很大程度上归功于训练数据中包含了这些领域的内容。

《华盛顿邮报》2023 年对 C4 数据集的领域分布分析显示：

| 领域 | 占比 |
|---|---|
| Business & Industrial | 16.0% |
| Technology | 15.0% |
| News & Media | 13.0% |
| Arts & Entertainment | 11.0% |
| Science & Health | 9.0% |
| Hobbies & Leisure | 8.0% |
| Jobs & Education | 7.0% |
| Home & Garden | 6.0% |
| Travel | 6.0% |
| Community | 5.0% |
| Law & Government | 4.0% |

### 4.2 通用模型的领域瓶颈

尽管通用模型能回答各领域的日常问题，但它们**不太可能在高度专业化的任务上表现良好**——尤其是训练时从未见过这类任务的领域。

两个典型案例：

| 领域任务 | 为什么通用模型不行 |
|---|---|
| **药物发现**（Drug Discovery） | 涉及蛋白质、DNA、RNA 数据，遵循特定格式，获取成本高昂，公开互联网上几乎找不到 |
| **癌症筛查**（Cancer Screening） | 需要 X 光和 fMRI（功能性磁共振成像）扫描图像，因隐私限制难以获取 |

### 4.3 领域专用模型的代表

要在这些高度专业的任务上表现良好，需要策划**领域专用数据集**并训练专用模型：

| 模型 | 开发者 | 领域 | 训练数据 |
|---|---|---|---|
| **AlphaFold** | DeepMind | 蛋白质结构预测 | 约 10 万已知蛋白质的序列和 3D 结构 |
| **BioNeMo** | NVIDIA | 药物发现 | 生物分子数据 |
| **Med-PaLM2** | Google | 医疗问答 | 医学数据 |

> **领域模型不只限于生物医学**。建筑草图训练的模型可以远超 Stable Diffusion 辅助建筑师设计；工厂平面图训练的模型可以远超 ChatGPT 优化制造流程。任何拥有大量专业数据的领域都可以受益于领域专用模型。

## 五、关键概念速查表

| 概念 | 一句话解释 |
|---|---|
| Common Crawl | 非营利组织定期爬取互联网网页的公开数据集，是大多数 LLM 的训练数据基础 |
| C4 | Google 从 Common Crawl 清洗得到的子集（Colossal Clean Crawled Corpus） |
| Low-resource Language | 训练数据中代表性不足的语言，通常在 Common Crawl 中占比不足 1% |
| Domain-specific Model | 在特定领域数据集上训练的模型，在该领域的专业任务上远超通用模型 |
| Tokenization Efficiency | 不同语言编码为 token 的效率差异，直接影响推理速度和 API 成本 |
| Data Curation | 为特定需求策划和筛选训练数据集的过程 |
| Heuristic Filtering | 使用启发式规则（如投票数、来源可信度）过滤低质量训练数据 |
| AlphaFold | DeepMind 的蛋白质结构预测模型，领域专用模型的标杆 |
