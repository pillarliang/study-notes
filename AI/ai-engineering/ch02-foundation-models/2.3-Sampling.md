# Sampling — 采样策略与 AI 的概率本质

> **核心定位**：采样（Sampling）是模型从所有可能的输出中**选择一个**的过程。它是 AI 工程中最被低估的概念之一——不仅能解释幻觉（hallucination）和不一致性（inconsistency）等令人困惑的 AI 行为，还能通过选择合适的采样策略，以极低的成本显著提升模型性能。本节涵盖采样基础、采样策略、测试时计算、结构化输出，以及 AI 概率本质带来的挑战。

---

## 一、采样基础（Sampling Fundamentals）

### 1.1 从概率分布到输出

模型生成输出的核心机制是**概率计算**：

- **分类模型**：给定输入，模型计算每个类别的概率。例如垃圾邮件分类器输出 `spam=90%, not spam=10%`，然后根据决策规则（如概率 > 50% 即标为 spam）做出判断
- **语言模型**：生成下一个 token 时，模型计算词表中**每个 token** 的概率分布，然后从中采样

> **关键区别**：分类任务通常直接选最高概率（贪婪采样），而语言模型的贪婪采样会导致输出**单调乏味**——想象一个模型无论你问什么，都只用最常见的词回答。

### 1.2 Logits → Probabilities → Logprobs

模型内部的概率计算经过三个阶段：

```
Input → Neural Network → Logits → Softmax(+Temperature) → Probabilities → Log → Logprobs
```

#### Logits（原始分数）

神经网络对每个输入产生一个 **logit 向量**，其中每个 logit 对应词表中的一个 token。Logit 的特点：

- **不是概率**：logit 可以是负数，且不需要加和为 1
- **大小有意义**：更大的 logit 对应更高的概率
- **向量维度 = 词表大小**：如果词表有 100,000 个 token，logit 向量就有 100,000 维

#### Softmax（归一化为概率）

将 logit 转换为概率的标准方法是 **softmax 函数**。假设模型词表大小为 N，logit 向量为 [x₁, x₂, ..., xₙ]，第 i 个 token 的概率为：

$$p_i = \text{softmax}(x_i) = \frac{e^{x_i}}{\sum_j e^{x_j}}$$

Softmax 保证所有输出值非负且加和为 1，满足概率的基本性质。

#### Logprobs（对数概率）

许多模型提供商以 **logprobs**（log probabilities，对数概率）的形式返回概率，而非原始概率。原因是语言模型的词表可能有 100,000+ 个 token，大多数 token 的概率极小，直接用原始概率会遇到**下溢问题**（underflow）——数值太小，被机器舍入为 0。对数刻度能有效缓解这个问题。

> **实践**：截至写作时，许多模型提供商不暴露其模型的 logprobs，或者 logprobs API 功能有限。例如 OpenAI API 仅展示最可能的 20 个 token 的 logprobs。Anthropic 不暴露其模型的 logprobs。这可能是出于安全考虑——暴露的 logprobs 使他人更容易复制模型。

> **调试技巧**：查看模型对给定输入计算的概率是常用的调试手段。如果概率看起来是随机的，说明模型还没有学到有用的东西。

## 二、采样策略（Sampling Strategies）

合适的采样策略可以引导模型生成**更符合应用需求**的输出——例如一种策略让输出更有创造力，另一种让输出更可预测。你甚至可以设计自己的采样策略（前提是能访问模型的 logits）。

### 2.1 Temperature（温度）

#### 问题

如果直接按概率分布采样，常见 token（如 "red""green""purple"）概率最高，模型倾向于生成像五岁小孩一样的句子——"My favorite color is green"。低概率 token（如 "the"）几乎不会被选中，因而模型难以生成创意性表达，如 "My favorite color is the color of a still lake on a spring morning"。

#### 机制

Temperature 是在 softmax 之前对 logits 进行调整的常数。给定温度 T，第 i 个 token 的调整后 logit 为 `xᵢ / T`，然后再对调整后的 logits 应用 softmax。

#### 效果

| Temperature | 效果 | 适用场景 |
|---|---|---|
| **T → 0** | 概率分布**极度尖锐**，几乎总是选最高 logit 的 token（等效于 greedy sampling / argmax） | 需要**一致性**和**可重复性**的任务 |
| **T = 1** | 等同于不使用 temperature | 默认行为 |
| **T > 1** | 概率分布**趋于平坦**，高概率 token 的优势被削弱，低概率 token 有更多机会被选中 | 需要**创造力**和**多样性**的任务 |

#### 数值示例

假设模型有两个可能输出 A 和 B，logits 分别为 [1, 2]：

- **T = 1**（默认）：softmax 概率为 [0.27, 0.73]，B 被选中 73% 的时间
- **T = 0.5**：softmax 概率为 [0.12, 0.88]，B 被选中 88% 的时间（更确定）
- **T < 0.1**：B 几乎 100% 被选中

> **实践建议**：模型提供商通常将 temperature 限制在 0 到 2 之间。0.7 是创意类任务常用的推荐值，因为它在创造力和可预测性之间取得了较好平衡。但最终应该通过实验找到最适合你应用的值。

### 2.2 Top-k

#### 问题

Softmax 需要对词表中**所有** token 进行指数运算，包括两遍遍历：一遍计算 `∑ⱼ eˣʲ`，一遍对每个 token 计算 `eˣⁱ / ∑ⱼ eˣʲ`。对于拥有大词表的语言模型，这在计算上非常昂贵。

#### 机制

模型计算 logits 后，只保留 **top-k 个最大的 logit**，丢弃其余，然后仅在这 k 个 token 上执行 softmax 并采样。

#### 参数选择

- k 的典型范围为 **50 到 500**——远小于词表大小
- **较小的 k**：文本更可预测，但更受限于一小组常见词
- **较大的 k**：文本更多样，但可能引入不太相关的 token

#### 局限

k 是固定值，不随上下文变化。例如 "Do you like music? Answer with only yes or no." 只需考虑 2 个候选；而 "What's the meaning of life?" 应该考虑更多候选。固定的 k 无法适应这种变化。

### 2.3 Top-p（Nucleus Sampling）

#### 核心思想

Top-p（又称 **nucleus sampling**，核采样）提供了比 top-k 更**动态**的候选集选择方式。它不固定候选数量，而是固定**累积概率阈值**。

#### 机制

1. 将所有 token 按概率从高到低排序
2. 从概率最高的 token 开始，依次累加概率
3. 当累积概率达到阈值 p 时停止
4. 只从这些被选中的 token 中采样

#### 示例

假设 token 概率分布为：yes=60%, maybe=31%, no=8%, the=0.5%, ...

- **top-p = 0.90**：只考虑 "yes"（60%）和 "maybe"（31%），因为它们的累积概率 91% > 90%
- **top-p = 0.99**：考虑 "yes""maybe" 和 "no"，累积概率 99%

#### Top-p 的常用范围

语言模型中 top-p 的典型值为 **0.9 到 0.95**。

#### Top-p vs Top-k

| 维度 | Top-k | Top-p |
|---|---|---|
| 候选集大小 | 固定为 k | 动态，随上下文变化 |
| 是否减少 softmax 计算 | 是（只对 k 个 token 计算） | 不一定（仍需先计算所有概率） |
| 上下文适应性 | 差 | 好——聚焦于最相关的候选集 |

> **相关策略**：**min-p** 是另一种采样策略，通过设定一个 token 被考虑的**最低概率阈值**来筛选候选集。

### 2.4 停止条件（Stopping Condition）

自回归语言模型会逐 token 生成，理论上可以无限生成。需要设置**停止条件**来控制输出长度，原因有二：

1. **成本**：更长的输出序列消耗更多计算资源（API 按 token 收费）
2. **延迟**：更长的输出意味着用户等待更久

#### 三种常见方式

| 方式 | 说明 | 优点 | 缺点 |
|---|---|---|---|
| **最大 token 数** | 生成到固定数量的 token 后停止 | 简单 | 容易在句中被截断 |
| **停止 token / 停止词** | 遇到特定 token（如 end-of-sequence）时停止 | 输出更完整 | 需要模型学会在合适时机生成停止 token |
| **格式验证** | 持续生成直到输出符合某种格式要求（如有效 JSON） | 保证输出可用性 | 可能消耗更多资源 |

> **注意**：过早停止可能导致输出格式不完整。例如要求模型生成 JSON，提前停止可能导致缺少闭合括号，使 JSON 无法解析。

## 三、测试时计算（Test Time Compute）

### 3.1 核心思想

前面讨论的是模型如何采样**下一个 token**。Test time compute 关注的是模型如何采样**整个输出**。

核心策略很简单：**生成多个候选输出，然后从中选出最好的**。这种 "best of N" 策略可以显著提升输出质量，代价是更多的推理计算。

> **术语说明**：作者使用 "test time compute" 而非更直觉的名称，是为了与现有文献保持一致。在 AI 研究中，"test time" 通常指推理（inference），因为研究者主要在测试阶段做推理。但该技术同样适用于生产环境。其核心含义是：**你可以分配多少推理时计算来提升单次输出的质量**。

### 3.2 生成多个候选的策略

| 策略 | 说明 |
|---|---|
| **随机采样多次** | 对同一输入，独立生成 N 个输出，然后选最佳 |
| **Beam Search** | 在每个生成步保留 beam 个最有前途的候选序列，而不是独立生成完整输出。比随机采样更有策略性 |
| **多样化采样** | 调整采样变量（如 temperature、top-k）来生成不同风格的候选，增加输出多样性 |

> **多样性的价值**：如果用同一个模型生成多个候选，调整采样变量来增加输出多样性是好习惯。更多样的候选集更可能包含高质量输出。

### 3.3 从多个候选中选择最佳

生成多个候选后，如何选出"最好的"？有以下方法：

#### 方法一：基于概率选择

语言模型的每个输出是 token 序列，每个 token 都有模型计算的概率。输出的总概率是所有 token 概率的乘积：

```
p("I love food") = p("I") × p("love" | "I") × p("food" | "I", "love")
```

在对数刻度下更方便计算（乘法变加法）：

```
logprob("I love food") = logprob("I") + logprob("love" | "I") + logprob("food" | "I", "love")
```

> **长度偏差修正**：logprob 值在 0 和 1 之间取对数后为负数，因此较长序列的总 logprob 通常更低。为避免偏向短序列，可以用**平均 logprob**（总 logprob / 序列长度）来选择。OpenAI API 的 `best_of` 参数就使用此方法。

#### 方法二：使用奖励模型（Reward Model）/ 验证器（Verifier）

用一个独立的模型对每个候选输出打分，选得分最高的。这正是 Stitch Fix、Grab、Nextdoor 等公司采用的策略——生成多个输出，由奖励模型筛选最佳。

**验证器的威力**：
- OpenAI 训练了验证器来帮助模型筛选最佳数学题解答（Cobbe et al., 2021）
- 使用验证器带来的性能提升，**约等于将模型规模扩大 30 倍**——即一个 1 亿参数的模型 + 验证器，可以媲美 30 亿参数的无验证器模型
- DeepMind 进一步证明：**扩展 test time compute（推理时生成更多输出）可能比扩展模型参数更高效**（Snell et al., 2024）

#### 方法三：多数投票（Majority Voting）

对于有确定答案的任务特别有效。例如：
- 数学题：生成多个解，选**最常出现的最终答案**（Wang et al., 2023 称之为 self-consistency）
- 多选题：生成多个回答，选**最频繁的选项**。Google 评估 Gemini 在 MMLU 上的表现时，每题采样 32 个输出，取多数结果

#### 方法四：应用特定启发式

根据应用特性选择最佳输出：
- 如果应用偏好**短回答** → 选最短的候选
- 如果应用是 text-to-SQL → 持续生成直到产生**有效 SQL 查询**

### 3.4 Test Time Compute 的扩展性

OpenAI 的实验发现，增加采样数量可以提升性能，但**只到一定程度**：

- 在他们的实验中，**400 个输出**是性能拐点
- 超过 400 个后性能反而下降——可能因为大量采样增加了"欺骗验证器的对抗性输出"的概率
- 但斯坦福的 "Monkey Business" 实验（Brown et al., 2024）发现不同结论：从 1 到 10,000 个样本，解题成功率**对数线性增长**

> **现实考量**：在生产环境中，没有人会为每个输入采样 400 甚至 10,000 个候选——成本将是天文数字。

### 3.5 应用：克服延迟

Test time compute 的一个巧妙应用是**降低感知延迟**：对于某些需要长时间思考的查询（如 chain-of-thought），让模型并行生成多个响应，**展示最先完成的那个**。

## 四、结构化输出（Structured Outputs）

### 4.1 为什么需要结构化输出

在生产环境中，许多场景需要模型输出遵循特定格式：

| 场景 | 说明 | 示例 |
|---|---|---|
| **语义解析任务** | 将自然语言转换为结构化、机器可读的格式 | Text-to-SQL：用户输入 "过去 6 个月的月均收入" → 模型输出有效 SQL 查询 |
| **下游应用消费** | 模型输出将被其他程序解析和处理 | 邮件生成后需要以 `{"title": "...", "body": "..."}` 的 JSON 格式传给下游系统 |

> **对 Agentic Workflow 尤其重要**：在智能体工作流中，模型的输出经常作为工具调用的输入参数（详见 Chapter 6）。格式错误会直接导致工具调用失败。

### 4.2 结构化输出框架与 API

支持结构化输出的框架包括 **guidance**、**outlines**、**instructor** 和 **llama.cpp**。模型提供商也提供原生支持，例如 OpenAI 的 **JSON mode** 保证输出是有效 JSON 对象（但不保证内容正确性）。

> **注意**：JSON mode 保证格式有效，但如果生成在最大 token 长度处被截断，JSON 仍可能不完整（缺少闭合括号等）。

### 4.3 五种实现方式

可以在 AI 技术栈的不同层面引导模型生成结构化输出。前三种更像是"创可贴"，适用于模型已经基本能生成正确格式、只需小幅修正的情况；后两种是更深层的方案。

#### （1）Prompting（提示词引导）

最简单的方法——在提示词中明确指定输出格式。但模型不保证 100% 遵循指令，即使只有几个百分点的错误率，对某些应用也不可接受。

增强手段：用 AI 来**验证和/或修正**输出（AI-as-a-judge，详见 Chapter 3），但每个输出至少需要两次模型查询（一次生成 + 一次验证），成本和延迟翻倍。

#### （2）Post-processing（后处理）

模型倾向于犯**重复的、可预测的错误**。如果能识别出常见错误模式，可以写脚本自动修正。

> **实例**：LinkedIn 发现其 YAML 输出经常缺少闭合括号，编写了防御性 YAML 解析器，将正确输出比例从 90% 提升到 **99.99%**（Bottaro and Ramgopal, 2020）。

> LinkedIn 选择 YAML 而非 JSON 作为输出格式，因为 YAML 更简洁，需要的输出 token 更少，从而降低成本。

#### （3）Test Time Compute（测试时计算）

持续生成输出，直到某个输出符合预期格式。例如持续生成直到产生有效 JSON。

#### （4）Constrained Sampling（约束采样）

在采样过程中**直接过滤**不符合约束的 token：

```
Input → Model → Logits(全词表) → Filter(约束条件) → Logits(有效token) → Softmax → Probabilities
```

例如，如果约束是"输出只能是 [red, green, blue] 之一"，则在每一步只保留能组成这三个词的 token 的 logits。

**局限**：
- 需要为每种输出格式（JSON、YAML、regex、CSV 等）构建独立的语法规则
- 构建和集成语法到采样流程中**并不简单**
- 语法验证可能**增加生成延迟**（Brandon T. Willard, 2024）
- 有人认为，与其投入资源做约束采样，不如投入到让模型更好地遵循指令上

#### （5）Finetuning（微调）

在遵循目标格式的样本数据上微调模型，是**最有效、最通用**的方法。微调后的模型不保证 100% 输出正确格式，但比单纯 prompting 可靠得多。

对于分类等特定任务，可以在微调时修改模型架构——在基础模型后面添加一个**分类器头**（classifier head），确保模型只能输出预定义的类别。这种方法也被称为 **feature-based transfer**（基于特征的迁移学习，详见 Chapter 7）。

> **展望**：结构化输出技术本质上是因为模型还不够擅长遵循指令。随着模型能力提升，未来可能只需要极少的提示就能获得格式正确的输出，这些技术会逐渐变得不那么重要。

## 五、AI 的概率本质（The Probabilistic Nature of AI）

### 5.1 概率性 vs 确定性

AI 模型的采样机制使其输出具有**概率性**（probabilistic）——对同一个输入，模型可能给出不同的输出。这与**确定性**（deterministic）系统形成对比，后者对相同输入总是产生相同输出。

> **类比**：如果你问朋友"世界上最好的菜系是什么"——间隔一分钟问两次，朋友两次回答应该一样。但 AI 模型可能不一样：如果模型认为越南菜 70% 概率最好、意大利菜 30%，那么两次回答可能分别是"越南菜"和"意大利菜"。

这种概率性是 AI 的**双刃剑**：
- **优点**：赋予 AI 创造力——能够探索非常规路径，进行无限头脑风暴，生成前所未见的设计
- **缺点**：导致不一致性和幻觉——对需要事实准确性的应用是致命缺陷

> AI 工程的很大一部分工作，就是围绕这种概率本质**构建工作流**——让 AI 工程即使不能完全确定性，至少做到系统化。

### 5.2 不一致性（Inconsistency）

#### 两种表现形式

| 类型 | 说明 | 示例 |
|---|---|---|
| **相同输入，不同输出** | 对完全相同的 prompt，模型两次给出截然不同的回答 | 同一篇作文，ChatGPT 第一次打 3/5，第二次打 5/5 |
| **微小输入差异，巨大输出差异** | 对几乎相同的 prompt（如仅一个字母大小写不同），模型输出完全不同 | 比第一种更难处理 |

#### 缓解方案

| 方案 | 说明 | 效果 |
|---|---|---|
| **缓存（Caching）** | 缓存响应，相同问题直接返回缓存结果 | 解决"同输入不同输出"问题 |
| **固定采样变量** | 设定 temperature=0、固定 top-k/top-p、固定 seed | 有帮助，但**不能保证 100% 一致**——底层硬件的数值运算差异也会影响输出 |
| **精心设计 Prompt** | 通过更精确的 prompt 引导模型趋向一致的输出（Chapter 5） | 缓解"微小输入差异"问题 |
| **记忆系统（Memory）** | 让模型记住之前的回答（Chapter 6） | 提升跨会话一致性 |

> **底层限制**：即使固定了所有采样变量，也不能保证模型输出 100% 一致。不同的硬件对相同的浮点运算可能产生略微不同的结果，从而导致输出差异。如果使用 OpenAI 或 Google 等 API，你甚至无法控制底层硬件。

### 5.3 幻觉（Hallucination）

#### 定义与严重性

**幻觉**是指模型生成**不基于事实**的回答。对于依赖事实准确性的任务（如医疗建议、法律研究），幻觉是致命的。

> **真实案例**：2023 年 6 月，一家律师事务所因向法庭提交了 ChatGPT 编造的虚假法律案例而被罚款。他们使用 ChatGPT 准备案件材料，却不知道 ChatGPT 有幻觉倾向。

#### 幻觉的历史

幻觉并非 LLM 时代才出现的现象——早在 2016 年，文本生成领域就已经提到了幻觉（Goyal et al., 2016）。在 NLG（自然语言生成）领域，检测和度量幻觉一直是核心研究课题。

#### 为什么会发生幻觉？两种假说

**假说一：自我妄想（Self-Delusion）**

最先由 DeepMind 的 Ortega et al.（2021）提出。核心观点：语言模型**无法区分**输入数据（用户的 prompt）和自己生成的数据。

机制：
1. 用户问："Who's Chip Huyen?"
2. 模型生成第一句："Chip Huyen is an architect."（错误）
3. 模型在生成后续内容时，将之前的输出视为**既定事实**
4. 基于这个错误前提，模型进一步展开，生成更多离谱的"事实"

> **实例**：Zhang et al.（2023）称之为 **snowballing hallucinations**（雪球式幻觉）——一个初始错误假设可以让模型持续在错误方向上编造，甚至在本可正确回答的问题上犯错。

> **视觉模型也有此问题**：LLaVA-v1.5-7B 被要求从洗发水瓶标签上读取成分，但模型错误地认定瓶子是牛奶瓶，进而编造了以 "Milk" 开头的成分列表。

DeepMind 论文提出两种缓解技术：
- 让模型学会区分用户提供的 prompt（世界的**观察**）和模型自己生成的 token（模型的**行为**）——利用强化学习的框架
- 在训练数据中包含**反事实信号**（counterfactual signals），通过监督学习让模型识别事实

**假说二：内部知识不匹配（Mismatched Internal Knowledge）**

由 OpenAI 研究者 Leo Gao 首先提出，John Schulman（OpenAI 联合创始人）在 2023 年 UC Berkeley 演讲中进一步阐述。

核心观点：在 SFT（监督微调）阶段，模型被训练去模仿标注者写的回答。但标注者使用了**他们自己拥有而模型并不拥有的知识**——这等于在教模型**编造**。

> **如果这个假说正确**——即 LLM 知道自己知道什么——那么幻觉可以通过**强制模型只基于其内部知识**来回答而解决。Schulman 提出两个方向：
> 1. **验证**：要求模型为每个回答检索其所依据的来源
> 2. **强化学习**：训练一个更好的奖励函数，对编造行为施加更大惩罚

#### 两种假说的关系

| 假说 | 关注点 | 原因 |
|---|---|---|
| **自我妄想** | 自监督学习如何导致幻觉 | 模型把自己生成的内容当作事实 |
| **内部知识不匹配** | 监督学习如何导致幻觉 | 模型被教导回答它不知道的事情 |

两种假说**互补而非矛盾**——幻觉可能同时由这两种机制造成。

#### RLHF 与幻觉的关系（2026 更新）

InstructGPT 论文（Ouyang et al., 2022）的数据表明，RLHF **反而加剧了幻觉**——尽管 RLHF 改善了其他方面（如毒性、恰当性），但在幻觉指标上，RLHF 模型不如仅用 SFT 的模型。总体而言，人类标注者仍然更偏好 RLHF 模型。

截至 2026 年，更新的 RLHF/DPO 技术和更精细的奖励建模已经在缓解这个问题上取得进展，但幻觉仍然是 AI 工程中最大的挑战之一。

#### 缓解幻觉的实践建议

- 在 prompt 中加入"尽可能如实回答，如果不确定就说不知道"
- 要求模型给出**简洁回答**——生成的 token 越少，编造的机会越小
- 利用 Prompting 和上下文构建技术（Chapters 5 & 6）引导事实性输出
- 检测和度量幻觉的方法详见 Chapter 4

## 六、全章总结（Summary）

本章讨论了构建基础模型的核心设计决策。以下是关键要点回顾：

### 训练数据

- 模型性能的关键影响因素是**训练数据**。大模型需要海量数据，获取成本高昂
- 模型提供商往往"有什么用什么"，导致模型在训练数据中已有的任务上表现好，但未必在你关心的任务上表现好
- 针对**特定语言**和**特定领域**策划数据集，是训练专用模型的关键

### 模型架构与规模

- 语言基础模型的主导架构是 **Transformer**
- 模型规模通过三个数字衡量：**参数量**、**训练 token 数**、**训练所需 FLOPs**
- **Scaling law** 帮助在给定计算预算下确定最优的参数量和 token 数
- 目前，扩大模型规模通常能带来更好的性能——但这能持续多久？

### 后训练（Post-Training）

- 预训练数据质量低和自监督学习可能导致模型输出**不符合用户期望**
- 后训练通过两个步骤解决：**监督微调（SFT）** 和 **偏好微调（Preference Finetuning）**
- 人类偏好多样且难以用单一数学公式捕获，现有方案远非完美

### 采样（Sampling）

- 采样是模型**生成输出 token 的过程**，使 AI 模型具有概率性
- 这种概率性使模型适合**创意任务**，但也带来了**不一致性**和**幻觉**
- AI 工程需要围绕这种概率本质构建工作流——如果不能完全确定性，至少要做到**系统化**
- 建立**可靠的评估管道**来检测故障和意外变化，是系统化 AI 工程的第一步——本书后续两章专门讨论评估

## 七、关键概念速查表

| 概念 | 一句话解释 |
|---|---|
| **Logits** | 神经网络输出的原始分数向量，每个值对应词表中一个 token，经 softmax 转换为概率 |
| **Softmax** | 将 logits 归一化为非负且加和为 1 的概率分布的函数 |
| **Logprobs** | 对数概率，用对数刻度表示概率以避免数值下溢，是模型调试和输出选择的关键工具 |
| **Temperature** | 在 softmax 前除以 logits 的常数，控制输出的创造力（高温）vs 确定性（低温） |
| **Top-k** | 只保留概率最高的 k 个 token 进行采样，减少计算量并控制多样性 |
| **Top-p（Nucleus Sampling）** | 按累积概率阈值动态选择候选 token 集，比 top-k 更适应上下文 |
| **Greedy Sampling** | 始终选择概率最高的 token，等价于 temperature=0 |
| **Test Time Compute** | 推理时生成多个候选输出并选最佳的策略，可等效于数十倍模型规模扩展 |
| **Beam Search** | 在每步保留多个最有前途的候选序列，而非独立生成完整输出 |
| **Constrained Sampling** | 在采样时过滤不满足约束的 token，保证输出符合指定格式 |
| **Self-Delusion** | 幻觉假说之一：模型把自己生成的内容当作事实，导致雪球式错误 |
| **Snowballing Hallucination** | 初始错误假设引发的连锁幻觉，模型在错误方向上越走越远 |
| **Inconsistency** | 相同或相似输入产生截然不同输出的现象，源于采样的随机性 |
| **Self-Consistency** | 生成多个输出并选最常见答案的策略，用于提升确定性任务的准确率 |
| **Structured Outputs** | 确保模型输出遵循特定格式（JSON、SQL、YAML 等）的各种技术 |
