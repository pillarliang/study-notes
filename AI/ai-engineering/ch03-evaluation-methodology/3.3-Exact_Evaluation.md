# Exact Evaluation（精确评估）

> **核心定位**：模型评估分为**精确评估（Exact Evaluation）和主观评估（Subjective Evaluation）**。精确评估产出无歧义的判断结果——同一输入、同一标准，任何人得到的结论一致。主观评估（如 AI as a Judge）的结果则取决于评审模型和 prompt 的选择。

---

## 精确 vs 主观的分界


| 维度        | 精确评估              | 主观评估             |
| --------- | ----------------- | ---------------- |
| **判断确定性** | 无歧义，结果确定          | 结果依赖评审者（人或 AI）   |
| **典型例子**  | 选择题答对/答错；代码能否通过测试 | 作文评分；AI Judge 打分 |
| **可重复性**  | 完全可重复             | 同一评审者两次打分可能不同    |


> 精确评估并非只适用于封闭式任务。本节聚焦的是**开放式生成（open-ended response）** 场景下的精确评估方法——因为封闭式任务（如分类）的评估方法已经成熟。

---

## 1. 功能正确性（Functional Correctness）

### 第一性原理

功能正确性衡量的是：**系统是否完成了它本应完成的功能**。这是评估任何应用的终极指标，因为它直接回答"你的系统到底行不行"。

### 适用场景

功能正确性适用于**结果可被客观验证**的任务：

- **代码生成**：生成的代码能否通过单元测试？（执行准确率 / Execution Accuracy）
- **游戏 Bot**：Tetris bot 得了多少分？
- **任务型 AI**：让 AI 订餐厅，订上了没？让 AI 调度工作负载以节省能耗，节省了多少？

### 代码生成评估详解

代码生成是功能正确性可自动化评估的典型场景。评估流程：

1. **问题**：给模型一个编程任务（如实现 `gcd(num1, num2)`）
2. **生成**：模型生成 k 个代码样本
3. **测试**：每个样本在 Python 解释器中执行，用 assert 语句验证输出
4. **指标**：**pass@k** —— 在 k 个样本中，只要有 1 个通过所有测试用例，该题即"解决"

$$\text{pass@k} = \frac{\text{被解决的问题数}}{\text{总问题数}}$$

- k 越大，模型有更多机会解决每道题，因此 pass@1 ≤ pass@3 ≤ pass@10
- 代表性 Benchmark：**HumanEval**（OpenAI）、**MBPP**（Google）、**Spider** / **BIRD-SQL** / **WikiSQL**（Text-to-SQL）

### 局限

功能正确性是最理想的指标，但**不总是可以自动化测量**。很多复杂任务虽然有可衡量的目标，但 AI 只完成了解决方案的一部分——评估"部分解决方案"往往比评估"最终结果"更难。就像评价一个人的棋力：评估整盘棋的输赢容易，评估单步棋的好坏很难。

## 2. 基于参考数据的相似度度量（Similarity Measurements Against Reference Data）

### 第一性原理

当任务无法用功能正确性自动评估时，退而求其次：**将模型输出与"标准答案"（参考数据）做比较**。生成的响应越接近参考响应，就被认为越好。

### 参考数据的性质

- 格式：`(input, reference_responses)`，一个输入可以有多个参考响应
- 别名：ground truth、canonical response
- 来源：人工标注（昂贵但高质量）或 AI 生成（便宜但需审核）
- **以人类表现为金标准**：用人工标注数据作为参考，本质上是以人类表现衡量 AI 表现

### 四种比较方式概览


| 方式        | 描述                | 得分类型          |
| --------- | ----------------- | ------------- |
| 评审者判断     | 人或 AI 判断两段文本是否相同  | 依赖评审者         |
| **精确匹配**  | 生成响应是否与参考响应完全一致   | 二值（匹配/不匹配）    |
| **词汇相似度** | 生成响应在字面上多大程度与参考重叠 | 连续值（如 0–1）    |
| **语义相似度** | 生成响应在含义上多接近参考     | 连续值（如 -1 到 1） |


> **相似度度量的通用性**：除了评估生成质量，相似度度量还广泛用于检索与搜索、排序、聚类、异常检测、数据去重等场景。

---

### 2.1 精确匹配（Exact Match）

**定义**：生成的响应与某个参考响应**完全一致**即为匹配。

**适用范围**：期望**短且唯一**答案的任务——

- "2 + 3 = ?" → "5"
- "第一位获得诺贝尔奖的女性是谁？" → "Marie Curie"
- "我的账户余额是多少？" → "¥1,234.56"

**变体：包含匹配（Contains Match）**

接受任何"包含"参考答案的输出。例如 "2+3=?" 的参考答案为 "5"，则 "The answer is 5" 也算正确。

**缺陷**：包含匹配可能误判。问 "Anne Frank 出生于哪一年？"（正确答案 1929），模型输出 "September 12, 1929" 包含了 "1929"，但整个回答是错误的。

**根本局限**：对于复杂任务，精确匹配几乎不可用。以翻译 "Comment ça va?" 为例，合法英文翻译有无数种（"How are you?" / "How is everything?" / "How is it going?" …），不可能穷举所有正确响应。

---

### 2.2 词汇相似度（Lexical Similarity）

#### 核心思想

衡量两段文本在**字面形式上的重叠程度**——将文本拆分为 token，计算共有 token 的比例。

#### 方法一：模糊匹配（Fuzzy Matching / Approximate String Matching）

通过**编辑距离（Edit Distance）** 衡量：将一段文本转换为另一段文本需要多少次编辑操作。

三种基本编辑操作：


| 操作               | 示例             |
| ---------------- | -------------- |
| 删除（Deletion）     | "brad" → "bad" |
| 插入（Insertion）    | "bad" → "bard" |
| 替换（Substitution） | "bad" → "bed"  |


部分匹配器还支持**置换（Transposition）**：交换两个相邻字符（"mats" → "mast"），也可视为一次删除 + 一次插入。

编辑距离越小 → 两段文本越相似。

#### 方法二：N-gram 相似度

不只比较单个 token，而是比较**连续 token 序列（n-gram）** 的重叠：

- **Unigram (1-gram)**：单个 token
- **Bigram (2-gram)**：两个连续 token。例如 "My cats scare the mice" 的 bigram 为 {"my cats", "cats scare", "scare the", "the mice"}

计算方式：参考响应中的 n-gram 有多少也出现在生成响应中。

#### 常用指标


| 指标           | 典型用途               |
| ------------ | ------------------ |
| **BLEU**     | 机器翻译（衡量 precision） |
| **ROUGE**    | 文本摘要（衡量 recall）    |
| **METEOR++** | 翻译（改进了 BLEU 的一些局限） |
| **TER**      | 翻译（基于编辑距离）         |
| **CIDEr**    | 图像描述               |


> 基础模型兴起后，使用词汇相似度的 benchmark 越来越少。仍在使用的包括 WMT、COCO Captions、GEMv2。

#### 两大缺陷

1. **参考数据瓶颈**：如果参考集不包含与模型输出"长得像"的正确答案，好的输出也会被低估。Adept 发现其模型 Fuyu 在某些 benchmark 上得分低，不是因为输出错误，而是因为参考数据缺少了某些正确答案。此外参考数据本身也可能有误（WMT 2023 发现了大量错误的参考翻译）。
2. **词汇相似度高 ≠ 质量高**：OpenAI 在 HumanEval 上发现，正确代码和错误代码的 BLEU 分数相近——优化 BLEU 并不等同于优化功能正确性。

---

### 2.3 语义相似度（Semantic Similarity）

#### 核心思想

词汇相似度衡量"看起来像不像"，语义相似度衡量**"意思一不一样"**。

- "What's up?" 和 "How are you?" 词汇完全不同，但语义接近
- "Let's eat, grandma" 和 "Let's eat grandma" 词汇几乎一样，但意思天差地别

#### 实现方式

1. **Embedding**：用模型将文本转为数值向量（embedding），使含义相近的文本在向量空间中距离更近
2. **余弦相似度（Cosine Similarity）**：计算两个 embedding 向量的夹角余弦

$$\text{cosinesimilarity}(A, B) = \frac{A \cdot B}{A \cdot B}$$

- 完全相同的 embedding → 相似度 = 1
- 完全相反的 embedding → 相似度 = -1

#### 常用指标


| 指标             | 特点                   |
| -------------- | -------------------- |
| **BERTScore**  | 使用 BERT 生成 embedding |
| **MoverScore** | 使用混合算法生成 embedding   |


#### 优势与局限


| 优势                | 局限                                             |
| ----------------- | ---------------------------------------------- |
| 不需要像词汇相似度那样穷举参考响应 | 依赖 embedding 模型的质量——同义文本如果 embedding 不好，相似度也会低 |
| 能捕捉同义表达           | embedding 模型的运行需要非平凡的计算资源和时间                   |


> 虽然语义相似度被归入"精确评估"，但严格来说有一定主观性——不同 embedding 算法会产生不同 embedding。不过，给定两个确定的 embedding，其相似度分数是精确计算的。

## 3. Embedding 简介

> 语义相似度的核心在于 embedding，embedding 也是向量搜索（Chapter 6）和数据去重（Chapter 8）的基础。

### 什么是 Embedding

- 计算机处理的是数字，所以需要将数据转换为数值表示
- **Embedding = 用数值向量捕捉原始数据的语义含义**
- 例如 "the cat sits on a mat" → `[0.11, 0.02, 0.54]`（实际维度通常在 100–10,000 之间）

### 常见 Embedding 模型及维度


| 模型                                    | 维度          |
| ------------------------------------- | ----------- |
| BERT base / large                     | 768 / 1024  |
| OpenAI CLIP（图像/文本）                    | 512 / 512   |
| OpenAI text-embedding-3-small / large | 1536 / 3072 |
| Cohere embed-english-v3.0 / light     | 1024 / 384  |


### 好的 Embedding 的标准

**含义越相近的文本，embedding 之间的距离应越小。**

"the cat sits on a mat" 的 embedding 应比 "AI research is super fun" 更接近 "the dog plays on the grass"。

也可以通过 embedding 在下游任务（分类、主题建模、推荐、RAG）上的效果来评估其质量。代表性 benchmark：**MTEB**（Massive Text Embedding Benchmark）。

### 多模态联合 Embedding

Embedding 不限于文本。任何数据都可以有 embedding 表示。

**跨模态联合 Embedding 空间（Joint Embedding Space）** 是前沿方向——将不同模态的数据映射到同一个向量空间：


| 模型                                  | 支持模态                |
| ----------------------------------- | ------------------- |
| **CLIP**（Radford et al., 2021）      | 文本 + 图像             |
| **ULIP**（Xue et al., 2022）          | 文本 + 图像 + 3D 点云     |
| **ImageBind**（Girdhar et al., 2023） | 文本 + 图像 + 音频等 6 种模态 |


**CLIP 训练原理**：

1. 输入：`(image, text)` 配对数据
2. 分别用文本编码器和图像编码器生成各自的 embedding
3. 投影到联合 embedding 空间
4. 训练目标：让配对的图像 embedding 和文本 embedding 在联合空间中尽可能接近

联合 embedding 空间使得**跨模态比较与检索**成为可能——例如用文本搜索图像：给定文本"一个钓鱼的人"，找到最接近的图像 embedding。

## 总结：三种精确评估方法的对比


| 方法        | 适用场景                 | 优势                | 劣势                    |
| --------- | -------------------- | ----------------- | --------------------- |
| **功能正确性** | 结果可客观验证（代码、游戏、可量化任务） | 最终极的指标；直接回答"行不行"  | 不总能自动化；评估部分解决方案困难     |
| **精确匹配**  | 短且唯一的答案              | 简单直接              | 长文本/开放式任务不适用          |
| **词汇相似度** | 翻译、摘要等有参考答案的任务       | 不依赖模型；计算高效        | 需要全面的参考集；相似度高 ≠ 质量高   |
| **语义相似度** | 同义表达多、参考集不全面的场景      | 捕捉语义而非字面；对参考集要求更低 | 依赖 embedding 质量；有计算开销 |


> **尽管 AI as a Judge 使用灵活，但手工设计的相似度度量因其精确性在工业界仍被广泛使用。** 精确评估和主观评估应相互补充，而非替代。
