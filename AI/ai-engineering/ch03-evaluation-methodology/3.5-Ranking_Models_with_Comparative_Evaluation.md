# Ranking Models with Comparative Evaluation（用比较评估排名模型）

> **核心定位**：评估模型的最终目的往往不是获得一个分数，而是**排出优劣顺序**——哪个模型最适合我的应用？排名有两条路径：逐个打分再排序（Pointwise）vs 直接让模型两两对决再算排名（Comparative）。本节聚焦后者。

---

## 两种排名范式

| 维度 | 逐点评估（Pointwise） | 比较评估（Comparative） |
| --- | --- | --- |
| **过程** | 独立评估每个模型 → 打分 → 按分数排序 | 模型两两对决 → 评审选出赢家 → 用排名算法算出总排序 |
| **类比** | 舞蹈比赛：每位选手独立表演，评委打分，按分数排名 | 舞蹈比赛：选手并排 battle，评委选出更好的那位 |
| **核心前提** | 能为每个模型给出有意义的绝对分数 | 只需判断"A vs B 谁更好" |
| **难点所在** | 设计 benchmark 和评分标准 | 信号收集和排名计算都有挑战 |

### 为什么比较评估更适合主观任务

从第一性原理出发：**判断"哪个更好"比给出绝对分数更容易。**

- 你可以轻松判断两首歌哪首更好听，但很难给每首歌一个精确的分数
- 当模型能力超越人类时，人类可能无法给出有意义的绝对分数，但仍能**感知差异**。Llama 2 论文指出：当模型写作能力超过最好的人类标注员时，人类仍能在两个回答之间做出有效比较

## 比较评估的工作流程

### 基本流程

```
1. 对每个请求，选择两个（或多个）模型生成响应
2. 评审者（人或 AI）选出赢家（可允许平局）
3. 每次对决称为一个 match
4. 积累大量 match 结果后，用排名算法计算总排名
```

### Win Rate（胜率）

模型 A 对模型 B 的胜率 = A 在所有 A vs B 的 match 中获胜的比例。

两个模型时排名很简单——赢得多的排高。但当模型数量增多时，问题变得复杂。例如下表中 5 个模型的胜率矩阵，很难直接看出排名：

| 模型对 | A >> B 胜率 |
| --- | --- |
| Model 1 vs Model 2 | 90% |
| Model 1 vs Model 3 | 40% |
| Model 1 vs Model 4 | 15% |
| Model 1 vs Model 5 | 10% |
| Model 2 vs Model 3 | 60% |
| Model 2 vs Model 4 | 80% |
| Model 2 vs Model 5 | 80% |
| Model 3 vs Model 4 | 70% |
| Model 3 vs Model 5 | 10% |
| Model 4 vs Model 5 | 20% |

此时需要**排名算法**将胜率矩阵转化为一个全局排名。

### 排名算法

这些算法并非 AI 领域发明，而是借鉴了已有近百年历史的体育和游戏排名系统：

| 算法 | 来源 | 说明 |
| --- | --- | --- |
| **Elo** | 国际象棋（1960s） | 经典排名系统；对评审者和 prompt 的顺序敏感 |
| **Bradley–Terry** | 统计学（1952） | LMSYS Chatbot Arena 当前使用的算法；比 Elo 更稳定 |
| **TrueSkill** | Xbox Live（Microsoft） | 支持多人比赛和团队对抗 |

> LMSYS Chatbot Arena 最初用 Elo，后因发现 Elo 对评审者和 prompt 顺序敏感而换为 Bradley–Terry。为兼容，其输出分数经过缩放处理，使其"看起来像" Elo 分数。

### 排名正确性的判定

一个排名是正确的，当且仅当：**对于任意一对模型，排名更高的模型在未来 match 中更可能获胜。**

换言之，模型排名本质上是一个**预测问题**——用历史对决结果预测未来对决结果。不同排名算法可能产生不同排名，没有客观的"正确排名"，只有**预测能力更强的排名**。

## 行业实践

- **Anthropic**（2021）：首次在 AI 领域使用比较评估来排名不同模型
- **LMSYS Chatbot Arena**：最知名的公开比较排行榜，由社区用户投票驱动
- **ChatGPT**：偶尔向用户并排展示两个输出，请用户选择更好的。这些输出可能来自不同模型，也可能是同一模型不同采样参数的结果
- 许多模型提供商在生产环境中使用比较评估来持续评估和改进模型

### 比较评估 vs A/B 测试

| 维度 | 比较评估 | A/B 测试 |
| --- | --- | --- |
| 用户看到的 | **同时看到**多个模型的输出 | **只看到**一个模型的输出 |
| 评估方式 | 直接对比，选出更好的 | 通过整体指标（转化率、满意度等）间接比较 |

## 三大挑战

### 1. 可扩展性瓶颈（Scalability Bottlenecks）

**根本问题**：模型对的数量随模型数**二次增长**。

- $n$ 个模型 → $\frac{n(n-1)}{2}$ 个模型对
- LMSYS 2024 年 1 月：57 个模型、244,000 次比较 → 平均每个模型对仅 153 次比较（考虑到 1,596 个模型对），对于基础模型的广泛任务范围来说远远不够

**缓解：传递性假设（Transitivity）**

排名算法通常假设传递性：如果 A > B 且 B > C，则 A > C。这意味着不需要直接比较所有模型对。

**传递性的风险**：
- 多篇论文质疑传递性在 AI 模型评估中是否成立（Boubdir et al.; Balduzzi et al.; Munos et al.）
- 人类偏好不一定满足传递性
- 不同模型对由不同评审者、不同 prompt 评估，进一步破坏传递性

**新模型评估问题**：

- Pointwise：新模型只需独立评估即可
- Comparative：新模型必须与已有模型对比，且可能改变已有模型的排名

**私有模型评估困难**：如果你用内部数据训练了私有模型，想与公开模型对比，要么自建排行榜，要么付费让公开排行榜做私有评估。

**更高效的匹配策略**：不必让所有模型对均匀比较。一旦对某个模型对的优劣关系足够确信，就可以停止对比。高效的匹配算法应优先安排**能最大程度减少整体排名不确定性**的 match。

### 2. 缺乏标准化与质量控制（Lack of Standardization and Quality Control）

以 LMSYS Chatbot Arena 的众包模式为例，任何人都可以提交 prompt、获得两个匿名模型的回答、投票选出更好的（投票后才揭示模型名称）。

**好处**：捕捉多样化信号；匿名机制使博弈较难。

**问题一：Prompt 质量无法保证**

- 没有"什么算更好的回答"的标准——志愿者不太可能去核实事实，可能偏好"听起来好"但事实错误的回答
- 不同用户偏好不同：有人喜欢礼貌温和的回答，有人喜欢无过滤的回答。用户可能因模型拒绝不当请求而给差评，但开发者可能希望模型拒绝
- 恶意用户可能故意选择有害回答，污染排名

**问题二：Prompt 不反映真实使用场景**

- 用户在评估平台上的 prompt 与实际工作中的 prompt 差异大
- LMSYS 2023 年发布的 33,000 条 prompt 中：180 条是 "hello" / "hi"（占 0.55%），还不算 "hello!" / "hola" / "hey" 等变体
- 大量脑筋急转弯类问题（如 "X has 3 sisters, each has a brother. How many brothers does X have?" 被问了 44 次）
- 简单 prompt 无法区分模型能力差异，反而稀释排名的区分度

**问题三：不支持高级使用模式**

如果排行榜不支持 RAG（用内部文档增强上下文）等场景，其排名就无法反映模型在你的 RAG 系统中的表现。生成能力和检索能力是两回事。

**缓解方案对比**：

| 方案 | 做法 | 优势 | 劣势 |
| --- | --- | --- | --- |
| **限定 prompt 集** | 只允许预定义 prompt | 标准化 | 无法覆盖多样化场景 |
| **过滤 hard prompt** | LMSYS 方式：允许任意 prompt，用模型过滤出"难 prompt"，仅用难 prompt 排名 | 兼顾多样性和质量 | 依赖过滤模型的质量 |
| **训练评审者** | Scale 方式：用训练过的专业评审者 | 判断质量高 | 昂贵；比较数量大幅减少 |
| **融入产品** | 在产品中让用户选择（如代码编辑器中推荐两段代码） | 最贴近真实场景 | 用户可能不具备判断能力；可能随意点击引入噪声 |
| **AI 替代人类评审** | 用 AI 做比较判断 | 可能比随机网民更可靠 | 不如训练过的人类专家 |

### 3. 比较信号无法转换为绝对性能（Comparative → Absolute Performance）

**根本问题**：比较评估只告诉你"B 比 A 好"，但不告诉你"B 到底好不好"，也不告诉你"B 是否对你够用"。

B 胜过 A 可能是以下任一情况：

1. B 好、A 差
2. A 和 B 都差
3. A 和 B 都好

**实际决策困难**：

假设模型 A 用于客服，能解决 70% 的工单。模型 B 对 A 的胜率为 51%。这 51% 的胜率意味着 B 能解决多少工单？**无法推算。** 实践中，有人发现 1% 的胜率差异在某些应用中带来巨大提升，在另一些应用中几乎无感。

**成本决策无法支撑**：

如果 B 的成本是 A 的两倍，仅凭胜率无法判断性能提升是否值得加倍成本。比较评估缺乏做**成本-收益分析**所需的绝对性能信息。

## 比较评估的未来

尽管有上述局限，比较评估有几个不可替代的优势：

| 优势 | 说明 |
| --- | --- |
| **判断门槛低** | 比较"谁更好"比给出绝对分数容易得多，尤其当模型能力超越人类时 |
| **不会饱和** | 传统 benchmark 会被模型刷满分而失效（GLUE 一年饱和、MMLU 被 MMLU-Pro 替代）；比较评估只要有更强模型出现就永远有效 |
| **难以博弈** | 没有参考数据可以提前训练，不像 benchmark 可能被数据污染 |
| **捕捉真实偏好** | 直接衡量人类偏好，不需要将偏好抽象为具体分数 |
| **减轻 benchmark 军备竞赛** | 不需要持续创建新 benchmark 来跟上模型能力的扩展 |

### 与后训练对齐的共生关系

比较评估和后训练对齐（RLHF）都依赖**偏好信号**，且偏好信号获取昂贵。这推动了**偏好模型（Preference Model）** 的发展——用专用 AI 预测人类偏好，既服务于评估，也服务于训练。

### 最佳使用方式

比较评估不是独立方案，而是评估体系的**补充组件**：

- **离线评估**：作为 benchmark 评估的有力补充
- **在线评估**：作为 A/B 测试的互补手段
- 需要与 pointwise 评估结合，才能获得绝对性能的判断

## 总结

```
                                ┌─ 逐点评估：独立打分 → 排序
    模型排名两条路径 ─────────────┤
                                └─ 比较评估：两两对决 → 排名算法 → 总排序

                                ┌─ 可扩展性：模型对二次增长；传递性假设未必成立
    比较评估三大挑战 ─────────────┼─ 标准化：prompt 质量参差、不反映真实场景、可被污染
                                └─ 相对 vs 绝对："B 比 A 好"≠"B 够好"

                                ┌─ Elo（国际象棋）
    排名算法 ───────────────────┼─ Bradley-Terry（LMSYS 当前方案）
                                └─ TrueSkill（Xbox Live）
```

### 核心原则

1. **排名是预测问题**——没有"正确排名"，只有预测未来对决更准的排名
2. **比较评估只给出相对信号**，需要其他评估方法补充绝对性能判断
3. **偏好投票只适用于投票者有判断能力的场景**——让用户选不懂的问题的答案不仅无用，还产生错误信号
4. **不是所有问题都应该用偏好来回答**——事实性问题应以正确性而非偏好来评判
5. 比较评估最适合作为**AI 作为助手（而非专家）** 场景下的评估手段
