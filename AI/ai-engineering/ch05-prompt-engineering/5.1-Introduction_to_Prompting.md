# Introduction to Prompting（Prompting 入门）

> **核心定位**：Prompt Engineering 的本质是**人与 AI 的沟通**——任何人都能沟通，但并非任何人都能高效沟通。本节从 prompt 的基本结构出发，逐层展开 in-context learning、system/user prompt 分工、chat template 机制、以及上下文长度与利用效率等核心概念，建立 prompt engineering 的第一性原理框架。

---

## Prompt 的本质与组成

### 什么是 Prompt

Prompt 是给模型的一条**指令**，用于驱动模型完成任务。任务的复杂度跨度极大：从回答"谁发明了零？"到"分析竞品并搭建网站"。

### 三个核心组成部分

一个 prompt 通常由以下一个或多个部分构成：

| 组成部分 | 作用 | 示例 |
|---|---|---|
| **Task Description** | 告诉模型要做什么，包括角色设定和输出格式 | "从以下文本中提取所有人名，以 JSON 列表输出" |
| **Example(s)** | 通过示例消除歧义，展示期望的行为模式 | 给出毒性/非毒性文本的标注样例 |
| **The Task** | 模型需要处理的具体输入 | 待回答的问题、待总结的文章 |

### Prompt Engineering 的第一性原理

Prompt engineering 不是无脑调词，而是**结构化的人机沟通工程**：

- **前提**：模型必须具备**指令遵循能力**——如果模型本身无法理解指令，再好的 prompt 也无效
- **类比**：就像与人沟通一样，清晰的指令 + 充分的上下文 + 恰当的示例 = 高效沟通
- **工程化要求**：prompt 实验应当与任何 ML 实验一样，具备系统化的实验设计和评估流程

> OpenAI 一位研究经理的原话："问题不在于 prompt engineering——它确实是一项真实且有用的技能。问题在于，有些人**只会** prompt engineering。"要构建生产级 AI 应用，还需要统计学、工程和经典 ML 知识来支撑实验追踪、评估和数据管理。

## 指令遵循能力与 Prompt 鲁棒性

### 模型能力决定 Prompting 的天花板

Prompting 生效的**必要条件**是模型能够遵循指令。如果模型的指令遵循能力差，无论 prompt 写得多好都无济于事。模型的指令遵循能力评估详见 Chapter 4。

### 鲁棒性：对 Prompt 微扰的敏感程度

Prompt 鲁棒性衡量的是：当 prompt 发生微小变化时（如将 "5" 改为 "five"、增加换行、改变大小写），模型的输出是否会**剧烈变化**。

- 鲁棒性与模型整体能力**强正相关**——越强的模型越能理解 "5" 和 "five" 是同一回事
- 模型越强 → 鲁棒性越高 → 需要的 prompt 微调（fiddling）越少 → **选择更强的模型本身就能节省 prompt engineering 的时间**
- 可通过随机扰动 prompt 来度量鲁棒性

### 位置偏好

不同模型对 task description 的位置有不同偏好：

| 模型 | 偏好位置 |
|---|---|
| GPT-4 | prompt **开头** |
| Llama 3 | prompt **结尾** |

实际使用时应针对目标模型实验不同的 prompt 结构。

## In-Context Learning：Zero-Shot 与 Few-Shot

### 核心概念

**In-context learning**（上下文学习）指模型仅通过 prompt 中的示例就能学会新行为，**无需更新权重**。这一术语由 Brown et al. (2020) 在 GPT-3 论文 "Language Models Are Few-shot Learners" 中提出。

传统范式下，模型的行为在训练阶段（pre-training / post-training / finetuning）通过权重更新来学习。GPT-3 论文的突破在于：一个仅被训练做 next token prediction 的模型，可以通过 prompt 中的示例学会翻译、阅读理解、简单数学甚至回答 SAT 题目。

### 术语定义

| 术语 | 含义 |
|---|---|
| **Shot** | Prompt 中的一个示例 |
| **Zero-shot** | 不提供示例，直接给出任务 |
| **Few-shot** / **N-shot** | 提供 N 个示例 |

### 为什么 In-Context Learning 重要

1. **持续学习能力**：模型可以通过上下文获取训练数据截止日期之后的新信息，避免过时。例如，将新版 JavaScript 文档放入 context，模型即可回答关于新特性的问题——无需重新训练
2. **灵活性**：同一个模型通过不同的 prompt 可以执行完全不同的任务

### Few-Shot 的权衡

| 因素 | 影响 |
|---|---|
| **示例数量** | 通常越多越好，但受限于 context length |
| **推理成本** | 更多示例 → 更长的 prompt → 更高的推理成本（按 token 计费） |
| **模型能力** | 强模型（如 GPT-4）的 zero-shot 已接近 few-shot 性能；弱模型（如 GPT-3）few-shot 相比 zero-shot 有显著提升 |
| **领域特殊性** | 即使是强模型，在训练数据稀少的领域（如 Ibis dataframe API）仍然显著受益于 few-shot |

### Chollet 的心智模型

Keras 创始人 François Chollet 将 foundation model 比作一个**程序库**：模型内部包含大量"程序"（如写俳句的程序、写打油诗的程序），每个程序可以被特定的 prompt 激活。在这个视角下，**prompt engineering 就是找到能激活正确程序的钥匙**。

## 术语辨析：Prompt vs Context

"Prompt" 和 "Context" 在不同语境下含义不同，容易混淆：

| 来源 | Prompt 的含义 | Context 的含义 |
|---|---|---|
| **GPT-3 论文** (Brown et al., 2020) | — | 模型的全部输入（= prompt） |
| **社区惯用** | 模型的全部输入 | prompt 中为模型提供的**上下文信息**（prompt 的子集） |
| **Google PaLM 2 文档** | 模型的全部输入 | **任务描述**（角色设定、主题限制、回复风格等） |

**本书约定**：

- **Prompt** = 模型的完整输入
- **Context** = 模型执行任务所需的上下文信息（prompt 的一部分）

## System Prompt 与 User Prompt

### 基本分工

许多模型 API 支持将 prompt 拆分为 system prompt 和 user prompt：

| 组成 | 职责 | 典型内容 |
|---|---|---|
| **System Prompt** | 任务描述 + 角色设定 + 行为约束 | "你是一位经验丰富的房产经纪人，阅读披露文件并简洁专业地回答买家问题" |
| **User Prompt** | 具体任务 + 上下文数据 | 披露文件内容 + 用户提出的具体问题 |

> 实际上可以灵活安排：把所有内容放进 system prompt 或 user prompt 均可，应通过实验确定最优结构。

### Chat Template：从两段文本到最终 Prompt

模型在内部将 system prompt 和 user prompt **按模板拼接**为单一 prompt。不同模型使用不同的 chat template。

**Llama 2 chat template**：

```text
<s>[INST] <<SYS>>
{{ system_prompt }}
<</SYS>>
{{ user_message }} [/INST]
```

**Llama 3 chat template**（Meta 在版本间更换了模板）：

```text
<|begin_of_text|><|start_header_id|>system<|end_header_id|>
{{ system_prompt }}<|eot_id|><|start_header_id|>user<|end_header_id|>
{{ user_message }}<|eot_id|><|start_header_id|>assistant<|end_header_id|>
```

其中 `<|begin_of_text|>`、`<|start_header_id|>` 等特殊标记各被模型视为**单个 token**。

### Chat Template 的常见陷阱

使用错误的 chat template 会导致**静默性能下降**——模型不会报错，仍然会生成看似合理的输出，但质量悄然变差。即使是微小的偏差（如多一个换行符）也可能显著改变模型行为。

**最佳实践**：

1. 确保输入严格遵循模型的 chat template
2. 使用第三方工具构造 prompt 时，**验证工具是否使用了正确的 chat template**（模板错误非常常见）
3. 发送查询前，**打印最终 prompt** 进行检查

### 为什么 System Prompt 能提升性能

从模型的视角看，system prompt 和 user prompt 被拼接后以相同方式处理。性能提升可能来自两个因素：

| 因素 | 机制 |
|---|---|
| **位置优势** | System prompt 排在最终 prompt 的开头，模型通常对开头的指令理解更好 |
| **训练偏向** | 模型在 post-training 阶段被专门训练为**更重视 system prompt**（如 OpenAI 的 "The Instruction Hierarchy" 论文），这也有助于抵御 prompt 攻击 |

## Context Length 与 Context Efficiency

### 上下文窗口的演进

模型的最大上下文长度决定了 prompt 能包含多少信息。近年来增长极为迅速：

| 模型 | 发布时间 | Context Length |
|---|---|---|
| GPT-2 | 2019.02 | 1K |
| GPT-3 | 2020.06 | 2K |
| GPT-3.5 | 2022.11 | 4K |
| Gemini-1.5 Pro | 2024.05 | **2M** |

5 年内增长 **2000 倍**。100K context 约可容纳一本中等篇幅的书；2M context 约可容纳 2000 个 Wikipedia 页面或一个中等复杂度的代码库（如 PyTorch）。

### Needle In A Haystack (NIAH)：上下文利用效率并不均匀

并非 prompt 中的所有位置都被模型同等对待。Liu et al. (2023) 的研究表明：模型对 prompt **开头和结尾**的信息理解远优于**中间部分**。

**NIAH 测试方法**：在 prompt（haystack）的不同位置插入一条随机信息（needle），要求模型找到它。结果：所有被测模型在信息位于开头或结尾时表现显著优于中间。

**实践注意事项**：

- 测试用的信息应为**私有数据**，避免模型依赖内部知识而非上下文来回答
- 除人工生成的 needle 外，也可使用真实场景中的问答（如从长篇医生问诊记录中提取用药信息）

### 相关评估工具

| 工具 | 用途 |
|---|---|
| **NIAH** | 评估模型在不同上下文位置的信息检索能力 |
| **RULER** (Hsieh et al., 2024) | 更系统地评估模型处理长 prompt 的能力 |

### 实践建议

1. **将关键信息放在 prompt 的开头或结尾**，避免放在中间
2. 如果模型在长上下文下性能显著下降，考虑**缩短 prompt** 或换用长上下文能力更强的模型
3. 使用 NIAH / RULER 等 benchmark 评估你所用模型的长上下文能力

## 小结

| 知识点 | 核心要点 |
|---|---|
| **Prompt 组成** | Task Description + Example(s) + The Task |
| **生效前提** | 模型的指令遵循能力；模型越强 → 鲁棒性越高 → fiddling 越少 |
| **In-Context Learning** | 无需更新权重，通过 prompt 中的示例学习新行为；强模型 zero-shot 已足够好，领域特定场景仍需 few-shot |
| **Prompt vs Context** | Prompt = 完整输入；Context = 执行任务所需的上下文信息 |
| **System / User Prompt** | System prompt 用于任务描述和角色设定，性能优势来自位置和训练偏向；必须严格遵循模型的 chat template |
| **Context Length** | 5 年增长 2000 倍；模型对开头和结尾的信息利用效率高于中间 |
