# RAG（Retrieval-Augmented Generation，检索增强生成）

> **核心定位**：模型要完成任务，需要两样东西：**指令**（instructions，告诉它怎么做）和**信息**（context，做这件事需要的知识）。指令对所有查询通用，而上下文是每个查询独有的。上一章讨论了如何写好指令（Prompt Engineering），本章聚焦于如何为每个查询构造相关的上下文。RAG 是最主要的上下文构造模式之一——它让模型能从外部数据源检索出与当前查询最相关的信息，再据此生成回答。

---

## 一、RAG 的起源与核心价值

### 为什么需要 RAG？

人类在缺乏信息时更容易犯错，AI 模型也一样——**当模型缺少上下文时，更容易产生幻觉（hallucination）**。RAG 的核心价值就是：在生成回答之前，先从外部记忆源（external memory source）中检索出与查询相关的信息，补充给模型，让模型"有据可答"。

外部记忆源可以是：内部数据库、用户的历史聊天记录、互联网等。

将通用大模型应用于实际业务场景时，三大核心痛点驱动了 RAG 的需求：

| 痛点 | 说明 |
|---|---|
| **知识的局限性** | 大模型无法获取实时数据、非公开数据或离线数据。训练数据有截止日期，之后的信息模型完全不知道 |
| **幻觉问题** | 所有 AI 模型的底层原理都基于数学概率，模型有时会"一本正经地胡说八道"，尤其在模型自身不具备某领域知识时。更棘手的是，幻觉难以识别——需要使用者自身具备相应领域知识 |
| **数据安全性** | 企业不愿将私域数据上传第三方平台训练。RAG 允许数据留在本地，只在查询时检索相关片段，避免数据泄露风险 |

### RAG vs Fine-tuning

两者都是让通用大模型适应特定业务的手段，但路径不同：

| 维度 | RAG（检索增强生成） | Fine-tuning（微调） |
|---|---|---|
| **核心思路** | LLM + 外部知识：先将文档 embedding 存入向量库，查询时检索相关知识，结合 prompt 让 LLM 生成答案 | 用特定数据集对 LLM 局部参数进行调整，使其更理解业务逻辑，提升 zero-shot 能力 |
| **知识更新** | 只需更新外部知识库，无需重新训练 | 需要重新训练才能吸收新知识 |
| **适用场景** | 知识密集型任务、需要引用来源、数据频繁更新 | 需要模型学习特定风格/格式/推理模式 |
| **数据安全** | 数据留在本地检索系统中 | 数据需用于训练过程 |
| **成本** | 运行时有检索开销，但无需训练成本 | 训练成本高，但推理时无额外检索开销 |

两者并非互斥——可以先 fine-tune 模型使其更好地利用检索结果，再用 RAG 补充实时知识。

### 历史脉络

| 时间 | 里程碑 | 论文/来源 |
|---|---|---|
| 2017 | **Retrieve-then-generate 模式**首次提出：系统先检索 5 篇最相关的 Wikipedia 页面，再由模型（当时用的是 LSTM）阅读这些页面并生成答案 | Chen et al., "Reading Wikipedia to Answer Open-Domain Questions" |
| 2020.5 | Facebook 研究表明，用检索系统增强预训练语言模型能显著提升事实问答性能 | Petroni et al., "How Context Affects Language Models' Factual Predictions" |
| 2020 | **RAG 一词正式提出**。作为知识密集型任务的解决方案——当所有知识无法直接输入模型时，只检索与查询最相关的部分。实验表明 RAG 能生成更详细的回答并减少幻觉 | Lewis et al., "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" |

### RAG 的本质

RAG 可以理解为一种**为每个查询定制上下文**的技术，而非对所有查询使用相同的上下文。这与传统 ML 中的**特征工程（feature engineering）**本质相同——都是为模型提供处理输入所需的必要信息。

### RAG 不会被长上下文取代

有人认为，随着模型上下文窗口不断增大，RAG 将变得多余。这种观点有两个盲点：

1. **数据量永远在增长**——无论上下文有多长，总有应用需要超过这个长度的上下文。人们不断产生新数据但很少删除数据。上下文长度增长虽快，但追不上任意应用的数据需求
2. **长上下文 ≠ 有效使用上下文**——上下文越长，模型越可能聚焦在错误的部分。每多一个 token 都意味着额外的成本和潜在的延迟增加。RAG 让模型只使用最相关的信息，**减少输入 token 同时可能提升性能**

> Anthropic 建议：如果知识库小于 200,000 tokens（约 500 页），可以直接将全部内容放入 prompt，无需 RAG。
> source： https://www.anthropic.com/engineering/contextual-retrieval

## 二、RAG 架构与 Pipeline 全流程

### 两大核心组件

| 组件 | 职责 |
|---|---|
| **Retriever（检索器）** | 从外部记忆源中检索与查询相关的信息。包含两个功能：**Indexing（索引）**——预处理数据使其可被快速检索；**Querying（查询）**——从索引中检索最相关的数据 |
| **Generator（生成器）** | 基于检索到的信息生成回答 |

在 Lewis et al. 的原始论文中，检索器和生成模型是**联合训练**的。在今天的 RAG 系统中，这两个组件通常**分开训练**，很多团队直接使用现成的检索器和模型组装 RAG 系统。但**端到端微调整个 RAG 系统**可以显著提升性能。

> 本章中 "document" 同时指完整文档和文档块（chunk），因为一个 chunk 本质上也是一个 document，这与经典 NLP/IR 术语一致。

### Pipeline 三阶段全景

一个完整的 RAG 系统分为三个阶段，每个阶段都有可调优的组件：

```text
┌──────────────────────────────────────────────────────────────────┐
│  Ingestion（数据摄取 / 离线阶段）                                  │
│  数据清洗 → 文档解析 → Chunking → Embedding → Metadata → Indexing │
└─────────────────────────────┬────────────────────────────────────┘
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│  Retrieval（检索 / 在线阶段）                                     │
│  Query 优化(重写/HyDE/子查询) → 检索(term/embedding/hybrid)      │
│  → Reranking(精排) → Advanced Retrieval(句子窗口/自动合并)        │
└─────────────────────────────┬────────────────────────────────────┘
                              ▼
┌──────────────────────────────────────────────────────────────────┐
│  Generation（生成）                                               │
│  Prompt 构造(query + 检索结果) → LLM 生成 → Response 合成         │
└──────────────────────────────────────────────────────────────────┘
```

### 贯穿全流程的核心认知：「最相似 ≠ 最相关」

这是理解 RAG 系统中许多设计决策的关键。语义搜索根据 embedding 相似度检索文档，但**语义相似的文档未必是用户真正需要的**。这一认知贯穿于 RAG 每个环节的设计动机：

- **为什么需要 Reranking？** 初始检索按相似度排序，但例如查询"苹果发布会"时，"苹果种植技术"在 embedding 空间中也有较高相似度却完全不相关。Reranker 用更精细的模型重新评估"相关性"而非仅仅"相似性"
- **为什么需要 Metadata Filtering？** 相似度无法捕捉时间、作者、类别等结构化约束。一篇 2020 年的旧文档可能在语义上与查询非常相似，但用户要的是最新信息
- **为什么需要 Hybrid Search？** 纯语义检索可能遗漏精确关键词匹配（如错误码、产品名），纯关键词检索又缺乏语义理解。混合搜索弥补两者的盲区
- **为什么需要 Query 优化？** 用户的原始查询可能措辞模糊、不完整或与文档的表述风格不同，导致高相关文档的相似度反而不高

理解了这一点，后续每个环节的设计动机就很清晰了。

## 三、Ingestion——数据摄取阶段

在检索发生之前，数据必须先经过一系列预处理步骤。这个阶段是**离线执行**的，决定了后续检索和生成的质量上限。

### 3.1 数据清洗（Data Cleaning）

在执行任何后续步骤之前，确保数据达到以下标准：

| 标准 | 说明 |
|---|---|
| **Clean** | 应用常见 NLP 数据清理技术：确保特殊字符正确编码、去除 HTML 标签、统一编码格式等 |
| **Correct** | 确保信息一致且事实准确。相互矛盾的信息会混淆 LLM，导致幻觉 |

### 3.2 文档解析（Document Parsing）

不同格式的文档（PDF、Word、PPT、网页、代码文件等）需要用不同的解析器提取文本内容。解析质量直接影响后续所有步骤——如果解析丢失了表格结构或段落层次，后续的分块和检索都会受到影响。

### 3.3 Chunking（分块策略）

#### 为什么分块很重要？

文档长度从 10 token 到百万 token 不等，不分块直接检索整篇文档可能导致上下文过长。分块策略直接影响检索系统的性能。有效的分块策略能：提高检索效率（更精准的分块减少搜索空间）、提高生成质量（相关性更高的 chunk 提供更丰富的上下文）、支持特定任务（如情感分析、主题分类）。

分块技术的选择取决于数据类型。例如在 LangChain 中，不同的 text splitter 会根据不同逻辑拆分文档——代码与 Markdown 文件需要不同的分块技术。

#### 五级分块策略

| 级别 | 策略 | 说明 |
|---|---|---|
| **Level 1** | **Fixed Size Chunking（等长分块）** | 按字符（如 2,048）、单词（如 512）、句子（如 20 句）或段落切分。chunk_size 取决于用例：问答需要较短的具体分块，摘要需要较长分块 |
| **Level 2** | **Recursive Chunking（递归分块）** | 先按大单元（如章节）分割，太长就按段落分，段落太长再按句子分，逐步缩小，减少将相关内容截断的概率 |
| **Level 3** | **Document-Based Chunking（基于文档结构）** | 利用文档自身结构切分：代码按编程语言分割，Q&A 按问答对分割，Markdown 按标题层级分割，中英文可能需要不同策略 |
| **Level 4** | **Semantic Chunking（语义分块）** | 不按固定规则，而是根据文本语义来确定分块边界。通过计算相邻句子的 embedding 相似度，在语义断裂处切分，确保每个 chunk 内部语义连贯 |
| **Level 5** | **Agentic Chunking（智能体分块）** | 利用 LLM 根据上下文动态确定每个块的内容和长度。与传统的固定规则不同，Agentic Chunking 基于特定目标/任务创建文本块 |

**Agentic Chunking 详解**：以电商平台分析客户评价为例，最好的做法是将与特定主题相关的评价放在同一个块中，批评性评价和积极性评价分别放入不同块。实现这种智能分块需要情感分析、实体提取和聚类，可通过多 agent 系统来实现。

**Token 分块**：用生成模型的 tokenizer 对文档分词后按 token 数切分。优点是与下游模型直接对齐；缺点是换模型（不同 tokenizer）需要重新索引。

#### 重叠（Overlap）

不重叠的切分可能在关键信息中间截断。例如 "I left my wife a note" 被切成 "I left my wife" 和 "a note"，两个块都丢失了原始含义。设置重叠（如 chunk 2,048 字符、overlap 20 字符）确保边界信息至少出现在一个 chunk 中。

#### Chunk 大小的权衡

| 小 chunk | 大 chunk |
|---|---|
| 同等上下文空间能装更多 chunk → 信息更多样 | 保留更完整的语境 |
| 可能丢失重要信息（如某话题仅在文档前半部分提到，后半 chunk 未被检索到） | chunk 更少，计算开销更低 |
| 计算开销更大（更多 chunk 要索引、更多 embedding 要生成和存储、搜索空间更大） | — |

> chunk 大小不应超过生成模型的最大上下文长度；对于 embedding-based 方法，也不应超过 embedding 模型的上下文限制。没有普遍最优的 chunk 大小和重叠大小，需要实验确定。

### 3.4 Embedding 模型选型

通常，生成 embedding 的**维度越高，精度越高**，但存储和计算成本也越高。

**选型参考**：
- [MTEB Leaderboard](https://huggingface.co/spaces/mteb/leaderboard)（Massive Text Embedding Benchmark）——在检索、分类、聚类等多任务上对 embedding 模型进行综合排名
- 中文场景需要额外关注中文文本嵌入模型的表现

**微调 Embedding 模型**：LlamaIndex 的实验表明，针对特定领域**微调 embedding 模型可在检索效果上提升约 5-10%**。可以用合成数据（synthetic data）来构造微调训练集，降低标注成本。

### 3.5 Metadata 管理

给向量嵌入附加 metadata 标注，在后续检索中发挥重要作用。

#### Metadata Filtering（元数据过滤）

添加时间戳、作者、类别等元数据，检索时**先根据元数据过滤，再做相似性搜索**。这能提高检索效率并减少噪音。

**示例**：用户搜索"最新旅行指南"，通过时间戳过滤只检索最近的指南，避免过时信息。

#### Metadata Enrichment（元数据增强）

除了文档自带的元数据（时间戳、作者、页码等），还可以构建更有价值的元数据：

| 增强方式 | 说明 |
|---|---|
| **摘要** | 为每个 chunk 生成摘要 |
| **标签提取** | 自动提取实体和关键词 |
| **反向假设文档嵌入（Reverse HyDE）** | 用 LLM 为每个 chunk 生成它可能回答的潜在查询，将这些查询加入 metadata。检索时将用户查询与这些生成的查询比较，找到最相关的 chunk |

> 元数据架构设计需谨慎——避免冗余，管理好处理和存储成本。

### 3.6 Multi-indexing（多重索引）

当 metadata 无法充分区分不同上下文类型时，可以为**不同文档类型采用不同的索引策略**，同时在检索时加入**索引路由机制**（由 LLM 决定查询哪个索引）。

#### 结构化索引设计

| 索引结构 | 说明 |
|---|---|
| **父子索引（Parent-Child）** | 文档按层级组织：父文档包含总体主题或摘要，子文档深入具体细节。检索时先定位最相关的子文档，需要时参考父文档获取更多上下文。提高精确度的同时保留广泛上下文，但层级结构可能增加内存和计算开销 |
| **知识图谱索引（Knowledge Graph）** | 将数据存储为实体及其关系。图形索引通过建立二级/三级连接，增强上下文理解和推理能力。适合具有多对多关系或层次结构的数据 |
| **表格专用索引** | 对于表格数据，可提取每个表并生成适合检索的摘要，但将原始表返回给 LLM 用于答案生成 |

### 3.7 索引算法与参数调优

ANN 算法通常提供可调参数。以 HNSW 为例：

| 参数 | 含义 |
|---|---|
| `ef` | 查询时探索的邻居数量——越大越精确但越慢 |
| `efConstruction` | 构建索引时探索的邻居数量——越大索引质量越高但构建越慢 |
| `maxConnections` | 每个节点的最大连接数——影响图的密度和搜索质量 |

还可以为索引算法启用**向量压缩**技术。虽然压缩会牺牲一定精度，但通过合理选择压缩算法并调参，可以在效率和精度之间取得平衡。

## 四、Retrieval——检索阶段

### 4.1 检索算法基础

信息检索（Information Retrieval, IR）有百年历史，是搜索引擎、推荐系统、日志分析等的核心。很多传统检索算法可以直接用于 RAG。检索的核心操作是：**根据查询与文档的相关性对文档进行排序**。不同检索算法的区别在于如何计算相关性分数。

#### Sparse vs Dense 检索

文献中常将检索算法分为 sparse（稀疏）和 dense（稠密）两类：

| 类型 | 表示方式 | 典型方法 |
|---|---|---|
| **Sparse** | 稀疏向量（大部分值为 0），如 one-hot 向量 | 基于词项的检索（term-based） |
| **Dense** | 稠密向量（大部分值非 0），如 embedding | 基于嵌入的检索（embedding-based） |

但本书选择按 **term-based vs embedding-based** 分类，原因是：SPLADE（Sparse Lexical and Expansion，Formal et al., 2021）虽然使用稀疏 embedding，但它的工作原理、优缺点更接近 dense embedding 检索而非 term-based 检索。用 sparse/dense 分类会导致 SPLADE 被误归类到 term-based。

#### Term-based Retrieval（基于词项的检索）

**核心思想**：通过关键词匹配来查找相关文档（也称 lexical retrieval，词法检索）。

Term-based 检索需要解决两个问题：**(1) 怎么快速找到包含关键词的候选文档？(2) 怎么给这些候选文档排序？** 分别对应 Indexing 和 Scoring 两个环节：

```text
Term-based Retrieval = Indexing（索引） + Scoring（评分）
│
├── Indexing：怎么快速找到候选文档？
│   └── Inverted Index（倒排索引）—— 核心数据结构
│
├── Scoring：怎么给候选文档排序？
│   ├── TF-IDF —— 基础评分算法
│   └── BM25  —— TF-IDF 的改进版（工业界标准）
│
└── 生产系统：把 Indexing + Scoring 组装在一起的工程实现
    └── Elasticsearch（内置倒排索引 + BM25）
```

##### (1) Indexing——倒排索引（Inverted Index）

倒排索引是 Term-based 检索的**数据结构基础**。它反转了传统"文档 → 词"的关系，变成"词 → 文档列表"——以词（term）为键，存储包含该词的文档列表（称为倒排列表，Posting List），其中记录文档 ID、词频、位置等信息。

```text
假设有三个文档：
  文档1: "the cat sat on the mat"
  文档2: "the cat lay on the rug"
  文档3: "the dog barked at the cat"

构建的倒排索引：
  词项(Term)    倒排列表(Posting List)
  ─────────    ──────────────────────
  cat          文档1, 文档2, 文档3
  sat          文档1
  mat          文档1
  lay          文档2
  rug          文档2
  dog          文档3
  barked       文档3
```

查询 "cat dog" 时，直接从索引中取出 cat → {1,2,3} 和 dog → {3}，无需遍历所有文档。倒排索引解决的是**速度**问题——从百万文档中瞬间定位包含关键词的候选集。

##### (2) Scoring——TF-IDF 与 BM25

找到候选文档后，需要给它们**排序**——哪些文档与查询最相关？这就是评分算法要解决的问题。

**TF-IDF（基础评分算法）**

直接用关键词匹配有两个问题，TF-IDF 用两个指标分别解决它们：

| 问题 | 指标 | 原理 |
|---|---|---|
| 太多文档包含查询词，需要区分谁更相关 | **TF（Term Frequency，词频）** | 一个词在文档中出现次数越多，该文档与该词越相关 |
| 查询中有些词比其他词重要（如 "for" vs "vietnamese"） | **IDF（Inverse Document Frequency，逆文档频率）** | 一个词出现在越多文档中，它的信息量越低。IDF = log(N / C(t))，N 是文档总数，C(t) 是包含词 t 的文档数 |

TF-IDF 评分公式：Score(D, Q) = Σᵢ IDF(tᵢ) × f(tᵢ, D)，其中 f(tᵢ, D) 是词 tᵢ 在文档 D 中的词频。

**BM25（TF-IDF 的改进版，工业界标准）**

TF-IDF 有一个明显缺陷：长文档天然包含更多词，词频更高，导致评分偏向长文档。BM25（全称 Okapi BM25，Best Matching 算法第 25 代，Robertson et al., 1980s）在 TF-IDF 基础上引入了**文档长度归一化**和**词频饱和函数**来修正这个问题：

```text
Score(D, Q) = Σᵢ IDF(qᵢ) × [ f(qᵢ,D) × (k₁+1) ] / [ f(qᵢ,D) + k₁ × (1 - b + b × |D|/avgdl) ]
```

BM25 相比 TF-IDF 的两个改进：

| 改进 | 说明 |
|---|---|
| **文档长度归一化** | 通过 \|D\|/avgdl（文档长度 / 平均文档长度）校正长文档的不公平优势。参数 b（通常 0.75）控制归一化强度：b=0 不归一化，b=1 完全归一化 |
| **词频饱和函数** | 一个词出现 10 次和 100 次的区别不应像 1 次和 10 次那样大。参数 k₁（通常 1.2~2.0）控制饱和速度——k₁ 越大，高频词的边际收益衰减越慢 |

> BM25 及其变体（BM25+、BM25F）至今在工业界广泛使用，是评估更复杂检索算法的强基线。Perplexity CEO Aravind Srinivas 曾表示："要真正超过 BM25 或全文检索是很难的。"

##### (3) 生产系统——Elasticsearch

**Elasticsearch**（Shay Banon, 2010）是最主流的 Term-based 检索生产系统。它基于 Apache Lucene 构建，**把倒排索引（Indexing）和 BM25（Scoring）组装在了一起**：

- 写入文档时 → 自动构建倒排索引
- 查询时 → 通过倒排索引快速定位候选文档 → 用 BM25 对候选文档评分排序 → 返回 top-k 结果

除了基础检索，Elasticsearch 还支持分布式部署、近实时搜索、聚合分析等生产级功能。

##### Tokenization 的注意事项

将查询拆分为词项的过程叫 tokenization（分词）。最简单的方法是按空格拆分，但这会破坏多词术语的语义（如 "hot dog" 被拆成 "hot" 和 "dog"）。解决方法：把常见 n-gram 当作整体词项。此外通常还需要：小写化、去标点、去停用词（"the"、"and"、"is" 等）。

#### Embedding-based Retrieval（基于嵌入的检索）

**为什么需要语义检索？** Term-based 检索在**词法层面**计算相关性，而非语义层面。查询 "transformer architecture" 可能返回关于电力设备或电影《变形金刚》的文档。Embedding-based 检索（也称 semantic retrieval，语义检索）的目标是根据**语义相似度**对文档排序。

**工作流程**：索引阶段比 term-based 多一步——将原始数据块转为 embedding 并存入**向量数据库（vector database）**。查询阶段分两步：(1) 用与索引时相同的 embedding 模型将查询转为 embedding；(2) 从向量数据库中取出 k 个 embedding 最接近查询 embedding 的数据块。实际生产系统还可能包含 reranker（重排序器）和 cache（缓存）。

**向量数据库与向量搜索**：向量数据库存储向量，但存储是简单部分——**难点是向量搜索（vector search）**：给定查询 embedding，快速找到数据库中最接近的向量。向量数据库早期用于推荐系统，近年也广泛用于问答应用。主流向量数据库包括 FAISS（Facebook）、Milvus/Zilliz、Pinecone、Weaviate、Qdrant 等。

**相似度度量（Similarity Measures）**：向量搜索依赖于相似度度量来评判两个向量的"接近程度"：

| 度量 | 公式 | 特点 |
|---|---|---|
| **Cosine Similarity** | cos(A,B) = A·B / (\|A\|×\|B\|) | 衡量方向相似性，不受向量长度影响。范围 [-1, 1]，1 表示完全同向 |
| **Euclidean Distance** | d(A,B) = √Σ(Aᵢ-Bᵢ)² | 衡量空间中的绝对距离。**分数越低相似度越高**（注意：与 cosine 相反） |
| **Dot Product** | A·B = ΣAᵢBᵢ | 同时考虑方向和长度。当向量已归一化时等价于 cosine similarity |

> 选择哪种度量取决于 embedding 模型——大多数现代 embedding 模型输出归一化向量，此时 cosine similarity 和 dot product 等价。

向量搜索通常被建模为**近邻搜索（nearest-neighbor search）问题**。**朴素方案 k-NN**：计算查询 embedding 与数据库中所有向量的相似度 → 排序 → 返回 top-k。精确但计算量大，仅适用于小数据集。

**实际方案 ANN（Approximate Nearest Neighbor，近似近邻）**：牺牲少量精度换取大幅加速。主要算法：

| 算法 | 原理 | 实现 |
|---|---|---|
| **LSH**（Locality-Sensitive Hashing，Indyk & Motwani, 1999） | 把相似向量哈希到同一桶中，加速相似度搜索，以少量精度换效率 | FAISS, Annoy |
| **HNSW**（Hierarchical Navigable Small World，Malkov & Yashunin, 2016） | 构建多层图，节点是向量，边连接相似向量，通过遍历图的边来做近邻搜索 | Hnswlib（作者开源）, FAISS, Milvus |
| **Product Quantization**（Jégou et al., 2011） | 将每个向量分解为多个子向量，用低维表示计算距离，大幅加速 | FAISS（核心组件），几乎所有主流库都支持 |
| **IVF**（Inverted File Index，Sivic & Zisserman, 2003） | 用 K-means 聚类将相似向量分到同一簇（通常每簇 100-10,000 个向量）。查询时先找最近的簇质心，再在这些簇内搜索 | FAISS（与 PQ 共同构成核心） |
| **Annoy**（Approximate Nearest Neighbors Oh Yeah，Bernhardsson/Spotify, 2013） | 树方法：构建多棵二叉树，每棵树用随机标准把向量分成两支。查询时遍历这些树收集候选邻居 | Spotify 开源实现 |

其他算法还有 Microsoft SPTAG（Space Partition Tree And Graph）和 FLANN（Fast Library for Approximate Nearest Neighbors）等。

> 虽然向量数据库随 RAG 的兴起成为独立品类，但任何能存储向量的数据库都可以称为向量数据库。很多传统数据库已经或将要扩展以支持向量存储和搜索。

**向量搜索中的索引 vs 查询权衡**：更详细的索引 → 更精确的检索，但索引构建更慢、更耗内存。HNSW：高精度 + 快速查询，但构建耗时耗内存；LSH：构建快 + 省内存，但查询较慢且精度较低。**ANN-Benchmarks** 网站用 Recall / QPS / Build time / Index size 四个指标评估不同 ANN 算法。

#### 两种检索方法对比

| 维度 | Term-based Retrieval | Embedding-based Retrieval |
|---|---|---|
| **查询速度** | 快得多 | 查询 embedding 生成 + 向量搜索可能较慢 |
| **性能** | 开箱即用表现好，但难以进一步提升；可能因词汇歧义检索到无关文档 | 通过微调可超越 term-based；支持更自然的查询，聚焦语义而非词汇 |
| **成本** | 便宜得多 | Embedding 生成、向量存储、向量搜索都可能昂贵 |
| **可调优空间** | 少（简单性的代价） | 可微调 embedding 模型和检索器，单独/联合/与生成模型一起微调 |
| **已知局限** | — | 转为 embedding 后可能丢失关键词信息（如错误码 EADDRNOTAVAIL、产品名），可通过混合搜索弥补 |

### 4.2 Hybrid Search（混合搜索）

生产级检索系统通常**组合多种检索方法**。组合 term-based 和 embedding-based 称为 **hybrid search（混合搜索）**。

**召回（Recall）的本质**：召回是快速准确获取相关信息的过程，可理解为**建立索引 + 相似度匹配**的结合。索引负责缩小查找范围（把相似数据放在一起，查询时无需遍历全库），相似度匹配负责在缩小的范围内做精细排序。

#### 串行组合（Sequential）

先用便宜但不够精确的检索器（如 term-based）获取候选集，再用更精确但更贵的机制（如 k-NN）从候选集中挑选最优结果。第二步也叫 **reranking（重排序）**。

**示例**：
- 查询 "transformer" → 先用关键词检索出所有包含 "transformer" 的文档（无论是电力设备、神经网络还是电影），再用向量搜索在候选中筛选出真正相关的
- 查询 "Who's responsible for the most sales to X?" → 先用关键词 X 检索所有关联文档，再用向量搜索筛选与 "sales responsibility" 相关的

#### 并行组合——结果融合策略

多个检索器同时获取候选，然后**融合不同的排序**得到最终排序。有两种主要融合策略：

**策略一：加权融合（Weighted Fusion）**

将两路召回的得分按权重加权求和：

```text
hybrid_score = (1 - α) × sparse_score + α × dense_score
```

`alpha = 1` 纯语义搜索；`alpha = 0` 纯关键词搜索；`0 < alpha < 1` 混合搜索。

**前提——得分归一化（Score Normalization）**：两路检索的分数量纲完全不同（BM25 可能是 0~200，cosine 是 -1~1），加权前**必须**归一化到同一区间（如 [0, 1]）：

| 检索方式 | 归一化方法 |
|---|---|
| **BM25** | 不能用简单的 Min-Max（最大值未知），通常用 `score / (score + C)`（C 为常数）或 Z-Score |
| **Cosine similarity** | 如果范围是 [-1, 1]，可用 `(score + 1) / 2` 转到 [0, 1] |

> **警告**：归一化非常脆弱且难以做好，这也是业界逐渐转向 RRF 的原因。注意对于 FAISS 等默认使用欧式距离的系统（分数越低相似度越高），计算 hybrid_score 时需要先将欧氏距离转换为相似度得分。

**如何确定 α 权重？**

| 方法 | 说明 |
|---|---|
| **启发式** | 凭直觉设定（如认为语义更重要就设 α=0.6）。简单但效果无法保证 |
| **网格搜索（Grid Search）** | 专业方案：准备评测集（N 个查询 + 标准答案文档）→ 选择评估指标（如 NDCG@10、MRR）→ 遍历 α ∈ {0.1, 0.2, ..., 0.9}，对每个 α 计算评估指标 → 选择指标最高的 α 值 |

---

**策略二：RRF（Reciprocal Rank Fusion）——推荐方案**

RRF 不是检索算法，而是一种**数据融合/结果重排策略**。它的诞生就是为了解决加权融合中归一化和权重调优的复杂性。

**核心哲学：放弃得分，只看排名**。BM25 得分是 20 还是 200，Cosine 是 0.8 还是 0.9，RRF 都不关心。它只问一个问题：**这个文档在各自的列表里排第几？**

**RRF 工作流程**：

1. **多路召回**：同时发起多个检索请求（如 1 个向量 + 2 个 BM25）
2. **获取排名**：查看每个文档在每个检索列表中的排名
3. **计算 RRF 分数**：

```text
RRFScore(d) = Σᵢ 1 / (k + rankᵢ(d))
```

其中 d = 文档，rankᵢ(d) = 文档 d 在检索器 i 中的排名（从 1 开始），k = 平滑参数（默认 60）

4. **最终排序**：按 RRFScore 从高到低排序

**参数 k 的作用**——降低高排名结果的绝对权威性：

```text
k=0 时（标准倒数排名）：
  Rank 1 → 1/1 = 1.0     Rank 2 → 1/2 = 0.5     → 第1名是第2名的 2 倍
  
k=60 时（RRF 默认值）：
  Rank 1 → 1/61 ≈ 0.01639  Rank 2 → 1/62 ≈ 0.01612  → 第1名和第2名几乎相等
```

k 越大，排名之间的分数差异越小，融合越"民主"——奖励在**多个列表中都表现良好**的文档，而非只在某个列表中侥幸排第一的文档。

**具体示例**（k=60，向量检索 + BM25 两路）：

| 文档 | 向量排名 | BM25 排名 | RRFScore | 结论 |
|---|---|---|---|---|
| **文档 A** | Rank 2 | Rank 5 | 1/62 + 1/65 ≈ **0.03150** | **胜出** |
| **文档 B** | Rank 1 | Rank 30 | 1/61 + 1/90 ≈ **0.02750** | 落败 |

文档 A 在两个系统里都排得靠前（第 2、第 5），RRF 认为它比那个在一个系统排第 1 但另一个系统排第 30 的文档 B 更值得信赖。这就是 RRF 鲁棒性的体现。

**RRF 优缺点**：

| 优点 | 缺点 |
|---|---|
| **无需归一化**：从根本上解决不同得分量纲的问题 | **"民主"假设**：隐含假设所有检索器同等重要，无法表达"向量检索比 BM25 重要 3 倍" |
| **简单**：只有一个超参数 k（甚至直接用默认 60） | **最终得分无意义**：RRFScore 仅用于排序，不代表相似度等物理含义 |
| **鲁棒**：对"坏"检索器有强容忍度，评测中通常最好或接近最好 | |
| **易于实现**：公式简单，计算开销小 | |

---

**两种融合策略对比**：

| | 加权融合（Weighted Fusion） | RRF（推荐） |
|---|---|---|
| **是否需要归一化** | 必需（且脆弱） | 不需要 |
| **超参数** | α 权重（需要评测集做 Grid Search 确定） | k 平滑参数（默认 60 通常足够） |
| **鲁棒性** | 低——归一化质量直接影响结果 | 高——对坏检索器有强容忍度 |
| **业界趋势** | 逐渐被 RRF 取代 | 主流选择（Milvus、Weaviate 等默认支持） |

#### 知识图谱混合搜索

混合搜索不仅限于 term-based + embedding-based 的组合，还可以**将知识图谱技术与向量搜索结合**。知识图谱擅长揭示实体之间的关系（如人物-公司-职位），弥补纯文本检索在关系推理上的不足。不同搜索系统各自擅长揭示文本中的不同联系：精确关系、语义关系、主题关系、结构关系、实体关系、时间关系和事件关系。没有一种单一搜索模式适用于所有场景，混合搜索通过整合多个搜索系统实现互补。

### 4.3 Reranking（重排序）

#### 为什么需要 Reranking？

检索器的初始排序往往是"粗排"——速度快但精度有限。Reranking 是在粗排结果之上做"精排"。**核心洞察：语义搜索根据相似度检索，但"最相似"并不总等于"最相关"**。Reranker 通过为每个检索结果计算与查询的相关性得分，帮助排除不相关的结果。

它在两个场景下特别有用：(1) 上下文空间有限，需要从 N 个候选中挑出最相关的 top-k；(2) 减少输入 token 成本。

#### 策略一：多阶段检索（Cascading Retrieval）

本质是一个**漏斗结构**：

```text
全量文档集（百万级）
    ↓ 第一阶段：便宜/快速的检索器（如 BM25 / term-based）
候选集（数百~数千篇）
    ↓ 第二阶段：精确/昂贵的检索器（如 k-NN 向量搜索 / cross-encoder）
最终结果（top-k，如 5~20 篇）
    ↓ 送入生成模型
```

**为什么不直接用精确检索器？** 精确方法计算量大，无法在百万级文档上实时运行。先用便宜的检索器大幅缩小候选范围，再让精确检索器在小范围内精排，是成本与精度的最佳平衡。

**常用的精排模型**：
- **Cross-encoder reranker**：不同于 bi-encoder（分别编码 query 和 document 再算相似度），cross-encoder 将 query 和 document 拼接后一起输入模型，让 token 之间做完整的 attention 交互，精度更高但速度更慢。代表如 [Cohere Rerank](https://cohere.com/rerank)、[sentence-transformer cross-encoder](https://www.sbert.net/examples/applications/cross-encoder/README.html)、BGE Reranker
- **向量搜索重排**：在粗排结果上用 k-NN（精确近邻）替代 ANN（近似近邻）做重排

**需要调整的参数**：Re-rank 输入数量（初始检索返回多少结果送入 reranker）和 Re-rank 输出数量（重排序后保留多少结果送入 LLM）。与 embedding 模型一样，可以针对特定场景**微调重排序模型**。

Reranking 的最大优势是提供了一种**简单、低复杂度**的方法来改善搜索结果，并允许用户将语义相关性集成到现有搜索系统中，无需重大基础设施改造。

#### 策略二：基于时间的重排序（Time-Weighted Reranking）

给更新的数据更高的权重分数。适用于时效敏感场景（新闻聚合、邮件问答、股市分析、技术文档等）。实现方式通常是 `final_score = relevance_score × decay(age)`，其中 `decay(age)` 是一个随文档年龄递减的函数。

#### 上下文重排 vs 搜索重排

| 维度 | 搜索重排（Search） | 上下文重排（Context） |
|---|---|---|
| **排名位置的重要性** | 极高——第 1 名和第 5 名的点击率差距巨大 | 相对较低——只要文档被包含在上下文中，模型就能利用它 |
| **排序的影响** | 决定用户体验和点击率 | 影响模型的理解效果——模型更容易理解位于**上下文开头和结尾**的信息（"Lost in the Middle" 效应，Liu et al., 2023） |

**实践建议**：Reranking 并非必须——如果初始检索已经足够好，增加 reranking 只会徒增延迟。先跑基线看检索质量，只在精度不够时才引入。

### 4.4 Query Optimization（查询优化 / Pre-retrieval）

查询优化发生在检索之前，目的是让查询更适合检索系统理解。Query 的措辞会直接影响搜索结果。

#### Query Rewriting（查询重写）

也称 query reformulation / query normalization。多轮对话中，后续问题往往不完整（如 "How about Emily Doe?" 需重写为 "When was the last time Emily Doe bought something from us?"）。可用 LLM 重写（Rephrasing），prompt 类似 "Given the following conversation, rewrite the last user input to reflect what the user is actually asking"。

**复杂情况**：如果用户问 "How about his wife?"，系统需要先查数据库确认 "his wife" 是谁。如果查不到，重写模型应承认无法解答，而非幻觉一个名字。

#### HyDE（Hypothetical Document Embeddings，假设性文档嵌入）

不直接用原始查询去检索，而是**先让 LLM 生成一个假设性的答案**，再用这个答案的 embedding 去搜索。原理是：假设性答案在语义空间中通常比问题更接近真正的相关文档。

```text
用户查询: "What causes aurora borealis?"
    ↓ LLM 生成假设性答案
假设性答案: "Aurora borealis is caused by charged particles from the sun..."
    ↓ 用假设性答案的 embedding 去检索
检索到的真实文档（与假设性答案语义接近）
```

#### Sub-queries（子查询分解）

将复杂查询拆解为多个子问题，分别检索后合并结果。例如 "Compare the economic policies of China and the US in 2024" 拆为两个子查询分别检索后合并上下文。LlamaIndex 提供了 Sub Question Query Engine 来自动化这个过程。

#### Hypothetical Questions（假设性问题生成）

与 HyDE 方向相反——不是用答案检索，而是在索引阶段为每个 chunk **预先生成它可能回答的问题**。查询时对问题向量执行搜索，匹配后路由到原始文本 chunk。这种 query-to-query 的匹配通常比 query-to-document 更精确。

#### SlimPlm 方法

用小的代理模型生成问题的启发式答案 → LLM 根据启发式答案生成领域知识 → 用领域知识和 query 从知识库中召回文档 → 最终将原始 query、启发式答案和召回文档输入 LLM 生成响应。

### 4.5 Contextual Retrieval（上下文化检索）

**核心思想**：给每个 chunk 补充相关上下文信息，使其更容易被正确检索。

#### 元数据增强

| 增强方式 | 示例 |
|---|---|
| **标签和关键词** | 给产品块附加描述和评论 |
| **实体提取** | 自动从 chunk 中提取特定术语（如错误码 EADDRNOTAVAIL），加入 metadata，即使 chunk 被转为 embedding 后仍可通过关键词检索 |
| **潜在问题增强** | 给每个 chunk 附加它能回答的问题。如密码重置文章附加 "How to reset password?", "I forgot my password", "I can't log in" 等 |
| **图片/视频** | 通过标题、标签、描述进行检索 |

#### Anthropic 的上下文化方法

当文档被拆成多个 chunk 后，某些 chunk 可能缺失理解其内容所需的背景。Anthropic 用 AI 模型为每个 chunk 生成 50-100 token 的短上下文，解释该 chunk 及其与原始文档的关系：

```text
<document>
{{WHOLE_DOCUMENT}}
</document>

Here is the chunk we want to situate within the whole document:
<chunk>
{{CHUNK_CONTENT}}
</chunk>

Please give a short succinct context to situate this chunk within
the overall document for the purposes of improving search retrieval
of the chunk. Answer only with the succinct context and nothing else.
```

生成的上下文被拼接到 chunk 前面，增强后的 chunk 再被检索算法索引。

### 4.6 Advanced Retrieval Strategies（高级检索策略）

核心思想：**用于检索的 chunk 不一定是用于生成的 chunk**。理想做法是嵌入更小的数据块以提高检索精度，同时检索更广泛的上下文供 LLM 推理。

#### Sentence-Window Retrieval（句子窗口检索）

检索时不只找到相关的单个句子，还要**获取该句子前后的相关句子**，形成一个"窗口"。这样既保持了检索的精准性（用单句匹配），又为 LLM 提供了足够的上下文。

#### Auto-Merging Retrieval（自动合并检索 / Parent Document Retriever）

文档被分割成**层次化的 chunk 结构**（树状），最小的叶子块被送入索引：

```text
       [Parent Chunk A]           [Parent Chunk B]
      /       |        \         /       |        \
  [Child 1] [Child 2] [Child 3] [Child 4] [Child 5] [Child 6]
```

检索时找到 k 个叶子 chunk，如果其中 n 个 chunk 指向同一个父 chunk，就用**父 chunk 替换它们**，将父 chunk 送给 LLM 用于答案生成。这样既精确检索（用小 chunk），又提供了完整上下文（用父 chunk）。

### 4.7 Query Construction & Routing（查询构建与路由）

#### Query Construction（查询构建）

自然语言查询需要转换为检索系统的查询语法：

| 转换类型 | 说明 |
|---|---|
| **Text-to-Metadata Filter** | 将自然语言转为向量存储的结构化查询。例如用户说"最近一周的文档"，系统自动构造时间戳过滤条件 |
| **Text-to-SQL** | 将自然语言翻译为 SQL 请求，查询关系型数据库 |
| **Text-to-Cypher** | 将自然语言转为图数据库查询语言 Cypher。适用于知识图谱场景——特别适合多对多关系或层次结构数据 |

#### Query Routing（查询路由）

接收到用户查询后，由 LLM 决定**接下来的操作步骤**——选择对特定数据索引进行搜索，或尝试多个不同的处理方法并合成答案。当系统拥有多个数据源（如文档库、SQL 数据库、知识图谱）时，Query Routing 根据查询意图自动选择最合适的检索路径。

### 4.8 检索质量评估

#### Context Precision 与 Context Recall

这两个指标从两个不同角度衡量检索质量。假设一个场景：

> 数据库中共有 100 篇文档。用户问 "如何重置密码？"，其中有 **4 篇**文档真正相关（A、B、C、D）。检索器返回了 **5 篇**文档（A、B、E、F、G）。

```text
数据库中的 100 篇文档
├── 真正相关的 4 篇: A, B, C, D
└── 不相关的 96 篇: E, F, G, H, ...

检索器返回的 5 篇: A, B, E, F, G
├── 命中(相关): A, B        → 2 篇
└── 未命中(不相关): E, F, G → 3 篇
```

**Context Precision（上下文精确率）**——检索回来的东西有多"纯净"？

```text
Precision = 返回结果中相关的数量 / 返回结果的总数量 = 2 / 5 = 40%
```

Precision 低 = 检索器带回了太多无关噪音 → 浪费 token + 可能误导 LLM。

**Context Recall（上下文召回率）**——该找到的都找到了吗？

```text
Recall = 返回结果中相关的数量 / 数据库中所有相关文档的总数量 = 2 / 4 = 50%
```

Recall 低 = 检索器遗漏了重要信息 → LLM 回答不完整或产生幻觉。

| | Context Precision（精确率） | Context Recall（召回率） |
|---|---|---|
| **问的问题** | 检索结果**有多干净**？ | 相关文档**有没有漏掉**？ |
| **分母** | 检索器**实际返回**的文档数 | 数据库中**所有相关**的文档数 |
| **低分意味着** | 返回了太多垃圾（噪音高） | 遗漏了太多有用信息 |
| **计算难度** | 较简单——只需评判返回的那几篇是否相关 | 困难——需要对全库做相关性标注 |

两者天然存在张力：多返回 → Recall 升 Precision 降；少返回 → 反之。通过调整 **top-k** 来平衡。

#### 排序质量指标

| 指标 | 含义 |
|---|---|
| **NDCG**（Normalized Discounted Cumulative Gain） | 排在前面的相关文档贡献更高分数，排在后面的打折 |
| **MAP**（Mean Average Precision） | 在每个相关文档位置计算 Precision 后取平均 |
| **MRR**（Mean Reciprocal Rank） | 只关注第一个相关文档出现的位置 |

#### Embedding 质量评估

核心标准：语义更相似的文档是否有更接近的 embedding 向量？MTEB benchmark（Muennighoff et al., 2023）是主要参考。

**BEIR**（Benchmarking IR，Thakur et al., 2021）是一个跨领域检索评估框架，支持 14 个常见检索 benchmark。

#### 评估检索方案的选型维度

| 维度 | 关键问题 |
|---|---|
| **检索机制** | 支持哪些检索方法？是否支持 hybrid search？ |
| **模型与算法** | （若为向量数据库）支持哪些 embedding 模型和向量搜索算法？ |
| **可扩展性** | 数据存储和查询流量的扩展能力如何？ |
| **索引速度** | 索引数据需要多长时间？一次能批量处理多少数据？ |
| **查询延迟** | 不同检索算法下的查询延迟分别是多少？ |
| **定价** | 按文档/向量量计费还是按查询量计费？ |

## 五、Generation——生成阶段

检索完成后，检索增强的 prompt 被输入到 LLM 中生成最终回答。这个阶段同样有多个可调优的维度。

### 5.1 LLM 选择与微调

与 embedding 模型和 reranker 一样，可以针对特定用例**微调 LLM**，使其合并特定的措辞或语气。例如客服场景可能需要礼貌正式的语气，技术文档问答需要精确的术语表达。

### 5.2 Prompt Engineering for RAG

RAG 场景下的 prompt 设计有其特殊性——需要明确指示模型**基于检索结果回答，而非使用自身知识**：

**关键 prompt 策略**：
- 明确约束："Please base your answer **only** on the search results and nothing else!"
- 要求引用来源："Your answer MUST be grounded in the search results provided. Please explain **why** your answer is grounded in the search results!"
- 使用 **few-shot 示例**提高完成质量
- **输入的上下文数量**是需要实验的参数——太少信息不够，太多引入噪音且增加成本

### 5.3 Chat Engine（对话引擎）

在多轮对话场景中，需要考虑聊天上下文。核心通过 **query compression 技术**解决，将聊天历史纳入考量：

| 模式 | 工作流程 |
|---|---|
| **ContextChatEngine** | 检索与当前查询相关的上下文，然后将其与内存缓冲区中的聊天历史一起发送给 LLM。即：`(query → context) + history → answer` |
| **CondensePlusContextMode** | 先将当前查询与聊天历史合并生成新查询，再用新查询检索上下文。即：`(query + history → new query) + context → answer` |

两种模式的关键区别：ContextChatEngine 直接用原始查询检索；CondensePlusContextMode 先用 LLM 将历史对话压缩进查询中再检索，检索精度通常更高。

### 5.4 Response Synthesiser（响应合成器）

基于初始查询和检索到的所有上下文生成 response。有多种合成策略：

| 策略 | 说明 |
|---|---|
| **一次性合成（Stuffing）** | 将所有获取到的 context 与查询一起拼接，一次性输入 LLM。简单直接，但受限于上下文窗口 |
| **迭代精炼（Iterative Refinement）** | 将检索到的 context 分块逐步发送给 LLM，迭代地完善答案。适合大量上下文的场景 |
| **摘要合成（Summarization）** | 将检索到的内容先总结成符合 prompt 的形式，再送入 LLM |
| **多答案合成（Multi-answer）** | 基于不同的 context 分别生成多个答案，然后将它们连接或总结为最终答案 |

## 六、RAG Beyond Texts（超越文本的 RAG）

### 6.1 Multimodal RAG（多模态 RAG）

如果生成模型是多模态的，上下文不仅可以包含文本，还可以包含**图像、视频、音频**等。

**示例**：查询 "What's the color of the house in the Pixar movie Up?" → 检索器取回 Up 中房屋的图片，帮助模型回答。

**检索图片的两种方式**：

| 方式 | 说明 |
|---|---|
| **通过 metadata** | 根据图片的标题、标签、描述等 metadata 检索 |
| **通过内容** | 需要多模态 embedding 模型（如 CLIP，Radford et al., 2021）能同时为图片和文本生成 embedding，然后在向量数据库中做跨模态搜索 |

使用 CLIP 的流程：
1. 为所有数据（文本和图片）生成 CLIP embedding 并存入向量数据库
2. 将查询转为 CLIP embedding
3. 在向量数据库中搜索与查询 embedding 接近的图片和文本

### 6.2 RAG with Tabular Data（结构化数据 RAG）

大多数应用不仅处理非结构化数据（文本、图片），还需要处理**表格数据**。这套工作流与经典 RAG 显著不同。

**工作流程（Text-to-SQL）**：

1. **Text-to-SQL**：根据用户查询和表结构，生成 SQL 查询（语义解析的一种）
2. **SQL Execution**：执行 SQL 查询
3. **Generation**：基于 SQL 结果和原始用户查询生成最终回答

如果可用的表太多，schema 无法全部放入模型上下文，还需要一个中间步骤来预测每个查询应该使用哪些表。Text-to-SQL 可以由最终生成器完成，也可以使用专门的 Text-to-SQL 模型。

## 七、Evaluation（RAG 系统评估）

RAG 系统应从**组件级**和**端到端**两个层面评估。

### 组件级评估

检索质量指标（Context Precision、Context Recall、NDCG、MAP、MRR）、Embedding 质量评估（MTEB）和检索基准（BEIR），详见 4.8 节。

### 端到端评估——RAGAS 框架

[RAGAS](https://docs.ragas.io/)（Retrieval Augmented Generation Assessment）是一个专为 RAG 系统设计的评估框架，评估维度涵盖检索和生成两个环节：

| 评估维度 | 含义 |
|---|---|
| **Faithfulness（忠实度）** | 生成的答案是否忠实于检索到的上下文？（有没有编造不在上下文中的信息） |
| **Answer Relevancy（答案相关性）** | 生成的答案是否与用户的问题相关？ |
| **Context Precision（上下文精确率）** | 检索到的上下文中相关信息的占比 |
| **Context Recall（上下文召回率）** | 回答所需的信息是否都被检索到了 |

### 索引层面的评估

通过调整索引算法参数（如 HNSW 的 `ef`、`efConstruction`、`maxConnections`）来优化系统性能，也是评估的一部分。Weaviate 提供了详细的 RAG 评估指南，涵盖从索引到生成的全链路调优。

## 小结

| 主题 | 核心要点 |
|---|---|
| **核心价值** | RAG = LLM + 外部知识。解决知识局限、幻觉、数据安全三大痛点。不会被长上下文取代 |
| **核心认知** | **最相似 ≠ 最相关**——这一认知驱动了 Reranking、Metadata Filtering、Hybrid Search、Query 优化等所有设计决策 |
| **RAG vs Fine-tuning** | RAG 补充实时知识，Fine-tuning 学习特定模式。两者可互补 |
| **Ingestion** | 数据清洗 → 文档解析 → Chunking（5 级策略）→ Embedding（可微调提升 5-10%）→ Metadata（过滤+增强+Reverse HyDE）→ Multi-indexing（父子索引/知识图谱）→ 索引参数调优 |
| **Retrieval** | 检索算法（Term-based/Embedding-based/Hybrid Search）→ Reranking（cross-encoder/时间加权）→ Query 优化（重写/HyDE/子查询）→ 上下文化检索 → 高级检索（句子窗口/自动合并）→ 查询构建与路由 |
| **Generation** | LLM 微调、RAG 专用 Prompt（grounding 约束）、Chat Engine（ContextChat/CondensePlusContext）、Response Synthesiser（一次性/迭代/摘要/多答案） |
| **Evaluation** | 组件级（Precision/Recall/NDCG/MAP/MRR/MTEB/BEIR）+ 端到端（RAGAS：Faithfulness/Answer Relevancy/Context Precision/Context Recall） |
| **Beyond Texts** | 多模态 RAG（CLIP 跨模态检索）、结构化数据 RAG（Text-to-SQL/Cypher） |
