# Sampling（采样与推理）详解

> 模型训练完成后，如何从概率分布中生成实际文本？Sampling 决定了模型输出的**质量、多样性、可控性与可靠性**。

---

## 1. Sampling 基础

### 1.1 自回归生成过程

LLM 是逐 token 生成文本的：每一步模型输出当前词表上所有 token 的概率分布，然后从中"采样"一个 token，拼接到已有序列后，再预测下一个。

```
P(token_t | token_1, token_2, ..., token_{t-1})
```

这个循环不断重复，直到模型生成终止符（EOS）或达到最大长度限制。

### 1.2 概率性（Stochasticity）：Sampling 的本质属性

Sampling 的核心特性是**概率性**——同一个 prompt，每次生成的结果可能不同。这一特性源于从概率分布中**随机抽样**的过程，而非确定性地选取最优解。

#### Logits → Probabilities → Logprobs 流水线

模型实际输出的并不是"概率"，而需要经过一条完整的转换链：

```
Neural Network → logits（原始未归一化分数）
    → softmax → probabilities（归一化概率分布）
        → log → logprobs（对数概率，常用于评估和调试）
```

- **Logits**：模型最后一层的原始输出，每个 token 对应一个标量分数，值域为 $(-\infty, +\infty)$，不构成概率分布。
- **Softmax**：将 logits 转换为合法概率分布（所有值为正且和为 1）。公式：$p_i = \frac{\exp(z_i)}{\sum_j \exp(z_j)}$
- **Logprobs**：对概率取对数，即 $\text{logprob}_i = \log(p_i)$。值域为 $(-\infty, 0]$，越接近 0 表示模型越确信。API 中常返回 logprobs 供开发者分析模型置信度。

#### 概率性的影响

- **同一 prompt 多次生成 → 不同结果**：这既是创造力的来源，也是不可控性的根源。
- **Temperature = 0 时趋近确定性**：消除随机性，退化为 greedy decoding。
- **概率性 ≠ 随机性**：模型不是"随机乱猜"，而是在一个有结构的概率分布上采样——高概率 token 仍然被优先选择。

### 1.3 核心权衡：确定性 vs 多样性

模型每步都输出一个概率分布，必须从中选出一个 token。**怎么在"不重复无聊"和"不胡说八道"之间找到平衡**，是所有 decoding 策略的根本困境。三种朴素策略各有明显缺陷：

- **Greedy Decoding（贪心解码）**：每步选概率最高的 token。简单但输出重复、缺乏多样性，且局部贪心不保证全局最优。
- **Beam Search（束搜索）**：同时维护 k 条候选序列，扩展并保留得分最高的 k 条。比贪心更优但仍倾向于保守、重复的输出，且计算开销更大。
- **随机采样（Stochastic Sampling）**：完全按概率分布随机抽取 token，引入多样性但低概率的"离谱" token 也可能被选中，导致不连贯。

> 正因以上策略都不完美，实践中通过 **temperature、top-k、top-p** 等技术精细调控这一权衡。

### 1.4 停止条件（Stopping Condition）

生成不会无限进行，由以下条件控制何时终止：

- **Max tokens**：达到预设的最大 token 数即停止。注意可能导致输出被截断在不完整的位置。
- **Stop tokens / Stop words**：遇到特定终止符（如 EOS token、`\n\n`、自定义停止词）时终止。Chat 模型通常使用特殊的 end-of-turn token。
- **结构化输出的终止**：当使用 JSON Mode 等约束时，生成完整的合法结构后自动停止。

## 2. 关键采样参数

### 2.1 Temperature（温度）

- **原理**：调整概率分布的"陡峭度"。
- **方法**：将 logits（未归一化的预测分数）除以温度参数后再进行 softmax。

$$
p_i = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}
$$

| Temperature 值 | 效果 |
|----------------|------|
| **T → 0** | 分布趋近于 one-hot，几乎等同于 greedy decoding，输出确定性高 |
| **T = 1** | 原始分布，不做调整 |
| **T > 1** | 分布变平，低概率 token 被赋予更多机会，输出更随机/有创意 |

**使用建议**：事实性问答/代码生成用低温（0.0–0.3）；创意写作/头脑风暴用高温（0.7–1.2）。

> **公式解释：**
>
> - 其中，$z_i$ 是第 i 个词对应的未归一化的 logits（预测分数）。
> - $T$ 是 Temperature 参数，用来控制输出的随机性。(OpenAI API 中取值范围为 0–2)
> - Numerator（分子）：将每个词的 logits 除以 Temperature 参数后取指数。
> - Denominator（分母）：将所有词的 logits 除以 Temperature 后取指数，再相加以进行归一化，确保结果是一个概率分布。

- **效果**：低温使高概率词更突出，高温则使分布更均匀。

### 2.2 Top-k Sampling

- 每步只保留概率最高的 **k 个** token，将其余 token 的概率置零后重新归一化。
- 优点：排除长尾中的低质量 token。
- 缺点：**k 是固定的**——当分布很集中时 k 可能太大（引入噪声）；当分布很平坦时 k 可能太小（截断合理选项）。

### 2.3 Top-p（Nucleus Sampling，核采样）

- 动态选择最小的 token 集合，使其**累积概率 ≥ p**，然后在该集合内重新归一化后采样。

- 与 **Top-k** 不同，Top-p 不再限制一定数量的词，而是根据概率的累积值进行限制。

  首先对模型生成的所有候选词按概率进行排序（从大到小），设定一个概率阈值（如70%），从概率最高的词开始，累加它们的概率。当累加和首次超过设定的 P 值时，停止添加词。在这组词中进行选择。

- **自适应性**是核心优势：分布集中时自动选少量 token，分布分散时自动包含更多 token。

### 2.4 Min-p Sampling

- 设定一个**最小概率阈值**（相对于最高概率 token 的比例），低于该阈值的 token 被过滤掉。
- 例如 min_p=0.1 且最高 token 概率为 0.8，则只保留概率 ≥ 0.08 的 token。
- 与 Top-p 类似具备自适应性，但逻辑更直观——直接从"最低可接受概率"的角度出发。

### 2.5 参数组合实践

这些参数可以叠加使用，典型组合：

```
temperature=0.7 + top_p=0.9       → 通用对话
temperature=0.0                    → 确定性任务（数学/代码）
temperature=1.0 + top_k=50        → 创意生成
```

**注意**：同时使用 top-k 和 top-p 时，取两者的**交集**（先 top-k 截断，再在 k 个 token 中做 top-p 筛选）。

## 3. Structured Outputs（结构化输出）

### 3.1 动机

许多下游应用需要模型输出**符合特定格式**的内容（JSON、SQL、代码、XML 等），自由采样无法保证格式正确。

### 3.2 实现方式

| 方法 | 原理 | 特点 |
|------|------|------|
| **Prompt Engineering** | 在提示中明确要求输出格式并给示例 | 简单但无硬保证 |
| **Constrained Decoding（受限解码）** | 在采样时用语法/模式约束过滤非法 token | 保证格式正确，但可能降低内容质量 |
| **JSON Mode** | 部分 API 提供原生 JSON 模式，强制输出合法 JSON | 易用，但灵活性有限 |
| **Function Calling** | 模型输出函数名+参数的结构化调用，API 层解析并执行 | 工具调用场景的标准方案 |

### 3.3 Constrained Decoding 细节

- 在每一步解码时，根据当前已生成的部分输出和目标语法（如 JSON schema），计算当前位置**合法的 token 集合**。
- 将不合法 token 的概率设为 0，仅在合法 token 中采样。
- 可用 **有限状态自动机（FSA）** 或 **上下文无关文法（CFG）** 来定义约束。

## 4. Test-Time Compute（推理时计算）

### 4.1 核心洞察

模型在推理阶段投入更多计算，可以显著提升输出质量——让模型"多想一会儿"。DeepMind 研究发现：**扩展推理时计算在某些场景下比扩展模型参数量更高效**（即小模型 + 多推理 > 大模型 + 单次推理）。

### 4.2 主要策略

#### Chain-of-Thought（CoT，思维链）

- 让模型在给出最终答案前，先**逐步展示推理过程**。
- 通过提示（如 "Let's think step by step"）或训练来诱导。
- 对数学推理、逻辑推理、多步问题效果显著。

#### Self-Consistency（自一致性）

- 让模型对同一问题**生成多条推理路径**（通过较高温度采样）。
- 对最终答案进行**多数投票（majority voting）**，选择出现最多的答案。
- 用"冗余推理"换取准确性，显著提升了推理任务的鲁棒性。

#### Best-of-N 选择

- 生成 N 个候选回复，使用**评分机制**选出最佳。
- 评分方式可以是：模型自身的 logprobs（序列概率）、单独训练的 Reward Model 打分、或外部验证器（Verifier）。
- 与 Self-Consistency 的区别：Self-Consistency 靠投票选答案，Best-of-N 靠打分选回复。

#### Verifier（验证器）

- 训练专门的验证模型来**判断候选回复的正确性**，而不仅仅是打"偏好分"。
- 关键发现：使用 Verifier 筛选小模型的多个输出，效果可以**等效于 30 倍参数量的大模型**直接输出——即用计算换参数。
- Verifier 特别适用于有明确正确答案的任务（数学、代码、事实性问答）。

#### Tree of Thought（思维树）

- 将推理过程组织为**树结构**，每个节点是一个中间思考步骤。
- 模型可以**回溯、探索不同分支**，类似搜索算法。
- 适用于需要规划和探索的复杂问题（如博弈、数学证明）。

#### 比较总结

| 方法 | 策略 | 计算开销 | 适用场景 |
|------|------|----------|----------|
| **CoT** | 串行逐步推理 | 低（1 次生成） | 多步推理、数学 |
| **Self-Consistency** | 多次采样 + 多数投票 | 中（N 次生成） | 需要高可靠性的推理 |
| **Best-of-N** | 多次采样 + 评分选优 | 中（N 次生成 + 评分） | 有质量评估标准的任务 |
| **Verifier** | 多次采样 + 验证器筛选 | 中–高 | 有明确正确答案的任务 |
| **Tree of Thought** | 树搜索 + 回溯 | 高（指数级分支） | 复杂规划/探索问题 |

### 4.3 Scaling Test-Time Compute 的意义

这些策略共同指向一个范式转变：**模型的能力不仅取决于训练阶段投入了多少算力（参数量、数据量），还取决于推理阶段愿意投入多少算力**。这为"小模型 + 高推理预算"提供了与大模型竞争的路径。

## 5. Hallucination（幻觉）

### 5.1 定义

模型生成的内容**看似流畅合理，但事实上是错误的、虚构的或无法验证的**。这是 LLM 最核心的可靠性挑战之一。

### 5.2 幻觉的类型

- **事实性幻觉（Factual Hallucination）**：生成与现实世界事实矛盾的内容（如虚构人物、错误日期）。
- **忠实性幻觉（Faithfulness Hallucination）**：生成与输入上下文不一致的内容（如摘要中加入原文没有的信息）。

### 5.3 幻觉的根因

| 根因 | 说明 |
|------|------|
| **训练数据噪声** | 互联网数据本身包含错误/矛盾信息，模型在预训练中学到了这些 |
| **知识截断** | 模型只知道训练截止前的信息，对之后的事实一无所知 |
| **概率生成本质** | LLM 的目标是生成"看起来合理"的文本而非"正确"的文本 |
| **长尾知识不足** | 对低频/冷门知识的记忆不可靠 |
| **上下文遗忘** | 在长文本中可能"遗忘"前面的关键信息，导致不一致 |

### 5.4 缓解策略

- **Retrieval-Augmented Generation（RAG）**：在生成前检索外部知识库中的相关文档，将其注入上下文，使模型基于实际证据回答，而非仅靠参数记忆。
- **引用与归因（Citation & Attribution）**：要求模型在回答中标注信息来源，便于验证。
- **Self-Consistency 多次采样**：对于事实性问题，多次采样并交叉验证。
- **Post-hoc 验证**：用外部工具（搜索引擎、知识图谱、代码执行）验证模型输出的事实性。
- **训练阶段优化**：在 SFT/RLHF 中特别标注和惩罚幻觉回复，训练模型"不知道就说不知道"。

## 6. 关键 Takeaways

1. **概率性是 Sampling 的根基**：LLM 输出的是概率分布而非确定答案，理解 logits → softmax → probabilities → logprobs 这条流水线是理解一切采样策略的前提。
2. **Sampling 不是简单的"随机抽取"**：temperature、top-k、top-p、min-p 的组合极大影响了输出的确定性、创意性和质量，需要根据任务精心调节。
3. **结构化输出是生产部署的刚需**：Constrained Decoding 和 JSON Mode 让 LLM 能可靠地嵌入工程系统。
4. **推理时多花计算 = 更好的答案**：CoT、Self-Consistency、Best-of-N、Verifier、Tree of Thought 是"用算力换智力"的核心范式，Verifier 甚至可以让小模型媲美 30 倍参数的大模型。
5. **幻觉是 LLM 的结构性缺陷，而非偶发 bug**：概率生成的本质决定了模型不可能 100% 避免幻觉，只能通过 RAG、引用归因、验证等组合策略持续缓解。
6. **采样策略应与任务匹配**：没有万能参数——代码要确定性，创意要随机性，推理要冗余性。
