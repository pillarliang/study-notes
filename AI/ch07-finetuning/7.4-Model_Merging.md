# Model Merging（模型合并）

> **核心思想**：如果微调让你从**一个**模型创建定制模型，那么模型合并让你从**多个**模型组合出定制模型。模型合并不需要 GPU（不涉及训练），对独立开发者尤其有吸引力。其三大主流方法是**求和（Summing）**、**层堆叠（Layer Stacking）** 和**拼接（Concatenation）**。模型合并与微调互补——先微调专用模型，再合并它们，是一种强大的多任务模型构建流程。

---

## 一、为什么要合并模型？

### 1.1 核心价值

| 价值 | 说明 | 场景 |
|------|------|------|
| **提升性能** | 两个各自擅长 60% 问题的模型，合并后可能解决 80% | 互补的专用模型 |
| **减少内存** | 两个不同任务的模型合并为一个多任务模型 | 端侧部署 |
| **避免灾难性遗忘** | 各任务独立微调再合并，无需顺序训练 | 多任务微调 |
| **无需 GPU** | 合并本身不涉及训练 | 独立开发者/算力受限 |

### 1.2 多任务微调的三种方式

不使用模型合并时，多任务微调有两种传统方法，各有严重局限：

| 方式 | 做法 | 问题 |
|------|------|------|
| **同时微调** | 创建包含所有任务数据的数据集，一次性训练 | 多任务同时学习更难，需要**更多数据和训练** |
| **顺序微调** | 先训练任务 A，再训练任务 B... | **灾难性遗忘**（Kirkpatrick et al., 2016）：学新任务时忘掉旧任务 |
| **模型合并** ★ | 各任务独立并行微调，完成后合并 | 独立学习→更好的单任务效果；并行训练→无遗忘风险 |

### 1.3 其他应用场景

- **端侧部署**：手机、笔记本、汽车、手表等设备内存有限，与其挤入多个模型，不如合并为一个多任务模型
- **联邦学习**（McMahan et al., 2016）：同一模型部署到多设备，各设备用本地数据训练，之后合并所有副本
- **隐私保护**：数据不能离开设备时，通过联邦学习 + 模型合并实现分布式训练

### 1.4 模型合并 vs 模型集成

| 维度 | 模型合并（Merging） | 模型集成（Ensemble） |
|------|---------------------|---------------------|
| 做什么 | 混合模型**参数** | 仅组合模型**输出** |
| 推理成本 | 与单模型相同 | 需要多次推理 |
| 最终产物 | 一个模型 | 多个独立模型 + 聚合逻辑 |

> Hugging Face Open LLM Leaderboard 上的许多顶级模型都是合并模型——正如过去模型集成曾统治各种排行榜一样。

## 二、求和（Summing）方法

### 2.1 线性组合（Linear Combination）

最简单的合并方式：将两个模型的参数做加权平均。

```
Merge(A, B) = (w_A × A + w_B × B) / (w_A + w_B)
```

- **令人惊讶地有效**——这个简单方法的研究可追溯到 1990 年代初（Perrone, 1993）
- 常用于联邦学习（Wang et al., 2020）
- **Model Soups**（Wortsman et al., 2022）证明：将多个微调模型的全部权重取平均，可以提升精度而**不增加推理时间**

#### 任务向量（Task Vectors）

线性组合对**共享同一基座模型**的微调模型效果最佳。此时可以引入**任务向量**的概念：

```
任务向量 = 微调模型 - 基座模型
```

任务向量（也称 delta parameters）捕捉了微调对模型行为的改变。如果用 LoRA 微调，任务向量就是 LoRA 权重本身。

**任务算术（Task Arithmetic）**（Ilharco et al., 2022）：

| 操作 | 效果 |
|------|------|
| **加法**：基座 + 任务向量 A + 任务向量 B | 组合两个任务的能力 |
| **减法**：基座 - 任务向量 C | 移除特定能力（如面部识别）或减少偏见 |

> **不同架构/大小的模型也可以线性组合**——通过将某一层投影到相同维度来对齐。也有人提出先对齐参数再求平均（如 "Git Re-Basin"，Ainsworth et al., 2022），但对齐操作本身较复杂，因此不如朴素线性组合常用。

### 2.2 球面线性插值（SLERP）

**SLERP（Spherical LinEar inteRPolation）** 是另一种常见的求和方法。

**直觉理解**：
- 将每个待合并的向量看作球面上的一个点
- 沿球面**最短路径**（类似地球表面两城之间的弧线）找到合并点
- 插值因子（0 到 1）控制合并点在路径上的位置
  - < 0.5：偏向第一个模型
  - = 0.5：正中间
  - \> 0.5：偏向第二个模型

**限制**：SLERP 一次只能合并**两个**向量。合并三个以上需要顺序执行（先合并 A 和 B，再将结果与 C 合并）。

### 2.3 剪枝冗余参数

微调过程中，大部分参数的调整幅度很小且对任务贡献不大。Yadav et al.（2023）发现：**保留前 20% 变化最大的任务向量参数即可获得与 100% 相当的性能**。

这些"冗余"参数虽然不影响单个模型，但合并时可能**干扰其他模型的有效参数**。因此：

| 方法 | 全称 | 策略 |
|------|------|------|
| **TIES** | TrIm, Elect Sign, and merge | 修剪微小变化 + 解决符号冲突 + 合并 |
| **DARE** | Drop And REscale | 丢弃 90-99% 的任务向量权重再合并 |

> 两者都证明：**合并前剪枝冗余参数可显著提升合并模型质量**。合并的模型越多，剪枝越重要——因为不同任务间的参数干扰机会更多。

## 三、层堆叠（Layer Stacking）

### 3.1 基本原理

从一个或多个模型中取出不同层，**叠放在一起**。也称 **passthrough** 或 **frankenmerging**（弗兰肯合并）。

特点：
- 可以创建具有**独特架构和参数量**的新模型
- 通常需要**进一步微调**才能达到好效果
- 可与求和方法混合使用

**案例**：**Goliath-120B**（alpindale, 2023）——从两个微调的 Llama 2-70B 模型（Xwin 和 Euryale）各取 72/80 层合并而成。

### 3.2 MoE 训练

层堆叠可用于创建 **MoE（混合专家）** 模型（"Sparse Upcycling"，Komatsuzaki et al., 2022）：

1. 取预训练模型，复制某些层/模块
2. 添加**路由器（Router）** 将输入分配到最合适的副本
3. 进一步训练合并模型和路由器

> **Together AI** 用此方法混合 6 个开源模型创建了 **Mixture-of-Agents**，在部分基准上达到了 GPT-4o 水平（Wang et al., 2024）。

### 3.3 模型上缩放（Model Upscaling）

层堆叠的一个有趣应用：**用现有模型创建更大的模型**。

**深度缩放（Depthwise Scaling）** 示例——SOLAR 10.7B（Kim et al., 2023）：

```
原始模型：7B 参数，32 层
    │
    ├── 1. 复制一份模型
    ├── 2. 选择性地求和某些层（16 层被合并），其余层堆叠
    │       最终层数 = 32 × 2 - 16 = 48 层
    └── 3. 进一步训练上缩放后的模型

结果：10.7B 参数，48 层
```

## 四、拼接（Concatenation）

将多个模型的参数直接**拼接**在一起。例如合并两个秩为 r1 和 r2 的 LoRA adapter，结果秩为 r1 + r2。

> **不推荐**：拼接不减少内存（合并后参数量等于各模型参数之和），性能提升也未必值得额外参数量的开销。

## 五、模型合并最新进展（2025 更新）

### 5.1 工具生态

| 工具 | 特点 |
|------|------|
| **MergeKit**（Arcee AI） | 当前最主流的开源合并工具，支持 SLERP、TIES、DARE、任务算术、线性合并、passthrough 等 |
| **Mergenetic**（ACL 2025） | 基于遗传算法的合并优化库 |

### 5.2 进化式模型合并

**Sakana AI**（发表于 Nature Machine Intelligence, 2025）提出用**遗传算法自动搜索最优合并配方**，面向特定目标能力优化。产出的模型在数学和视觉能力上超越了更大的模型。

### 5.3 重要警示

2025 年 11 月的系统性研究发现：当前合并技术**在大规模现代 LLM 上并不总是可靠**——许多在小模型上有效的方法在大模型上失败。模型合并仍然是一个**实验性**较强的领域，需谨慎使用。

```
┌──────────────────────────────────────────────────────────┐
│                  模型合并方法对比                           │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  求和（Summing）                                          │
│  ├── 线性组合    简单有效，适合共享基座的模型               │
│  ├── SLERP      球面插值，一次合并两个模型                  │
│  └── TIES/DARE  合并前剪枝冗余参数，提升合并质量            │
│                                                          │
│  层堆叠（Layer Stacking）                                 │
│  ├── 创建新架构/参数量                                     │
│  ├── MoE 构建（Sparse Upcycling）                         │
│  └── 模型上缩放（Depthwise Scaling）                      │
│                                                          │
│  拼接（Concatenation）→ 不推荐                             │
│                                                          │
├──────────────────────────────────────────────────────────┤
│  实践建议：                                                │
│  • 多任务模型 → 各任务独立 LoRA 微调 → 合并 adapter         │
│  • 端侧部署 → 合并减少内存占用                              │
│  • 使用 MergeKit 作为默认工具                               │
│  • 合并前用 TIES/DARE 剪枝冗余参数                          │
│  • 大模型合并效果不稳定，需充分验证                          │
└──────────────────────────────────────────────────────────┘
```
